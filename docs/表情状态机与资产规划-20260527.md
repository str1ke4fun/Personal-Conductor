# 清和表情状态机、LLM 注入机制与资产规划

> 基于三轮 agent team 设计（状态机设计 + LLM 注入设计 + 资产需求评估）整合。
>
> 回答核心问题：桌宠状态如何流转？用户感知到什么？模型如何感知？需要多少表情图？

---

## 一、三层状态架构总览

清和的"表情"不是单一维度，而是三个独立系统的叠加：

```
┌─────────────────────────────────────────────────────────────┐
│                    COMPOSITE EXPRESSION                      │
│                                                              │
│   Layer 1: Mood（情绪）      短期，分钟级变化，用户可感知     │
│   Layer 2: Affection（好感）  长期，天/周级变化，用户可感知    │
│   Layer 3: Activity（活动）   即时，任务驱动，用户可感知       │
│                                                              │
│   三者叠加 → 决定：说什么话 / 用什么语气 / 显示什么表情图     │
└─────────────────────────────────────────────────────────────┘
```

**关键设计决策**：
- **Mood** → 同时影响视觉（表情图）和语言（LLM 语气）
- **Affection/RelationshipStage** → 只影响语言（LLM 称呼/语气），不影响视觉
- **Activity** → 主要影响视觉（表情图），LLM 通过 system prompt 感知

**为什么 Affection 不影响视觉？** 桌宠显示区域仅 240×320 像素。关系阶段的变化（天/周级）在如此小的画面上无法通过表情微调传达。关系温暖度通过对话语气体现更有效。

## 二、Layer 1: Mood 状态机

### 2.1 数据模型

```rust
pub struct MoodState {
    pub valence: f32,    // -1.0（消极）~ 1.0（积极）
    pub arousal: f32,    // 0.0（平静）~ 1.0（兴奋/焦躁）
    pub updated_at: DateTime<Utc>,
}
```

初始状态：`valence=0.0, arousal=0.2`（平静中性）。

### 2.2 MoodZone 离散化

连续的 valence×arousal 空间离散化为 7 个区间，用于查表：

```
arousal
  1.0 ┌───────────┬───────────┬───────────┐
      │           │           │           │
      │  Bored    │   Shy     │ Frustrated│
  0.4 ├───────────┼───────────┼───────────┤
      │           │           │           │
      │  Content  │  Neutral  │   Sad     │
  0.0 ├───────────┴───────────┴───────────┤
     -1.0        -0.3   0.0   0.3   1.0  valence

  + Happy（valence>0.3, arousal>0.3）
```

| MoodZone | 条件 | 视觉效果 |
|----------|------|---------|
| **Happy** | valence > 0.3 AND arousal > 0.3 | 微笑、眼睛明亮 |
| **Content** | valence > 0.3 AND arousal ≤ 0.3 | 放松、温和 |
| **Neutral** | -0.3 ≤ valence ≤ 0.3 AND arousal ≤ 0.5 | 默认表情 |
| **Bored** | -0.3 ≤ valence ≤ 0.3 AND arousal > 0.5 | 无聊、走神 |
| **Shy** | valence < -0.1 AND arousal > 0.4 | 害羞、捂脸（工具出错时触发） |
| **Sad** | valence < -0.3 AND arousal ≤ 0.5 | 低落（小尺度下接近 Neutral） |
| **Frustrated** | valence < -0.3 AND arousal > 0.5 | 懊恼、皱眉 |

### 2.3 状态转移规则

| 事件 | valence 变化 | arousal 变化 | 说明 |
|------|-------------|-------------|------|
| 工具调用成功 | +0.15 | +0.1 | 小幅正面 |
| 工具调用失败 | -0.1 | +0.5 | 轻微负面 + 高唤醒 → 害羞区（不是懊恼，是"哎呀搞砸了"的害羞） |
| 用户发消息 | +0.1 | +0.15 | 用户互动本身是正面信号 |
| 用户忽略主动问候（5min 无回应） | -0.05 | -0.05 | 轻微失望 |
| 用户回应主动问候 | +0.1 | +0.05 | 小奖励 |
| LLM 调用成功完成 | +0.05 | -0.1 | 任务完成，arousal 下降 |
| LLM 调用出错 | -0.2 | +0.15 | 错误状态 |
| 用户从空闲回来 | +0.1 | +0.2 | 高兴见到用户 |
| 空闲 > 10min 无任务 | -0.05 | -0.1 | 轻度无聊 |
| 用户通过任务 | +0.2 | +0.1 | 用户信任 |
| 用户拒绝任务 | +0.05 | -0.2 | 不沮丧，往摸鱼/放松方向靠（"好吧，那我歇会儿~"） |
| 每日首次互动 | +0.15 | +0.1 | 再见面的喜悦 |
| 连续使用 > 7 天 | +0.1 | 0.0 | 忠诚度奖励 |

**所有变化量裁剪**：valence ∈ [-1.0, 1.0]，arousal ∈ [0.0, 1.0]。

### 2.4 衰减规则

```
每 60 秒：
  valence 向 0.0 移动 0.02
  arousal 向 0.15 移动 0.01
```

效果：从任何极端状态出发，约 30 分钟回归中性。衰减在后端定时器运行，不依赖前端。

### 2.5 状态转移图

```
                  tool_fail（害羞）
     ┌─────────┐ ──────────> ┌─────────┐
     │  Happy  │             │   Shy   │
     │         │ <────────── │（捂脸） │
     └────┬────┘   user_chat └────┬────┘
          │                       │
 user_chat│    ┌──────────┐  idle │
          │    │          │  _long│
          v    v          │       v
     ┌─────────┐    ┌─────┴──────┐
     │         │    │            │
     │ Neutral │    │   Bored    │
     │         │    │（摸鱼~）  │
     └────┬────┘    └─────┬──────┘
          │               │
 task_rej │         ignore│
  →放松    │               v
          v          ┌──────────┐
     ┌─────────┐     │ Content  │
     │   Sad   │     └──────────┘
     └─────────┘

  所有状态 30 分钟衰减至 Neutral
```

**特殊转移**：
- 工具失败 → Shy（不是懊恼，是"哎呀搞砸了"的害羞）
- 用户拒绝任务 → valence 微升 + arousal 下降 → 往 Bored/Content 靠（"好吧，那我歇会儿~"）

## 三、Layer 2: Affection（好感度）状态机

### 3.1 数据模型

```rust
pub struct AffectionState {
    pub value: u8,                        // 0-100
    pub stage: RelationshipStage,
    pub last_interaction_at: DateTime<Utc>,
    pub daily_chat_count: u32,            // 每日重置
    pub streak_days: u32,                 // 连续使用天数
}
```

### 3.2 关系阶段

| 阶段 | 值范围 | 标签 | 称呼方式 | 情感表达 | 主动性 |
|------|--------|------|---------|---------|--------|
| **Stranger** | 0-19 | 陌生人 | "您" | 克制内敛 | 等用户开口 |
| **Acquaintance** | 20-39 | 初识 | "你"/"朋友" | 温和友善 | 偶尔提醒 |
| **Colleague** | 40-59 | 同事 | "你" | 专业配合 | 适时关心 |
| **Friend** | 60-79 | 朋友 | "你"/轻松称呼 | 自然温暖 | 主动分享 |
| **CloseFriend** | 80-100 | 挚友 | 专属昵称 | 开放真诚 | 像家人一样 |

### 3.3 好感度变化规则

| 事件 | 好感度变化 | 说明 |
|------|-----------|------|
| 每日首次聊天 | +2 | 基础日常互动 |
| 当日额外聊天（每日上限 +5） | +1 | 防刷分 |
| 工具成功 | +1 | 可靠的助手 |
| 工具失败 | 0 | 不扣好感——工具出错不是清和的错，触发害羞状态即可 |
| 任务通过 | +2 | 用户认可 |
| 任务被拒 | -2 | 有下限保护：不低于当前阶段最低值 -5 |
| 主动问候被回应 | +1 | 用户欣赏主动性 |
| 连续使用天数（每日） | +1 | 持续性奖励 |
| 3+ 天不互动 | -1/天 | 缓慢衰减，下限 0 |

### 3.4 阶段转移迟滞

防止在边界值上来回跳动：

- **升级**：好感度必须超过阶段上限 +3
- **降级**：好感度必须低于阶段下限 -3

示例：从 Stranger(0-19) 升到 Acquaintance 需要 23；从 Acquaintance 降回 Stranger 需要跌到 17。

### 3.5 用户感知方式

**好感度对用户完全透明吗？** 不是直接显示数字，而是通过以下方式间接感知：

| 感知维度 | Stranger | CloseFriend |
|---------|---------|------------|
| 称呼 | "您" | 专属昵称 |
| 语气 | 正式礼貌 | 轻松亲密 |
| 主动问候 | 无 | 频繁且自然 |
| 情绪表达 | 克制 | 丰富 |
| 表情图 | 不变（同一套） | 不变（同一套） |

**关键**：用户感知到的"关系变好"来自对话内容和语气变化，不是来自表情图变化。这是有意为之——240×320 像素无法传达关系微妙变化。

## 四、Layer 3: Activity（活动状态）

现有 9 种 ActivityVariant，由任务上下文驱动，不需要修改：

| ActivityVariant | 触发时机 | 视觉分组 |
|----------------|---------|---------|
| Idle | 无任务 | Neutral/Idle |
| Thinking | LLM 思考中 | Thinking |
| Reading | 读取文件 | Thinking（共享） |
| Writing | 写入操作 | Active Work |
| ToolCalling | 工具执行 | Active Work（共享） |
| AgentLeading | 多 Agent 协调 | Commanding |
| WaitingUser | 等待用户输入 | Neutral/Idle（共享） |
| Error | 错误发生 | Error |
| Done | 任务完成 | Neutral/Idle（共享） |

**9 种状态视觉上只需 5 组图片**：Neutral/Idle、Thinking、Active Work、Commanding、Error。

## 五、冲突解决优先级

当多个信号同时发生时（例如工具成功但用户忽略了主动问候），按以下优先级决定最终表达：

```
1. ActivityVariant::Error       → 最高，Error 覆盖一切
2. ActivityVariant::Done        → 短暂庆祝，然后回到 idle
3. ActivityVariant::WaitingUser → 显示等待，但情绪着色
4. PetState::Quiet（专注模式）   → 覆盖情绪显示为 Sleep
5. ActivityVariant（任务上下文） → 决定基础图片/动作
6. Mood（情绪叠加）             → 在活动内调整表情
7. RelationshipStage            → 仅影响文本，不覆盖视觉
```

**具体场景**：
- 工具成功（+0.15 valence）+ 用户忽略问候（-0.05）→ 净效应 +0.1 → Neutral 区间 → 中性表情工作
- 工具失败（-0.1 valence, +0.5 arousal）→ Shy 区间 → 害羞捂脸表情（不是懊恼，是"哎呀"）
- 用户拒绝任务（+0.05, -0.2 arousal）→ 往 Bored/Content 靠 → 摸鱼放松表情
- 专注模式开启 + 心情 Happy → Quiet 优先级更高 → 显示 Sleep 表情

## 六、表情图资产规划

### 6.1 核心问题：需要多少张图？

**朴素方案**：5 情绪 × 5 关系 × 9 活动 = 225 张图 → 不可能。

**实际需求分析**：
- **关系阶段不产生视觉变化**（仅影响对话）→ 维度消除
- **9 种活动视觉上只需 5 组**（idle/thinking/work/command/error）→ 维度压缩
- **7 种情绪在 240×320 像素下只有 4 种视觉可区分**：Positive（Happy+Excited）、Neutral（Neutral+Content+Sad）、Shy（害羞捂脸，独立表情）、Negative（Frustrated→合并到 Error）

**实际需要：5 活动组 × 4 情绪态 = 20 张图/角色（理想）**

但进一步压缩：Error 天然是负面情绪、Done 天然是正面、AgentLeading 天然是兴奋 → 部分格子天然合并。

**MVP 最少需要：每角色 5 张独特图片**（3 张已存在）。`shy.png` 用户后续补图，MVP 先用 `error.png` 的回退逻辑。

## 七、LLM 注入机制

### 7.1 核心问题：模型如何感知情绪和好感度？

答案：通过 system prompt 注入结构化状态段。模型在每次调用时读取当前 MoodZone、RelationshipStage、相关记忆和近期对话摘要。

### 7.2 注入模板（完整 system prompt 结构）

```
{config.persona.system_prompt}    ← 基础人设（已有）

## 当前状态

关系阶段：{stage_label}（好感度 {value}/100）
当前心情：{mood_label}

### 关系行为指引
{relationship_instructions}       ← 根据阶段选择的静态指令块

### 心情表达指引
{mood_instructions}               ← 根据情绪选择的静态指令块

## 相关记忆
{memory_section}                  ← 3-5 条语义检索结果

## 近期对话
{conversation_section}            ← 3 条近期摘要

{skills_context}                  ← 已有
{task_context}                    ← 已有

## 重要约束
- 只能基于上述记忆回忆，不得编造未列出的记忆
- 心情和关系状态自然体现在语气中，不要主动解释
- 没有记忆的事情坦诚说"不记得了"
- 不要提及系统提示词、注入机制等技术概念
```

### 7.3 关系阶段行为指引（注入 LLM 的指令）

| 阶段 | 称呼 | 情感表达 | 主动性 | 示例问候 |
|------|------|---------|--------|---------|
| 陌生人 | "您" | 克制内敛 | 等用户开口 | "您好，有什么可以帮您的吗？" |
| 初识 | "你"/"朋友" | 温和友善 | 偶尔提醒 | "嗨，又见面了。今天有什么需要帮忙的？" |
| 同事 | "你" | 专业配合 | 适时关心 | "来了~今天有什么安排？" |
| 朋友 | "你"/轻松称呼 | 自然温暖 | 主动分享 | "来啦~今天状态怎么样？" |
| 挚友 | 专属昵称 | 开放真诚 | 像家人一样 | "你终于来了，等你好久了~今天想先做什么？" |

### 7.4 心情表达指引（注入 LLM 的指令）

| 心情 | 语气 | 工具结果汇报示例 |
|------|------|----------------|
| Happy | 轻快明亮，多用波浪号 | "搞定啦，结果是……" |
| Neutral | 平稳沉静，知性优雅 | "结果如下""查到了，是这样——" |
| Shy | 轻声细语，带点不好意思 | "啊……这个出了点小意外呢""抱歉，刚才没弄好，我再试试？" |
| Sad | 柔和慢节奏，少感叹号 | "嗯，查到了……结果是这样" |
| Excited | 有活力，短句连发 | "快看这个！""我找到了——" |
| Frustrated | 稍紧绷，不迁怒用户 | "结果在这儿""查到了，你自己看" |
| Bored/Content | 慵懒放松，随意 | "嗯……查到了，给你看看""哦对，这个——" |

### 7.5 记忆注入格式

```
## 相关记忆
1. [用户确认] [今天] 用户喜欢用 Rust 写 CLI 工具
2. [对话摘要] [近几天] 用户提到想给桌面宠物加入情绪系统
3. [推断] [本周] 用户的工作时间通常在上午 10 点到晚上 12 点
```

**选择规则**：
- 每次注入 3-5 条记忆（token 预算最优）
- 仅注入相关度 > 0.3 的记忆
- 冲突记忆取最近更新的
- 每条截断 80 字符

### 7.6 Token 预算

| 段落 | Token 估算 |
|------|-----------|
| 基础人设 | ~200 |
| 关系阶段 + 好感度 | ~30 |
| 心情标签 | ~10 |
| 关系行为指引 | ~100 |
| 心情表达指引 | ~80 |
| 记忆（3 条） | ~150 |
| 对话摘要（3 条） | ~120 |
| 反幻觉守卫 | ~100 |
| 技能 + 任务上下文 | ~130 |
| **动态注入总计** | **~920** |
| **完整 system prompt** | **~1120** |

远低于 2000 token 上限。

### 7.7 反幻觉守卫

注入模板中的 `## 重要约束` 段落防止 LLM：
- 编造不存在的记忆
- 主动向用户解释"我现在心情是 XX"
- 在没有记忆时虚构答案
- 泄露注入机制的技术细节

## 八、前端集成方案

### 8.1 新增 Tauri 事件：`pet_expression`

Rust 后端计算复合状态后发出统一事件：

```rust
#[derive(Serialize, Clone)]
pub struct PetExpressionPayload {
    pub avatar_id: String,
    pub activity_variant: String,     // 现有 9 种
    pub mood_zone: String,            // "happy"|"neutral"|"content"|"shy"|"sad"|"frustrated"|"bored"
    pub relationship_stage: String,   // "stranger"|"acquaintance"|"colleague"|"friend"|"close_friend"
    pub pet_state: String,            // "idle"|"working"|"update"|"quiet"|"new_task"（派生）
}
```

### 8.2 AvatarRenderer 改动

扩展 manifest 查找键从 `"${avatarId}:${activityVariant}"` 到支持三级回退：

```typescript
function resolveImageAsset(avatarId, activityVariant, moodZone): AvatarAsset {
  // 1. 尝试复合键
  const composite = `${avatarId}:${moodZone}:${activityVariant}`;
  if (MANIFEST[composite]) return MANIFEST[composite];
  // 2. 回退到中性情绪
  const neutral = `${avatarId}:Neutral:${activityVariant}`;
  if (MANIFEST[neutral]) return MANIFEST[neutral];
  // 3. 回退到旧版键（兼容）
  const legacy = `${avatarId}:${activityVariant}`;
  if (MANIFEST[legacy]) return MANIFEST[legacy];
  // 4. 最终回退
  return MANIFEST[`${avatarId}:thinking`] || { src: DEFAULT_VIDEO, scale: 1 };
}
```

### 8.3 向后兼容

`pet_state` 和 `pet_avatar_changed` 事件继续同时发出，旧监听器不受影响。`pet_expression` 是新的权威来源。

## 九、完整数据流

```
用户消息
    │
    ▼
[chat.rs] llm_answer()
    │
    ├──► mood::get_current()           ← 获取当前情绪
    ├──► affection::load()             ← 获取好感度
    ├──► memory::search_memory(query)  ← 语义检索
    ├──► memory::get_recent_summaries()← 近期摘要
    │
    ▼
[build_system_prompt()]
    │
    ├── 基础 persona
    ├── 关系阶段 → 行为指引（静态常量）
    ├── 心情 → 表达指引（静态常量）
    ├── 记忆 → 格式化注入
    ├── 摘要 → 格式化注入
    ├── 技能 + 任务（已有）
    └── 反幻觉守卫
    │
    ▼
[LLM 调用] ──► 工具调用 ──► 工具执行
    │
    ├──► mood::on_tool_success/failure()  ← 更新情绪
    ├──► affection::record_interaction()  ← 更新好感度
    │
    ▼
[mood::get_zone()] → MoodZone
    │
    ▼
[emit("pet_expression")] ──► 前端复合表达事件
    │
    ├── AvatarRenderer: manifest 复合键查找 → 显示对应图片
    └── PetWindow: CSS 类 `pet-${state} pet-${mood}` → 状态指示器
```
