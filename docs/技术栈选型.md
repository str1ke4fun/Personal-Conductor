# Personal Conductor Agent — 技术栈选型

> 本文回答三个新增体验需求的技术选型：弹窗显示 task list、桌宠常驻桌面、桌宠提示任务完成与待审。
>
> 结论：**Tauri v2 + React/TypeScript + Rust 本地核心 + SQLite + Live2D Cubism**。
>
> 最后更新: 2026-05-27 — 补充 conductor-sense、fastembed、Live2D 替代 Lottie

---

## 1. 选型结论

| 层 | 选择 | 说明 |
|----|------|------|
| 桌面壳 | Tauri v2 | 轻量、适合常驻、支持多窗口、托盘、系统通知、全局快捷键 |
| 前端 UI | React 18 + TypeScript + Vite | task list 弹窗、桌宠气泡、设置页都用同一套组件体系 |
| 本地核心 | Rust crate (conductor-core) | 事件消费、任务状态、情绪/好感度、LLM 调用、智能监控 |
| 感知层 | conductor-sense | OS 级窗口焦点检测、用户空闲检测 |
| Hook 入口 | Rust CLI (conductor-cli) | Claude Code hook 只调用 CLI 写入事件，避免阻塞 Claude |
| 状态库 | SQLite (sqlx) | 适合多窗口、后台 worker、hook/CLI 并发读写 |
| 向量嵌入 | fastembed (BGE-small-en-v1.5) | 记忆语义搜索，384 维 |
| LLM | OpenAI 兼容 / Anthropic 兼容 | 对话、摘要、智能监控决策 |
| 原始日志 | `events.ndjson` | append-only，保留 hook 原始事件，方便排错和重放 |
| 人读镜像 | `tasks.md` | 从 SQLite 生成，方便不打开 UI 时直接看 |
| 桌宠动画 | Live2D Cubism SDK | 替代 Lottie，支持更丰富的角色表情和动作 |
| 系统集成 | Tauri tray / notification / global-shortcut 插件 | 托盘常驻、系统通知、快捷键呼出 task list |

## 2. 为什么不是 Electron / WPF / Python Qt

### 2.1 Electron

优点是前端生态最成熟、开发快。问题是它对这个项目偏重：Conductor 需要长期常驻，主要功能是小窗口、通知和本地状态，不需要完整 Chromium 应用的体量。

结论：**不作为首选**。除非后面发现 Tauri 透明窗口或桌宠交互成本过高，再回退到 Electron。

### 2.2 WPF / WinUI 3

优点是 Windows 原生能力强，窗口和通知控制细。问题是 UI/动画迭代不如 Web 技术舒服，后续如果要做更丰富的桌宠和任务面板，会比较慢。

结论：**不作为首选**。如果 Tauri 的透明窗口、点击穿透、置顶行为在 Windows 上遇到硬问题，可以写少量 Windows native 扩展，而不是整套改成 WPF。

### 2.3 Python + PySide/PyQt

适合原型，但长期桌面常驻、打包、自动启动、系统托盘和透明窗口稳定性都不如 Tauri 方案清晰。

结论：**不作为首选**。

## 3. 目标窗口设计

### 3.1 `task-panel`：task list 弹窗

定位：人的主要审阅入口。

行为：
- 由桌宠点击、托盘点击、全局快捷键呼出。
- 默认显示当前 1 小时 active queue。
- backlog 折叠显示，不抢注意力。
- 支持快速操作：通过、跳过、推后、进入专注模式。
- 不做普通 modal，而是一个可独立显示/隐藏的轻量窗口。

窗口建议：
- 宽度 360-460px。
- 高度随内容增长，但限制最大高度。
- 默认靠屏幕右侧或桌宠附近弹出。
- 失焦后可以自动收起，但“正在操作任务”时不自动收。

### 3.2 `pet-window`：桌宠窗口

定位：常驻提示入口，不承担复杂管理。

行为：
- 透明、无边框、置顶的小窗口。
- 只占角色实际尺寸附近，不做全屏透明 overlay。
- 点击桌宠打开 `task-panel`。
- 右键打开菜单：查看清单、专注 30 分钟、静默到明天、退出。

MVP 状态：
- `idle`：无待审或没有新变化。
- `new_task`：刚有新产出。
- `pending_review`：待审任务累计较多。
- `quiet`：专注模式，不打扰。

注意：点击穿透是 Windows 桌宠的关键细节。第一版不要做全屏透明窗口；如果透明区域挡点击，再用 Windows native API 补点击穿透。

### 3.3 `settings-window`：设置与调试

第一版可以没有完整设置页，但需要保留入口。

最小内容：
- 当前 hook 状态。
- 最近 20 条事件。
- 当前静默/专注状态。
- 通知权限状态。
- 数据目录路径。

## 4. 数据流

```text
Claude Code Stop/SubagentStop hook
  -> conductor-cli hook stop
  -> append state/events.ndjson
  -> conductor desktop worker 消费事件
  -> 生成摘要 / 创建 task
  -> 写入 state/conductor.sqlite
  -> 生成 state/tasks.md
  -> emit task_created / task_changed
  -> pet-window 气泡提示
  -> task-panel 刷新
  -> 必要时发系统通知
```

原则：
- hook 入口只做轻活：写事件，快速退出。
- 摘要、排序、通知判断都在后台 worker 里做。
- UI 不直接解析 transcript，只读结构化 task。
- `events.ndjson` 是事实来源，SQLite 是当前状态，`tasks.md` 是人读镜像。

## 5. SQLite 数据模型草案

### 5.1 `tasks`

| 字段 | 说明 |
|------|------|
| `id` | task 唯一 ID |
| `source_agent` | `claude` / `codex` / `manual` |
| `kind` | `review_doc` / `review_ppt` / `test_atom` / `manual` |
| `artifact_uri` | 文件路径、会话 ID、网页 URL 等 |
| `artifact_label` | 人能看懂的产物名 |
| `summary` | 一句话摘要 |
| `focus_hint` | 人需要重点看的 1-2 句话 |
| `estimated_minutes` | 预计审阅时长 |
| `priority` | `low` / `normal` / `high` |
| `status` | `pending` / `in_progress` / `passed` / `rejected` / `skipped` |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `due_at` | 建议审阅时间，可为空 |
| `snoozed_until` | 推后到什么时候，可为空 |

### 5.2 `events`

| 字段 | 说明 |
|------|------|
| `id` | event 唯一 ID |
| `event_type` | `claude_stop` / `task_updated` / `quiet_started` 等 |
| `source` | 事件来源 |
| `payload_json` | 原始 payload |
| `created_at` | 创建时间 |
| `processed_at` | 消费时间，可为空 |

### 5.3 `notification_state`

| 字段 | 说明 |
|------|------|
| `id` | 状态 ID |
| `quiet_until` | 静默到什么时候 |
| `last_notified_at` | 上次提醒时间 |
| `pending_minutes` | 当前待审累计分钟数 |
| `pending_count` | 当前待审数量 |

## 6. 提醒策略

提醒分两层：

1. 桌宠气泡：默认提醒方式，轻量、可忽略。
2. 系统通知：只用于需要跨应用打断人的场景。

最小规则：
- 新 task 生成时，桌宠显示气泡。
- 如果 active queue 累计超过 30 分钟，允许系统通知。
- 如果 30 分钟内已经提醒过，不重复系统通知。
- 专注模式期间不提醒，只更新桌宠状态。
- 静默结束后，如果仍有待审任务，聚合提醒一次。

提示文案结构：

```text
Claude 完成了「文档 A 第 3 段」
预计审阅 12 分钟，重点看口径是否和第 2 段一致。
```

不要让桌宠评价质量，只说明发生了什么、为什么需要看、预计花多久。

## 7. 工程结构建议

```text
I:/personal-agent/
├── apps/
│   └── desktop/                 # Tauri + React 桌面壳
│       ├── src/                 # React UI
│       └── src-tauri/           # Tauri Rust 入口
├── crates/
│   ├── conductor-core/          # 任务、事件、提醒策略
│   └── conductor-cli/           # hook 调用的 CLI
├── state/
│   ├── conductor.sqlite         # 当前状态
│   ├── events.ndjson            # 原始事件日志
│   ├── tasks.md                 # 人读镜像
│   └── summaries/               # 摘要 Markdown
├── docs/
│   ├── MVP-实施方案.md
│   ├── 技术架构.md
│   ├── 感知层调研.md
│   └── 技术栈选型.md
└── PRD.md
```

说明：
- `conductor-cli` 和 `desktop` 共用 `conductor-core`，避免 hook 逻辑和 UI 逻辑分裂。
- 桌面壳退出时，hook 仍能写 `events.ndjson`；下次启动后 worker 补消费。
- `tasks.md` 不作为主写入源，只作为镜像。

## 8. 实施顺序

### Phase 0：核心管道

目标：不用 GUI 也能跑通。

- 建 `conductor-core` 数据模型。
- 建 `conductor-cli hook stop`。
- 写 `events.ndjson`。
- 生成 SQLite task。
- 同步 `tasks.md`。

### Phase 1：task list 弹窗

目标：先解决“我现在该看什么”。

- 建 Tauri desktop app。
- 接 SQLite。
- 做 `task-panel`。
- 托盘常驻。
- 快捷键呼出。

### Phase 2：桌宠

目标：让桌宠成为状态入口。

- 建 `pet-window`。
- 接 Lottie 动画。
- 根据任务状态切换 `idle/new_task/pending_review/quiet`。
- 点击桌宠打开 task panel。
- 加气泡提示。

### Phase 3：提醒策略打磨

目标：减少打扰感。

- 系统通知阈值。
- 专注模式。
- 静默结束聚合提醒。
- 任务堆积提示。

## 9. 风险与验证

| 风险 | 验证方式 | 对策 |
|------|----------|------|
| 透明窗口挡住桌面点击 | 做 200x200 pet-window 原型测试 | 不做全屏 overlay；必要时加 Windows native click-through |
| 系统通知在开发态显示名称/图标不理想 | 打包安装后再测一次 | 开发态接受瑕疵，发布包配置正式图标 |
| SQLite 和 `events.ndjson` 状态不一致 | 用事件重放测试 | SQLite 可从日志重建关键状态 |
| 桌宠提示太烦 | 连续使用 3 天记录打扰点 | 默认气泡，系统通知加阈值和冷却时间 |
| Tauri 插件权限漏配 | 每个窗口最小权限配置 | 不给桌宠窗口数据库写权限，只读 task 状态 |
