# 场景化记忆系统细粒度落地方案

> 日期：2026-05-30
> 范围：`crates/conductor-core/src/memory.rs`、`chat/prompt.rs`、`db.rs`、`scene.rs`、`expression.rs`、`affection.rs`、桌宠前端记忆管理 UI
> 目标：把当前“KV 记忆 + 未接通的向量表”升级为可支撑桌宠场景化表达的长期记忆系统。

---

## 1. 现状校准

### 1.1 真实已存在的数据表

当前代码已经创建以下记忆相关表：

| 表 | 当前角色 | 真实使用状态 |
|---|---|---|
| `memory_entries` | KV 记忆主表，保存 key/value/category/source/confidence/sensitivity/status 等 | 已被 `memory_set`、`memory_save_preferences` 使用 |
| `conversation_summaries` | 对话摘要表，保存 summary、keywords、timestamp | 有 IPC 写入/读取接口，prompt 会注入最近 3 条 |
| `memory_chunks` | 可检索记忆片段表，保存 content/summary/scope/category/source/confidence 等 | 表和 API 存在，但生产写入链路未接通 |
| `memory_embeddings` | chunk 对应 embedding，BLOB 保存 f32 little-endian 向量 | 表和 API 存在，但生产写入链路未接通 |

### 1.2 当前最关键缺口

当前系统的问题不是“没有向量表”，而是：

```text
用户/工具/对话产生记忆
  -> memory_entries 或 conversation_summaries
  -> 没有稳定进入 memory_chunks
  -> 没有稳定生成 memory_embeddings
  -> search_memory() 很可能检索不到真实长期记忆
```

因此第一阶段优先级不是更换向量库，也不是直接扩大 prompt，而是打通：

```text
MemoryEntry / ConversationSummary
  -> MemoryChunk
  -> MemoryEmbedding
  -> search_memory()
  -> prompt 注入
```

### 1.3 当前检索和注入限制

`search_memory()` 当前逻辑：

- 候选：从 `memory_chunks` 按 `updated_at DESC` 取 `limit * 10`。
- 评分：`0.55 * vector + 0.20 * keyword + 0.15 * recency + 0.10 * authority`。
- 向量：fastembed v4 + `BGESmallENV15`，失败后 hash 伪向量 fallback。
- 注入：`chat/prompt.rs` 中取 top 5，分数大于 0.3，每条 `content` 截断 80 字符。
- 近期对话：另取 `conversation_summaries` 最近 3 条，每条摘要截断 80 字符。

这套机制可作为 MVP 基础，但不够支撑“像有记忆的桌宠伙伴”。场景化需要记忆不仅能被搜到，还要能被按时间、情绪、关系阶段、工作场景、交互模式召回。

## 2. 场景化记忆的目标定义

### 2.1 产品目标

桌宠需要做到：

1. 记得用户稳定偏好：称呼、语气、常用工具、偏好的提醒方式。
2. 记得近期上下文：最近几天用户在忙什么、卡在哪里、已经聊过什么。
3. 识别场景：工作、编码、文档、休息、长时间空闲、深夜、高压任务、会议后整理。
4. 记得交互模式：用户什么时候容易接受主动提醒、什么时候不喜欢被打断。
5. 关系感知表达：同一条记忆在不同好感/心情阶段下，表达方式不同。
6. 可审计可遗忘：用户能看到、编辑、删除、禁用敏感记忆。

### 2.2 非目标

第一阶段不做：

- 独立向量数据库服务。
- 大规模 ANN 索引。
- 完整心理画像。
- 未经确认的敏感信息长期沉淀。
- 自动替用户做高风险操作。
- 把所有聊天原文永久保存进向量库。

## 3. 记忆分层

### 3.1 五类记忆

| 类型 | 说明 | 示例 | 默认保留期 | 是否进向量检索 |
|---|---|---|---|---|
| `preference` | 用户明确偏好 | “用户喜欢简短直接的回复” | 长期 | 是 |
| `fact` | 稳定事实 | “用户当前项目是 personal-agent” | 长期或按 workspace | 是 |
| `episode` | 具体一次事件/对话摘要 | “5 月 30 日讨论了记忆系统升级” | 30-90 天 | 是 |
| `interaction_pattern` | 交互规律 | “用户夜间更常要求低干扰模式” | 90 天滚动 | 是，但低权重 |
| `scene_signal` | 场景状态信号 | “最近 30 分钟在编辑文档” | 1-7 天 | 主要用于场景判断，不一定进向量 |

### 3.2 记忆强度

新增概念：`memory_strength`，用于控制长期保留和召回优先级。

| 强度 | 含义 | 进入条件 |
|---|---|---|
| `pinned` | 用户手动固定 | 用户明确说“记住这个” |
| `stable` | 稳定长期记忆 | 多次出现或用户确认 |
| `recent` | 近期上下文 | 最近对话/任务摘要 |
| `weak` | 低置信推断 | 工具/模型推断，未确认 |
| `decayed` | 已衰减 | 长期未使用、过期或被新事实覆盖 |

MVP 可以先不新增枚举字段，而是用 `category`、`source`、`confidence`、`expires_at` 组合表达。后续再迁移为显式字段。

## 4. 数据模型方案

### 4.1 第一阶段不破坏现有表

先复用当前表：

- `memory_entries` 继续作为事实源。
- `conversation_summaries` 继续作为对话摘要源。
- `memory_chunks` 作为统一检索片段。
- `memory_embeddings` 作为 chunk 向量。

第一阶段不强制新增表，优先新增写入链路和少量字段。

### 4.2 建议新增字段

给 `memory_chunks` 增补：

| 字段 | 类型 | 用途 |
|---|---|---|
| `origin_table` | TEXT | `memory_entries` / `conversation_summaries` / `scene_events` |
| `origin_id` | TEXT | 源记录 id |
| `tags` | TEXT JSON | 主题、情绪、场景标签 |
| `importance` | REAL | 0-1，控制召回和保留 |
| `last_reinforced_at` | TEXT nullable | 最近一次被重复确认/使用 |

如果想先保持迁移极小，也可以暂不加字段，用 `memory_id` 指向源 id，`category` 表达类型，`summary` 放标签摘要。但长期建议补字段，否则后续难以追踪 chunk 来源。

### 4.3 关系约束

目标关系：

```text
memory_entries.id
  -> memory_chunks.origin_id / memory_id
  -> memory_embeddings.chunk_id

conversation_summaries.id
  -> memory_chunks.origin_id / memory_id
  -> memory_embeddings.chunk_id
```

删除或遗忘时：

- `memory_entries.status = forgotten` 后，对应 chunk 不直接物理删除，先标记为不可检索或删除 embedding。
- `purge_forgotten()` 执行时，再清理 chunk 和 embedding。
- conversation summary 暂不提供忘记接口时，先不做级联删除。

## 5. 写入链路

### 5.1 统一入口

新增内部函数：

```rust
pub async fn index_memory_entry(entry: &MemoryEntry) -> anyhow::Result<()>;
pub async fn index_conversation_summary(summary: &ConversationSummary) -> anyhow::Result<()>;
pub async fn reindex_memory_chunk(chunk_id: &str) -> anyhow::Result<()>;
```

职责：

1. 把源对象转成一个或多个 `MemoryChunk`。
2. 对 chunk.content 生成 embedding。
3. upsert 到 `memory_chunks` 和 `memory_embeddings`。
4. 保持幂等：同一源对象更新时覆盖旧 chunk。

### 5.2 MemoryEntry 到 chunk

转换规则：

```text
content = "类别: {category}\n键: {key}\n内容: {value}"
summary = value 的 80-160 字符摘要
source = entry.source
sensitivity = entry.sensitivity
confidence = entry.confidence
scope = entry.scope
workspace_id = entry.workspace_id
expires_at = entry.expires_at
```

写入触发：

- `set()` 新增或更新 active 记忆后触发。
- `classify(key, "active")` 后触发。
- `archive/forget/quarantine` 后更新 chunk 可检索状态或清理。
- `save_preferences()` 间接走 `set()`，因此自动进入检索。

### 5.3 ConversationSummary 到 chunk

转换规则：

```text
content = "近期对话摘要: {summary}\n关键词: {keywords}"
category = "conversation"
source = Summary
sensitivity = Normal
confidence = 0.8
expires_at = now + 90 days
```

写入触发：

- `add_conversation_summary()` 成功插入后触发。

后续增强：

- 每个 chat session 结束或空闲一段时间后自动生成 summary。
- summary 分两层：短摘要用于 prompt，长摘要用于检索。

### 5.4 自动摘要触发

MVP 触发策略：

| 触发点 | 条件 | 动作 |
|---|---|---|
| chat session 消息数达到阈值 | 最近未生成摘要且新增 8-12 条消息 | 生成对话摘要 |
| 用户明确切换话题 | LLM 或简单规则检测到主题变化 | 生成上一主题摘要 |
| 会话空闲 | 15 分钟无新消息 | 生成 session 摘要 |
| 工具任务完成 | agent/task/tool run 结束 | 生成任务型摘要 |

第一版可以先做非 LLM 摘要：

```text
用户最后 N 条问题 + assistant bubble_summary + tool summary
```

后续再接 LLM 摘要器。

## 6. 嵌入模型策略

### 6.1 第一阶段策略

先保留 fastembed + fallback，不阻塞写入链路。

理由：

- 当前最大断点是没有真实索引写入。
- 先打通链路后才能评估召回质量。
- 模型更换会引入维度、重建索引、下载体积和兼容性问题。

### 6.2 第二阶段更换中文模型

候选：

| 模型 | 用途 | 备注 |
|---|---|---|
| `bge-small-zh-v1.5` | 中文优先 | 适合当前中文对话和偏好记忆 |
| `bge-m3` small/quantized | 中英混合 | 适合代码、中文说明、英文技术词混合 |
| 继续 `BGESmallENV15` | 英文/代码辅助 | 中文语义弱，不适合作为主模型 |

切换要求：

1. `memory_embeddings.model` 记录模型名。
2. `dims` 必须参与校验，不同维度不混算。
3. 提供 `rebuild_embeddings(model)` 后台任务。
4. prompt 检索时只使用当前模型或兼容模型。

## 7. 检索方案

### 7.1 多路召回

替代单次 `search_memory(user_message, None, 5)`，新增场景化召回：

```rust
pub struct MemoryRecallRequest {
    pub user_message: String,
    pub workspace_id: Option<String>,
    pub scene_id: Option<String>,
    pub mood_zone: Option<String>,
    pub relationship_stage: Option<String>,
    pub recent_days: u32,
    pub limit: usize,
}
```

召回通道：

| 通道 | 目标 | 候选来源 |
|---|---|---|
| semantic | 当前问题相关记忆 | `search_memory(query)` |
| recent | 最近发生的上下文 | `conversation_summaries` + recent chunks |
| preference | 用户长期偏好 | `category = preference` |
| scene | 当前场景相关 | scene/persona/workspace tags |
| relationship | 关系阶段表达约束 | affection/persona，不一定查向量 |

### 7.2 排序公式

建议从当前公式扩展为：

```text
score =
  0.40 semantic_similarity
+ 0.15 keyword_match
+ 0.15 recency
+ 0.10 authority
+ 0.10 scene_match
+ 0.05 importance
+ 0.05 relationship_fit
```

MVP 可以先只实现：

- 保留原 4 维评分。
- 增加 category boost：
  - 当前 query 命中偏好类问题时，`preference +0.1`。
  - 当前是工作区问题时，workspace chunk +0.1。
  - 最近对话类 chunk 在 24 小时内 +0.1。

### 7.3 候选集扩大

当前 `limit * 10` 对 top 5 只取 50 条，场景化时偏少。建议：

- MVP：`candidate_limit = max(limit * 20, 100)`。
- 数据量超过 5000 条后再考虑 FTS/Vec1/LanceDB。
- 先加 `category`、`scope`、`updated_at`、`workspace_id` 的过滤，避免无谓扫描。

## 8. Prompt 注入策略

### 8.1 分层注入

当前每条 80 字符太短。建议改成：

```text
## 长期偏好
- ...

## 近期上下文
- ...

## 当前场景相关记忆
- ...

## 表达约束
- 只能基于上述记忆回忆，不得编造
- 敏感记忆不要主动提及，除非用户明确询问
```

### 8.2 注入预算

MVP 建议：

| 段落 | 条数 | 每条长度 |
|---|---:|---:|
| 长期偏好 | 3 | 120 字符 |
| 近期对话 | 3 | 160 字符 |
| 场景相关 | 4 | 200 字符 |
| 用户明确要求回忆时 | 8 | 300 字符 |

不要无限扩大系统 prompt。正常对话只放高置信、高相关、低敏感的信息。

### 8.3 记忆使用规则

prompt 中应明确：

- 不要把“可能是”说成“我记得”。
- `source=inferred` 或 `confidence < 0.7` 的记忆只能委婉使用。
- `sensitivity=private` 只能在上下文强相关时使用。
- `sensitivity=secret` 默认不注入。
- 用户问“不记得了吗”时，可以回答“我目前没有查到相关记忆”。

## 9. 场景化桌宠表达

### 9.1 场景状态输入

可用输入：

| 输入 | 当前来源 | 用途 |
|---|---|---|
| foreground app/title | sense/worker | 判断编码、文档、浏览器、会议等场景 |
| workspace | workspaces | 判断项目上下文 |
| task status | tasks/tasklist | 判断是否有进行中任务 |
| mood | expression | 决定语气强度 |
| affection | affection | 决定亲近程度 |
| chat history | chat_messages | 生成近期摘要 |
| memory | memory_* | 长期偏好和历史上下文 |

### 9.2 场景标签

第一批标签：

| 标签 | 触发示例 | 桌宠行为 |
|---|---|---|
| `coding_focus` | IDE 前台、长时间无聊天、agent run 进行中 | 少打扰，回答更短 |
| `document_work` | Word/PPT/Markdown 编辑 | 可主动建议整理结构 |
| `planning` | docs/PRD/roadmap/task 相关 | 注重条理和待办 |
| `debugging` | error/test/fail/build 日志 | 提醒保留上下文和复现步骤 |
| `idle_short` | 1-5 分钟空闲 | 轻提示 |
| `idle_long` | 30 分钟以上空闲 | 可建议休息或回顾 |
| `late_night` | 深夜活跃 | 降低打扰，语气更柔和 |

### 9.3 场景记忆写入

不要把每次前台窗口变化都写入长期记忆。只沉淀聚合结果：

```text
过去 7 天，用户多次在深夜处理 personal-agent 文档。
用户在编码场景下更偏好简短直接的建议。
用户在文档场景下接受结构整理建议。
```

这些属于 `interaction_pattern`，默认 `source=inferred`、`confidence=0.5-0.7`，需要多次强化或用户确认后才升为 stable。

## 10. 隐私和治理

### 10.1 写入门槛

| 来源 | 默认状态 | 说明 |
|---|---|---|
| 用户明确“记住” | active | 可直接进入长期记忆 |
| 用户设置/偏好 UI | active | 用户显式配置 |
| 对话摘要 | active，但有过期时间 | 不是逐字稿 |
| 工具结果 | candidate | 需要分类或低置信 |
| 模型推断 | candidate | 不直接当事实使用 |
| 敏感信息 | private/secret | 默认不主动注入 |

### 10.2 管理 UI

最小 UI：

- 按 category 查看记忆。
- 查看来源、置信度、更新时间、是否进入检索。
- 删除/忘记。
- 标记为 private/secret。
- 固定为长期偏好。
- 对 candidate 记忆确认或拒绝。

### 10.3 安全规则

- secret 不进入 prompt，不参与默认 search。
- private 可检索但默认不主动说出。
- forgotten 必须从检索结果排除。
- quarantined 必须从检索结果排除。
- 所有自动推断记忆都要能在 UI 中看到来源。

## 11. 分阶段落地计划

### Phase 1：打通真实索引链路

目标：所有 active 的 `memory_entries` 和新增 `conversation_summaries` 都能进入 `memory_chunks` 和 `memory_embeddings`。

任务：

1. 新增 `index_memory_entry()`。
2. 新增 `index_conversation_summary()`。
3. 在 `set()` / `set_with_source()` 成功后触发索引。
4. 在 `add_conversation_summary()` 成功后触发索引。
5. 在 `archive/forget/quarantine` 后让对应 chunk 不再被检索。
6. 增加单元测试：写 KV 后 `search_memory()` 能搜到。
7. 增加单元测试：写 conversation summary 后 `search_memory()` 能搜到。

验收：

- `memory_set("favorite_reply_style", "简短直接", "preference")` 后，`search_memory("你应该怎么回答我", None, 5)` 能召回。
- `memory_add_conversation()` 后，相关关键词能召回。
- forgotten/quarantined 记忆不再出现在召回结果。

### Phase 2：自动对话摘要

目标：桌宠能记住最近对话主题，而不是只记得用户手动写入的 KV。

任务：

1. 为 chat session 增加摘要生成触发器。
2. 第一版使用规则摘要，不依赖 LLM。
3. 摘要写入 `conversation_summaries`。
4. 摘要自动进入 chunk/embedding。
5. prompt 注入近期摘要时从 80 字符提升到 160 字符。

验收：

- 连续对话后无需手动调用 `memory_add_conversation`，系统能生成摘要。
- 新会话中询问“刚才我们讨论了什么”，能基于摘要回答。

### Phase 3：场景标签和多路召回

目标：同一条用户消息在不同场景下召回不同记忆。

任务：

1. 新增 `MemoryRecallRequest`。
2. 新增 `recall_for_prompt()`，替代 prompt 中直接调用 `search_memory()`。
3. 引入 category boost 和 workspace boost。
4. prompt 分段注入长期偏好、近期上下文、场景记忆。
5. 根据 foreground app/workspace/task 状态生成 scene tag。

验收：

- 文档场景下更容易召回文档偏好。
- coding 场景下更容易召回项目和代码协作偏好。
- 普通闲聊不会塞入过多工作记忆。

### Phase 4：交互模式记忆

目标：系统能从重复行为中形成低置信模式记忆。

任务：

1. 聚合最近 7/30 天的主动提醒接受/拒绝记录。
2. 生成 `interaction_pattern` candidate 记忆。
3. 多次强化后提升 confidence。
4. 在 UI 中展示并允许用户确认/删除。

验收：

- 用户多次拒绝某类提醒后，系统降低该类主动提示频率。
- 用户确认某个模式后，后续表达明显遵守。

### Phase 5：中文 embedding 和重建索引

目标：提升中文语义召回质量。

任务：

1. 抽象 embedding provider。
2. 支持当前模型名和 dims 校验。
3. 接入中文 embedding 模型。
4. 增加 `rebuild_embeddings` 后台任务。
5. 提供模型切换配置。

验收：

- 中文近义表达召回优于英文模型。
- 旧 embedding 不和新 embedding 混算。
- 模型不可用时 fallback 不影响基本关键词检索。

### Phase 6：记忆管理 UI

目标：长期记忆不再是黑盒。

任务：

1. 增加记忆列表页。
2. 支持 category/status/sensitivity/source 过滤。
3. 支持编辑、忘记、归档、隔离。
4. 支持查看 chunk 是否已索引。
5. 支持手动重建 embedding。

验收：

- 用户能看见系统记住了什么。
- 用户能删除不想保留的记忆。
- 用户能知道哪些记忆会影响桌宠回复。

## 13. 风险和取舍

| 风险 | 影响 | 处理 |
|---|---|---|
| 自动记忆过多 | prompt 噪音、隐私风险 | 只摘要不存原文，candidate 默认低置信 |
| embedding 下载失败 | 检索质量下降 | 保留关键词和 hash fallback |
| 中文模型切换导致旧向量不可用 | 召回混乱 | model/dims 严格隔离，后台重建 |
| 场景判断误判 | 打扰用户 | 主动行为只到“询问”，不自动执行高风险动作 |
| 用户无法审计 | 信任下降 | Phase 6 必做记忆管理 UI |

## 14. 最小可执行切片

如果只做一轮，建议切片如下：

1. `memory_entries -> memory_chunks -> memory_embeddings` 写通。
2. `conversation_summaries -> memory_chunks -> memory_embeddings` 写通。
3. prompt 注入从 `search_memory()` 改为一个简单 `recall_for_prompt()`。
4. 注入长度从 80 字符提升到：
   - 记忆：160 字符。
   - 近期摘要：160 字符。
5. 增加 4 个测试：
   - KV 记忆可召回。
   - 对话摘要可召回。
   - quarantined 不可召回。
   - private 可召回但 secret 不注入。

这轮完成后，桌宠才真正具备“从真实长期记忆中取上下文”的基础。后续再做中文模型和复杂场景策略才有评估意义。
