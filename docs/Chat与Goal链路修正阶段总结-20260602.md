## 1. 这轮修复要收口什么

这轮改动不是单点修 bug，而是在收口一条用户可见状态机链路：

```text
短聊天 / Goal 输入
-> 选择执行入口
-> 运行中状态投影
-> 结果写回
-> review_ready / blocked / completed
```

前一阶段的主要问题是，`chat`、`goal`、`projected run`、`reply_stored` 各自能工作，但它们对“同一轮执行”没有统一锚点，导致用户视角上出现以下错位：

1. Goal 首发会同时出现两个 `Working`
2. projected run 完成后，最终结果会插到 live block 上方
3. Goal 跟进输入的行为不稳定，终态 Goal 和非终态 Goal 的处理不一致
4. v2 链路在“工具已执行但最终收尾失败”时，会丢失可审阅答复
5. 普通短聊天里只要提到 agent/team，就容易被错误升级成更重的 Goal/能力申请路径
6. projected placeholder 会污染后续 LLM 历史上下文

这几个问题本质上都在破坏同一个约束：同一轮用户动作只能对应一个清晰、可回写、可收尾的用户可见运行态。

## 2. 状态机口径下的目标约束

本轮收口后的目标约束如下：

- 短聊天仍然默认停留在 `chat.*` 状态，不因提到 agent/team 关键字就自动升级
- Goal 首次输入进入 `goal.planning -> goal.working`，只能有一个前台可见运行块
- 后台 Goal task 的投影必须挂靠到时间线中的稳定占位消息，而不是游离在消息列表外
- projected run 完成后，结果必须原位收口到同一消息，不允许出现“结果在上、运行块在下”的插位
- 即便 final answer 收尾失败，只要工具结果或部分文本已产生，就必须给用户落一条可审阅 assistant reply
- 运行态占位信息不能再次回灌到 LLM prompt

可简化为下面这条用户可见链路：

```text
goal.composing
-> goal.created
-> goal.projected_working
-> goal.synthesizing
-> goal.review_ready | goal.blocked | goal.failed
```

## 3. 根因归纳

### 3.1 Goal 首发双入口

Goal 模式首条输入原来会同时触发：

- 创建 Goal / 后台编排执行
- 前台 `chat.sendMessage(...)`

这会制造两个不同 `request_id`，前端自然会渲染出两个并行 `Working`。

### 3.2 projected run 没有稳定消息锚点

后台 Goal task 的 live 投影原来只是 `request_id` 级别的运行态，没有先在时间线里创建一个稳定 assistant message 占位。结果完成时又是追加一条新消息，而不是更新原位置，所以 UI 只能表现成“上面是最终结果，下面还挂着旧 live block”。

### 3.3 前端时间线把 `messages` 和 `projectedRuns` 当成两个平行列表

即使后端写回成功，前端仍会先渲染 `messages`，再渲染 `projectedRuns`。如果 projected run 没有内联锚点，只能永远处于“补丁层”。

### 3.4 `goals_changed` 和 projected run 的竞态

后台写回期间，如果前端因为 `goals_changed` 提前刷新历史，会把尚未收尾的 projected run 和新消息叠在一起，进一步放大插位问题。

### 3.5 v2 后置收尾失败时没有恢复答复

原来的 `send_v2` 在工具已经执行、甚至已经积累部分最终文本后，只要 closing turn 报错，外层仍可能直接返回裸失败并发出 `message_id = null` 的 `reply_stored`，用户只能看到失败，看不到可审阅产出。

### 3.6 Goal 升级判定过宽

短聊天里只要显式提到 agent/team/subagent 等词，就容易被误判为需要 Goal 或能力升级，这与“用户只是要求普通对话里调用某种执行能力”的语义不一致。

## 4. 已落地修复

### 4.1 Goal 会话入口收口

已完成的入口侧修复包括：

- Goal 首条输入不再同时触发前台聊天发送；首条消息成为 Goal objective，由编排器接管执行
- 首条用户消息仍会追加进 Goal 会话，保证时间线完整
- 非终态 Goal 的 follow-up 改为 append-only，不再意外启动前台长任务发送
- `blocked` Goal 允许 resume
- `rework_required` 会回退到 `planning`
- 终态 Goal 会话中的 follow-up 不再硬接着旧 Goal 跑，而是直接创建新 Goal

这部分修复的核心意义，是让“Goal 会话”重新成为单入口状态机，而不是前台 chat run 与后台 goal run 并行抢占 UI。

关键文件：
- `apps/desktop/src/windows/AgentWorkspacePanel.tsx`
- `apps/desktop/src/windows/AgentWorkspacePanel.test.tsx`
- `apps/desktop/src/windows/AgentWorkspacePanel.goalMount.test.tsx`

### 4.2 projected run 改为“占位消息 + 原位更新”

这是本阶段最关键的结构性修复。

新增能力：

- 新增 content block：`runtime_projection`
- 新增消息内容更新接口：`update_message_content(...)`
- Goal task 启动时，先在目标会话中落一条 durable assistant placeholder
- placeholder 创建后立即发 `reply_stored`
- 后台执行完成、失败、或写回失败时，统一更新同一条消息内容，而不是另起一条 assistant message

这样，projected run 从“时间线外的悬浮状态”变成了“时间线内一条可以不断进化的消息”。

关键文件：
- `crates/conductor-core/src/chat/types.rs`
- `crates/conductor-core/src/chat/session.rs`
- `apps/desktop/src/ipc/invoke.ts`
- `apps/desktop/src-tauri/src/worker.rs`

### 4.3 projected run 前端交接收口

前端侧配套修复包括：

- 在 `thinking-update phase=done` 到最终写回之间，统一显示 `synthesizing` 文案，避免用户误以为卡死
- 只要仍有 active projected run，`goals_changed` 就不触发历史强刷，避免竞态
- `ChatTimelinePane` 现在优先从消息内容中识别 `runtime_projection` placeholder，并把对应 projected run 内联渲染到该消息位置
- 时间线尾部的 `projectedRuns` 区域只保留给没有 placeholder 的孤儿运行态

这部分修复之后，时间线模型从：

```text
messages + projectedRuns
```

变成：

```text
messages(其中部分消息携带 projected runtime 占位) + fallback projectedRuns
```

关键文件：
- `apps/desktop/src/windows/useChatSession.ts`
- `apps/desktop/src/windows/ChatTimelinePane.tsx`
- `apps/desktop/src/windows/useChatSession.projectedRuns.test.tsx`

### 4.4 Goal projected writeback 失败也能闭环

即使执行本身成功，如果结果写回 Goal 会话失败，原来 placeholder 也可能一直挂住。现在 worker 会把同一个 placeholder 更新成明确的失败投影，至少保证用户能看到这轮执行停在什么阶段、为什么没能写回。

关键文件：
- `apps/desktop/src-tauri/src/worker.rs`

### 4.5 v2 晚期失败恢复为可审阅答复

新增 `build_error_reviewable_message(...)`，在出现“工具阶段已经有产物，但最终 closing answer 失败”的情况下：

- 优先从 `TimeoutRecoveryState` 提取已累积的最终文本和工具记录
- 生成一条可审阅 assistant reply
- 写库并发出正常 `reply_stored`
- 记录 `failed_recovered` stage
- 向上返回 `Ok(ChatReply)`，而不是裸 `Err`

这保证了用户看到的是“失败但有可审阅结果”，而不是“只失败、不知道执行到了哪”。

关键文件：
- `crates/conductor-core/src/chat/send_v2.rs`

### 4.6 普通短聊天不再被 agent/team 关键字误拦截

`should_offer_goal_upgrade(...)` 已收窄，只在明确 Goal/long-task 语义下触发升级，不再因为用户提到 agent/team/subagent 就直接走重路径。

这一步的目标不是放松能力边界，而是纠正入口语义：显式要求“在普通聊天里调用某种执行方式”不应自动被解释为“我要进入 Goal 生命周期”。

关键文件：
- `crates/conductor-core/src/chat/send_v2.rs`

### 4.7 runtime placeholder 不再污染 LLM 历史

`plain_text_for_llm(...)` 已忽略 `runtime_projection` block，因此后台占位 JSON 不会被重新拼进后续 prompt。

同时补了回归测试，确保 flatten 后只保留真正可读文本。

关键文件：
- `crates/conductor-core/src/chat/util.rs`

## 8. 2026-06-02 补充收口

在上一轮主链路收口之后，又继续补了一轮与用户直观体验强相关的修复，重点不再是扩状态机分支，而是把多余控制面、被截断的投影输出、以及桌宠状态回写这几类“破坏用户感知一致性”的问题清干净。

### 8.1 右侧 Goal 面板改成只读的当前会话状态

此前右侧 `GoalConsole` 实际上是另一套并行控制面，里面同时承载了：

- Goal review verdict 操作
- review queue
- team / lane 级生命周期展示
- task / cycle / transcript 明细

这会让用户除了聊天时间线之外，还能从侧栏直接推动 Goal 生命周期，形成第二条 UI 控制路径。补充收口后的处理方式是：

- 右侧面板只展示当前 Goal、当前 cycle、当前任务状态、heartbeat
- 不再暴露 review / rework / accept / reject 按钮
- 不再暴露 task id、result ref、team lane transcript 等实现细节
- 过滤到当前 goal/session，仅保留“这轮正在发生什么”的只读视图

这一步的核心意义不是“弱化功能”，而是重新确立用户可见生命周期的单入口：聊天时间线负责交互与收尾，右侧 Goal 面板只负责状态观察，不再承担状态推进职责。

关键文件：
- `apps/desktop/src/windows/GoalConsole.tsx`
- `apps/desktop/src/windows/GoalConsole.test.tsx`
- `apps/desktop/src/windows/GoalConsole.integration.test.tsx`

### 8.2 Goal 输出被截断的根因与修复

这轮确认的明确截断点在 `crates/conductor-core/src/adapters/claude_p.rs`。

原先 Goal 子运行结果投影回聊天时，存在两层摘要化截断：

- `parse_output(...)` 的 fallback summary 会截前 500 字符
- `project_run_to_chat_session()` 在拿不到 summary 时，会退化成 `output.raw_stdout.chars().take(300)`

这意味着即使底层子运行已经产出了完整 stdout，投影到聊天里的用户可见结果仍可能只剩一小段摘要文本。补丁后的处理是：

- `project_run_to_chat_session()` 优先直接使用完整 `output.raw_stdout.clone()`
- completion card 不再重复塞一份截断 summary，而是把该字段置空，避免和完整正文形成双轨内容

修完之后，Goal 子运行的可见投影不再默认被压缩成 300 字符摘要，聊天侧能拿到完整 stdout 文本。

关键文件：
- `crates/conductor-core/src/adapters/claude_p.rs`

### 8.3 桌宠主形象状态 hook 恢复为以后端状态为准

桌宠形象回退问题的根因有两层：

- `usePetVisualState.ts` 监听 `pet_avatar_changed` 时按 camelCase 读 `avatarId` / `activityVariant`，但后端事件实际发的是 snake_case：`avatar_id` / `activity_variant`
- 前端挂载后没有先从后端主动拉取当前 avatar 和 mood，导致首屏只能靠本地默认值起步

这会让桌宠在真实状态已经切换后，前端仍停在占位状态，或者被 `original/idle` 占位状态重新覆盖。补充修复包括：

- `usePetVisualState.ts` 挂载时主动调用 `api.getCurrentAvatar()` 与 `api.getMoodState()`
- 同时兼容 camelCase 与 snake_case 事件载荷
- `AvatarRenderer.tsx` 增加占位态保护，避免尚未 hydrate 的默认 `original/idle` 把真实后端 avatar 冲掉
- `avatarId` 加入 effect 依赖，确保后端真实值到达后重新判定

这一步把桌宠视觉状态重新拉回“以后端状态为真源”的模型。

关键文件：
- `apps/desktop/src/windows/usePetVisualState.ts`
- `apps/desktop/src/windows/usePetVisualState.test.ts`
- `apps/desktop/src/windows/AvatarRenderer.tsx`

### 8.4 去掉额外的“教用户怎么走 Goal 状态机”的侧栏路径

这轮没有继续强化“review 一个个 team/task”的交互，而是反过来收掉这条用户路径。

当前主路径上的变化是：

- `GoalConsole` 已不再渲染 review queue、agent lane、transcript、verdict action
- 右侧面板不再承担“继续推进 Goal”的职责
- 如果旧的 `AgentLanes` 组件被误渲染，review action 入口也已被额外封死
- 对话时间线里的 `CapabilityRequestCard`、空白态文案、任务抽屉空态文案也已改成只描述当前状态，不再额外教用户如何使用 Goal
- Goal 状态投影与 cycle 收口文案改成纯状态描述，不再写入“继续补充要求就会沿用当前 Goal”这类固定指导语

换句话说，这轮对“不要在 UI 上硬编码 user guidance / state machine coaching”的落实，不是再改几句提示词，而是直接把那条并行交互面从主路径移开。用户现在只会看到当前 Goal 正在发生什么，而不是被引导去逐队、逐任务地做审批。

补充说明：

- `AgentLanes.tsx`、`TaskDrawerBody.tsx`、`CapabilityRequestCard.tsx` 里仍可能残留少量旧说明性文案
- 这些文案当前已不在主链路控制面上，但仍值得后续继续清理

## 9. 阶段结论

本阶段最重要的变化，不是“又修了几个 bug”，而是把 Chat/Goal/projected run/reply writeback 重新收拢成了一条可解释的状态机链路。

如果用一句话概括当前成果，就是：

```text
同一轮 Goal 执行现在终于有了单入口、单锚点、可原位收尾、失败可恢复的用户可见生命周期。
```

这为后续继续做 AgentTeam、能力入口、设置面板、Prompt 简化和更强的 runtime 治理，提供了一个可以继续演进的稳定基线。
