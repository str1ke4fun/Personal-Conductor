## 四、工作台面板拆分派工

### 产品口径

不做独立前端壳、不新建独立项目。目标是在现有 Tauri 桌宠应用内，把“对话面板 + 任务列表”升级为一个独立功能面板：左侧是与桌宠的历史聊天记录和上下文 resume，中间是单个 chat 的完整类终端时间线，右侧是可收起任务抽屉，展示 Claude / Codex / 桌宠自身正在完成、待审和失败的任务。

当前事实：
- 已有独立入口：`apps/desktop/src/chat.tsx` 挂载 `ChatPanel standalone`，`apps/desktop/src/tasks.tsx` 挂载 `TaskPanelContent standalone`。
- 已有 Tauri 窗口：`tauri.conf.json` 中存在 `chat`、`tasks`、`settings` 三个窗口，但 `chat` 仍是 450x600 小面板，`tasks` 是独立 400x500 列表。
- 当前缺口：两个窗口并列存在，不能在同一工作台里完成“选历史会话 -> resume 上下文 -> 看工具/终端输出 -> 审阅/调度任务”的闭环。

### 非目标 / 禁止误解

- 不要把工作台继续塞进 `SettingsPanel`。
- 不要新建独立 Web 项目、Electron 壳或外部网页。
- 不要只做三栏静态 UI mock；左侧 session、中间 timeline、右侧 task drawer 都必须接真实 IPC 或明确标注阻塞。
- 不要用 `pet_message` 作为 ChatPanel 历史同步通道。
- 不要把当前 `codex.*` stub 误标为“Codex 已打通”。
- 不要继续在前端手工拼 `listTasks(false)` + `listAgentTasks(true)` + `proposals` 作为最终任务状态。

### BUG-007：对话时间线错乱 + 假流式传输

**现象**：对话面板时间线错乱，上下文顺序不对；没有真正的流式输出，用户等待期间看不到逐步生成的内容。

**根因分析**（5 层问题）：

1. **LLM 调用是批量的，不是流式的**：`llm.rs:235` `call_with_tools_with_messages` 发送标准非流式 HTTP 请求，无 `stream: true`，无 SSE 解析。整个响应体 `await` 完成后才返回。

2. **"stream-chat-token" 是假流式**：`chat.rs:327-330` 在每轮 LLM 返回**完整文本**后，将整段文本作为单个 `stream-chat-token` 事件 emit。不是逐 token，是逐轮次（一轮 = 一整块文本）。

3. **assistant 消息只在整个 query loop 结束后才写入 DB**：`chat.rs:511` `insert_message` 在 `for _turn in 0..max_turns` 循环退出后才调用。用户消息在 `chat.rs:232`（循环前）写入，assistant 消息在循环后写入。多轮工具执行期间 DB 中无 assistant 消息 → `created_at` 反映的是完成时间，不是首次响应时间。

4. **前端完成后丢弃流式内容**：`ChatPanel.tsx:195-198`，`sendChatMessageV2` resolve 后调用 `setMessages(reply.history)` 替换整个消息列表 + `setStreamTokens([])` 清空临时流式显示 → 等待期间展示的内容被丢弃。

5. **时间线按 DB `created_at` 排序**：`chat.rs:996-1016` `history()` 按 `created_at ASC` 排序。user 和 assistant 的 created_at 差异大（循环前 vs 循环后），排序本身正确，但用户在命令返回前看不到任何增量消息。

**影响**：
- 用户发送消息后，长时间无反馈（LLM 批量调用期间）
- 多轮工具调用时，每轮文本一次性涌出，不是逐字显示
- 刷新或重渲染时，等待期间的上下文丢失
- 多条消息的 `created_at` 可能非常接近（循环内多轮快速完成），排序不稳定

**修复方向**：
1. `llm.rs`：实现真正的 SSE 流式解析（`stream: true` + 逐 token 事件）
2. `chat.rs`：assistant 消息在每轮 LLM 返回后立即增量写入 DB（不等 loop 结束）
3. `ChatPanel.tsx`：流式 token 不在完成后清空，而是作为消息内容的一部分保留
4. 前端消息排序：除 `created_at` 外增加 `seq` 或 `turn` 字段保证稳定排序

**状态**：✅ 已修复（2026-05-28）— `llm.rs` 新增 OpenAI-compatible / Anthropic-compatible SSE 真流式解析；`send_message_v2` 改走 `call_with_tools_with_messages_streaming()`，`stream-chat-token` 由 SSE delta 逐段 emit，不再等完整响应后一次性发送；`chat_messages` 新增 `seq` 并按 `seq, created_at, id` 稳定排序；前端 `useChatSession` 将流式 token 合并进临时 assistant 消息，完成后用持久化 history 替换，避免等待期内容丢失。验证：`rtk cargo check -p conductor-core --features tauri-events`、`rtk cargo test -p conductor-core llm`、`rtk cargo test -p conductor-core chat`、`rtk npm.cmd run build`、`rtk cargo check -p conductor-desktop` 均通过（保留既有 4 个 warning）。

### BUG-008：codex::test_read_output_with_offset 测试 panic

**现象**：`cargo test` 运行时，`codex::tests::test_read_output_with_offset` panic：`range start index 250 out of range for slice of length 0`。

**根因**：`codex.rs:479` 的 `read_output` 函数在 session output 为空时，仍尝试按 offset 切片。新增的测试用例暴露了 `offset > stdout.len()` 时未做边界检查的问题。

**影响**：测试套件 1/230 失败；不影响运行时（需要空 session + 非零 offset 才触发）。

**修复方向**：`read_output` 中对 offset 做 `min(offset, stdout.len())` 防御，或在 offset 超出时返回空字符串。

**状态**：✅ 已修复（2026-05-28 19:30）— `read_output` 改为先转 UTF-8 再按字符串偏移切片，避免 GBK→UTF-8 转换时的替换字符不一致。230 测试全通过。

### BUG-009：设置面板关闭后无法再次打开

**类型**：Bug（回归）
**提交时间**：2026-05-28
**涉及模块**：`apps/desktop/src/windows/PetWindow.tsx`、`apps/desktop/src/windows/SettingsPanel.tsx`

**问题描述**：
右键菜单打开设置面板后，关闭面板（无论用关闭按钮还是窗口 X），再次右键点击设置时面板无法重新打开。

**根因分析**：
`PetWindow.tsx:138` `openPanelWindow()` 使用 `WebviewWindow.getByLabel(label)` 获取窗口引用。当窗口被关闭（destroy）后，`getByLabel` 返回一个 stale 引用，`win.show()` 抛异常，catch 块尝试 `new WebviewWindow(label, ...)` 创建新窗口，但 Tauri v2 可能因 label 未完全释放而创建失败。

关键代码路径：
- `PetWindow.tsx:144-154`：getByLabel → show() → catch → new WebviewWindow
- `SettingsPanel.tsx:292`：`getCurrentWindow().hide()` 只隐藏不销毁
- 但窗口 X 按钮走 Tauri 默认 close（销毁），与 hide() 行为不一致

**修复方向**：
1. `openPanelWindow` 中 `show()` 失败后，先 `try { await win.destroy() }` 清理 stale 引用，再创建新窗口
2. 或在 SettingsPanel 中监听 `onCloseRequested`，阻止默认销毁行为，改为 `hide()`
3. 或使用 `WebviewWindow.getAll()` 检查窗口是否实际存在，而非依赖 `getByLabel` 的返回值

**状态**：✅ 已修复（2026-05-28）— PetWindow.tsx `openPanelWindow()` 在 `show()` 失败后先 `destroy()` 清理 stale 引用，再创建新窗口。`npm run build` 通过。

### AUDIT-008：桌宠子形象频闪与状态仲裁

**断点**：
- `pet_state`、`pet_avatar_changed`、`pet_expression` 三条事件源并行驱动前端。

2. 状态协调器实现去重：相同 `avatarId + activityVariant + moodZone + petState` 不触发 setState。
3. 状态协调器实现优先级：`error/update > tool_calling/writing > thinking/working > done/waiting_user > idle`。
4. 状态协调器实现最短展示时间：
   - `tool_calling`：1200ms
   - `writing`：1500ms
   - `done/update`：1800ms
   - `idle`：延迟 2000ms，且不能抢占 active chat / hook
5. `AvatarRenderer` 的 `img key` 改为 `avatarId + asset.src`，不要把 raw `activityVariant/moodZone` 放进 key。

10. hook 活动改为显式 `pet_activity` 事件，带 `source/session/priority/ttl`，不要只靠 pending task 间接推导。

**验收标准**：

**状态**：✅ 大部分完成（2026-05-29）— usePetVisualState.ts(199行)统一消费 3 事件源+去重+优先级+最短展示时间(tool_calling 1200ms/writing 1500ms/done 1800ms/idle 2000ms)；AvatarRenderer key 改为 avatarId+asset.src 减少无意义 remount；avatar.rs set_activity_variant 后端去重；worker.rs 状态去重+focus watcher 不再直接发 idle。剩余：hook pet_activity 事件改造、Live2D 路径去重。
