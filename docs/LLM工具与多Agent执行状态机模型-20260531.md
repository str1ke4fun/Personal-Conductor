## 1. 当前应用内 LLM 状态机

当前 chat v2 的核心流程是：

```text
request_received
  -> session_resolved
  -> context_loaded
  -> system_prompt_built
  -> tool_definitions_built
  -> llm_turn_running
  -> llm_text_streaming?
  -> llm_tool_calls_returned?
  -> tool_calls_executing
  -> tool_results_appended
  -> llm_turn_running
  -> final_reply_stored
  -> done

任意阶段 -> failed / timeout
```

关键代码证据：

- `send_v2.rs` 构造 system prompt 和工具定义。
- `build_tool_definitions(config, user_prompt)` 先生成本轮工具列表。
- LLM 返回 `tool_calls` 后，后端逐个执行 `execute_tool_call()`。
- 工具结果追加回 messages 后再进入下一轮 LLM 调用。

这意味着：当前不是“LLM 需要时再查工具列表”，而是“后端先筛出一批工具定义，LLM 在这些工具里决定是否调用”。

## 2. 当前工具暴露状态机

```text
tool_registry_registered
  -> allowed_tool_ids_loaded
  -> contextual_tool_ids_matched
  -> skill_packages_matched
  -> policy_filter_applied
  -> legacy_fallback?
  -> final_tool_defs
```

当前逻辑：

1. `register_builtin_tools()` 注册内置工具。
2. 读取 `config.llm.allowed_tool_ids`。
3. 根据 `tool_tiers` 的关键词命中得到 `context_ids`。
4. 通过 `match_enabled_skills -> collect_capabilities -> PolicyEngine::filter_tools` 尝试新路径。
5. 如果 policy 结果为空且 `allowed_tool_ids` 非空，走 legacy fallback。
6. 最终工具 id 转成 LLM function name，例如 `agent.start` -> `agent__start`。

当前默认配置的重要事实：

- `default_allowed_tool_ids` 包含 `agent.start`、`agent.read_output`、`agent.stop`。
- 默认不包含 `subagent.claude_p`。
- `tool_tiers` 命中 `agent/启动/运行/子任务` 时会暴露 agent 工具。
- `tool_tiers` 命中 `团队/协作/team` 时会暴露 `agent.team.*` 和 mailbox 工具。
- deprecated `coding_assistant` skill 的 allowed tools 包含 `subagent.claude_p`。

## 3. 目标态：渐进式工具暴露状态机

用户之前设计的是渐进式暴露。更合理的 LLM 状态机应是：

```text
llm.minimal_context
  -> llm.intent_classified
  -> llm.answer_directly

llm.intent_classified
  -> llm.needs_tool
  -> llm.query_tool_catalog
  -> llm.select_tool_intent
  -> policy.authorize_tool
  -> tool.execute
  -> llm.observe_tool_result
  -> llm.continue_or_finish

llm.needs_agent
  -> llm.propose_mode_upgrade
  -> user.approves_agent_or_goal
  -> agent_or_goal.start
```

这个目标态的关键不是“工具越少越好”，而是把风险和工作时长分层。前端只需要区分短任务和长任务；普通聊天、文档协作、代码协作是短任务内的子场景，不需要拆成三个模块。

- 短任务：默认给低风险上下文、记忆、任务、只读工具；按意图渐进暴露文档、Office、Lark、文件读写、命令、测试、代码 skill。
- 长任务：在用户确认后暴露 Goal、AgentRun、AgentTeam、Runtime API、Review Queue 等能力。

LLM 显式提到 `agentteam`、`claude`、`subagent` 时，应先进入 `llm.query_tool_catalog` 或“建议升级模式”，而不是直接执行。

## 4. ToolCall 状态机

当前 `tool_calls` 表中的状态流：

```text
pending
  -> executing
  -> succeeded

pending
  -> executing
  -> failed

pending
  -> executing
  -> approval_required
```

工具调用需要被投影为用户可见事件：

| ToolCall 状态 | LLM 视角 | 用户视角 |
|---|---|---|
| `pending` | 工具调用已提出 | 准备调用 |
| `executing` | 等工具结果 | 正在执行工具，显示工具名和 elapsed |
| `approval_required` | 等用户授权 | 等待授权，不是卡住 |
| `succeeded` | 可继续推理 | 已完成，结果可展开 |
| `failed` | 需要解释/重试/降级 | 失败，显示错误和下一步 |

## 5. `claude -p` 三条执行路径

### 5.1 `agent.start`

```text
llm.tool_call(agent.start)
  -> tool_call.pending
  -> tool_call.executing
  -> agent_run.queued
  -> agent_run.running
  -> agent_run.succeeded / failed / stopped
  -> tool_call.succeeded
  -> llm.observe(run_id)
```

用途：

- 应用内推荐的后台 Agent 路径。
- 有 `AgentRun`、pid、stdout/stderr sidecar、output_ref、stop 能力。

问题：

- 当前默认工具配置已经包含 `agent.start`，短任务也可能拿到它。
- 需要前端模式和后端工具策略共同限制：只有用户确认升级为长任务、明确后台 Agent 或明确多 Agent 场景时才暴露或允许执行。

### 5.2 `subagent.claude_p`

```text
llm.tool_call(subagent.claude_p)
  -> tool_call.pending
  -> tool_call.executing
  -> process.running_without_agent_run
  -> process.exited
  -> tool_call.succeeded / failed
```

用途：

- 一次性同步子 Agent。
- CLI 调试路径。

问题：

- 不创建完整 `agent_runs`。
- 中间缺少 pid、queued/running/succeeded 状态。
- 用户只能看到工具执行中，无法看到真实后台 Agent lane。

结论：

- 应用内长任务应收敛到 `agent.start` 或 `ClaudePAdapter`。
- `subagent.claude_p` 应标注为 debug/legacy 路径，或封装成可观察 AgentRun。

### 5.3 `ClaudePAdapter`

```text
agent_task.proposed
  -> queued
  -> claimed
  -> running
  -> adapter.spawn
  -> agent_run.queued
  -> agent_run.running
  -> child_process.running
  -> task.review_ready / accepted / failed
```

用途：

- Goal/AgentTask 场景的内置 Claude 执行器。
- 子进程环境注入 `RUNTIME_API_URL`、`RUNTIME_TOKEN`、`WORKSPACE_ID`、`TASK_ID`。

理想通信方式：

- 子进程通过 Runtime API 写 heartbeat、message、task result、permission request。
- UI 通过 Tauri event 或 SSE 投影实时状态。

当前需要注意：

- Adapter 已经注入 Runtime API 环境变量。
- 但桌面端没有看到 `RuntimeApiServer` 的启动入口；如果 runtime 未启动，子进程无法真正回写。

## 6. AgentTeam 状态机

AgentTeam 当前是协作容器，不是执行器。

生命周期：

```text
draft
  -> planning
  -> awaiting_plan_approval
  -> executing
  -> awaiting_review
  -> accepted
  -> archived

awaiting_plan_approval -> rework_required -> planning
awaiting_review -> rework_required -> planning
```

AgentTeam 工具能力：

- `agent.team.create`
- `agent.team.list`
- `agent.team.add_member`
- `agent.team.snapshot`
- `agent.mailbox.send`
- `agent.mailbox.list`
- `agent.mailbox.mark_read`

LLM 何时应调用 AgentTeam：

| 条件 | 动作 |
|---|---|
| 用户显式提到团队、协作、AgentTeam | 查询/暴露 team 工具，解释用途 |
| 任务可拆成多个并行角色 | 先提出 Dispatch Plan，再创建 team |
| 已有多个 AgentRun 需要协作 | add_member，把 run 绑定到 team |
| 需要跨 agent 通知/同步 | 使用 mailbox/message |
| 只是普通问答或单文件修改 | 不创建 AgentTeam |

关键约束：

- `agent.team.create/add_member` 不会启动 `claude -p`。
- `team.executing` 必须关联至少一个真实 executor：`agent_run_id`、`task_id` 或外部 session。
- 用户看到 team executing 时，也必须看到 Agent lane。

## 7. Goal/OODA 状态机

当前后端已实现 GoalRun 和 GoalCycle 基础状态机。

GoalRun：

```text
draft
  -> planning
  -> awaiting_plan_approval
  -> running
  -> awaiting_review
  -> accepted
  -> archived

awaiting_plan_approval -> rework_required -> planning
running -> blocked -> running
running -> failed / cancelled
failed -> archived
cancelled -> archived
```

GoalCycle：

```text
observing
  -> orienting
  -> deciding
  -> dispatching
  -> executing
  -> reviewing
  -> summarizing
  -> completed

任意非终态 -> failed / blocked / cancelled
blocked -> 任意非终态恢复
```

GoalOrchestrator 当前核心约束：

- Act 阶段不直接执行工具。
- Act 只创建 task、lease、message、event。
- 真实执行应由 Adapter 或外部 agent 完成。

这和参考方案中的边界一致。

## 8. 多进程通信与同步模型

目标同步模型：

```text
SQLite runtime_events = canonical event ledger
Runtime API HTTP = external agent write path
Runtime API SSE = external agent subscribe path
Tauri event = desktop UI refresh path
agent_messages / mailbox = agent-to-agent messages
heartbeats = liveness and current stage
leases = claim and write-scope conflict control
agent_runs / command_runs = process execution evidence
```

当前代码已有的对象：

- `AgentMessage`
- `AgentHeartbeat`
- `WorkLease`
- `AgentTask`
- `GoalRun` / `GoalCycle`
- `AgentRun`
- `ToolCall`
- `runtime_events`
- Runtime API endpoints：messages、heartbeats、task claim/complete/fail/block、goals、permission requests、events SSE。

当前尚未证明贯通的地方：

- Runtime API server 未看到桌面端启动入口。
- 前端 Goal/Agent lanes 组件未挂载到主界面。
- LLM 短任务不使用 Runtime API 作为工具发现或 agent 协作总线。
- Lark connector 存在，但没有看到运行时注册到 ConnectorRegistry 的入口。

## 9. LLM 决策边界

LLM 可以自主决定“是否需要工具”，但不应自主越过“工作模式边界”。

建议边界：

| LLM 判断 | 允许直接做 | 需要用户感知/确认 |
|---|---|---|
| 只需回答 | 直接回复 | 无 |
| 需要读上下文 | 读文件/任务/记忆 | 显示工具摘要 |
| 需要写入文档/代码 | 生成计划或 diff | 用户确认写入 |
| 需要运行命令 | 生成命令意图 | 权限审批或明确模式 |
| 需要后台 Agent | 提议升级为长任务 | 用户确认后启动 `agent.start` 或创建 Goal |
| 需要 AgentTeam | 提出 team plan | 用户确认 team/executor 关系 |

结论：

- LLM 自主选择工具是合理的。
- 但“工具是否暴露”和“是否允许启动外部进程”必须由模式、policy、权限和用户确认共同决定。
