# 用户侧 Agent 使用场景与状态机模型

日期：2026-05-31

定位：用户体验状态机 / 能力入口 / 可感知投影模型

参考：
- `docs/多Agent共享工作区与自治Goal调度方案-20260530.md`
- `docs/运行时可观察性与Agent链路收口-20260531.md`
- 当前代码：`chat/send_v2.rs`、`chat/tools.rs`、`goals.rs`、`goal_tasks.rs`、`goal_orchestrator/*`、`agent_runs.rs`、`agent_teams.rs`、`runtime_api.rs`、桌面端 `AgentWorkspacePanel/ChatComposer/TaskPanelContent/TaskDrawerPane`

## 1. 建模目标

这里建模的是“用户以为系统正在做什么”的状态机，不是单个后端表的状态机。

用户侧需要回答三个问题：

1. 我现在处于哪种工作形态：短任务，还是长任务 / Goal。
2. 系统现在是否正在工作：LLM 回复中、工具执行中、后台 Agent 运行中、等待我处理。
3. 结果落在哪里：聊天记录、文件、飞书文档、代码变更、AgentRun 输出、Goal Review 队列。

因此用户侧状态必须用“模式 + 运行态 + 结果投影”三层表示。

```text
UserVisibleState =
  ModeState          // 这是什么工作模式
  + ActivityState    // 当前是否在工作、卡住、等待审批、完成
  + ArtifactState    // 结果和副作用落在哪里
```

## 2. 全局用户状态机

```text
idle
  -> composing
  -> mode_selected
  -> context_bound
  -> working
  -> reviewing
  -> completed

working -> awaiting_user_action -> working
working -> blocked
working -> failed
reviewing -> rework_requested -> working
reviewing -> accepted -> completed
completed -> archived
```

| 状态 | 用户看到什么 | 后端证据 |
|---|---|---|
| `idle` | 没有活动任务，只有历史会话/任务列表 | 无 active chat run、无 running agent run、无 active hook task |
| `composing` | 用户正在输入 | 前端输入框状态 |
| `mode_selected` | 已明确是短任务 / 长任务 | 前端显式模式字段，当前缺失 |
| `context_bound` | 已绑定当前文件、工作区、飞书文档、代码仓库或 Goal | workspace/session/doc/task 元数据 |
| `working` | 系统正在处理，必须显示 elapsed、阶段、最近动作 | `active_run`、`tool_calls`、`agent_runs`、`tasks`、`heartbeats` |
| `awaiting_user_action` | 等待授权、选择方案、确认写入、处理冲突 | permission request、proposal、review queue |
| `reviewing` | 系统完成了一轮，需要用户验收结果 | pending hook review、Goal review、AgentTask `review_ready` |
| `completed` | 本轮已完成并有结果引用 | assistant message、file diff、output_ref、summary_ref |
| `blocked` | 系统无法继续，显示阻塞原因和下一步 | task/goal blocked、failed permission、stale heartbeat |
| `failed` | 执行失败，显示错误和可恢复入口 | tool_call failed、agent_run failed、command_run timed_out |

关键约束：

- `pending` 不能表示“正在工作”。`pending` 只表示等待用户审阅或处理。
- 任意超过 5 秒的后台动作都必须有 `working/running/in_progress` 可见状态。
- 任意写入、外部进程、飞书写操作、代码修改，都必须在 `ArtifactState` 有落地记录。

## 3. 场景一：普通 LLM 聊天 / 记录

### 3.1 用户目标

用户只是和 LLM 对话，通常不需要后台 subagent。它可以读取少量上下文，也可以把结论写成记录、任务或记忆，但不应默认启动 `claude -p` 或 AgentTeam。

### 3.2 推荐入口

入口应是默认 `短任务` 模式。普通聊天、文档协作、代码协作不需要拆成三个前端模块，它们是短任务内的不同子场景。

用户可见控件：

- 模式：`短任务`
- 权限：`只读` / `询问后写入`
- 可选：`记录到任务/记忆`

当前代码现状：

- `AgentWorkspacePanel` 有 `capabilityMode` 和 `planOnly` 前端状态。
- `ChatComposer` 显示这些 chip。
- `useChatSession.sendMessage()` 实际只调用 `sendChatMessageV2(content, sessionId)`，没有把模式传到后端。

### 3.3 用户状态机

```text
chat.idle
  -> chat.composing
  -> chat.submitted
  -> chat.llm_working
  -> chat.tool_working?
  -> chat.responding
  -> chat.recording?
  -> chat.done

chat.tool_working -> chat.awaiting_permission
chat.awaiting_permission -> chat.tool_working
chat.tool_working -> chat.failed
```

| 状态 | 用户需要感知 | 允许能力 |
|---|---|---|
| `chat.llm_working` | “正在生成回复”与流式 token | LLM 文本生成 |
| `chat.tool_working` | 工具摘要，例如“读取 3 个文件” | 低风险读取、任务/记忆写入 |
| `chat.awaiting_permission` | 明确等待授权 | 写文件、发消息、运行命令 |
| `chat.responding` | 最终自然语言回复 | assistant message |
| `chat.recording` | 记录到哪里 | task、memory、chat history |

### 3.4 不应发生的事

短任务模式不应静默：

- 调 `agent.start` 启动后台 `claude -p`。
- 调 `subagent.claude_p` 长时间运行。
- 创建 AgentTeam。
- 运行 shell/python 修改文件。

如果 LLM 判断需要这些能力，应进入“升级建议”状态：

```text
chat.needs_more_capability
  -> ask_user_upgrade_to_long_task
  -> user_accepts
  -> mode_selected
```

## 4. 场景二：文档协作

### 4.1 用户目标

用户希望 LLM 协助读文档、写作、整理、生成或修改 Office/PPTX/Markdown/飞书文档。这里通常会涉及文件读取、文件写入、Office 工具、飞书 CLI、简单 Python 脚本。

### 4.2 推荐入口

入口仍然是 `短任务` 模式，不需要单独的 Document 前端模块。系统可在短任务内识别文档子场景，来源可以是：

- 用户在聊天里选择“文档协作”。
- 当前 workspace 被识别为 Office/文档。
- 用户显式提到 Word、PPT、Excel、Markdown、飞书文档。
- 用户拖入文件或选择当前文档。

用户可见控件：

- 当前文档或目录。
- 输出目标：聊天草稿 / 本地文件 / 飞书文档 / Office 文件。
- 写入策略：先预览 diff / 直接写入需审批。

### 4.3 用户状态机

```text
doc.idle
  -> doc.context_selected
  -> doc.inspecting
  -> doc.planning_edit
  -> doc.drafting
  -> doc.awaiting_write_approval
  -> doc.applying
  -> doc.verifying
  -> doc.reviewing
  -> doc.done

doc.inspecting -> doc.missing_context
doc.applying -> doc.failed
doc.reviewing -> doc.rework_requested -> doc.planning_edit
```

| 状态 | 用户需要感知 | 能力入口 |
|---|---|---|
| `doc.context_selected` | 当前处理哪个文档 | workspace/file/doc id |
| `doc.inspecting` | 正在读取、解析或导出文本 | `file.read`、`office.inspect_document`、`office.export_text`、Lark read |
| `doc.planning_edit` | 改动计划 | LLM 计划，必要时 `planOnly` |
| `doc.drafting` | 正在写草稿 | LLM 文本生成 |
| `doc.awaiting_write_approval` | 等用户确认写入 | permission/proposal |
| `doc.applying` | 正在写文件、调用飞书、运行脚本 | `file.write/edit`、office patch、shell/python、Lark write |
| `doc.verifying` | 正在检查结果 | 读取输出文件、命令结果、飞书返回 |
| `doc.reviewing` | 等用户审阅 | diff、summary、artifact path |

### 4.4 文档场景的落地记录

每次文档协作结束，用户至少要看到：

- 输入文档：路径或飞书 doc token。
- 生成内容：聊天摘要、草稿、diff 或文件路径。
- 外部动作：执行了哪些工具、是否运行脚本、是否调用飞书。
- 风险：未写入、已写入、写入失败、需要人工继续。

## 5. 场景三：代码协作

### 5.1 用户目标

用户希望 LLM 分析代码、修改代码、运行测试、解释失败、做较复杂推理。复杂任务可能需要 subagent、AgentTeam 或专用 skill 辅助，但这应该是有感知的升级，而不是普通聊天中突然启动。

### 5.2 推荐入口

入口仍然是 `短任务` 模式，不需要单独的 Code 前端模块。系统可在短任务内识别代码子场景；当任务需要持续推进、后台 Agent、多 Agent 或跨轮 Review 时，再升级为长任务。来源可以是：

- 当前 workspace 是代码仓库。
- 用户显式要求“改代码、跑测试、分析错误、实现功能”。
- 用户显式提到 agent、subagent、Claude、Codex、AgentTeam。

用户可见控件：

- 当前仓库。
- 工作范围：只读分析 / 可写实现 / 升级为长任务。
- 执行策略：LLM 自己做 / 启动后台 Agent / 创建 AgentTeam。

### 5.3 用户状态机

```text
code.idle
  -> code.context_bound
  -> code.analyzing
  -> code.plan_ready
  -> code.awaiting_plan_approval?
  -> code.editing
  -> code.testing
  -> code.summarizing
  -> code.reviewing
  -> code.done

code.plan_ready -> code.spawn_agent_suggested
code.spawn_agent_suggested -> code.agent_running
code.agent_running -> code.agent_review_ready
code.agent_review_ready -> code.merging_results

code.editing/testing -> code.blocked
code.reviewing -> code.rework_requested -> code.analyzing
```

| 状态 | 用户需要感知 | 能力入口 |
|---|---|---|
| `code.analyzing` | 读取了哪些文件、当前假设 | `file.glob/grep/read` |
| `code.plan_ready` | 修改计划和影响面 | LLM 计划 |
| `code.editing` | 正在写哪些文件 | `file.write/edit` |
| `code.testing` | 正在运行什么命令 | shell/command_run |
| `code.spawn_agent_suggested` | 为什么需要后台 agent | `agent.start` 或 Goal |
| `code.agent_running` | 哪个 AgentRun、PID、elapsed、输出入口 | `agent_runs` |
| `code.agent_review_ready` | 后台 agent 产出待审 | `output_ref`、summary、review task |
| `code.merging_results` | 合并哪个 agent 的结果 | diff、artifact、final summary |

### 5.4 代码场景中 AgentTeam 的位置

AgentTeam 不是“执行器”。它是协作容器。

用户侧应看到：

```text
team.planning
  -> team.awaiting_plan_approval
  -> team.executing
  -> team.awaiting_review
  -> team.accepted / team.rework_required
```

同时必须看到每个成员背后的真实 executor：

- `agent_run_id`
- `task_id`
- `heartbeat`
- `last_message`
- `output_ref`

没有 executor 的 AgentTeam 只能算“计划/协作容器”，不能显示成“LLM 正在工作”。

## 6. 场景四：长任务 / Goal / 多 Agent 工作区

### 6.1 用户目标

用户不是要一个长 prompt，而是要一个可恢复、可审计、可暂停/继续的目标。Goal 应把任务拆解为 OODA 循环，并派给合适的执行器。

### 6.2 推荐入口

入口应是显式 `长任务` 模式；Goal 是长任务的承载对象，与默认短任务模式区分。

用户可见控件：

- `新建 Goal`
- `启动 / 暂停 / 继续 / 取消`
- 是否自动派工。
- 是否需要计划审批和 Review Gate。
- 最大运行时间、最大 Agent 数、写入范围。

当前代码现状：

- 后端已有 `GoalRun`、`GoalCycle`、`AgentTask`、`GoalOrchestrator`、`Heartbeat`、`Lease`、`Runtime API`。
- 前端已有 `GoalConsole`、`AgentLanes`、`OodaTimeline`、`ReviewQueue`、`AgentTranscript` 组件。
- 但这些组件没有挂到主界面；短任务也没有显式升级到 Goal 的入口。

### 6.3 用户状态机

```text
goal.none
  -> goal.draft
  -> goal.planning
  -> goal.awaiting_plan_approval
  -> goal.running
  -> goal.awaiting_review
  -> goal.accepted
  -> goal.archived

goal.awaiting_plan_approval -> goal.rework_required -> goal.planning
goal.running -> goal.blocked -> goal.running
goal.running -> goal.failed
goal.running -> goal.cancelled
goal.awaiting_review -> goal.rework_required -> goal.planning
```

Goal 运行时的一个 cycle：

```text
cycle.observing
  -> cycle.orienting
  -> cycle.deciding
  -> cycle.dispatching
  -> cycle.executing
  -> cycle.reviewing
  -> cycle.summarizing
  -> cycle.completed

任意非终态 -> blocked / failed / cancelled
blocked -> 任意非终态恢复
```

### 6.4 Goal 用户感知

Goal 模式不应该把 OODA 术语直接丢给普通用户，但需要有可展开层级：

默认视图：

```text
正在推进：文档状态机建模
阶段：执行中
已运行：12 分钟
当前：2 个 Agent 运行中，1 个任务待审
需要你处理：批准写入 docs/...
```

展开视图：

- Goal 状态。
- 当前 cycle 阶段。
- Dispatch Plan。
- Agent lanes。
- Review Queue。
- Recent Events。
- Workspace Projection。

## 7. 用户所需能力入口汇总

| 场景 | 默认入口 | 子场景能力 | 需要升级为长任务的条件 |
|---|---|---|---|
| 短任务 | Chat composer / 当前上下文 | 普通聊天、记录、文档读写、代码分析、小范围修改、脚本辅助 | 需要后台 Agent、跨多轮推进、多 Agent 协作、长时间运行、复杂 Review Gate |
| 长任务 | 长任务入口 / Goal Console | Goal、AgentRun、AgentTeam、Runtime API、Review Queue、Workspace Projection | 不应只靠聊天窗口假装长任务 |

## 8. 最小可感知标准

每一种场景都必须满足：

1. 模式可见：用户知道现在是短任务还是长任务。
2. 活动可见：LLM、工具、AgentRun、hook session、Goal cycle 的运行中状态可见。
3. 等待可见：系统在等用户审批、等输入、等 Review 时要明确显示。
4. 结果可见：文件、summary、output_ref、command_run、tool_call、飞书对象必须有统一入口。
5. 升级可见：从短任务升级到长任务、后台 Agent 或 AgentTeam 必须有用户可感知的确认点。
