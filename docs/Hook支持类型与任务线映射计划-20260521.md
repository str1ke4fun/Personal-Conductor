# Hook 支持类型与任务线映射计划

> 核对日期：2026-05-21  
> 目标：为后续把 Claude Code / Codex 的 hook 事件接入 Conductor task list 留一份实现参考。默认原则是：hook 只做旁路状态同步和上下文读取，不向 agent 注入额外指令，不阻塞正常工作。

## 1. 参考来源

- OpenAI Codex Hooks: https://developers.openai.com/codex/hooks
- OpenAI Codex Advanced Config / Notifications: https://developers.openai.com/codex/config-advanced
- Claude Code Hooks reference: https://code.claude.com/docs/en/hooks
- Claude Code Hooks reference 旧入口： https://docs.anthropic.com/en/docs/claude-code/hooks

## 2. Hook 类型

### 2.1 Codex

Codex 当前可落地运行的 handler 类型：

| 类型 | 当前状态 | 本项目建议 |
| --- | --- | --- |
| `command` | 官方文档标明当前实际运行的类型 | 使用。调用 `conductor hook codex ...`，stdin 读 JSON，stdout 默认保持空 |
| `prompt` | 解析但当前跳过 | 不接入 |
| `agent` | 解析但当前跳过 | 不接入 |
| plugin-bundled hooks | 需开启 `plugin_hooks`，且仍需信任审核 | 后置，不作为 MVP 入口 |

Codex hook 配置位置：

| 位置 | 用途 | 本项目建议 |
| --- | --- | --- |
| `~/.codex/hooks.json` | 用户全局 | 不默认写入，避免污染全局环境 |
| `~/.codex/config.toml` | 用户全局 inline hooks | 不默认写入 |
| `<repo>/.codex/hooks.json` | 仓库级 | 首选，便于随项目闭环 |
| `<repo>/.codex/config.toml` | 仓库级 inline hooks | 可选，但避免与 `hooks.json` 混用 |

Codex 还有 `notify`，目前支持 `agent-turn-complete` 这类外部通知脚本。它适合桌面 toast / webhook，不适合作为完整任务线闭环入口，因为缺少 prompt submit、permission request、tool use 等关键阶段。

### 2.2 Claude Code

Claude Code 支持的 handler 类型更多：

| 类型 | 当前状态 | 本项目建议 |
| --- | --- | --- |
| `command` | 稳定基础能力 | MVP 使用。调用 `conductor hook claude ...` |
| `http` | 支持 POST 事件 JSON 到服务端 | 后置；适合未来常驻 Conductor 服务 |
| `mcp_tool` | 支持调用 MCP 工具 | 后置；不用于本地任务列表最小闭环 |
| `prompt` | 可做 prompt-based hook | 不用于任务线，避免注入上下文打扰 Claude |
| `agent` | 实验性，适合复杂验证 | 后置；生产流程优先 command hook |

Claude Code hook 配置位置：

| 位置 | 用途 | 本项目建议 |
| --- | --- | --- |
| `~/.claude/settings.json` | 用户全局 | 不默认写入 |
| `.claude/settings.json` | 项目级，可提交 | 闭环验证后显式启用 |
| `.claude/settings.local.json` | 项目级本机配置 | 可放本机权限，不等价于启用 hooks |
| plugin / skill / subagent frontmatter | 组件生命周期 | 后置 |

## 3. 关键事件矩阵

### 3.1 Codex 事件

| 事件 | 触发点 | 关键字段 | stdout 风险 | Conductor 映射 |
| --- | --- | --- | --- | --- |
| `SessionStart` | 会话启动、恢复、清理 | `session_id`, `transcript_path`, `cwd`, `source`, `model` | 文本 stdout 会作为额外 developer context | 可记录 session/cwd，但 MVP 可不建任务 |
| `UserPromptSubmit` | 用户 prompt 即将提交 | `session_id`, `turn_id`, `prompt`, `cwd`, `transcript_path` | 文本 stdout 会进入 developer context | 创建或更新任务为 `in_progress` |
| `PreToolUse` | Bash / `apply_patch` / MCP 工具执行前 | `turn_id`, `tool_name`, `tool_use_id`, `tool_input` | 文本 stdout 忽略；JSON 可影响工具调用 | 只记录将要做什么，不返回决策 |
| `PermissionRequest` | Codex 即将发起提权审批 | `turn_id`, `tool_name`, `tool_input`, `description` | 文本 stdout 忽略；JSON 可 allow/deny | 记录提权原因，任务仍 `in_progress` 或标记 `needs_permission` 上下文 |
| `PostToolUse` | 支持的工具执行后 | `turn_id`, `tool_name`, `tool_input`, `tool_response` | JSON 可给模型额外上下文或替换结果 | 记录修改/命令结果摘要 |
| `Stop` | turn 结束 | `turn_id`, `stop_hook_active`, `last_assistant_message` | exit 0 时 stdout 必须是 JSON；纯文本无效 | 汇总为 `pending`，显示“Codex 已完成什么” |

Codex 当前 matcher 支持重点：

- `SessionStart` matcher 过滤 `startup` / `resume` / `clear`。
- `PreToolUse` / `PermissionRequest` / `PostToolUse` matcher 过滤工具名，如 `Bash`、`apply_patch`、MCP 工具名；`apply_patch` 也可用 `Edit` / `Write` 别名。
- `UserPromptSubmit` / `Stop` 当前不使用 matcher。

### 3.2 Claude Code 事件

| 事件 | 触发点 | 关键字段 | stdout 风险 | Conductor 映射 |
| --- | --- | --- | --- | --- |
| `SessionStart` | 新会话、恢复、清理、压缩后 | `session_id`, `transcript_path`, `cwd`, `source`, `model`, `agent_type` | stdout 可加入 Claude 上下文 | 只记录 session/cwd，不默认注入上下文 |
| `UserPromptSubmit` | 用户 prompt 提交后、Claude 处理前 | `session_id`, `prompt`, `cwd`, `transcript_path` | stdout / `additionalContext` 会进 Claude 上下文 | 创建或更新任务为 `in_progress` |
| `PreToolUse` | 工具调用前 | `tool_name`, `tool_input`, `tool_use_id` | JSON 可 allow/deny/ask/defer/改写输入 | 只记录即将执行的工具和目标 |
| `PermissionRequest` | 权限弹窗即将展示 | `tool_name`, `tool_input`, `permission_suggestions` | JSON 可代替用户 allow/deny | 只记录提权上下文，绝不自动批准 |
| `PostToolUse` | 工具成功执行后 | `tool_name`, `tool_input`, `tool_response`, `duration_ms` | JSON 可影响 Claude 看到的结果 | 记录文件修改、命令输出、Agent 子任务结果 |
| `Notification` | 权限、空闲、认证、elicitation 等通知 | `message`, `title`, `notification_type` | 不能阻塞，只做副作用 | 记录“需要权限/空闲”等状态 |
| `SubagentStart` | Agent 工具启动子 agent | `agent_id`, `agent_type` | 可给子 agent 注入上下文 | 记录哪个子 agent 在处理什么 |
| `SubagentStop` | 子 agent 完成 | 同 Stop 类似，含子 agent 语境 | 可阻止子 agent 停止 | 汇总子 agent 产出，更新同一任务或子任务 |
| `TaskCreated` | agent team 创建任务 | `task_id`, `task_subject`, `task_description`, `teammate_name`, `team_name` | 可阻止任务创建 | 可映射为 Conductor 子任务 |
| `TaskCompleted` | agent team 标记任务完成 | `task_id`, `task_subject`, `teammate_name`, `team_name` | 可阻止完成 | 可把子任务转 `pending` 待审 |
| `TeammateIdle` | agent team teammate 即将空闲 | `teammate_name`, `team_name` | 可阻止进入 idle | 记录 teammate 当前结束点 |
| `Stop` | 主 Claude Code agent 完成响应 | `stop_hook_active`, `last_assistant_message` | JSON 可阻止 Claude 停止 | 汇总为 `pending`，显示“Claude 已完成什么” |
| `StopFailure` | API 错误导致未正常 Stop | `error`, `error_details`, `last_assistant_message` | 输出和退出码被忽略 | 记录失败，不改成待审 |

## 4. 任务线映射计划

### 4.1 统一事件模型

Conductor 内部建议先把不同来源归一为统一结构：

```json
{
  "source": "claude|codex",
  "event": "UserPromptSubmit|PreToolUse|PermissionRequest|PostToolUse|Stop|...",
  "session_id": "...",
  "turn_id": "...",
  "agent_id": "...",
  "agent_type": "...",
  "team_name": "...",
  "teammate_name": "...",
  "cwd": "...",
  "transcript_path": "...",
  "terminal_id": "...",
  "prompt": "...",
  "tool_name": "...",
  "tool_input_summary": "...",
  "tool_output_summary": "...",
  "last_assistant_message": "...",
  "occurred_at": "..."
}
```

字段原则：

- `source` 必填，用于区分 Claude Code / Codex。
- `session_id` 是第一匹配键。
- `turn_id` 有则记录，Codex 比 Claude Code 更稳定提供 turn 维度。
- `agent_id` / `agent_type` / `team_name` / `teammate_name` 用于区分子 agent 或 agent team。
- `terminal_id` 无法从 hook JSON 稳定拿到时，用 `cwd + session_id + 最近活跃窗口` 推断。

### 4.2 状态映射

| 事件组 | 任务动作 | 状态 |
| --- | --- | --- |
| `UserPromptSubmit` | 创建新任务，或按 `session_id` 更新当前活跃任务 | `in_progress` |
| `PreToolUse` | 追加“准备执行”上下文，不改变主状态 | 保持原状态 |
| `PermissionRequest` / `Notification(permission_prompt)` | 追加“需要权限/提权原因”上下文 | 保持 `in_progress`，UI 可显示权限提示 |
| `PostToolUse` | 追加工具结果、文件路径、命令摘要 | 保持原状态 |
| `SubagentStart` / `TaskCreated` | 创建或关联子任务，记录 agent/team 归属 | `in_progress` |
| `SubagentStop` / `TaskCompleted` | 汇总子 agent 产出，子任务转待审 | `pending` |
| `Stop` | 读取 transcript tail 和 last assistant message，更新完成摘要 | `pending` |
| `StopFailure` | 记录失败原因，不把任务误标待审 | `in_progress` 或 `failed_context` |
| 人工 `pass` / `reject` / `skip` | 人审闭环 | `passed` / `rejected` / `skipped` |

### 4.3 任务匹配优先级

多终端、多 agent 并行时，任务归属按以下顺序匹配：

1. `source + session_id`
2. `source + session_id + turn_id`
3. `source + agent_id`
4. `source + team_name + teammate_name`
5. `source + terminal_id + cwd`
6. `source + cwd + 最近活跃窗口`

不允许仅按 `cwd` 合并任务；同一仓库多个终端并行时必须保留独立任务线。

## 5. 非打扰实现约束

这些约束比“能不能拦截”更优先：

- `UserPromptSubmit` stdout 默认必须为空；不输出 plain text，不返回 `additionalContext`。
- `Stop` 默认 stdout 也保持为空；如 Codex Stop 需要 exit 0 且输出，必须输出合法 JSON，不能输出普通文本。
- 不使用 hook 自动 allow/deny 权限；`PermissionRequest` 只记录上下文。
- 不使用 hook 自动改写 tool input / tool output。
- 不使用 `decision: "block"` 阻止 agent 停止，除非未来明确进入质量门禁模式。
- 单次 hook 必须快速返回；重摘要、LLM 摘要、文件扫描应异步或带短超时。
- hook 失败不能影响 Claude Code / Codex 正常推进；失败只写本地事件日志。
- `.claude/settings.json` / `.codex/hooks.json` 只有在本地 smoke test 通过后再显式启用。

## 6. 推荐落地顺序

1. 保持现有 Claude Code hook 闭环稳定：`UserPromptSubmit` -> `in_progress`，`Stop/SubagentStop` -> `pending`。
2. 新增 Codex CLI hook 子命令：
   - `conductor hook codex user-prompt-submit`
   - `conductor hook codex pre-tool-use`
   - `conductor hook codex permission-request`
   - `conductor hook codex post-tool-use`
   - `conductor hook codex stop`
3. 新增 repo-local `.codex/hooks.json` 草案，但默认不要启用到全局。
4. 新增 Codex smoke script，用临时 `CONDUCTOR_ROOT` 喂 stdin JSON，验证：
   - `UserPromptSubmit` stdout 为空，并创建 `in_progress`。
   - `PermissionRequest` stdout 为空，并写入权限摘要。
   - `PostToolUse` stdout 为空，并写入工具摘要。
   - `Stop` stdout 为空或合法 JSON，并把同一任务转 `pending`。
5. UI 继续按 `source/session/terminal/agent/team` 展示任务归属，让用户能看出“哪个终端/哪个 teammate 正在做什么”。

## 7. Repo-local Codex hooks 草案

后续验证通过后，可在 `<repo>/.codex/hooks.json` 中使用类似结构。命令路径需替换为实际 `conductor.exe` 或 `cargo run -p conductor-cli --` 入口。

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "I:/personal-agent/target/debug/conductor.exe hook codex user-prompt-submit",
            "timeout": 5
          }
        ]
      }
    ],
    "PermissionRequest": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "I:/personal-agent/target/debug/conductor.exe hook codex permission-request",
            "timeout": 5
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash|apply_patch|Edit|Write|mcp__.*",
        "hooks": [
          {
            "type": "command",
            "command": "I:/personal-agent/target/debug/conductor.exe hook codex post-tool-use",
            "timeout": 5
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "I:/personal-agent/target/debug/conductor.exe hook codex stop",
            "timeout": 5
          }
        ]
      }
    ]
  }
}
```

注意：Codex 会要求项目 `.codex/` 配置层被信任后才加载 repo-local hooks。多个 hook 来源会合并运行，不是覆盖关系，所以实现时要避免全局和项目级重复写同一条任务。
