## 2. 评审补强点（原文档考虑不到位之处）

### A.【最关键】Goal 不通的根因不止"UI+Runtime"，而是**根本没有执行器在 claim 任务**

原文档把 Goal 不通归因于 UI 未挂 + Runtime API 未起。核验发现更深一层：

- `goal_orchestrator/act.rs` 只做 `proposed → queued`，**不 dispatch**（act.rs:48）。
- `ClaudePAdapter::spawn`（`claude_p.rs:126`）**零生产调用方**。
- `advance_cycle_phase`（`goals.rs:433`）只在 test 被调用，**无生产驱动循环**。
- 生产中唯一能 `claim_task` 的路径是 Runtime API HTTP handler（`runtime_api.rs:395`）——它假设**外部 agent 进程**轮询/claim/执行/回写。

即：当前 Goal 架构隐含"外部 agent 驱动"前提，但没人写那个 runner。**走方案二即正面补齐这条链**：启动 Runtime API + 写 `conductor agent run` CLI loop。

### B.【优先级】渐进式工具暴露被高估 → 已降 P2

P0（默认工具过强、`agent.start` 暴露给短任务）用静态模式分层即可解，不需要发现协议。渐进式发现每轮多一次 LLM 往返，对桌宠延迟不友好。本期只做静态分层。

### C.【UX 命脉】chat → Goal 升级交接被一笔带过 → 纳入 P0

原文档命名了 `chat.needs_more_capability → ask_user_upgrade_to_long_task`，但未定义交接机制。最难的一步——**把当前对话上下文作为 Goal 种子描述**传给 `createGoal`——是空的。本方案定义：LLM 返回结构化 `capability_request{reason, suggested_mode, goal_seed}` → 前端渲染确认卡 → 确认后 `createGoal(goal_seed, 继承 session 上下文)`。

### D.【迁移风险】改 `default_allowed_tool_ids` 是 schema 变更 → 改为运行时 gate

存量 `config.json` 已持久化旧 `allowed_tool_ids`，改默认值对存量无效且动到 deprecated skill 路径。改为：在 `build_tool_definitions` 层按模式 gate，把 `allowed_tool_ids` 当"长任务上限"而非"短任务实集"。零迁移、不破坏 skill。

### E.【正确性 bug】`subagent.claude_p` 同步阻塞 + 120s 超时 → 孤儿进程

`send_v2` 整体超时 120s，`subagent.claude_p` 同步阻塞（semaphore=1）。子进程超 120s 时 chat turn 报 failed，**但子进程仍在跑** → 孤儿 + UI 假失败。原文档只从可观察性角度建议收敛，漏了此正确性问题。

### F.【过程教训】"accepted" ≠ "integrated"

dispatch board 记 112/112 accepted（含 Goal 调度 + UI），但这些组件孤立、Runtime 未起、无执行器。根因是验收口径：每个 task 作为独立代码单元过了 tsc/cargo check/单测，但从未验"挂载到主界面、端到端能跑"。**本方案所有 task 验收强制含集成/挂载/端到端验证。**

### G.【一致性】四份文档缺统一 backlog → 本文档即收口

观察性文档的"5 步优先改动"第一批已 DONE，其余与差距文档批次 2/3 重叠。本文档合并为带依赖图的单一 backlog。

### H.【可测性】除可观察性外缺可测验收

模式边界、Goal 入口两批原只有"整改要求"无可测断言。本方案从 FSM 每个状态迁移反推单测断言。

## 3. 目标态三条贯通链（设计基线）

```text
链1 用户模式 -> 工具策略
  ChatComposer(mode) -> sendChatMessageV2(task_mode,capability,plan_only)
    -> send_message_v2(mode) -> build_tool_definitions(mode-gated allowlist)

链2 Goal/Runtime -> 主界面投影
  GoalConsole(mount) -> createGoal/start/pause/cancel
    -> RuntimeApiServer(started at desktop boot)
    -> `conductor agent run` CLI loop: claim -> ClaudePAdapter::spawn -> heartbeat/result 回写
    -> SSE/Tauri event -> AgentLanes/OodaTimeline/ReviewQueue 刷新

链3 Artifact -> 统一落地记录
  chat turn / doc job / code job / goal cycle = 统一 activity id
    -> assistant message + tool_call + command_run + agent_run + file artifact + summary_ref
    -> 前端"正在进行" + "落地记录"两个稳定入口
```

### 工具暴露分层（链1 核心，运行时 gate）

| 模式 | 暴露工具集 | 显式排除 |
|---|---|---|
| 短任务（默认） | 记忆/任务/只读 file.read/glob/grep、office.inspect/export、Lark read、按 intent 渐增 file.write/edit、shell（询问后） | `agent.start`、`subagent.claude_p`、`agent.team.*` |
| 长任务（确认后） | 短任务全集 + `agent.start`、Goal、AgentRun、AgentTeam、Runtime API、Review Queue | — |

用户显式提 `claude/subagent/agentteam` 时，短任务模式**不直接执行**，而是返回 `capability_request` 触发升级确认（链 C）。
