# 感知层增强：主形象与子形象状态方案

> 日期：2026-05-26
> 目标：在现有进程检测、任务状态和桌宠形象切换能力基础上，明确“主形象由 LLM/用户/Skill 决定，子形象由系统 hook 决定”的状态分层方案，降低形象切换冲突，让桌宠能稳定表达当前工作进度。

---

## 1. 结论

当前感知层新基线：

```text
暂时放弃 Live2D，不接入 Live2D runtime。
桌宠渲染采用“视频渲染 + 图片展示”混合方案。
```

采用双层形象状态：

```text
主形象 = 领域身份
子形象 = 当前动作/工作状态
```

主形象只表示桌宠当前处于哪个领域身份：

```text
original
document_secretary
programmer
```

子形象只表示当前动作状态。MVP 固定资产基线为：

```text
original:
  original.video

document_secretary:
  document_secretary.thinking
  document_secretary.writing

programmer:
  programmer.thinking
  programmer.agent_leader
  programmer.coding
```

最终前端渲染使用：

```text
resolved_asset = main_avatar_id + activity_variant
```

核心边界：

```text
LLM / 用户 / Skill 可以切主形象。
Hook 只能切子形象。
Hook 永远不能切主形象。
```

这样能避免“系统进度 hook 抢走产品语义”。例如当前主形象是 `document_secretary` 时，即使正在调用工具，hook 也只能设置 `activity_variant = tool_calling`，不能把主形象切成 `programmer`。

说明：

- `original` 当前只有视频作为唯一子形象，常驻/摸鱼/默认状态都落到这个视频。
- `document_secretary.thinking` 表示文档秘书思考、摸鱼、常驻等待。
- `document_secretary.writing` 表示文档秘书正在写作、整理、生成文档内容。
- `programmer.thinking` 表示程序员思考、摸鱼、常驻等待。
- `programmer.agent_leader` 表示正在控制 Claude Code / Codex / subagent。
- `programmer.coding` 表示正在写代码。

## 2. 背景

当前桌宠方案已确定三套主形象：

```text
original：原设计 / 日常形象
document_secretary：文档秘书形象
programmer：程序员形象
```

同时，当前阶段不再推进 Live2D 接入。原因：

- Live2D runtime、模型 rigging、动作管理和透明窗口适配会显著抬高 MVP 复杂度。
- 现阶段更需要验证“感知 -> 场景判断 -> 三选一待审任务 -> 工具/agent 进度反馈”的闭环。
- 视频和图片混合渲染已经能覆盖当前所需的动作表达。

后续随着工具调用、subagent、Claude/Codex 派工、文档处理等能力增强，只靠三张主形象会不够表达当前状态。

例如：

- 用户正在写代码，适合程序员主形象。
- LLM 正在生成代码，适合程序员写代码子形象。
- LLM 正在调用 subagent / Claude / Codex，适合程序员 agent leader 子形象。
- 用户正在写文档，适合文档秘书主形象。
- 系统正在分析文档，适合文档秘书阅读/整理子形象。

如果所有状态都交给 LLM 通过 MCP 工具切换，会有几个问题：

- LLM 不一定能及时感知工具执行进度。
- 工具执行中的短状态变化不适合每次都问模型。
- LLM 可能把“动作状态”误判为“领域身份”。
- 高频状态更新会增加 token 和工具调用成本。

因此需要把主形象和子形象拆开。

## 3. 状态来源分工

### 3.1 主形象来源

主形象只由以下来源切换：

| 来源 | 是否允许切主形象 | 示例 |
|---|---:|---|
| 用户显式要求 | 是 | “切程序员形象” |
| LLM chat 判断后调用 MCP / 内部工具 | 是 | 用户说“帮我看这个代码问题” |
| Skill router | 是 | `coding_assistant` 命中后临时切 `programmer` |
| 进程检测 hook | 否 | 只能作为 Skill router 的输入信号 |
| 工具执行 hook | 否 | 只能切子形象 |
| subagent 进度 hook | 否 | 只能切子形象 |

主形象切换工具仍使用：

```text
pet.set_avatar
conductor.pet.set_avatar
```

但后续建议把字段名明确为：

```json
{
  "main_avatar_id": "programmer",
  "mode": "temporary",
  "ttl_minutes": 60,
  "reason": "用户进入 coding 任务"
}
```

如果暂时沿用 `avatar_id`，语义也应限定为主形象 ID。

### 3.2 子形象来源

子形象由系统 hook 更新，不经 LLM。

允许来源：

- LLM response lifecycle。
- ToolExecutor lifecycle。
- subagent / Claude / Codex run lifecycle。
- TaskList 状态变化。
- 用户输入等待状态。
- 错误/失败事件。

子形象更新工具不应暴露给 MCP 外部 agent，建议仅内部调用：

```text
pet.set_activity_variant
```

输入：

```json
{
  "activity_variant": "agent_leading",
  "source": "subagent.run.started",
  "ttl_seconds": 120,
  "priority": 80,
  "reason": "正在调用 Claude 子任务"
}
```

## 4. 状态模型

建议新增或扩展 avatar state：

```rust
pub struct AvatarState {
    pub main_avatar_id: MainAvatarId,
    pub main_mode: AvatarMode,
    pub main_reason: Option<String>,
    pub main_expires_at: Option<DateTime<Utc>>,

    pub activity_variant: ActivityVariant,
    pub activity_source: ActivitySource,
    pub activity_reason: Option<String>,
    pub activity_expires_at: Option<DateTime<Utc>>,
    pub activity_priority: u8,

    pub updated_at: DateTime<Utc>,
}
```

枚举：

```rust
pub enum MainAvatarId {
    Original,
    DocumentSecretary,
    Programmer,
}

pub enum ActivityVariant {
    Idle,
    Thinking,
    Reading,
    Writing,
    ToolCalling,
    AgentLeading,
    WaitingUser,
    Error,
    Done,
}

pub enum AvatarMode {
    Temporary,
    Persistent,
}
```

注意：

- `main_avatar_id` 和 `activity_variant` 必须分开存储。
- 不允许用 `programmer_agent_leader` 这种单字段覆盖主状态。
- 前端可以把二者拼成资源名，但状态层不能混在一起。

## 5. 渲染解析规则

前端根据主形象和子形象解析最终资产。资产可以是视频，也可以是图片：

```text
asset_key = "{main_avatar_id}.{activity_variant}"
```

示例：

| main_avatar_id | activity_variant | resolved_asset | media_type |
|---|---|---|---|
| `original` | 任意 | `original.video` | video |
| `document_secretary` | `thinking` / `idle` / `waiting_user` | `document_secretary.thinking` | image |
| `document_secretary` | `writing` / `tool_calling` | `document_secretary.writing` | image |
| `programmer` | `thinking` / `idle` / `waiting_user` | `programmer.thinking` | image |
| `programmer` | `agent_leading` | `programmer.agent_leader` | image |
| `programmer` | `writing` / `tool_calling` | `programmer.coding` | image |

fallback 顺序：

```text
main_avatar_id + mapped activity asset
  -> main_avatar_id + thinking
  -> original.video
```

注意：

- `original` 是特例：当前只有视频，不再拆多个子形象。
- 子形象字段仍保留完整 activity 语义，但资源层可以多对一映射。
- 例如 `document_secretary.waiting_user` 和 `document_secretary.thinking` 可以使用同一张图。
- 例如 `programmer.tool_calling` 如果调用的是 Claude/Codex/subagent，映射到 `programmer.agent_leader`；如果是普通 coding 输出，映射到 `programmer.coding`。

## 6. Hook 事件映射

### 6.1 LLM 生命周期

| 事件 | activity_variant | 说明 |
|---|---|---|
| `llm.request.started` | `thinking` | 模型开始思考 |
| `llm.response.streaming` | `writing` | 模型正在生成回复 |
| `llm.response.completed` | `done` | 短暂完成态后回 idle |
| `llm.response.failed` | `error` | 展示错误态 |

### 6.2 ToolExecutor 生命周期

| 事件 | activity_variant | 说明 |
|---|---|---|
| `tool.validation.started` | `thinking` | 参数/权限判断 |
| `tool.read.started` | `reading` | 只读工具 |
| `tool.write.preview_started` | `writing` | 生成 dry-run / patch |
| `tool.call.started` | `tool_calling` | 调用普通工具 |
| `tool.call.completed` | `done` | 短暂完成态 |
| `tool.call.failed` | `error` | 工具失败 |

### 6.3 Agent / Claude / Codex 生命周期

| 事件 | activity_variant | 说明 |
|---|---|---|
| `subagent.spawn.started` | `agent_leading` | 派发子任务 |
| `subagent.running` | `agent_leading` | 子 agent 执行中 |
| `claude_p.started` | `agent_leading` | Claude 子任务 |
| `codex.run.started` | `agent_leading` | Codex 子任务 |
| `agent.output.received` | `reading` | 读取子 agent 输出 |
| `agent.completed` | `done` | 短暂完成态 |
| `agent.failed` | `error` | 子 agent 失败 |

### 6.4 TaskList 生命周期

| 事件 | activity_variant | 说明 |
|---|---|---|
| `task.pending_review` | `waiting_user` | 等用户三选一 |
| `task.claimed` | `thinking` | 已领取，准备执行 |
| `task.in_progress` | `writing` / `tool_calling` | 根据任务类型细分 |
| `task.completed` | `done` | 完成 |
| `task.blocked` | `waiting_user` | 需要用户或前置任务 |

## 7. 优先级与防抖

### 7.1 优先级

建议 priority：

| activity_variant | priority |
|---|---:|
| `error` | 100 |
| `waiting_user` | 90 |
| `agent_leading` | 80 |
| `tool_calling` | 70 |
| `writing` | 60 |
| `reading` | 50 |
| `thinking` | 40 |
| `done` | 30 |
| `idle` | 0 |

规则：

- 高优先级可以覆盖低优先级。
- 低优先级不能打断高优先级，除非高优先级过期。
- `error` 和 `waiting_user` 必须有明确结束事件或 TTL。

### 7.2 TTL

建议：

| 状态 | 默认 TTL |
|---|---:|
| `thinking` | 30 秒 |
| `reading` | 20 秒 |
| `writing` | 30 秒 |
| `tool_calling` | 60 秒 |
| `agent_leading` | 120 秒 |
| `waiting_user` | 不自动过期，或 10 分钟后降级 quiet badge |
| `error` | 30 秒 |
| `done` | 2 秒 |

### 7.3 防抖

必须防止子形象高频闪烁：

```text
同一 activity 最小展示 1.5 秒。
不同 activity 最小切换间隔 800ms。
agent_leading 最小展示 3 秒。
done 最多展示 2 秒。
```

主形象防抖：

```text
用户显式切换：立即生效。
LLM / Skill 自动切换：5-10 分钟 cooldown。
hook：不能切主形象。
```

## 8. MVP 资源规划

当前 MVP 资源基线是“视频渲染 + 图片展示”混合方案。

暂时不接入 Live2D：

- 不引入 Live2D runtime。
- 不做 Live2D motion 管理。
- 不做 model rigging 工作流。
- 不把当前资源规划设计成 Live2D 目录。

不要做 3 个主形象乘以 9 个子状态。MVP 只做已经明确的 1 + 2 + 3 个资源。

### 8.1 第一批子形象

```text
original:
  original.video

document_secretary:
  thinking
  writing

programmer:
  thinking
  agent_leader
  coding
```

语义：

```text
original.video:
  原设计唯一子形象；默认、摸鱼、常驻、等待都使用该视频。

document_secretary.thinking:
  文档秘书思考 / 摸鱼 / 常驻。

document_secretary.writing:

programmer.thinking:
  程序员思考 / 摸鱼 / 常驻。

programmer.agent_leader:
  控制 Claude Code / Codex / subagent，等待或汇总子 agent 输出。

programmer.coding:
  写代码 / 生成补丁 / 处理 coding 输出。
```

### 8.2 资源文件建议

### 8.4 未覆盖状态的表达

没有对应子形象时，使用：

- 状态 badge。
- 轻动画。
- 气泡文案。
- 前端 fallback 到该主形象的 thinking 图；`original` fallback 到 `original.video`。

不要为了覆盖所有状态而过早制作大量图片或视频。

### 8.5 透明图与抠图流程

MVP 选择“生成时要求透明 PNG + 本地 `isnet-anime` 抠图兜底”的流程，而不是依赖在线抠图服务或通用分割模型。

定调：

- 生成素材时仍然优先要求 `true transparent background, alpha channel PNG, no background`。
- 如果生成结果是假透明棋盘格、纯白底、浅灰底，当前优先使用 `rembg -m isnet-anime`。
- 通用 `rembg` / `u2net` 不作为本批次默认方案；它曾把深色衣服、桌面等区域处理成半透明水墨渐变。
- 本地边缘连通背景移除脚本保留为 fallback；它足够保守，但会留下部分角色内部白底。
- 每次抠图必须同时生成深色背景预览，用肉眼检查边缘、头发、配饰、桌面、纸张和道具。
- 只有在发丝、半透明纱、烟雾、玻璃、复杂阴影等场景明显失败时，才重新生成更干净的纯色背景图或升级到更强分割方案。

当前模型放置位置：

```text
C:\Users\徐笙祺\.u2net\isnet-anime.onnx
```

项目内批量命令：

```powershell
rtk powershell -NoProfile -ExecutionPolicy Bypass -File scripts\run-rembg-isnet-anime-batch.ps1
```

单张重跑命令：

```powershell
rtk powershell -NoProfile -Command "& 'tools\Rembg\rembg.exe' i -m isnet-anime '2Dworkspace\<source>.png' '2Dworkspace\rembg-cutouts\<source>.isnet-anime.png'"
```

验收命令：

```powershell
rtk powershell -NoProfile -Command "& 'C:\Program Files\ImageMagick-7.1.2-Q16-HDRI\magick.exe' identify -format '%f`t%[channels]`t%[opaque]`t%[pixel:p{0,0}]`n' '2Dworkspace\rembg-cutouts\*.isnet-anime.png'"
```

验收标准：

- ImageMagick 结果应包含 `srgba 4.0`，代表输出有 alpha 通道。
- `%[opaque]` 应为 `False`，代表图片不是全不透明。
- 角落像素 alpha 接近 0；本批次结果为 `srgba(1,1,1,0.00392157)`。
- 深色预览 `*.isnet-anime.preview-dark.png` 下不能出现大面积白底或棋盘格残留。
- 主体衣服、头发、桌面、纸张、道具不能被抠成透明或水墨渐变。
- 边缘允许有轻微抗锯齿，不追求 Live2D 级别精抠。
- 最终进入 `apps/desktop/public/avatar/` 的资源应是透明 PNG；后续如需减小体积，再批量转 WebP。

当前已验证的 5 个子形象透明输出：

- `2Dworkspace\rembg-cutouts\1DDsiXZ9p6hI.isnet-anime.png`
- `2Dworkspace\rembg-cutouts\7xX_7umRXHBq.isnet-anime.png`
- `2Dworkspace\rembg-cutouts\dBXs6I0R9GS9.isnet-anime.png`
- `2Dworkspace\rembg-cutouts\eAyfpPSM3fZV.isnet-anime.png`
- `2Dworkspace\rembg-cutouts\I_11CKWt8zlK.isnet-anime.png`

运行注意：

- `tools\Rembg\rembg.exe` 是项目内固定使用的 rembg 可执行文件，避免依赖系统 PATH。
- 如果 rembg 报 `pymatting` metadata 相关错误，先执行 `scripts\fix-rembg-pymatting-metadata.ps1` 修补项目内 Rembg 目录。
- 当前日志可能出现缺少 `cublasLt64_12.dll` 的 CUDA 警告；批处理已通过 CPU fallback 完成，属于性能问题，不代表抠图失败。
- 不覆盖原图；AI 输出统一放 `2Dworkspace\rembg-cutouts`，保守脚本输出保留在 `2Dworkspace\cutouts`。

保守 fallback 命令：


## 9. 与三选一待审任务的关系

当 TaskList 中存在待审任务时：

```text
activity_variant = waiting_user
```

桌宠展示：

```text
当前建议推进：整理当前方案文档结构

[可以推进] [不推进] [有什么不同]
```

此时：

- 主形象不因待审任务自动改变。
- 如果待审任务由 `document_assistant` 生成，Skill 可以先把主形象临时切到 `document_secretary`。
- 如果待审任务由 `coding_assistant` 生成，Skill 可以先把主形象临时切到 `programmer`。
- hook 只负责 `waiting_user` 子状态。

## 10. 与 MCP 工具的关系

主形象 MCP 工具：

```text
conductor.pet.set_avatar
```

允许：

- 用户显式要求。
- LLM chat 判断。
- Skill router 调用。

限制：

- 只能切 `original | document_secretary | programmer`。
- 不接受任意图片路径。
- 不接受 URL。
- 外部 agent 默认只能 temporary。

子形象不建议暴露 MCP 工具。子形象应由内部 hook 维护：

```text
internal: pet.set_activity_variant
```

如果未来必须对外暴露，也只能接受枚举 `activity_variant`，且不得接受 `main_avatar_id`。

```sql
INSERT INTO avatar_state (
```

## 12. 实施路线

### Phase 1：状态拆分

## 13. 风险与对策

| 风险 | 表现 | 对策 |
|---|---|---|
| hook 抢主形象 | 调用工具时突然变成程序员 | hook 永远不能切 `main_avatar_id` |
| 状态闪烁 | thinking/writing/tool_calling 高频切换 | 最小展示时间 + priority |
| 资源膨胀 | 3 主形象 x 多状态导致大量图或视频 | MVP 固定 1+2+3 资源，fallback |
| LLM 判断不稳定 | 主形象频繁切换 | cooldown + 用户显式优先 |
| 子形象语义混乱 | `programmer_agent_leader` 被当成主形象 | 状态字段分层，资源名只是渲染结果 |
| 外部 MCP 滥用 | 外部 agent 传任意图或乱切状态 | MCP 只允许枚举主形象，不暴露子形象 |
| Live2D 复杂度回流 | MVP 被 runtime/动作/rigging 拖慢 | 当前基线明确不接 Live2D |

---

## 14. 最终定调

本方案的核心不是“增加更多形象”，而是建立稳定的状态分层：

```text
主形象回答：我现在以什么身份陪你工作。
子形象回答：我此刻正在做什么。
```

主形象由用户、LLM、Skill 决定。

子形象由系统 hook 决定。

hook 不得切换主形象。

当前阶段暂不接入 Live2D。前端只负责把 `main_avatar_id + activity_variant` 解析为视频或图片资源，并在资源不存在时 fallback。

当前资源基线固定为：

```text
original:
  original.video

document_secretary:
  thinking
  writing

programmer:
  thinking
  agent_leader
  coding
```
