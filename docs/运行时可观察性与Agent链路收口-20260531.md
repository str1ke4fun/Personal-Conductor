# 运行时可观察性与 Agent 链路收口

日期：2026-05-31

## 结论

当前系统不是一条单一的“LLM 调用 Agent”链路，而是至少四条运行时路径并存：

1. 应用内对话 LLM：`send_message_v2_with_session` 驱动流式回复、thinking、工具调用。
2. 应用内工具式 Agent：LLM 调用 `agent.start` 后，由后台 `agent_runs` 启动并追踪 `claude -p`。
3. 一次性 subagent：`subagent.claude_p` 或 CLI `sub run` 直接执行 `claude -p`，结束后才有 stdout/stderr/log。
4. 外部 Claude/Codex hook：通过 `UserPromptSubmit/pre_tool_use/post_tool_use/stop` 写入 `tasks` 和 `runtime_events`。

用户现在感知不到“到底落地了什么”，主要不是底层没有记录，而是记录分散在 `tasks`、`agent_runs`、`tool_calls`、`command_runs`、`runtime_events`、`state/agent-runs`、`state/subagent-runs`，前端没有统一投影成“正在做什么”和“完成了什么”。

## 什么时候会调用 `claude -p`

### 1. `agent.start`：应用内推荐的后台 Agent 路径

代码入口：

- `crates/conductor-core/src/tools/agent.rs`
- `crates/conductor-core/src/agent_runs.rs`
- `apps/desktop/src-tauri/src/commands.rs`

触发条件：

- 应用内聊天 LLM 拿到 `agent.start` 工具定义并选择调用。
- 默认配置 `default_allowed_tool_ids` 已包含 `agent.start`、`agent.read_output`、`agent.stop`。
- `tool_tiers` 命中 agent 相关关键词时，也会暴露 `agent.start/read_output/stop`。

实际执行：

- `agent.start` 调 `agent_runs::start_claude_run`。
- 该函数创建 `agent_runs` 记录，状态从 `queued` 到 `running`。
- 子进程命令仍然是 `claude -p <prompt>`。
- stdout/stderr 写入 `state/agent-runs/*-stdout.log`、`*-stderr.log`，完成后写 `output_ref`。
- 前端已有 `list_agent_runs/read_agent_run_output/stop_agent_run` IPC。

判定：

这是当前应用内最适合用户感知的 Claude 后台执行路径，因为它有 `run_id`、`pid`、状态、输出引用和停止能力。

### 2. `subagent.claude_p`：一次性子 Agent 路径

代码入口：

- `crates/conductor-core/src/subagent.rs`
- `crates/conductor-core/src/tools/agent.rs`
- `crates/conductor-cli/src/main.rs`

触发条件：

- CLI `conductor sub run` 会直接调用 `subagent::run_claude_p`。
- 应用内 LLM 只有在 `subagent.claude_p` 被暴露到工具集中时才可能调用它。
- 默认 `llm.allowed_tool_ids` 不包含 `subagent.claude_p`。
- 但 `coding_assistant` skill 的 allowed tools 包含 `subagent.claude_p`，所以命中 skill/policy 后仍可能暴露。

实际执行：

- 直接 `Command::new("claude").arg("-p").arg(prompt)`。
- 有全局 semaphore，当前并发限制为 1。
- 进程结束后返回 stdout/stderr/exit_code/duration。
- 日志写入 `state/subagent-runs/*-claude-p.log`。

判定：

这条路径缺少运行中状态和实时输出。除非外层工具执行事件完整投影到 UI，否则用户只能看到“开始调用工具”和“结束结果”，中间会像卡住。

### 3. `ClaudePAdapter`：面向 AgentTask/Goal 的内置适配器

代码入口：

- `crates/conductor-core/src/adapters/claude_p.rs`

触发条件：

- 由内部 AgentTask/Goal 调度显式调用 `ClaudePAdapter::spawn`。
- 它不是普通聊天工具定义里的直接入口。

实际执行：

- 为 task 构造 prompt。
- 创建 `agent_runs` 记录。
- 注入 `RUNTIME_API_URL`、`RUNTIME_TOKEN`、`WORKSPACE_ID`、`TASK_ID` 给子进程。
- 子进程仍然是 `claude -p`。
- 完成后解析 stdout，更新 `AgentTask.result_ref/status`，发 `task.review_ready` 或 `task.failed` 事件。

判定：

这是“内置任务 Agent”更完整的执行器形态，但普通聊天里看到的不是它，而通常是 `agent.start` 或 `subagent.claude_p`。后续需要明确 Goal/AgentTask 调度是否统一走它。

## 什么时候用 AgentTeam

代码入口：

- `crates/conductor-core/src/tools/agent.rs`
- `crates/conductor-core/src/agent_teams.rs`
- `crates/conductor-core/src/adapters/agent_team_adapter.rs`

触发条件：

- 应用内 LLM 拿到 `agent.team.*`、`agent.mailbox.*` 工具定义并选择调用。
- `tool_tiers` 命中 team/团队/协作等关键词时，会暴露 team/mailbox 工具。

AgentTeam 做什么：

- 创建、列出、添加成员、快照。
- 发送、列出、标记 mailbox 消息。
- `AgentTeamAdapter` 把 team 生命周期和 mailbox 桥接到 Goal/Event runtime。

AgentTeam 不做什么：

- AgentTeam 本身不等于执行器。
- `agent.team.create/add_member/snapshot` 不会直接 spawn `claude -p`。
- 真正跑任务仍需要 `agent.start`、`ClaudePAdapter`、Codex adapter 或其他 executor。

判定：

AgentTeam 是协作容器和状态治理层，不是“模型正在工作的直接证据”。用户侧需要看到 team 进入 planning/executing/review，但还必须看到对应的 `agent_run`、task 或 executor。

## 哪些动作必须让用户感知

必须展示：

1. LLM 正在处理：thinking、planning、tool_calling、summarizing、done。
2. 工具生命周期：准备调用、执行中、成功、失败、需要审批、被阻塞。
3. 外部进程：`claude -p`、Codex、shell、长时间文件操作。
4. 后台 AgentRun：queued、running、succeeded、failed、stopped、pid、耗时、output_ref。
5. 外部 hook 会话：Claude/Codex 的 session、cwd、当前请求、last tool、last_event_at。
6. 权限/审批：proposal pending/approved/running/succeeded/failed。
7. 落地结果：写了什么文件、生成了什么 summary、哪个 output_ref、哪个 command_run/tool_call 产出。

`in_progress` 必须被用户感知。

原因很直接：当 LLM 没有继续吐 token，但后台工具或外部 Claude/Codex 仍在工作时，用户唯一能看到的推进信号就是 `in_progress/running`。如果这个状态不展示，用户视角就是“进程突然停了，没有输出”。

建议口径：

- `in_progress/running`：系统正在做事，显示 elapsed、last_event_at、last action。
- `pending`：不是运行中，而是等用户审阅或处理。
- `queued`：已排队但未开始。
- `blocked/approval_required`：等用户授权。
- `completed/succeeded`：完成且有结果引用。
- `failed/stopped`：结束但需要用户确认失败原因或输出。

## 当前前后端断点

### 断点 1：聊天侧 active_run 只覆盖应用内 LLM

代码：

- `crates/conductor-core/src/chat/send_v2.rs`
- `crates/conductor-core/src/chat/session.rs`
- `apps/desktop/src/windows/ChatSessionSidebar.tsx`

现状：

- `send_v2` 注册 `active_run`，并发出 `thinking-update`、`stream-chat-token`、`tool-execution-update`。
- `ChatSessionSummary.working` 来自内存里的 `active_run`。
- `ChatSessionSidebar` 只有在已有 session.working 时才每 5 秒刷新。

问题：

- 外部 Claude/Codex hook 不会进入这个 `active_run`。
- `agent_runs` 不会让聊天侧 session 自动变成 working。
- 如果 UI 没有其他入口展示 hook/agent_run，用户会以为会话停住。

### 断点 2：AgentRun 有后端状态，但前端刷新不够主动

代码：

- `crates/conductor-core/src/agent_runs.rs`
- `apps/desktop/src-tauri/src/commands.rs`
- `apps/desktop/src/windows/TaskPanelContent.tsx`

现状：

- `agent_runs` 会写 SQLite 和 `state/agent-runs`。
- `TaskPanelContent` 会每 10 秒轮询 `listAgentRuns`。
- `stop_agent_run` 会 emit `agent_runs_changed`。

问题：

- `agent_runs::start_claude_run` 和完成态变化只写 runtime events，没有直接驱动前端刷新。
- `TaskPanelContent` 当前没有监听 `agent_runs_changed` 来刷新。
- `TaskDrawerPane` 没有展示 `agent_runs`。

### 断点 3：hook 任务状态有记录，但抽屉没有展示运行中的 hook

代码：

- `crates/conductor-cli/src/main.rs`
- `apps/desktop/src/windows/TaskDrawerPane.tsx`
- `apps/desktop/src/windows/TaskPanelContent.tsx`

现状：

- `UserPromptSubmit` 会创建或更新 `source=claude/codex` 的 task，并设为 `InProgress`。
- `pre_tool_use/post_tool_use` 会更新 `permission_summary/last_event_at`。
- `stop/subagent_stop` 会把 task 设为 `Pending`，附带 `summary_ref/artifact/last_output_summary`。
- `TaskPanelContent` 计算了 `activeHookTasks`。

问题：

- `TaskDrawerPane` 的运行中只看 `agentTasks.in_progress`，不看 legacy hook tasks。
- `TaskPanelContent` 虽然计数包含 `activeHookTasks`，但没有形成足够明确的“外部 Claude/Codex 正在工作”卡片。
- 如果 hook 的 `UserPromptSubmit` 丢失，只收到 tool event，`update_permission_context` 不会创建运行中任务。
- 如果完成事件先把 task 置为 `Pending`，后续 tool event 只更新上下文，不会恢复 `InProgress`，状态语义容易混淆。

### 断点 4：一次性 `subagent.claude_p` 缺少运行中观察口

代码：

- `crates/conductor-core/src/subagent.rs`
- `crates/conductor-core/src/tools/agent.rs`

现状：

- 结束后写 `state/subagent-runs/*-claude-p.log`。
- 工具结果里返回 log_path/stdout/stderr。

问题：

- 没有 `agent_runs` 行。
- 没有 pid、queued/running/succeeded 状态。
- 没有实时 output tail。
- 如果这个工具跑很久，用户只能依赖工具调用 UI 的执行中状态。

## 修复方案记录

### P0：统一运行中状态语义

1. `Pending` 只表示等待用户审阅，不表示仍在执行。
2. hook 的 `UserPromptSubmit`、`pre_tool_use`、`post_tool_use` 都应能确保存在一条 active hook task。
3. tool event 到来时，如果没有 terminal completion 标记，应保持或恢复 `InProgress`。
4. `stop/subagent_stop/stop_failure` 才能把 hook task 转成 `Pending` 或失败待审。
5. task 需要记录 `last_event_kind`、`last_tool_name`、`last_event_at`，供 UI 展示“刚刚做了什么”。

### P0：把 AgentRun 变成主动可见

1. `agent_runs` 创建、queued->running、running->terminal 时，要有前端可监听的刷新信号。
2. Tauri 层增加或桥接 `agent_runs_changed`，不要只依赖 10 秒轮询。
3. `TaskDrawerPane` 也拉取并展示 active `agent_runs`。
4. AgentRun 卡片显示：status、elapsed、pid、agent_id/role、prompt_summary、output tail 入口、stop 按钮。

### P0：让 hook 运行态进入抽屉和任务面板

1. `TaskDrawerPane` 增加 `activeHookTasks`：`source in claude/codex` 且 `status=in_progress`。
2. 展示当前请求 `current_request`、cwd、session_id、last_event_at、permission_summary。
3. pending hook review 展示 `last_output_summary`、`summary_ref`、`artifact.file`。
4. `tasks_changed` 到来时，抽屉、任务面板、聊天侧状态都要刷新相关投影。

### P1：统一“落地记录”投影

新增一个前端聚合区，建议名为“落地记录”或“结果”：

- hook task：`artifact.file`、`summary_ref`、`last_output_summary`。
- agent run：`output_ref`、stdout/stderr tail、最终 status。
- tool call：tool id、status、duration、output/error。
- command run：command、cwd、exit_code、stdout/stderr ref。
- proposal：审批后的 result_ref。

目标是让用户能回答三个问题：

1. 现在谁在工作？
2. 刚才调用了什么？
3. 最终落到了哪里？

### P1：收敛 `subagent.claude_p`

建议减少直接使用一次性 `subagent.claude_p`：

- 应用内长期任务优先走 `agent.start` 或 `ClaudePAdapter`。
- 如果仍保留 `subagent.claude_p`，需要在执行前创建可观察 run，至少写入 `agent_runs` 或等价 runtime projection。
- CLI `sub run` 可以继续作为调试入口，但应明确标记为“非完整 UI 可观察路径”。

### P1：明确 AgentTeam 与执行器关系

1. AgentTeam 进入 `Executing` 时，必须关联至少一个 executor：`agent_run_id`、task id 或 external session id。
2. Team snapshot 里展示成员、mailbox、lifecycle 之外，也展示实际 executor 状态。
3. 没有 executor 的 AgentTeam 只算计划/协作容器，不算“LLM 正在工作”。

## 验收标准

1. 应用内 LLM 调用 `agent.start` 后，1 秒内前端出现 running/queued 状态，而不是等 10 秒轮询。
2. `agent_runs` 运行时，用户能看到 elapsed、pid、agent_id、prompt 摘要，并可读取 output tail。
3. 外部 Claude/Codex 触发 `UserPromptSubmit` 后，抽屉和任务面板都显示 active hook task。
4. 外部 Claude/Codex 触发 tool event 后，UI 展示 last tool/last_event_at，不出现“无输出但看起来停止”的状态。
5. `stop/subagent_stop` 后，运行态消失，出现 pending review，并展示 summary/artifact/output ref。
6. 任意超过 5 秒的后台动作，前端必须有可见状态。
7. 完成后用户能从一个统一入口看到“落地记录”：文件、summary、output_ref、tool_call 或 command_run。

## 当前优先改动建议

第一批代码收口建议只做最短闭环：

1. 后端：hook tool event upsert active hook task，并维护 `InProgress`。
2. 后端：AgentRun 状态变化桥接到 `agent_runs_changed` 或统一 runtime event watcher。
3. 前端：`TaskDrawerPane` 展示 active hook tasks 和 active agent runs。
4. 前端：`TaskPanelContent` 监听 `agent_runs_changed`，并在 AgentRun 卡片提供 output tail。
5. 前端：pending hook review 展示 `summary_ref/artifact.file/last_output_summary`。

完成这批后，用户至少能从 UI 判断：LLM 还在不在工作、调用了什么外部执行器、结果落在哪里。

本轮已完成第一批最短闭环整改：
