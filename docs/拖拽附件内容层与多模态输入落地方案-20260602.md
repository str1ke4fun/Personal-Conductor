# 拖拽附件内容层与多模态输入落地方案
> 日期: 2026-06-02
> 范围: Windows 资源管理器拖拽文件进入 Prompt、内容层抽象、Provider 编译层、UI 交互路线
> 关联文档: [Harness架构审查与演进计划-20260529.md](./Harness架构审查与演进计划-20260529.md)
> 当前结论: 不应把“拖入文件”退化成“把绝对路径塞进 prompt”；应新增一层显式的内容层，区分“工作区引用”和“直接附件”

---

## 1. 背景与问题

当前链路仍是“纯文本聊天”：

- 前端输入是单个 `textarea`，[apps/desktop/src/windows/ChatComposer.tsx](../apps/desktop/src/windows/ChatComposer.tsx)
- 前端 IPC 仍以 `message: string` 调用 `send_chat_message_v2`，[apps/desktop/src/ipc/invoke.ts](../apps/desktop/src/ipc/invoke.ts)
- Tauri 命令签名也是 `message: String`，[apps/desktop/src-tauri/src/commands.rs](../apps/desktop/src-tauri/src/commands.rs)
- 核心聊天入口是 `send_message_v2_with_session(content: String, ...)`，[crates/conductor-core/src/chat/send_v2.rs](../crates/conductor-core/src/chat/send_v2.rs)
- LLM 请求层当前仍以文本 content 为主，[crates/conductor-core/src/llm.rs](../crates/conductor-core/src/llm.rs)

这导致三个直接问题：

1. 从 Windows 资源管理器拖入图片、PDF、Office 文档时，没有统一入口承接文件本体。
2. 如果只把绝对路径注入 prompt，图片和 PDF 无法真正“看见”，代码/文档也会受 workspace 边界影响。
3. 当前 `file.read` 和 `office.export_text` 不足以覆盖多模态内容读取。

现状限制：

- `file.read` 读取图片只返回元数据，不读像素内容，[crates/conductor-core/src/tools/fs.rs](../crates/conductor-core/src/tools/fs.rs)
- `office.export_text` 本质是 `read_to_string`，不是真正的 Office/PDF 解析，[crates/conductor-core/src/tools/office.rs](../crates/conductor-core/src/tools/office.rs)
- 绝对路径默认要经过 workspace 边界校验，工作区外路径会触发 external access proposal，[crates/conductor-core/src/tools/mod.rs](../crates/conductor-core/src/tools/mod.rs) 和 [crates/conductor-core/src/chat/tools.rs](../crates/conductor-core/src/chat/tools.rs)

因此，本方案目标不是“让 LLM 会猜路径”，而是把“拖入文件”纳入一条正式的消息内容链路。

---

## 2. 设计前提

### 2.1 授权模型

本方案接受如下产品前提：

- 用户把文件拖入 composer，等于用户明确授权“本次消息可携带该文件内容给 LLM”。
- 这类授权是“消息级上传授权”，不是“放开宿主机任意路径访问”。
- 因此，拖入附件不需要再走 proposal。
- 但后续如果模型想主动读取别的工作区外路径，仍应走现有工具/权限模型。

换句话说：

- `拖入附件` 是用户主动提供内容。
- `file.read("C:\\...")` 是模型主动发起外部访问。

两者不应混为一谈。

### 2.2 产品判断

内容层的核心原则是：

- 对模型来说，优先给“内容对象”，而不是“宿主机路径字符串”。
- 对系统来说，保留绝对路径，用于 provenance、workspace 关系判断、缓存和审计。

### 2.3 Inline 引用前提

新增需求：

- 用户可以把文件直接拖拽到对话草稿文字的任意位置。
- 该动作语义不是“把附件挂到消息尾部”，而是“在该位置插入一个文件引用”。

因此需要区分两种交互：

- `append attachment`
- `inline reference insert`

这意味着：

- `textarea + 尾部附件 chip` 只能覆盖 append
- 内容层必须支持“有序混排的 text/file parts”

## 3. 目标与非目标

### 3.1 目标

- 支持从 Windows 资源管理器拖入代码、文本、图片、PDF、常见 Office 文件。
- 系统能识别文件类型、大小、workspace 归属、可上传性。
- 系统能把拖入内容编译成 provider-native 消息格式。
- 系统保留一条统一抽象，避免 OpenAI / Anthropic 分叉污染上层 UI。
- 对工作区内文本代码保留“路径引用”能力，不强制所有内容都上传成附件。

### 3.2 非目标

- v1 不实现本地图像识别算法、本地 OCR、本地 PDF 理解模型。
- v1 不实现任意二进制格式的深度解析。
- v1 不把“拖入文件”扩展成自动写回、自动编辑、自动执行命令。

---

## 4. 核心判断

## 4.1 不建议“只把绝对路径交给 LLM”

原因：

- 图片/PDF/Office 在当前工具面里并不能靠路径自然读通。
- 工作区外绝对路径天然会撞 workspace 边界。
- 宿主机路径泄露给模型的收益很低，副作用很高。
- 对 provider 来说，多模态能力本来就要求直接提供 image/file，而不是让模型再猜本地文件系统。

因此，绝对路径应只做内部控制元数据，不应成为主要语义输入。

## 4.2 应拆成两条内容通道

### A. Workspace Reference

适合：

- 工作区内代码
- 工作区内纯文本
- 用户希望模型基于“当前仓库文件”继续读写，而不是只看一次快照

特点：

- 给模型的是“工作区引用语义”
- 模型后续可继续调用 `file.read` / `file.edit`
- 不强制上传整个文件本体到 LLM

### B. Direct Attachment

适合：

- 图片
- PDF
- 工作区外文件
- 需要当前快照、无需后续路径追踪的文档
- Office 文件

特点：

- 直接把内容作为 provider-native attachment 喂给模型
- 路径仅作为系统元数据保留
- 不要求模型再去主动读本地路径

## 4.3 引用位置本身也是语义

对这个需求，文件不只是“被带上消息”，还要“出现在消息的某个位置”。

例如：

```text
请先看这个实现 [src/main.rs]，再对照这张截图 [error.png] 判断哪里有问题。
```

这里的 `src/main.rs` 和 `error.png` 都不是尾部附件，而是正文中的有序引用。

因此，系统的 canonical 表达不应是：

```text
text + unordered attachments[]
```

而应是：

```text
ordered content parts[]
```

也就是：

- 文本段
- workspace 文件引用段
- direct attachment 段

按用户草稿中的视觉顺序组成一个有序序列。

## 5. 最小抽象

## 5.1 `DroppedAsset`

`DroppedAsset` 是“拖拽/选择文件后，进入内容层之前”的宿主机对象。
它属于 UI/Native ingestion 层，不直接暴露给 LLM。

建议最小结构：

```ts
type WorkspaceRelation = 'in_workspace' | 'outside_workspace' | 'unknown';
type AssetKind =
  | 'text'
  | 'code'
  | 'image'
  | 'pdf'
  | 'office'
  | 'archive'
  | 'binary'
  | 'unknown';

type IngestDecision =
  | 'workspace_ref'
  | 'direct_attachment'
  | 'extract_text_then_attach'
  | 'reject';

interface DroppedAsset {
  id: string;
  absolutePath: string;
  displayName: string;
  extension: string | null;
  mimeType: string | null;
  sizeBytes: number;
  sha256?: string | null;
  kind: AssetKind;
  workspaceRelation: WorkspaceRelation;
  workspaceId?: string | null;
  relativePath?: string | null;
  uploadAuthorizedByUser: true;
  decision: IngestDecision;
  rejectReason?: string | null;
}
```

字段语义：

- `absolutePath`: 仅内部使用，用于读文件、缓存、审计、错误提示。
- `workspaceRelation`: 判断是否可转成 `WorkspaceFileRef`。
- `decision`: 内容路由器产物，不是 UI 猜测。
- `uploadAuthorizedByUser: true`: 明确标记“这是用户主动提供内容”。

## 5.2 `ComposerDraftNode`

为了支持“拖到正文任意位置插入引用”，草稿层不能只保留一个 `inputText` 字符串。

建议新增草稿内部抽象：

```ts
type ComposerDraftNode =
  | {
      type: 'text';
      id: string;
      text: string;
    }
  | {
      type: 'asset_reference';
      id: string;
      assetId: string;
      displayText: string;
      routeHint: 'workspace_ref' | 'attachment';
    };
```

语义：

- `ComposerDraftNode[]` 是用户当前草稿的可编辑、有序内容流
- 文件拖入到某个位置，本质上是在节点序列中插入一个 `asset_reference`
- 发送前，再把 `ComposerDraftNode[]` 编译成 `UserContentPart[]`

如果只保留 `inputText + draftAssets[]`，附件只能挂在尾部，无法表达正文中部引用。

## 5.3 `UserContentPart`

`UserContentPart` 是真正进入聊天消息层的统一抽象。
它面向编译器，不面向宿主机文件系统。

建议最小结构：

```ts
type UserContentPart =
  | {
      type: 'text';
      text: string;
    }
  | {
      type: 'workspace_file_ref';
      workspaceId: string;
      relativePath: string;
      mediaType: 'text' | 'code';
      snapshotPolicy: 'reference_only' | 'inline_excerpt';
    }
  | {
      type: 'attachment';
      assetId: string;
      filename: string;
      mimeType: string;
      kind: 'image' | 'pdf' | 'office' | 'text' | 'code';
      delivery:
        | { mode: 'bytes' }
        | { mode: 'extracted_text'; text: string }
        | { mode: 'provider_file'; fileId?: string };
      sourcePath?: string;
    }
  | {
      type: 'unsupported_attachment';
      assetId: string;
      filename: string;
      reason: string;
    };
```

判断标准：

- `DroppedAsset` 是宿主机对象。
- `UserContentPart` 是会话对象。

新增约束：

- `UserContentPart[]` 的数组顺序必须保留用户草稿中的视觉顺序
- compiler 不得按类型重排这些 part

判断标准：

- `DroppedAsset` 是宿主机对象。
- `ComposerDraftNode` 是草稿对象。
- `UserContentPart` 是发送对象。

## 5.4 为什么不把绝对路径直接作为 `UserContentPart`

因为路径本身不是能力。

- `C:\foo\bar.png` 不等于模型能看图
- `C:\foo\spec.docx` 不等于模型能读 Word
- `C:\repo\src\main.rs` 也不等于模型当前 turn 能调用 `file.read`

路径只能作为引用线索，不能替代内容层。

## 6. 内容路由规则

建议 v1 路由表如下：

| 文件类型 | 在 workspace 内 | 默认路由 | 说明 |
| --- | --- | --- | --- |
| `.rs/.ts/.tsx/.py/.js/.md/.json/.yaml/.toml/.txt` | 是 | `workspace_ref` | 让模型保留后续工具读写能力 |
| `.rs/.ts/.tsx/.py/.js/.md/.json/.yaml/.toml/.txt` | 否 | `direct_attachment` 或 `extract_text_then_attach` | 不走工具外部访问 |
| 图片 | 任意 | `direct_attachment` | 模型直接看图 |
| PDF | 任意 | `direct_attachment` | 优先交给模型原生文档能力 |
| `docx/xlsx/pptx` | 任意 | `direct_attachment`，必要时 fallback 为 `extract_text_then_attach` | 取决于 provider 支持 |
| `zip/7z/rar/exe/dll` | 任意 | `reject` | v1 不支持 |

补充规则：

- workspace 内文本文件默认不上传全文，而是保留引用能力。
- 如果文件是拖到正文中部插入，则最终 `workspace_ref` / `attachment` 也必须保持在对应正文位置。
- 如果用户明确说“按这份当前快照分析，不要再读仓库”，可把 workspace 文件也降级成 attachment snapshot。
- 超大文本文件可自动转 `inline_excerpt` 或拒绝。
- 拖入多个文件时，每个文件独立路由，不要求同一策略。

## 7. Provider Compiler

## 7.1 编译器职责

Provider compiler 负责把 `UserContentPart[]` 转成不同 LLM provider 需要的消息格式。

输入：

- `system prompt`
- `history`
- `user parts: UserContentPart[]`
- `tool definitions`

输出：

- OpenAI-compatible request
- Anthropic-compatible request

编译器是内容层的核心边界。UI 不应该知道 `input_image` / `image` / `file_id` 的差异。

## 7.2 推荐新增内部消息模型

当前 `OpenaiMessage.content: Option<String>` 过窄。

建议引入统一输入消息模型：

```rust
pub enum LlmInputPart {
    Text { text: String },
    ImageBytes { mime_type: String, data_base64: String },
    FileBytes { mime_type: String, filename: String, data_base64: String },
    WorkspaceRef { workspace_id: String, relative_path: String },
}

pub struct LlmInputMessage {
    pub role: String,
    pub parts: Vec<LlmInputPart>,
    pub tool_calls: Option<Vec<ToolCall>>,
    pub tool_call_id: Option<String>,
}
```

再由 provider compiler 把它下编译到：

- OpenAI message blocks
- Anthropic content blocks

关键要求：

- `LlmInputMessage.parts` 必须保持 `ComposerDraftNode[] -> UserContentPart[]` 的顺序
- 编译器不能把正文里的文件引用降级成尾部统一追加附件

## 7.3 OpenAI 路线

建议：

- `Text` -> text block
- `ImageBytes` -> `input_image`
- `FileBytes(pdf/text/code/office)` -> 优先 `input_file` / provider file；不支持时 fallback 为文本提取
- `WorkspaceRef` -> 编译成文本提示，不直接给绝对路径，例如：
  - “用户附带了工作区文件引用：`src/main.rs`，如需继续读取请使用 `file.read`”

这样可以保留工具调用闭环，而不是把 repo 文件一次性塞成附件。

## 7.4 Anthropic 路线

建议：

- `Text` -> `text`
- `ImageBytes` -> `image`
- `FileBytes(pdf)` -> 若 provider 原生支持 document/file，优先原生；否则 fallback 为提取文本
- `WorkspaceRef` -> 同样编译成受控文本提示，而非宿主机绝对路径

## 7.5 Fallback 策略

当 provider 不支持某类文件时：

1. 先尝试本地提取纯文本
2. 再降级成 `attachment.delivery = extracted_text`
3. 仍失败则标记 `unsupported_attachment`

v1 不需要本地图像算法，但需要本地“格式适配”。

## 8. UI 设计路线

## 8.1 Composer 不再只是 textarea

`ChatComposer` 需要从：

- `input: string`

升级为：

- `draftNodes: ComposerDraftNode[]`
- `draftAssets: DroppedAsset[]`
- `contentParts: UserContentPart[]`

视觉上应增加：

- 拖拽高亮态
- 可见的插入光标 / 插入锚点
- 正文内联引用 token
- 附件 chip 列表
- 每个附件的类型、大小、路由策略
- 不支持文件的 reject reason

实现判断：

- 如果需求是“拖到正文任意位置引用文件”，最终形态必须是 tokenized composer 或等价的有序节点编辑器
- 单纯 `textarea` 无法成为最终实现

## 8.2 文件入口顺序

推荐实现顺序：

1. 文件选择器按钮
2. 粘贴图片
3. Windows Explorer 拖拽

原因：

- 编译层和数据层先稳定
- UI 交互风险更低
- 便于逐步验证 provider compiler

## 8.3 Windows 拖拽路线

推荐采用双通路：

- 窗口级 `file-drop` 负责稳定接住来自 Windows 资源管理器的文件
- 编辑器级 `drop / selection` 负责把文件绑定到正文中的具体锚点

也就是说，不能只监听 textarea `onDrop`，也不能只依赖窗口级 `file-drop`。

原因：

- Windows 资源管理器拖入桌面壳时，窗口级事件更稳定
- 可在整个 chat panel 做 drag hover 高亮
- 编辑器级锚点解析才能满足“拖到正文任意位置即在该处引用”
- 便于后续扩展到 Goal panel

建议状态机：

```text
Idle
  -> DragHover
  -> AnchorResolved
  -> Inspecting
  -> ReadyToAttach
  -> Attached
  -> Sending
  -> Sent / Failed
```

其中：

- `AnchorResolved` 表示系统已经知道“文件将插入到草稿中的哪个位置”
- 如果没有 anchor，只能说明它还是尾部附件，不满足这个新增需求

## 9. 后端改造点

## 9.1 IPC / Tauri 命令

现有：

```rust
send_chat_message_v2(message: String, ...)
```

建议升级为兼容式签名：

```rust
send_chat_message_v3(
    message: Option<String>,
    user_content_parts: Option<Vec<UserContentPartDto>>,
    ...
)
```

兼容策略：

- 如果只有 `message`，沿用旧链路
- 如果提供 `user_content_parts`，走内容层编译
- 短期内保留 `send_chat_message_v2`

## 9.2 聊天主链路

现有入口：

- `send_message_v2_with_session(content: String, ...)`

建议新增：

- `send_message_with_parts(parts: Vec<UserContentPart>, ...)`

主链路内部：

1. 归一化文本和附件
2. 保留有序内容位置
3. 构造 `LlmInputMessage`
4. 经 provider compiler 下编译
5. 调用实际 provider

## 9.3 附件读入与缓存

建议新增服务：

- `asset_inspector.rs`
  - 路径规范化
  - MIME/扩展名判断
  - workspace 关系判断
  - 大小/类型校验
- `asset_loader.rs`
  - 读取字节
  - 文本提取
  - base64 编码
- `attachment_cache.rs`
  - provider file id 缓存
  - 避免同文件重复上传

## 9.4 持久化

建议新增独立表，而不是把附件信息塞回 message 文本。

建议表：

- `chat_message_assets`
  - `id`
  - `message_id`
  - `session_id`
  - `filename`
  - `absolute_path`
  - `mime_type`
  - `kind`
  - `workspace_relation`
  - `delivery_mode`
  - `provider_file_id`
  - `sha256`
  - `size_bytes`
  - `created_at`

用途：

- 回放消息
- 审计本次到底传了哪些文件
- provider file reuse

## 10. 与现有权限模型的关系

本方案的关键边界：

- 拖入附件不走 proposal
- 附件上传不等于工具放权
- 后续模型若要访问新的宿主机路径，仍走既有 proposal / workspace 校验

因此：

- `attachment` 是“用户显式提供内容”
- `file.read(abs_path)` 是“模型申请读取宿主机文件”

二者在审计和 UI 上都应分开呈现。

---

## 11. 风险与约束

### 11.1 隐私

- 绝对路径中可能带用户名、目录结构、项目代号
- 默认不把绝对路径直接作为模型语义输入
- 对模型只暴露必要文件名、相对路径、类型说明

### 11.2 Token / 上传成本

- 大文本文件不能无脑全文上传
- workspace 内代码默认用引用而不是附件
- 需要大小阈值和 excerpt 策略

### 11.3 内容陈旧

- `workspace_ref` 是“后续可再读”的动态引用
- `attachment` 是“当前拖入时刻”的静态快照
- UI 应明确这两者区别

### 11.4 Provider 差异

- OpenAI / Anthropic 对 file/image 的细节不同
- 差异不应泄漏到 UI 和业务层
- 必须由 compiler 吞掉

## 12. 分阶段落地

### Phase 0: 抽象与编译层

- 定义 `DroppedAsset`
- 定义 `UserContentPart`
- 定义 `LlmInputPart`
- 新增 provider compiler
- 保持旧 `message: string` 兼容

验收：

- 能用构造数据跑通“文本 + 图片附件”的 provider request 编译

### Phase 1: 手动附件入口

- Composer 增加“添加文件”按钮
- 支持图片、PDF、文本、代码
- 展示附件 chip
- `send_chat_message_v3` 打通

验收：

- 不靠拖拽，也能把附件送入模型

### Phase 2: Windows Explorer 拖拽

- 增加窗口级 file-drop
- drag hover 高亮
- 自动生成 `DroppedAsset`
- 自动做路由判断

验收：

- 从资源管理器拖图片、PDF、代码文件到 chat panel 可稳定附加

### Phase 3: 文档适配增强

- Office 文档解析 fallback

- 更好的超大文件策略

验收：

- docx/xlsx/pptx 在不支持原生上传时仍可走可解释的降级路径

## 13. 最终推荐方案

最终建议不是“路径 or 上传”二选一，而是：

- 把绝对路径保留在 `DroppedAsset`
- 把草稿顺序建模为 `ComposerDraftNode[]`
- 把消息语义提升到 `UserContentPart`
- 让 routing 决定“这是 workspace 引用还是 direct attachment”
- 让 provider compiler 决定“这是 image block、file block 还是文本降级”

一句话总结：

> 路径属于控制面，内容属于模型面，位置属于语义面；拖拽附件要打通的是“有序内容面”，而不是让模型自己去猜宿主机路径。
