# 对话面板与工具感知整改方案

> 日期：2026-05-30  
> 状态：待派工  
> 范围：LLM 聊天正文渲染、会话运行态、工具调用感知、超时与历史一致性

## 1. 背景

本轮专项审查源于两个用户感知问题：

1. 对话面板能看到工具调用，但 LLM 最终回复正文和“思考过程”不可见。
2. LLM 同时调用大量工具时，用户感知像“卡住”，缺少本轮仍在工作的明确反馈。

审查后确认：项目已有工具调用能力、流式事件、工具卡和阶段事件，不是能力缺失；主要问题在于**正文渲染缺陷、运行态表达不足、工具调用展示过细且缺少聚合**。

## 2. 已确认事实

### 2.1 最新会话记录

`state/state/events.ndjson` 中最近两轮 chat v2 事件显示：

- `request_id=1f9c4b38-2c2e-41a7-97d5-8fb6fd23f3bc`
  - `session_id=null`
  - 大量工具调用后触发 `timeout`
  - 未出现 `reply_stored`
  - 结果：用户消息可能已写入，但助手最终回复未写入

- `request_id=39185f97-362d-429b-9e79-f9f270e61977`
  - `session_id=088d8c6f-ce70-4c32-a367-344c144b268d`
  - 正常出现 `reply_stored`
  - `final_text_chars=1212`
  - `tool_records=27`
  - `history=2`
  - 结果：后端确认 LLM 正文已生成并写入回复

因此，“没有返回”的实际情况分两类：

- 一轮因为整体 120 秒超时，确实没有助手最终消息。
- 一轮已经正常返回，但前端只显示工具调用，没有显示正文。

### 2.2 当前链路

后端：

- `send_message_v2_with_session()` 进入 chat v2 查询循环。
- `send_message_v2_inner()` 按 `session_id` 写入 user/assistant 消息。
- LLM 文本 token 通过 `stream-chat-token` 实时发给前端。
- 工具生命周期通过 `tool-execution-update` 发给前端。
- 阶段提示通过 `thinking-update` 发给前端。
- 完成后使用 `build_content_blocks_for_db()` 将 `thinking/text/tool_use/tool_result` 组合写入 `chat_messages.content`。

前端：

- `useChatSession()` 监听 `stream-chat-token`、`tool-execution-update`、`thinking-update`。
- `ChatTimelinePane` 使用 `parseContentBlocks()` 将消息 content 拆成 blocks。
- `ContentBlocksRenderer` 渲染 `thinking`、`text`、`tool_use`。
- `ToolUseCard`、`CommandRunCard`、`PermissionCard` 负责不同工具展示。

## 3. 根因分析

### P0 根因：MarkdownRenderer 未渲染 children

代码位置：

- `apps/desktop/src/components/MarkdownRenderer.tsx`

当前逻辑中 `MarkdownRenderer({ content })` 接收了 content，但没有把 content 传给 `ReactMarkdown`。

当前形态：

```tsx
<ReactMarkdown
  remarkPlugins={[remarkGfm]}
  rehypePlugins={[rehypeHighlight]}
  components={{ ... }}
/>
```

应改为：

```tsx
<ReactMarkdown
  remarkPlugins={[remarkGfm]}
  rehypePlugins={[rehypeHighlight]}
  components={{ ... }}
>
  {content}
</ReactMarkdown>
```

影响：

- assistant 的 `text` block 会进入 `MarkdownRenderer`。
- 但由于 children 为空，正文被渲染成空白。
- 工具调用可见，因为 `tool_use` block 走 `ToolUseCard`，不依赖 MarkdownRenderer。

这与用户观察完全一致：**工具调用看得见，LLM 返回正文看不见**。

### P1 根因：思考过程语义混乱

当前存在两类“思考”：

1. `thinking-update`
   - 用于实时阶段提示：正在分析、正在调用工具、正在整理结果。
   - 完成后会清空。
   - 它不是持久化的模型思考内容。

2. `ContentBlock::Thinking`
   - 来自 LLM provider 的 reasoning/thinking content。
   - 后端只在 `response.reasoning_content` 存在时写入。
   - 当前 `send_v2.rs` 明确不把 reasoning token 通过可见流输出。

代码位置：

- `crates/conductor-core/src/chat/send_v2.rs`
- `crates/conductor-core/src/llm.rs`
- `apps/desktop/src/windows/ThinkingBlock.tsx`

结论：

- 不应承诺总能展示 raw reasoning。
- 产品上应将可见内容命名为“工作过程”或“执行轨迹”，而不是“思考过程”。
- 展示重点应是阶段、耗时、工具摘要、可展开 transcript。

### P1 根因：会话运行态缺少 session-level 表达

当前左侧会话列表主要基于最后消息时间显示“几分钟前”。当某个会话里的 LLM 正在工作时，用户更关心：

- 是否还在工作。
- 已经工作多久。
- 当前卡在哪个阶段。
- 是否正在大量调用工具。

现有 `ChatSessionSummary` 没有运行态字段，无法表达：

- `working`
- `working_since`
- `working_elapsed_ms`
- `working_stage`
- `active_tool_count`

代码位置：

- `crates/conductor-core/src/chat/session.rs`
- `apps/desktop/src/windows/AgentWorkspacePanel.tsx`

### P1 根因：工具感知过细，缺少聚合

当前每个工具调用都可以展示成单独卡片。对于 3-5 个工具调用，这是有用的；对于 20+ 个工具调用，会变成调试日志洪水。

用户需要的是 Codex/Claude Code 风格的 transcript：

```text
• Running file.read apps/desktop/src/windows/useChatSession.ts
• Ran file.glob apps/desktop/src/**/*.tsx 5ms
• Ran file.read × 12
  └ prompt.rs, send_v2.rs, useChatSession.ts...
```

而不是默认展开 27 张工具卡。

代码位置：

- `apps/desktop/src/windows/ChatTimelinePane.tsx`
- `apps/desktop/src/windows/ToolUseCard.tsx`
- `apps/desktop/src/components/cards/CommandRunCard.tsx`
- `apps/desktop/src/components/cards/PermissionCard.tsx`

## 4. 目标体验

### 4.1 对话正文

用户发送消息后：

- LLM 正文正常显示。
- Markdown 正常渲染。
- 工具调用不会遮掉最终回复。
- 历史消息再次打开后仍能看到正文和工具轨迹。

### 4.2 本轮工作状态

发送中区域顶部显示：

```text
Working 01:38
读取文件 18 次 · 搜索 5 次 · 工具调用 27 次
当前：正在整理结果...
```

阶段建议：

- `准备上下文`
- `思考中`
- `调用工具`
- `整理结果`
- `生成回复`
- `已完成`
- `已超时`

### 4.3 左侧会话列表

非运行中：

```text
3 分钟前
```

运行中：

```text
Working 01:38
```

可选补充：

```text
正在调用工具 · 27 次
```

### 4.4 工具展示

默认聚合，展开查看详情。

默认态：

```text
• Running file.read src/windows/useChatSession.ts
• Ran file.glob apps/desktop/src/**/*.tsx 5ms
• Ran file.read × 12
  └ prompt.rs, send_v2.rs, ChatTimelinePane.tsx...
```

展开态：

- 显示每个工具输入。
- 显示 stdout/stderr/result。
- 保留现有 `ToolUseCard`、`CommandRunCard`、`PermissionCard`。

## P1：左侧会话列表显示 Working 时长

### T5 后端维护 active chat run 状态

负责人建议：后端

改动文件：

- `crates/conductor-core/src/chat/send_v2.rs`
- `crates/conductor-core/src/chat/session.rs`
- 可能新增 `chat_run_state.rs`

建议数据结构：

```rust
struct ActiveChatRun {
    session_id: Option<String>,
    request_id: String,
    started_at: DateTime<Utc>,
    phase: String,
    tool_run_count: usize,
    active_tool_count: usize,
}
```

实现方式：

- MVP 可用进程内 `OnceLock<Mutex<HashMap<String, ActiveChatRun>>>`。
- key 使用 `session_id`，无 session 时使用 `__global__`。
- `send_message_v2_inner()` 开始时登记 active run。
- `thinking-update` 和 tool events 更新 phase/count。
- done/error/timeout 时移除 active run。

验收：

- 某会话工作中，后端能查询到 active run。
- 正常完成后 active run 被清理。
- timeout 后 active run 被清理。

## P2：历史一致性和超时体验

### T9 reply.history 合并策略优化

负责人建议：前端

改动文件：

- `apps/desktop/src/windows/useChatSession.ts`

当前风险：

- 收到 `reply.history` 后直接替换当前 messages。
- 如果 session 创建、切换、超时、流式占位存在时间差，容易造成 UI 闪烁或临时消息消失。

任务：

- 将直接替换改为按 message id merge。
- 保留本轮 optimistic user message，直到后端返回同内容消息后去重。
- 保留 error message，不被空 history 覆盖。
- 只有主动切换 session 时才整包替换。

验收：

- 超时后用户消息仍保留并显示错误态。
- 正常返回后临时用户消息被后端消息替换，不重复。
- 切换会话不串消息。

### T10 超时后写入可见 assistant error

负责人建议：后端

- `crates/conductor-core/src/chat/send_v2.rs`

当前行为：

- 整体 120 秒超时后没有 `reply_stored`。

验收：

- 超时会话不会只留下孤立用户消息。

## 7. 验收用例

### 用例 A：普通聊天

输入：

```text
你能做什么？
```

预期：

- 有 LLM 正文。
- 没有工具卡。
- 没有空白 assistant 消息。

### 用例 B：工具密集查询

输入：

```text
审查对话面板为什么没有显示最终回复
```

预期：

- 发送中显示 `Working mm:ss`。
- 左侧当前会话显示 `Working mm:ss`。
- 工具调用默认聚合。
- 完成后显示最终 LLM 正文。
- 展开后能看到完整工具调用记录。

### 用例 C：超时

人为将 timeout 调低或构造大量工具调用。

预期：

- 运行中持续显示计时。
- 超时后保留用户消息。
- 显示可见错误/超时助手消息。
- active working 状态被清理。

### 用例 D：历史回放

重启应用后打开同一会话。

预期：

- 正文仍显示。
- 工具 transcript 可回放。
- 历史中不显示空白 Markdown。

### 前端

- `ToolRunSummary` 聚合 transcript。

### 后端

- `ChatSessionSummary` 增加 working 字段。
- timeout/error 结构化返回或持久化。

### 联调

- 运行中事件与 session id 对齐。
- working 状态完成/失败/超时均清理。
- 20+ 工具调用下 UI 不刷屏、不空白、不丢最终文本。

## 9. 结论

优先级最高的是 `MarkdownRenderer`，它是 LLM 正文不可见的直接原因。

随后应把“思考过程”产品语义调整为“工作过程/执行轨迹”，并通过本轮计时、左侧会话 working 时长、工具聚合 transcript 来提升用户感知。

这套整改不需要削弱工具能力；相反，它让已有能力变得可观察、可理解、可恢复。
