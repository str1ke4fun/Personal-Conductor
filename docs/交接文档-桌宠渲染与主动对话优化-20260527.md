# 交接文档：桌宠渲染、形象状态与主动对话优化

> 日期：2026-05-27  
> 范围：`apps/desktop`、`apps/desktop/src-tauri`、`crates/conductor-core`

## 1. 本轮目标

本轮围绕三条闭环推进：

1. 桌宠窗口、视频/图片形象、对话气泡能随窗口尺寸稳定渲染。
2. 主形象/子形象状态方案在外观面板可见，并在锁定形象时屏蔽 AI/MCP 形象切换工具。
3. 前端对话、LLM 工具、skill prompt、主动对话和 Trae 检测形成可验证链路。

## 2. 已完成变更

### 2.1 前端渲染与窗口尺寸

- 默认桌宠窗口从 `240x320` 调整为 `320x420`：
  - `apps/desktop/src/windows/PetWindow.tsx`
  - `apps/desktop/src-tauri/src/window_state.rs`
  - `apps/desktop/src-tauri/tauri.conf.json`
- 修复 `PetWindow` resize 保存时闭包捕获旧 `windowState` 的问题，避免拖拽/缩放后覆盖 `locked`、`scale` 等状态。
- `AvatarRenderer` 补齐 `reading/error/done` 等 activity variant 映射。
- 图片子形象增加 per-asset scale 和 bottom align，改善透明边距导致的显示过小问题。
- 对话气泡改为基于 `100vw/100vh` 和底部输入栏高度计算宽高，减少被小窗口裁切。
- 设置页原设计预览从 `<img src="mp4">` 改为 `<video>`，避免视频预览失效。

### 2.2 形象系统与锁定

- 外观面板保留三套主形象：`original`、`document_secretary`、`programmer`。
- 子形象预览继续放在外观面板中，用户能看到文档秘书/程序员的状态差分。
- `pet.avatarLocked` 的执行期约束已补齐：`pet.set_avatar` 与 `conductor.pet.set_avatar` 都会在工具执行层检查锁定状态。
- Chat LLM 暴露工具时会同时过滤 `pet.set_avatar` 和 `conductor.pet.set_avatar`。

### 2.3 Chat -> LLM -> 工具链路

- 桌面启动和 chat 发送前会注册内置 ToolRegistry，避免 LLM 看不到工具。
- OpenAI-compatible tool function name 已做安全转换：`pet.set_avatar` 暴露为 `pet__set_avatar`，返回时再映射回原工具 ID。
- 修正 tool call 消息顺序：模型返回 tool calls 后，会先追加 assistant/tool_calls 消息，再追加 tool 结果消息。
- 最近 12 条对话历史会进入模型上下文，前端展示历史不再只是 UI 层记录。
- LLM HTTP 非 2xx 响应会返回明确错误，不再只表现为 JSON parse 失败。
- 桌宠内联输入读取 `sendChatMessage` 返回值的 bug 已修复：现在读取 `reply.message.content`。
- Tauri `send_chat_message` 增加 activity 轮询广播，LLM 思考、工具调用、完成/错误等子形象状态能推到前端。

### 2.4 Prompt 与 Skill 注入

- 默认人设 prompt 已按《新中式旗袍 Live2D 角色设定归档》重写，强调：
  - 新中式旗袍视觉设定。
  - 工作同伴定位。
  - 简洁、可执行、不过度承诺的回答方式。
  - 主形象/子形象边界。
  - 锁定形象时不主动切主形象。
- `config.persona.skills` 仍会拼入 chat system prompt；`crates/conductor-core/src/skills.rs` 的 skill spec 目前仍是规划/路由数据，尚未真正参与工具 allowlist 路由。

### 2.5 主动对话与 Trae

- `trae.exe` 已在以下位置落地：
  - `crates/conductor-sense/src/whitelist.rs`
  - `crates/conductor-core/src/config.rs`
  - `apps/desktop/src/windows/SettingsPanel.tsx`
  - `apps/desktop/src-tauri/src/worker.rs`
- “检测当前应用”是测试按钮，不是唯一入口。
- 真正主动入口在 `worker.rs`：
  - focus watcher 监听前台应用切换。
  - 匹配 `proactive.toolTriggers` 后生成主动问候。
  - 定时 initiative checker 每 5 分钟检查一次主动提案。
- 主动消息现在会写入 chat history，并通过 `pet_message` 同步给桌宠气泡和已打开的 Chat 面板。

## 3. 验证结果

- `npm.cmd run build`：通过。
- `npm.cmd test -- --run`：通过，2 个测试文件、15 个测试。
- `cargo check -p conductor-core`：通过，仍有既有 warning。
- `cargo check -p conductor-cli`：通过，仍有既有 warning。
- `cargo check -p conductor-desktop`：当前环境缺 Windows SDK `RC.EXE`，Tauri build script 在资源编译阶段阻断。
- `cargo test --workspace`：未完成，当前环境缺 MSVC `link.exe`；桌面 crate 还受 `RC.EXE` 阻断。

## 4. 仍需注意

- 当前没有真正的文件上传/附件链路；chat IPC 仍只传 `content: string`。
- `crates/conductor-core/src/skills.rs` 尚未成为真实 skill router，只是默认 skill spec 数据。
- MCP adapter 仍是 mapping/discovery scaffolding，不是完整远程 MCP runtime。
- Live2D runtime 按 2026-05-26 方案暂缓；当前主线是视频 + PNG 混合渲染。
- ToolExecutor 的完整生命周期 hook 还没有中心化事件总线，本轮通过 chat 层 activity 与 Tauri 轮询解决可见反馈。

## 5. 演进方向

建议下一步按“可见闭环优先”推进：

1. 做 Workspace/Proposal 详情 UI，让用户看到工具为什么执行、在哪里执行、风险是什么。
2. 把 `skills.rs` 接到真实路由：根据 foreground app、用户意图和 workspace 类型选择 skill、工具 allowlist 和默认主形象。
3. 完成附件输入：前端支持文件/截图，IPC 支持 attachment metadata，后端把可读摘要注入 LLM。
4. 为主动对话增加策略面板：最近触发记录、冷却命中原因、当前检测到的应用和匹配 trigger。
5. 做中心化 activity event bus：LLM、ToolExecutor、AgentRun、TaskList 都写同一套 avatar activity 事件。
6. MCP remote execution 单独成阶段，不要和本地 ToolRegistry 混写成“已生产完成”。
