# Harness 架构审查与演进计划

> 日期: 2026-05-29  
> 范围: harness 架构、工具能力、Codex/Claude Code 对齐度、AgentTeam 编排、前端聊天面板体验  
> 结论: 方向先进，但当前仍是 agent-capable workbench/prototype，未达到 Codex/Claude Code 级别 harness  
> 方法: 状态机驱动审查 + OODA-R AgentTeam 范式

---

## 1. 总体判断

当前项目已经具备成为高强度 coding-agent harness 的底座：

- 工具系统已模块化，`ToolSpec` 包含 provider、schema、risk、permissions、workspace_required。
- LLM 层支持 OpenAI/Anthropic 兼容协议，并支持流式 tool call。
- 已有 workspace、proposal、AgentRun、AgentTeam、MCP、任务抽屉等关键雏形。
- 前端可以展示流式 token、thinking、tool card 和 proposal 队列。

但它还没有形成真正的“运行时状态机 + 权限闸门 + 人机协作控制台”。核心问题不是工具数量不足，而是关键对象还没有成为一等状态实体：`ToolCall`、`PermissionGrant`、`CommandRun`、`AgentRunPhase`、`AgentTeamPhase`、`WorkspaceScope`、`AuditEvent`。

因此当前系统可以做基础对话和部分工具执行，但还不能稳定完成 Codex/Claude Code 级别的工作闭环：理解需求、规划、申请权限、执行文件修改、运行命令、流式反馈、处理阻塞、接受人工 review、恢复上下文、审计所有副作用。

## 2. 关键发现

### P0-1 工具调用缺少统一审批闸门

**证据**

- `crates/conductor-core/src/chat.rs:1367` 直接调用 `tools::execute_tool_with_workspace_async`。
- `crates/conductor-core/src/chat.rs:1266` 只有错误包含 `outside current workspace` 时才创建 proposal。
- `crates/conductor-core/src/tools/registry.rs:278` 将 `Trusted | AskWrite` 都视为 `Ok(())`，`AskWrite` 没有真正 ask。
- `crates/conductor-core/src/tools/fs.rs:487-522` `file.write` 和 `file.edit` 被标记为 `WorkspaceWrite`，但在已绑定 workspace 下可进入直接执行路径。

**影响**

风险标签存在，但不是硬权限边界。用户以为“需要同意的写操作”可能已经被执行。长期看会造成三类漂移：

- 设计意图漂移: `AskWrite` 语义和实际行为不一致。
- 安全边界漂移: 工具 risk/permission 成为展示信息，而非执行约束。
- 审计漂移: 事后只能看到 message 内嵌 tool record，缺少独立 ToolCall/PermissionGrant 账本。

**目标**

所有工具调用统一进入：

```text
Proposed -> RiskClassified -> AwaitPermission? -> Approved/Denied -> Executing -> Succeeded/Failed -> Recorded
```

任何 `WorkspaceWrite`、`ExternalSideEffect`、`SystemControl` 都必须生成 `ToolCall` 记录；是否需要审批由 `WorkspaceTrustLevel + CapabilityMode + ToolRisk` 决定，而不是由工具执行失败后补救。

---

### P0-2 Codex harness 仍是占位终端

**证据**

- `crates/conductor-core/src/codex.rs:396-406` 默认执行 `cmd.exe /C dir`。
- `crates/conductor-core/src/codex.rs:497` 有 `send_input`，但默认命令很快结束，无法形成真实交互。
- `crates/conductor-core/src/codex.rs:540` `resume_session` 明确不支持。
- `crates/conductor-core/src/tools/codex.rs:203-209` `codex.start` 是高风险工具，但 `workspace_required: false`。

**影响**

当前 `codex.*` 工具更接近“进程启动 demo”，不是 Codex 级 harness。它缺少：

- 真实 `codex` 命令启动与会话生命周期。
- PTY/伪终端。
- stdout/stderr 实时流。
- input/interrupt/resume 的稳定语义。
- 权限请求、工具事件、文件 diff、命令事件解析。
- 会话持久化与恢复。

**目标**

把 `codex` 从工具函数升级为 `InteractiveAgentSession`：

```text
Created -> Starting -> Ready -> Running -> AwaitInput -> Interrupted -> Resumable -> Completed/Failed/Stopped
```

并将其 stdout/stderr、工具调用、权限请求、文件变更都转换为统一 runtime event。

---

### P0-3 Shell/CommandRun 不具备 coding-agent 级控制能力

**证据**

- `crates/conductor-core/src/tools/shell.rs:63` 明确写着同步命令不能中途取消。
- `crates/conductor-core/src/shell/mod.rs:106-138` 等待命令结束后汇总 stdout/stderr。
- `crates/conductor-core/src/tools/shell.rs:111-114` `bash.execute` 是 `ExternalSideEffect + SystemControl`，但 `workspace_required: false`。
- `crates/conductor-core/src/shell/security.rs:273-293` 工作目录校验依赖 `CONDUCTOR_ROOT` 或进程 cwd，未显式绑定当前 ChatSession 的 WorkspaceRootGuard。

**影响**

运行构建、测试、安装依赖、启动服务时，用户无法像使用 Codex/Claude Code 那样：

- 看到实时输出。
- 中断卡住的命令。
- 复用 command id。
- 追踪命令属于哪个 agent/tool/user action。
- 对高风险命令进行逐次授权。

**目标**

引入 `CommandRun` 一等实体：

```text
Prepared -> AwaitPermission -> Starting -> Streaming -> Exited/TimedOut/Killed/Failed -> Archived
```

每个命令必须记录 `cwd`、`argv/raw_command`、`env_delta`、`risk`、`permission_grant_id`、`agent_run_id`、`tool_call_id`、`stdout_tail`、`stderr_tail`、`exit_code`。

### P1-1 AgentRun 状态太粗，无法表达真实认知生命周期

**证据**

- `crates/conductor-core/src/agent_runs.rs:16-21` 只有 `Queued/Running/Succeeded/Failed/Stopped`。
- `crates/conductor-core/src/agent_runs.rs:160-173` `claude -p` 作为子进程启动后即进入 Running。
- `crates/conductor-core/src/agent_runs.rs:253-268` 失败/停止会把关联 task 写回 `pending`。

**影响**

`AgentRunStatus` 当前是进程状态，不是 agent 工作状态。它无法表达：

- 正在观察上下文。
- 正在规划。
- 等待计划审批。
- 等待工具权限。
- 正在执行工具。
- 被外部阻塞。
- 等待人工 review。

这会让前端、任务系统、派工系统都只能看到“running/pending”，无法判断下一步该由谁行动。

**目标**

拆分两个状态：

- `ProcessStatus`: queued/running/exited/killed/failed。
- `AgentRunPhase`: observing/orienting/planning/awaiting_plan_approval/executing/awaiting_permission/reviewing/blocked/completed。

---

### P1-2 AgentTeam 是 mailbox 容器，不是 OODA-R 编排器

**证据**

- `crates/conductor-core/src/agent_teams.rs:11` team 只有 `Active/Archived`。
- `crates/conductor-core/src/agent_teams.rs:35` member 只有 `Active/Paused/Stopped`。
- `crates/conductor-core/src/agent_teams.rs:67-90` 有 `PlanApprovalRequest/Response` 消息类型，但不是硬状态转换。
- `crates/conductor-core/src/agent_teams.rs:264` 可以添加成员，`crates/conductor-core/src/agent_teams.rs:324` 可以发送消息，但没有写作用域、阶段闸门、冲突锁。

**影响**

多 agent 可以沟通，但系统不能保证：

- 先评估设计，再落地。
- 未经计划批准不得写代码。
- 多个 agent 不会同时改同一文件。
- review agent 的结论会阻断或放行 merge。
- 人工反馈能进入下一轮 OODA。

**目标**

AgentTeam 引入硬状态机：

```text
Draft -> Planning -> AwaitPlanApproval -> Executing -> AwaitReview -> ReworkRequired/Accepted -> Archived
```

成员必须带上：

- `role`
- `write_scope`
- `read_scope`
- `allowed_tools`
- `handoff_contract`
- `review_required`
- `conflict_lock_policy`

---

### P1-3 MCP 支持不完整，且缺少 per-tool 权限建模

**证据**

- `crates/conductor-core/src/mcp.rs:247-249` stdio transport 未实现。
- `crates/conductor-core/src/mcp.rs:267-268` stdio call 同样未实现。
- `crates/conductor-core/src/mcp.rs:494-513` MCP tools 继承 provider risk，但 `permissions: vec![]`。

**影响**

Claude Code 生态里大量 MCP server 使用 stdio。当前实现不能覆盖主流 MCP 使用方式。更重要的是 MCP provider 级 risk 过粗，无法区分一个 provider 内的 read/search/write/delete/send 等不同工具风险。

**目标**

MCP 工具注册必须进入工具权限映射流程：

```text
Discover -> PendingClassification -> RiskMapped -> PermissionMapped -> Enabled/Disabled
```

未分类工具默认 disabled，只允许显式启用。

---

### P2-1 前端聊天面板仍是普通聊天框，不是 coding-agent 控制台

**证据**

- `apps/desktop/src/windows/ChatComposer.tsx:11-23` 使用单行 `input`。
- `apps/desktop/src/windows/ChatComposer.tsx:15` Enter 直接发送，没有 Shift+Enter 多行语义。
- `apps/desktop/src/windows/ToolUseCard.tsx:6` tool 状态只有 `pending/running/success/error`。
- `apps/desktop/src/windows/ChatTimelinePane.tsx:34` 没有 tool result 时默认显示 `pending`。
- `apps/desktop/src/windows/TaskDrawerPane.tsx:269-289` proposal 操作在任务抽屉里，而不是嵌回当前对话上下文。
- `apps/desktop/src/windows/AgentWorkspacePanel.tsx:181-207` 工作区绑定暴露绝对路径输入和“绑定到当前会话”流程。

**影响**

用户可以聊天，但不容易管理一次复杂 agent 工作：

- 不知道当前 agent 是在想、在等权限、在跑命令，还是卡住了。
- 不知道下一步该自己 approve、补充信息、review diff，还是等待。
- 长需求输入体验弱。
- 工具调用、proposal、任务抽屉、workspace 之间的关系需要用户自己理解。

**目标**

聊天面板升级为“人类可读的运行控制台”：

- Composer 支持多行、Shift+Enter、附件、上下文 chips、能力模式、停止/继续。
- Tool card 支持 `awaiting_approval/blocked/retryable/cancelled/skipped`。
- Permission card 出现在对话主线中，可直接同意/拒绝/仅本次允许。
- Workspace 显示为上下文胶囊，而不是要求用户理解路径绑定。

## 4. 能力模式设计

面向用户不应该暴露一堆 tool id，而应该暴露能力模式。

| 模式 | 可做什么 | 必须审批 |
|---|---|---|
| Read-only | 读文件、搜索、总结、解释代码 | 无写入，无命令 |
| Workspace edit | 读写当前 workspace 文件 | 首次写入、跨目录写入、大批量写入 |
| Command run | 运行构建、测试、脚本 | 外部副作用、安装依赖、长命令 |
| Agent mode | 启动子 agent、AgentTeam、长任务 | 计划审批、工具权限、review |
| MCP mode | 调用外部 MCP 工具 | 未分类工具、写/发/删/外部访问 |

内部仍保留精细 tool permission，但用户只需要理解“本次我允许 agent 做到什么程度”。

## 5. 演进路线

### Phase 0: 止血，先把边界变硬

目标: 避免工具继续绕过用户意图。

- 将 `AskWrite` 改为真实 ask，不再等同 `Trusted`。
- 所有 tool call 先写入 `tool_calls`，再执行。
- `WorkspaceWrite/ExternalSideEffect/SystemControl` 默认进入 permission flow。
- proposal 不再只处理 `outside current workspace`，而是成为统一权限请求的一种 UI 表达。
- 前端 tool card 增加 `awaiting_approval/denied/blocked/cancelled`。

验收标准：

- `file.edit` 在 `AskWrite` workspace 下不会直接执行。
- shell/codex/agent.start 必须出现可审计的 permission grant。
- 聊天主线能直接显示并处理一次工具审批。

### Phase 1: CommandRun/PTY，补齐 coding-agent 执行力

目标: 让系统可以像 Codex/Claude Code 一样运行、观察、停止命令。

- 引入 `command_runs` 表。
- shell 执行返回 command id。
- stdout/stderr 实时事件推送到前端。
- 支持 cancel/kill/retry。
- 明确 cwd 必须落在 WorkspaceRootGuard 内。
- Codex session 改为真实可交互进程，不再默认 `dir`。

验收标准：

- 运行 `npm test`/`cargo test` 时前端能实时显示输出。
- 用户可以停止长命令。
- 命令输出、退出码、执行人、审批记录可追溯。

### Phase 2: AgentRun 认知状态机

目标: 让前端和派工系统知道 agent 现在到底处于哪一步。

- 拆分 `process_status` 和 `agent_phase`。
- `claude -p` 输出归档为 run events。
- 失败/停止不再简单写回 task `pending`，而是进入 `AwaitReview` 或 `Blocked`。
- 支持人工插入 feedback 后继续下一轮 OODA。

验收标准：

- 任意 AgentRun 都能回答：当前目标、当前阶段、是否等用户、是否等权限、最后一次工具调用、下一步动作。

### Phase 3: AgentTeam OODA-R 硬门禁

目标: 多 agent 协作不再只是 mailbox，而是可控的软件工程流程。

- Team 引入 `phase`。
- Plan approval 成为硬状态转换。
- Member 引入 `write_scope/read_scope/allowed_tools`。
- 引入文件级或模块级 write lock。
- Review agent 结论能阻断完成态。

验收标准：

- 没有 approved plan 时，成员不能执行写操作。
- 两个成员不能同时写同一 scope。
- Review failed 会自动进入 ReworkRequired，而不是 completed。

### Phase 4: 前端工作台人机体验

目标: 从“开发者控制台”变成“人能自然使用的 agent 工作台”。

- Composer 多行化，支持 Shift+Enter。
- 增加上下文 chips: workspace、能力模式、当前任务、选中文件。
- Tool/Permission/Command/Review 作为主线卡片展示。
- 任务抽屉只做总览，当前会话相关动作必须回到聊天主线。
- Workspace 绑定改为选择器 + 最近项目 + 拖拽打开，绝对路径输入放入高级入口。

验收标准：

- 用户无需理解 tool id、proposal、workspace_id，也能完成一次“让 agent 修改代码并跑测试”的闭环。
- 当前阻塞点在主界面一眼可见。

### Phase 5: MCP 与外部工具生态

目标: 支撑 Claude Code 级工具生态。

- 实现 stdio MCP。
- MCP tools 进入待分类状态。
- 每个 MCP tool 单独 risk/permission mapping。
- 默认禁用未分类 MCP tool。
- 支持 MCP 调用审计和权限撤销。

验收标准：

- 可接入常见 stdio MCP server。
- 同一个 MCP provider 内 read/write/delete/send 能被区分授权。

## 6. 前端聊天面板目标形态

当前问题不是视觉不够美，而是信息架构不匹配 agent 工作。

目标主线卡片：

```text
UserGoalCard
PlanCard
PermissionRequestCard
ToolCallCard
CommandRunCard
DiffReviewCard
BlockedCard
CompletionSummaryCard
```

Composer 必备能力：

- 多行输入。
- Shift+Enter 换行，Enter 发送可配置。
- Stop/Continue/Retry。
- 当前 workspace chip。
- 当前 capability mode chip。
- 附件/文件/目录引用。
- “让它先计划，不要执行”开关。

Tool card 必备状态：

```text
pending
awaiting_approval
approved
running
streaming
success
error
blocked
cancelled
denied
retryable
```

主界面必须回答四个问题：

- 它现在在做什么？
- 它需要我做什么？
- 它已经改了什么？
- 我如何停止、回滚、继续？

## 7. 文档化回归清单

后续每轮 harness 演进，都应检查以下问题：

- 是否新增了绕过 `ToolCall -> PermissionGrant -> Execution` 的执行路径？
- 是否新增了只有 process status、没有 agent phase 的运行态？
- 是否新增了高风险工具但未定义 risk/permission/workspace_required？
- 是否新增了前端状态但没有后端持久化来源？
- 是否新增了 proposal/task/run 三套状态之间的隐式映射？
- 是否让用户必须理解内部 id、路径、tool id 才能完成操作？
- 是否支持停止、重试、恢复和审计？
- 是否在 AgentTeam 中保留了 plan-before-act 和 review-after-act 的硬约束？

## 8. 建议优先级

| 优先级 | 事项 | 价值 |
|---|---|---|
| P0 | 统一 ToolCall/PermissionGrant 闸门 | 解决安全边界和设计意图漂移 |
| P0 | `AskWrite` 真实审批 | 修复权限语义错误 |
| P0 | CommandRun 实体与实时输出 | 支撑测试/构建/长任务 |
| P1 | 真实 Codex/PTY session | 对齐 Codex 类体验 |
| P1 | AgentRun phase 状态机 | 让派工、UI、review 可判断下一步 |
| P1 | AgentTeam OODA-R 硬门禁 | 支撑多 agent 混合开发 |
| P2 | 前端主线权限/工具/命令卡片 | 降低用户认知成本 |
| P2 | MCP stdio + per-tool permission | 对齐 Claude Code 生态 |
| P3 | 能力模式 UI | 让普通用户不必理解 tool id |

## 9. 一句话目标

把当前 harness 从“能调用工具的聊天工作台”升级为“状态机驱动、权限可审计、命令可控制、多 agent 可治理的人机协作运行时”。
