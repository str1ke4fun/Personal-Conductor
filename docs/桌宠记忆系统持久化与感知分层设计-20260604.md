## 一、当前持久化盘点

### 1.1 三类存储介质

| 介质 | 内容 | 写频率 | 读频率 |
|---|---|---|---|
| **SQLite**（主库，WAL, 5 conn） | tasks / chat / tool / agent / goal / memory / affection / mood / scene / workspace 等 25+ 张表 | 高（事务/会话级） | 高 |
| **JSON 文件**（`state/`） | config / persona_state / scene_state / initiative_state / proposals / tasks / pet_window / event_signal / inject.log 等 | 中（事件级） | 中 |
| **events.ndjson** | 审计事件流，一行一 JSON | 高（append-only） | 低（仅审计） |

### 1.2 向量库现状

- 表 `memory_embeddings`（BLOB，f32 little-endian，默认 384 dim）
- `EmbeddingProvider` 已抽象（见 [embedding.rs](../crates/conductor-core/src/embedding.rs)）：
  - `BgeSmallZhProvider`（HTTP `POST /embed`）
  - `BgeSmallEnProvider`
  - `HashFallbackProvider`（确定性伪向量，离线兜底）
  - `CompositeEmbeddingProvider`（primary + fallback 自动切换）
- 评分公式（[memory.rs](../crates/conductor-core/src/memory.rs) 当前实现）：

  ```text
  score = 0.55 · vector_similarity
        + 0.20 · keyword_match
        + 0.15 · recency
        + 0.10 · authority
  ```

- 注入策略（`chat/prompt.rs`）：top 5、每条 content 截 80 字符
- 摘要召回：取 `conversation_summaries` 最近 3 条，每条截 80 字符

### 1.3 核心断点（用户已观察到）

```text
memory_entries / conversation_summaries
        │
        │   ← 写入触发器未接通
        ▼
   memory_chunks          ← 表存在；生产链路未跑通
        │
        │   ← generate_embedding 多走 hash 伪向量
        ▼
   memory_embeddings      ← 表存在；真实数据几乎为空
        │
        │   ← 检索评分函数 OK
        ▼
   search_memory()        ← 真实环境下命中 ≈ 0
        │
        ▼
   prompt 注入            ← 实际 LLM 没有"被记忆"的内容
```

也就是说：**schema 都建好了，但没有任何真实数据能进 chunks 表**。向量库有壳没肉，记忆系统处于"接得通但用不上"的状态。

## 二、桌宠 + Agent Workflow 定位下的"记忆"边界

### 2.1 该持久化什么（四层）

| 层级 | 内容 | 介质 | 保留期 |
|---|---|---|---|
| **A. 身份层** | persona, avatar_state, persona_state.json | SQLite + JSON | 长期 |
| **B. 关系层** | affection_state, mood_state, scene_state, initiative_state | SQLite + JSON | 长期 + 衰减 |
| **C. 偏好/事实层** | memory_entries（preference / fact / episodic / pattern） | SQLite → chunks → embeddings | 长期 / 90d / 30d |
| **D. 工作记忆层** | conversation_summaries, episodic chunks, interaction_patterns | SQLite → embeddings | 30-90d 滚动 |

### 2.2 不该持久化（或短期滚动）的

| 数据 | 处理 |
|---|---|
| 单次 `tool_calls.output` / `command_runs.stdout/stderr` | 入库短期 30d；超期后转 NDJSON 压缩归档 |
| 鼠标位置、临时 UI state、bubble_queue | 仅内存，不入 SQLite |
| agent_runs 流式中间产物 | 落盘到 `agent-runs/<id>.jsonl`，可手动清理 |
| 失败 draft / retry 中间态 | 内存；commit 后才入库 |
| events.ndjson | 90d 后归档或按体积切片 |
| 同一 Goal 在 OODA 多次反复的 plan/act/review | 只保留最终 cycle；前面的在 `goal_cycles` 设 `status=abandoned` |

### 2.3 边界规则

1. **敏感度三档**（已有 `MemorySensitivity` 枚举）：
   - `Normal` — 偏好、对话、任务
   - `Private` — 个人信息、健康、财务
   - `Secret` — API key、token、密码片段、含 token 的工具输出

2. **Secret 必须离线**：`memory_embeddings` **不**收录 Secret；Secret 走 `runtime_token.txt`（不进库）；工具输出若含 Secret 内容自动 mark Secret 并限制保留期。

3. **工作区隔离**：`memory_chunks.workspace_id` 必须强制；跨工作区记忆默认不召回（[SearchFilter](../crates/conductor-core/src/memory.rs) 已实现 `path_prefix` / `workspace_id` 过滤）。

4. **路径前缀**：同一 workspace 内按 `path_prefix` 二次切分（`normalize_path_prefix` 已实现）。

5. **跨会话归属**：`memory_entries.source_session_id` / `source_turn_id` / `source_message_id` 强约束，防止"工作区 A 的会话记忆污染工作区 B"。

6. **写入门控**（[memory.rs::write_gate](../crates/conductor-core/src/memory.rs) 已实现）：
   - `user` → `UserConfirmed` / active / 1.0
   - `tool` → `Tool` / candidate / 0.7
   - `inferred` → `Inferred` / candidate / 0.5
   - `pattern_aggregation` → `PatternAggregation` / active / 0.8

## 三、用户感知层级（四层 UI 落点）

### 3.1 核心原则

> **角色"内在状态" ≠ 用户不可见**。mood / affection / scene 是桌宠的内在，但应该让用户看到（状态条、光晕、关系光环）。"内在"和"可见"是正交概念。

四层职责分得很清：

| 层级 | 角色 | 数据来源 | 触发频率 | 桌宠方案对应 |
|---|---|---|---|---|
| **L1 时刻感知** | "看一眼就知道她什么心情" | mood, affection, scene, idle_phase | 实时 | PNG 动态化方案 §5 状态→动画映射 |
| **L2 气泡故事** | "她主动讲给你听" | summaries + episodic + patterns | 事件触发 | PNG 动态化方案 §7 气泡重构 + §6 initiative |
| **L3 记忆簿** | "你翻她记的" | memory_entries + chunks + patterns | 手动 | 待派工（见 §7 P1） |
| **L4 审计日志** | "你查账" | tool_calls, events.ndjson, agent_runs | 手动 | 已有（TaskPanel / EventPanel） |

### 3.2 L1 时刻感知（实时）

- **位置**：桌宠状态条、头顶光环、表情层、鼠标侧头
- **数据**：mood PAD、affection 阶段、scene tag、idle phase
- **前端接入点**（PNG 动态化方案）：
  - `MoodAura`（头顶柔光环）
  - `MoodFace`（左上小表情）
  - `StatusBadge`（状态点）
  - `SceneOverlay`（环境 tint）
  - `--pet-anim-rate` / `--pet-brightness` CSS 变量

### 3.3 L2 气泡故事（事件触发）

- **位置**：`ChatBubble` / `SelfBubble` / `NotificationBubble`
- **数据**：`conversation_summaries` + `episodic chunks` + `interaction_patterns`
- **触发器**：
  - 用户问"还记不记得…"（直接召回）
  - 桌宠 initiative 触发（见 [initiative.rs](../crates/conductor-core/src/initiative.rs) 5 类）
  - 关系阶段跨越
  - 用户连续多天未上线后回归
- **写作要求**：summaries 内容**只用于辅助表达**，不直接复制到气泡；气泡文本由 persona 化改写

### 3.4 L3 记忆簿（手动）

- **位置**：记忆管理面板（待建）
- **数据**：`memory_entries`（KV 主）+ `memory_chunks`（可索引副本）+ `interaction_patterns`
- **能力**：浏览 / 编辑 / 删除 / 归档 / 隔离；按 category / scope / sensitivity / status 过滤；查看 chunk 是否已建索引；手动触发 embedding 重建
- **重要性**：让长期记忆不再是黑盒

### 3.5 L4 审计日志（手动）

- **位置**：任务面板、事件面板、Agent 面板
- **数据**：`tasks` / `tool_calls` / `command_runs` / `agent_runs` / `events.ndjson`
- **要求**：Secret 字段默认脱敏；提供"导出某段时间审计"能力

## 四、记忆类型 × 感知矩阵

> 同一份数据可能在多个 L 层出现；下表是当前设计的**默认感知位置**。

| 类别 | 持久 | 保留 | L1 状态 | L2 故事 | L3 记忆簿 | L4 审计 |
|---|---|---|---|---|---|---|
| 用户偏好（preference） | 长期 | 长期 | – | ✓ | ✓ | – |
| 关系阶段（affection） | 长期 | 长期 | ✓ | ✓ | ✓ | – |
| 连续登录天数 | 长期 | 长期 | ✓ | ✓ | – | – |
| 心情 PAD（mood） | 长期 | 长期 | ✓ | – | – | – |
| 场景状态（scene） | 30d | 30d | ✓ | ✓ | – | – |
| 对话摘要（summary） | 90d | 90d | – | ✓ | ✓ | – |
| 交互模式（pattern） | 90d | 90d | – | ✓ | ✓ | – |
| 工作区信任 | 长期 | 长期 | – | – | ✓ | ✓ |
| 任务列表 | 长期 | 长期 | – | – | ✓ | ✓ |
| 工具调用记录 | 长期 | 长期 | – | – | – | ✓ |
| API key / token | Secret | 长期 | – | – | ✓ | – |
| 关系升级事件 | 长期 | 长期 | ✓ | ✓ | ✓ | – |
| 工具失败模式 | 30d | 30d | – | – | ✓ | ✓ |
| 主动开口冷却 | 7d | 7d | ✓ | – | – | – |

**关键洞察**：

1. L1 不需要"看到"对话内容，只需看到**情绪 / 关系 / 场景**这些聚合信号
2. 桌宠**主动讲述**（L2）需要的是 summaries + episodic + patterns 三类，**不是**裸的 `memory_entries`
3. 同一份 affection 既是 L1 头顶光环、又是 L2 阶段升级气泡、又是 L3 记忆簿里可编辑项——一数三用

## 五、记忆的四种"心智类型"

> 这是定位判断。技术 schema 是承载；**心智类型**才是桌宠人格的根。

### 5.1 事实性记忆（KV 风）

- "用户喜欢简短直接的回复"
- "用户项目是 personal-agent"
- "用户在用 VSCode + 1Password + 飞书"
- 落 `memory_entries`（source=user, status=active, confidence=1.0）
- L3 记忆簿让用户管，长期保留

### 5.2 叙事性记忆（清和独有的"心法"）

> **这是桌宠人格的核心载体。**

- "5月30日晚上 11 点用户在改聊天面板，前两次 review 都打回了，她有点烦躁"
- "用户上周末加班赶一个 ddl，那阵子不爱被打扰"
- "用户第一次关系升到 Friend 那天，她给我们发了一个不常用的感谢语"
- 落 `conversation_summaries` + `episodic chunks`（90d 滚动）
- L2 气泡层由清和**主动**援引
- 表达方式：summaries 内容只用于辅助表达，气泡文本由 persona 化改写

### 5.3 模式性记忆（pattern）

- "用户晚上 9 点后被主动开口的接受率是 78%"
- "用户对工具失败的反应，67% 是 '嗯，再来'，33% 是 '先停一下'"
- 落 `interaction_patterns`（90d 滚动）
- **不进 L1/L2 提示词**；作为 `initiative.rs` 决策权重
- L3 记忆簿可查看，不让用户直接编辑（系统聚合）

### 5.4 工作记忆（agent workflow 用的）

- 当前 goal、当前 task 状态、最近 tool call、当前 OODA cycle
- 走 `goals` / `goal_cycles` / `tasks` / `chat_turn_events`
- **不进 L2 故事层**——那是 workflow 内部状态，不该让桌宠讲给用户听
- 仅 L4 审计可查

## 六、断点修复优先级

| # | 断点 | 工作量 | 收益 |
|---|---|---|---|
| 1 | `index_memory_entry` / `index_conversation_summary` / `reindex_memory_chunk` 三个 internal 函数未实现 | 1d | 检索命中从 0 → 1 |
| 2 | `generate_embedding` 多走 hash 伪向量 | 0.5d | 检索质量从 0 → 真 |
| 3 | `memory_chunks` 表空，跑一次回填脚本（既有 entries → chunks） | 0.5d | 历史数据可用 |
| 4 | `summary → chunks` 链路接通 | 1d | "记得我们聊过"生效 |
| 5 | 检索 → prompt 注入串联（已接，验证） | 0.5d | LLM "知道" |
| 6 | L3 记忆簿 UI | 2d | 用户可管理 |

**总投入 ≈ 5.5d**，可让"记忆"从"有 schema 没数据"变成"真有数据、真能召回、真能被用户感知"。

详细派工见 [场景化记忆系统细粒度落地方案-20260530.md](场景化记忆系统细粒度落地方案-20260530.md) Phase 1-6。

## 七、用户感知的 P0 / P1 / P2 落地

### 7.1 P0（必做，与"断点修复"对齐）

1. **L1 时刻感知接通**：
   - mood / affection / scene 已在 SQLite + JSON，且有钩子
   - 前端要的是把这三类信号"接上"状态条 / 光环（见 [桌宠静态 PNG 动态化与前端综合优化方案-20260604.md](桌宠静态PNG动态化与前端综合优化方案-20260604.md) §5 状态→动画映射表）
2. **L2 故事层小闭环**：
   - 修好 `add_conversation_summary` → `index_*` → embedding → 检索
   - 让桌宠"我看到你昨晚加班了"这条气泡走得通
3. **L4 审计脱敏**：
   - 任务面板 / 事件面板**确保 Secret 字段不被默认展示**

### 7.2 P1（建议做）

4. **L3 记忆簿 UI**：
   - 用户浏览 / 编辑 / 删除 / 禁用 `memory_entries`
   - 提供"清空某类"、"导出 JSON"、"暂停某类写入"开关
5. **隐私分级确认**：
   - 用户首次产生 Secret 内容时，弹一次性确认"这条要保存吗 / 标 Secret 吗"
6. **关系档案页**：
   - "我们认识 14 天 / 8 个对话 / 3 个任务 / 上次升级是 6/2"
   - 数据全来自 `affection_state` + `memory_entries`

### 7.3 P2（可选）

7. **场景化召回**：
   - 7 类 SceneTag × 5 类 RelationshipStage × 7 类 mood，召回权重动态调整
8. **Pattern 可视化**：
   - "你最近一周的写作模式：晚上 9-11 点最专注"
   - 数据来源 `interaction_patterns`
9. **跨工作区共享开关**：
   - 工作区 A 学到的偏好是否带到工作区 B
10. **数据导出/导入**：
    - 整个 SQLite + state 打包成 zip，方便迁移 / 重装

## 九、关键边界判断（一段话总结）

**该持久化**：身份 / 关系 / 偏好 / 事实 / 工作记忆（summaries / patterns）四层。
**不该持久化**：临时 UI 状态、单次工具输出、失败重试中间态、Secret 内容进向量。
**该让用户感知**：分 4 层——L1 状态条 / 光环、L2 故事气泡、L3 记忆簿、L4 审计面板。
**叙事性记忆是桌宠人格的核心**，必须用 `conversation_summaries` + `episodic chunks` + `interaction_patterns` 三件套支撑；事实性记忆用 `memory_entries` KV 即可。
**断点只有一处**——`memory_entries / conversation_summaries` 到 `memory_chunks / embeddings` 的写入链路没接通，修通后整个体系就活了。
