# 桌宠工具能力、记忆与工作区方案

> 日期：2026-05-26
> 目标：在不打破 Conductor 原有“调度人的时间、压缩其他 agent 产出”定位的前提下，设计桌宠自身调用工具、沉淀长期记忆、使用工作区边界的后续推进方案。

---

## 1. 背景

当前项目已经形成了几个关键基础：

- `chat.rs`：桌宠聊天终端已经能基于任务列表和 LLM 回答问题，但当前系统提示仍要求“不能声称已经执行系统操作”。
- `tasks.rs` / `db.rs`：任务、聊天历史、记忆、会话摘要已经落到 SQLite。
- `proposals.rs`：已经有“提案、批准、拒绝、使用”的数据模型。
- `subagent.rs`：已经有 `claude -p` 子进程执行通道。
- `conductor-sense`：已经有前台窗口、进程名、窗口标题等桌面感知能力。
- `worker.rs`：Tauri 桌面端已经常驻，并把感知事件写入统一事件流。

这些基础说明：项目已经不只是一个静态任务面板，而是具备“观察用户工作状态、与用户对话、维护任务状态、在必要时触发后续动作”的雏形。

新的需求是：

1. 桌宠除了调度 Claude Code / Codex，也能自己调用系统工具，例如 `office-cli`，帮助处理文档、PPT、表格等工作。
2. 桌宠不应只有单 session 上下文，而要拥有可控的长期记忆。
3. 桌宠需要类似 Codex / Claude Code 的工作区边界，用来确定读取、写入和提权范围，但不能让产品体验显得像“每个目录都装一个桌宠”。
4. 桌宠需要根据工作场景切换低风险形象，例如文档秘书、程序员、原设计三套静态形象，并允许外部 agent 通过受限 MCP 工具触发切换。

---

## 2. 核心判断

### 2.1 这个方向是否优秀

方向是优秀的。

桌宠型个人 agent 如果只能聊天和展示任务，很快会停留在“会说但不会做”。在用户明确同意后，它应当能调用有限工具完成低风险、可验证的动作，例如：

- 读取当前文档结构。
- 生成文档大纲或修订建议。
- 调用 `office-cli` 提取 Word / PPT / Excel 信息。
- 生成待插入内容。
- 创建草稿文件。
- 向 Claude / Codex 发起一个受限子任务。
- 把用户已批准的修改写回文件。

但这类能力必须以“工具提案 + 用户确认 + 执行日志”为边界。否则 Conductor 会从“人的调度器”变成“不透明地操作电脑的 actor”，错误影响会被放大。

### 2.2 是否会过重

会有变重风险，但可以控制。

不要把所有能力直接塞进聊天模型里，也不要让 LLM 自由决定调用任何命令。应拆成四层：

```text
观察层：知道用户正在做什么
规划层：判断是否值得提议帮忙
提案层：生成可审阅的 action proposal
执行层：在审批和权限策略下调用工具
```

只要执行层是显式、可审计、可回滚或可预览的，系统复杂度是可控的。

### 2.3 MCP 和 Skills 是否应该纳入

应该纳入，但角色不同：

- **Skills**：描述“什么时候用、怎么用、有什么限制、如何和用户交互”。它是行为策略和工作流知识。
- **MCP**：提供标准化外部工具接口。它是工具接入方式之一。
- **CLI / 内置 Rust 函数 / 子 agent**：仍然可以作为工具 provider，不必全部强行 MCP 化。

推荐顺序：

1. 先建设本地 `ToolRegistry` 和 `SkillRegistry` 抽象。
2. 先接 1-2 个确定工具，例如 `office-cli` dry-run、`claude -p`。
3. 再把 MCP 当成一种 `ToolProvider` 接入。
4. 最后让 Skills 按场景编排多个工具。

### 2.4 记忆选型决策

个人 agent 的持久化记忆不应在 SQLite 和向量库之间二选一。推荐决策是：

```text
SQLite = 事实源、状态源、权限源、记忆源数据
本地轻向量索引 = 语义召回辅助层
关键词 / FTS = 精确词、文件名、项目名兜底
```

MVP 选型：

```text
SQLite + 本地轻量向量记忆
```

具体含义：

- 继续使用现有 `state/conductor.sqlite` 作为唯一事实源。
- 新增 `memory_chunks` 表保存可检索记忆片段。
- 新增 `memory_embeddings` 表保存 embedding 向量。
- MVP 不引入 Qdrant / Chroma / Weaviate 这类独立向量数据库服务。
- MVP 先用 Rust 进程内精确 cosine 检索，数据量小于几千条时足够。
- 后续再评估 SQLite `Vec1` 扩展的 flat / ANN 模式，或 LanceDB 作为独立本地嵌入式备选。

选择理由：

- 当前项目已经有 SQLite、迁移逻辑和任务/聊天/记忆表，继续复用成本最低。
- 桌面端 `.exe` 不应额外管理一个向量数据库服务进程。
- MVP 的长期记忆规模不会大，精确检索比 ANN 更简单、更可控。
- 工作区、敏感级别、过期时间、权限策略都更适合先由 SQLite 管。
- 向量检索只是帮助“找相似记忆”，不能替代“该记什么、该忘什么”的策略。

参考方向：

- SQLite `Vec1` 是 SQLite 官方向量扩展方向，支持 NN / ANN 查询；官方用户手册也明确少量向量或缺少训练数据时可用 NN / exhaustive 模式。
- LanceDB 是本地嵌入式向量数据库备选，支持 vector search、full-text search 和 hybrid search，也有 Rust / TypeScript / Python SDK；但对当前 MVP 来说会引入第二套存储模型。

### 2.5 桌宠形象切换决策

MVP 支持三套静态形象：

```text
original：原设计 / 日常形象
document_secretary：文档秘书形象
programmer：程序员形象
```

三套形象建议理解为“同一桌宠的三套装扮”，不要设计成三个人格。这样能保留连续性，也方便后续从静态图升级到 Live2D。

MVP 只做每个形象一张透明 PNG / WebP，是可行的。表达力不足的问题由以下方式补：

- CSS 呼吸 / 轻微漂浮动画。
- 状态 badge：`idle` / `working` / `new_task` / `quiet`。
- 气泡文案。
- 主题色或小道具变化。

不要在 MVP 为每个形象做多套表情、动作或完整 Live2D rigging。先验证“进程检测 -> 场景判断 -> 桌宠回应 -> 形象切换”闭环。

## 3. 产品边界调整

原 PRD 里强调 Conductor “不写代码、不写文档”，这个边界在新方向下需要细化，而不是直接推翻。

新的边界建议：

### 3.1 Conductor 仍然不替人决策

它不能自行判断“这段文档最终就这样写”。它最多可以：

- 发现用户正在写文档。
- 询问是否需要帮助。
- 生成建议、草稿、修改预览。
- 在用户批准后调用工具写入。

### 3.2 Conductor 可以做受限执行

允许的执行类型：

- 只读分析：读取当前工作区、文档元数据、任务列表、最近事件。
- 草稿生成：生成新文件、临时文件、提案内容。
- 预览修改：调用工具产生 diff / patch / dry-run 输出。
- 批准后写入：用户明确批准后对目标文件写入。
- 子 agent 派工：用户批准后调用 `claude -p` / Codex / MCP agent 处理明确任务。

默认禁止：

- 未确认地覆盖用户文件。
- 未确认地删除文件。
- 未确认地向外部服务发送敏感内容。
- 未确认地发送邮件、IM、飞书消息。
- GUI 输入注入。
- 跨工作区读写。

### 3.3 桌宠主动性只到“询问”

例如：

> 我看到你正在写“项目方案.docx”，要不要我帮你先整理一个目录或检查结构？

这类主动询问可以自动触发。

但下面这类动作不能自动触发：

> 我已经改好了你的文档。

自动触发的上限是提出帮助，执行必须等用户回复。

### 3.4 用户审批压力最小化

所有待审任务可以进入桌宠队列，但用户不应该被要求理解工具、权限、参数、路径和执行链路。

统一产品定调：

```text
用户只做一件事：对当前推荐任务三选一。

[可以推进] [不推进] [有什么不同]
```

三选一的含义：

- **可以推进**：只批准当前这一小步。系统继续推进下一步前，如果风险等级变化或需要写入/外发/删除，必须再次给出三选一。
- **不推进**：拒绝或暂缓当前任务。默认不追问原因，避免把解释负担交给用户。
- **有什么不同**：解释推进与不推进的差异，包括收益、影响范围、风险、是否可回退、是否需要读取/写入/外发。

设计要求：

- 每次只推荐一条最值得处理的待审任务。
- 多个待审任务只显示摘要数量，不要求用户批量审批。
- 默认文案面向结果和影响，不面向工具实现。
- 工具名、JSON 参数、MCP server、workspace path、权限规则只在“有什么不同”里展示。
- 低风险只读任务也走同一套三选一，避免 UI 心智分裂。
- 高风险任务不得因为用户点过一次“可以推进”而获得后续永久授权。

示例：

```text
当前建议推进：整理当前方案文档结构

为什么：你正在编辑方案文档，这一步只读取标题层级和摘要。
影响：不会修改文件。

[可以推进] [不推进] [有什么不同]
```

## 4. 总体架构

推荐目标架构：

```text
                  ┌────────────────────┐
                  │   Desktop / Chat    │
                  └─────────┬──────────┘
                            │
                            ▼
┌──────────────┐    ┌────────────────────┐
│ Sense Events │───▶│  Context Builder   │
└──────────────┘    └─────────┬──────────┘
                              │
                              ▼
                    ┌────────────────────┐
                    │ Planner / Skill     │
                    │ Router              │
                    └─────────┬──────────┘
                              │
                  ┌───────────┴───────────┐
                  ▼                       ▼
        ┌──────────────────┐    ┌──────────────────┐
        │ Natural Reply     │    │ Action Proposal  │
        └──────────────────┘    └─────────┬────────┘
                                          │ approval
                                          ▼
                                ┌──────────────────┐
                                │ Tool Executor    │
                                └─────────┬────────┘
                                          │
                   ┌──────────────────────┼──────────────────────┐
                   ▼                      ▼                      ▼
            internal tool              CLI tool                 MCP tool
            tasks/memory              office-cli               filesystem
            proposals                 claude -p                browser
```

关键原则：

- Chat 不直接执行工具。
- Planner 可以生成提案。
- Proposal 才是用户可见的执行前对象。
- Executor 只执行已批准、权限允许的动作。
- 所有执行写入 `events.ndjson` 和 execution log。

## 5. Tool Registry 方案

### 5.1 目标

统一管理所有可调用能力：

- 内置能力：任务、记忆、配置、事件查询。
- 子 agent：`claude -p`、Codex。
- CLI：`office-cli`、`git`、自定义脚本。
- MCP：文件、浏览器、Office、飞书等外部工具。

### 5.2 Tool 定义

建议定义一个通用工具描述：

```rust
pub struct ToolSpec {
    pub id: String,
    pub name: String,
    pub description: String,
    pub provider: ToolProviderKind,
    pub input_schema: serde_json::Value,
    pub output_schema: serde_json::Value,
    pub risk_level: RiskLevel,
    pub permissions: Vec<ToolPermission>,
    pub supports_dry_run: bool,
    pub workspace_required: bool,
}
```

风险等级：

```rust
pub enum RiskLevel {
    ReadOnly,
    DraftOnly,
    WorkspaceWrite,
    ExternalSideEffect,
    Destructive,
}
```

权限类型：

```rust
pub enum ToolPermission {
    ReadWorkspace,
    WriteWorkspace,
    ReadExternalPath,
    WriteExternalPath,
    Network,
    SendMessage,
    SystemControl,
}
```

### 5.3 Tool Provider

不要让 MCP 垄断所有实现。建议 provider 分为：

```text
internal：Rust 函数，例如 tasks、memory、events
cli：本地命令，例如 office-cli
subagent：claude -p / codex
mcp：MCP server 暴露的工具
```

这样后续可以平滑迁移：某个 CLI 工具成熟后，也可以包成 MCP；但系统主逻辑不依赖具体接入方式。

### 5.4 office-cli 示例

文档协助场景可以先只接 dry-run：

```text
tool id: office.inspect_document
provider: cli
command: office-cli inspect --file <path> --json
risk: ReadOnly
workspace_required: true
```

```text
tool id: office.apply_patch
provider: cli
command: office-cli patch --file <path> --patch <patch> --dry-run/--apply
risk: WorkspaceWrite
supports_dry_run: true
workspace_required: true
```

第一阶段只允许：

- inspect
- export text
- generate patch dry-run

第二阶段才允许 apply。

### 5.5 桌宠形象切换工具

形象切换是低风险工具，适合最早进入 ToolRegistry 和 MCP wrapper。

内部工具：

```text
tool id: pet.set_avatar
provider: internal
risk: ReadOnly
workspace_required: false
```

输入：

```json
{
  "avatar_id": "programmer",
  "mode": "temporary",
  "ttl_minutes": 60,
  "reason": "用户进入 coding 模式"
}
```

字段：

- `avatar_id`：只能是 `original | document_secretary | programmer`。
- `mode`：`temporary | persistent`。
- `ttl_minutes`：临时切换持续时间；到期后回到 `original` 或用户设置的默认形象。
- `reason`：写入事件日志，方便后续回溯为什么切换。

禁止：

- 不接受任意 `image_path`。
- 不接受远程图片 URL。
- 不允许 MCP 端直接写 `state/config.json`。
- 不允许外部 agent 上传或覆盖形象资源。

权限策略：

| 调用来源 | 策略 |
|---|---|
| 用户明确说“切程序员形象” | 直接允许 |
| 内部 skill 根据 coding 场景触发 | 允许临时切换 |
| 外部 Claude / Codex MCP 调用 | 允许，但只限枚举 `avatar_id` |
| 任意图片路径 / URL | 禁止 |

实现路径：

```text
Tauri command: set_pet_avatar
  -> conductor-core avatar state
  -> app.emit("pet_avatar_changed")
  -> 前端切换图片资源

ToolRegistry: pet.set_avatar
  -> 调用同一内部函数

MCP: conductor.pet.set_avatar
  -> 只做参数校验和内部工具转发
```

这样桌宠自己、聊天终端、MCP 外部 agent 都走同一条安全路径。

## 6. Action Proposal 方案

现有 `proposals.rs` 已经能承载基础提案，但建议扩展为更通用的 action proposal。

### 6.1 提案内容

建议字段：

```json
{
  "id": "p-20260526-001",
  "workspace_id": "ws-xxx",
  "for_cwd": "I:/work/project-a",
  "source": "chat | proactive | task | hook",
  "title": "帮你整理当前 Word 文档目录",
  "reason": "检测到你正在编辑方案文档，且停留超过 8 分钟",
  "tool_id": "office.inspect_document",
  "tool_input": {
    "file": "I:/work/project-a/方案.docx"
  },
  "risk_level": "read_only",
  "dry_run": true,
  "status": "pending",
  "created_at": "2026-05-26T10:00:00+08:00"
}
```

### 6.2 提案生命周期

```text
pending
  -> approved
  -> running
  -> succeeded
  -> used

pending
  -> rejected

pending
  -> expired

running
  -> failed
```

现有状态 `pending / approved / rejected / expired / used` 可保留，但如果要进入执行层，需要新增 `running / succeeded / failed`。

### 6.3 用户体验

桌宠不应该弹出复杂权限表。所有待审任务都可以给到用户，但用户每次只做一件事：三选一。

统一选择：

```text
[可以推进] [不推进] [有什么不同]
```

含义：

| 选项 | 系统动作 | 用户负担 |
|---|---|---|
| 可以推进 | 批准当前待审任务进入下一步；只对当前任务有效 | 用户不需要理解工具参数 |
| 不推进 | 拒绝或暂缓当前待审任务，并记录原因/场景 | 用户不需要解释，除非主动补充 |
| 有什么不同 | 展开“做/不做”的差异说明、影响范围、风险和可回退性 | 用户只在需要时看细节 |

定调：

- 默认不让用户在“允许一次 / 永久允许 / 默认允许 / 拒绝 / 查看权限”之间做复杂选择。
- 默认不暴露 `tool_id`、JSON 参数、权限规则、MCP server、workspace path 等实现细节。
- 用户看到的是“这件事要不要推进”，不是“这个工具能不能获得某个权限”。
- 只有用户点“有什么不同”时，才展开技术细节、影响范围、预览和替代方案。
- 写入、外发、删除等高风险动作仍必须确认，但确认话术仍保持三选一；细节放在“有什么不同”里。

只读/低风险动作：

```text
我可以帮你整理当前文档结构：

动作：读取“方案.docx”的标题层级和摘要
范围：当前项目 I:/work/project-a
影响：只读，不会修改文件

[可以推进] [不推进] [有什么不同]
```

写操作：

```text
我可以把这段摘要写入文档末尾：

动作：修改“方案.docx”
范围：当前项目 I:/work/project-a
影响：会写入文件，已生成预览

[可以推进] [不推进] [有什么不同]
```

其中“有什么不同”展开后再给预览、diff、风险和回退方式：

```text
推进：把摘要写入文档末尾，并保留修改记录。
不推进：不改文件，只保留这次建议。
风险：会修改当前文档；执行前会保存 dry-run 结果。
回退：可根据预览 diff 手动撤回，后续再做自动备份。
```

## 7. Skills 方案

### 7.1 Skill 的职责

Skill 不等于工具。Skill 是场景化工作流。

例如“文档协助 Skill”负责：

- 判断用户是否正在写文档。
- 何时询问。
- 询问用什么话术。
- 收到用户同意后调用哪些工具。
- 工具失败时怎么降级。
- 生成结果后如何让用户确认。

工具只负责执行一个具体动作。

### 7.2 Skill 定义

建议结构：

```json
{
  "id": "document_assistant",
  "name": "文档协助",
  "description": "在用户编辑文档时，提供结构整理、摘要、润色和检查帮助。",
  "enabled": true,
  "triggers": [
    {
      "source": "desktop.focus_changed",
      "process": ["WINWORD.EXE", "wps.exe", "Code.exe"],
      "title_patterns": ["*.docx", "*.md", "方案", "PRD"]
    }
  ],
  "cooldown_minutes": 20,
  "allowed_tools": [
    "office.inspect_document",
    "office.export_text",
    "office.patch_dry_run"
  ],
  "approval_policy": "ask_before_execute",
  "prompt": "当用户正在写文档时，优先询问是否需要帮忙整理结构、提炼摘要或检查表达。不要直接修改文件。"
}
```

### 7.3 初期 Skills

建议先做 3 个：

1. `task_triage`
   - 使用现有任务列表。
   - 回答“现在该看什么”。
   - 不调用外部工具。

2. `document_assistant`
   - 触发 Word/WPS/Markdown/飞书文档等场景。
   - 初期只做只读分析和草稿。

3. `coding_reviewer`
   - 触发 VS Code/Cursor/终端场景。
   - 初期只生成测试建议或派工给 Codex/Claude。

不要一开始做太多泛化 skill，否则会变成 prompt 配置堆积。

## 8. MCP 纳入方案

### 8.1 MCP 的定位

MCP 是工具提供层，不是桌宠的主架构。

它适合：

- 标准文件系统工具。
- 浏览器工具。
- Office 工具。
- 飞书 / Lark 工具。
- 数据库、知识库、搜索等外部能力。

不适合一开始承载：

- 所有本地状态读写。
- 任务列表核心逻辑。
- 权限策略。
- 工作区解析。

这些应该放在 Conductor core。

### 8.2 MCP 接入边界

MCP tool 必须被 Conductor 包装后才能暴露给 Planner：

```text
MCP raw tool
  -> MCP Adapter
  -> ToolSpec
  -> ApprovalPolicy
  -> Executor
```

不要让 LLM 直接看到所有 MCP 原始工具。否则工具面会过宽，安全边界不可控。

### 8.3 MCP 配置

建议在 `state/config.json` 或单独 `state/tools.json` 中配置：

```json
{
  "mcp": {
    "enabled": true,
    "servers": [
      {
        "id": "office",
        "name": "Office MCP",
        "command": "office-mcp-server",
        "enabled": true,
        "allowed_tools": [
          "inspect_document",
          "export_text",
          "patch_document"
        ]
      }
    ]
  }
}
```

再由 Conductor 映射成内部工具：

```text
office.inspect_document
office.export_text
office.patch_document
```

### 8.4 桌宠形象 MCP 工具

建议开放一个低风险 MCP 工具：

```text
conductor.pet.set_avatar
```

输入 schema：

```json
{
  "type": "object",
  "properties": {
    "avatar_id": {
      "type": "string",
      "enum": ["original", "document_secretary", "programmer"]
    },
    "mode": {
      "type": "string",
      "enum": ["temporary", "persistent"],
      "default": "temporary"
    },
    "ttl_minutes": {
      "type": "integer",
      "minimum": 1,
      "maximum": 240
    },
    "reason": {
      "type": "string",
      "maxLength": 200
    }
  },
  "required": ["avatar_id"]
}
```

输出：

```json
{
  "ok": true,
  "avatar_id": "programmer",
  "expires_at": "2026-05-26T11:00:00+08:00"
}
```

这个工具不需要工作区，也不应需要弹审批。但它必须：

- 只允许枚举形象。
- 写入 `events.ndjson`。
- 触发 Tauri 前端事件。
- 尊重 quiet mode，不额外弹系统通知。

典型用途：

```text
用户：进入 coding 模式
LLM / skill：调用 conductor.pet.set_avatar({ avatar_id: "programmer", mode: "temporary", ttl_minutes: 60 })
桌宠：切到程序员形象，并用气泡回应“我切到 coding 模式了”
```

## 9. 记忆方案

### 9.1 目标

桌宠不应该只记得当前聊天 session。它应长期记得：

- 用户偏好。
- 常用工作区。
- 当前项目目标。
- 最近推进到哪里。
- 哪些任务被用户拒绝/跳过。
- 哪些提醒方式让用户觉得烦。
- 某类文档/项目的约定。

但也不能把所有信息永久塞进上下文。核心是“分层存储、按需检索、可遗忘”。

### 9.2 记忆分层

推荐四层：

```text
L0 当前上下文
  - 当前聊天消息
  - 当前任务
  - 当前前台应用
  - 当前工作区
  - 不长期保存或短期保存

L1 事件日志
  - hook 事件
  - focus_changed
  - idle_started / idle_ended
  - tool execution
  - 用于回溯和摘要
  - 可配置 TTL

L2 工作区记忆
  - 项目目标
  - 文档约定
  - 当前进度
  - 常用命令
  - 风险点
  - 只在对应 workspace 激活时检索

L3 用户画像
  - 提醒偏好
  - 回答风格
  - 工作习惯
  - 常见时间安排
  - 全局可用，但要小而稳定
```

### 9.3 不该长期记的内容

默认不要长期保存：

- 大段原始聊天。
- 屏幕截图。
- 私聊内容。
- 密码、token、密钥。
- 完整文档正文。
- 未经用户确认的推断。
- 一次性临时任务细节。

如果需要保存，必须降级为摘要或结构化 metadata。

### 9.4 MemoryEntry 扩展

当前 `memory_entries` 是 key/value/category。建议后续扩展：

```sql
ALTER TABLE memory_entries ADD COLUMN scope TEXT;
ALTER TABLE memory_entries ADD COLUMN workspace_id TEXT;
ALTER TABLE memory_entries ADD COLUMN source TEXT;
ALTER TABLE memory_entries ADD COLUMN confidence REAL DEFAULT 1.0;
ALTER TABLE memory_entries ADD COLUMN sensitivity TEXT DEFAULT 'normal';
ALTER TABLE memory_entries ADD COLUMN expires_at TEXT;
ALTER TABLE memory_entries ADD COLUMN last_used_at TEXT;
```

字段含义：

- `scope`：`global | workspace | document | session`
- `workspace_id`：对应工作区。
- `source`：`user_confirmed | inferred | tool | summary`
- `confidence`：推断可信度。
- `sensitivity`：`normal | private | secret`
- `expires_at`：临时记忆自动过期。
- `last_used_at`：用于清理长期不用的记忆。

### 9.5 记忆写入策略

只允许以下信息进入长期记忆：

1. 用户明确说“记住”。
2. 用户设置中保存的偏好。
3. 多次重复出现且低敏感的信息。
4. 完成任务后的稳定摘要。
5. 工作区级别的项目状态。

LLM 推断的记忆必须标记为 `source=inferred`，并降低 confidence。必要时应先问：

> 我注意到你经常希望文档方案先给目录再写正文，要我记住这个偏好吗？

### 9.6 记忆检索策略

每次聊天构造上下文时，不应加载全部记忆。建议只加载：

- 当前 workspace summary。
- 当前活跃任务 top 3。
- 最近事件摘要。
- 用户全局偏好 3-5 条。
- 与当前问题关键词匹配的记忆少量。

MVP 明确采用 SQLite + 本地轻量向量检索：

```text
1. 先用 SQLite 过滤 scope / workspace_id / sensitivity / expires_at。
2. 对过滤后的 memory_chunks 做本地 cosine 相似度检索。
3. 同时做关键词匹配，补足技术名词、文件名、项目名。
4. 合并向量分数、关键词分数、时间新鲜度、用户确认权重。
5. 只把 top 3-8 条短记忆放入 LLM 上下文。
```

这不是“把向量库直接交给 LLM”。LLM 只看到检索后的少量记忆。检索层负责控制边界。

### 9.7 记忆压缩

建议引入定时“记忆整理”任务：

```text
每日：
  - 汇总当天事件为 daily summary
  - 从已完成任务提取 workspace progress
  - 清理过期事件

每周：
  - 合并重复偏好
  - 降权长期未使用记忆
  - 生成 workspace weekly summary
```

### 9.8 SQLite + 本地轻向量记忆

#### 9.8.1 选型

MVP 选：

```text
SQLite 事实源 + SQLite 表内保存 embedding + Rust 进程内精确 cosine 检索
```

暂不选：

- 独立 Qdrant / Chroma / Weaviate 服务。
- 云向量库。
- 一开始就把 LanceDB 作为主存储。

备选路线：

- SQLite `Vec1`：等本地打包、Windows 加载扩展、检索质量验证通过后，可替换进程内 cosine。
- LanceDB：如果后续记忆规模变大、需要更强 hybrid search，可作为本地嵌入式向量库备选。

#### 9.8.2 数据表

新增 `memory_chunks`：

```sql
CREATE TABLE memory_chunks (
  id TEXT PRIMARY KEY,
  memory_id TEXT,
  workspace_id TEXT,
  scope TEXT NOT NULL,
  category TEXT NOT NULL,
  content TEXT NOT NULL,
  summary TEXT,
  source TEXT NOT NULL,
  sensitivity TEXT NOT NULL DEFAULT 'normal',
  confidence REAL NOT NULL DEFAULT 1.0,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  expires_at TEXT,
  last_used_at TEXT
);
```

新增 `memory_embeddings`：

```sql
CREATE TABLE memory_embeddings (
  chunk_id TEXT PRIMARY KEY,
  model TEXT NOT NULL,
  dims INTEGER NOT NULL,
  vector BLOB NOT NULL,
  created_at TEXT NOT NULL
);
```

`vector` MVP 可用 little-endian `f32` 数组序列化为 BLOB。这样不依赖 SQLite 扩展，Rust 端读取后计算 cosine。

建议索引：

```sql
CREATE INDEX idx_memory_chunks_scope_workspace
ON memory_chunks(scope, workspace_id, category);

CREATE INDEX idx_memory_chunks_expires
ON memory_chunks(expires_at);

CREATE INDEX idx_memory_chunks_updated
ON memory_chunks(updated_at DESC);
```

如果后续启用 SQLite FTS5：

```sql
CREATE VIRTUAL TABLE memory_chunks_fts
USING fts5(content, summary, content='memory_chunks', content_rowid='rowid');
```

#### 9.8.3 写入哪些内容

MVP 只 embedding：

- `conversation_summaries`
- `workspace_summaries`
- 用户确认过的长期偏好
- 已完成任务摘要
- 工具执行后的短摘要

MVP 不 embedding：

- 原始聊天全文
- 截图
- 完整文档正文
- 私聊内容
- 密钥、token、敏感配置

#### 9.8.4 检索算法

伪流程：

```text
input: user_message, current_workspace_id

query_embedding = embed(user_message)

candidates = SELECT chunks
  WHERE sensitivity != 'secret'
    AND (expires_at IS NULL OR expires_at > now)
    AND (
      scope = 'global'
      OR workspace_id = current_workspace_id
      OR scope = 'session'
    )
  LIMIT 500

for each candidate:
  vector_score = cosine(query_embedding, candidate.embedding)
  keyword_score = keyword_match(user_message, candidate.content)
  recency_score = decay(candidate.updated_at)
  authority_score = source_weight(candidate.source, candidate.confidence)
  final = 0.55 * vector_score
        + 0.20 * keyword_score
        + 0.15 * recency_score
        + 0.10 * authority_score

return top 3-8
```

权重可以先写死，后续根据体验调整。

#### 9.8.5 Embedding 模型

MVP 可先走两种模式：

```text
云端 embedding：配置 OpenAI 或兼容接口，开发最快
本地 embedding：后续接 fastembed / ONNX / sentence-transformers 小模型
```

隐私策略：

- `sensitivity = secret` 永不 embedding。
- `sensitivity = private` 默认不发云端 embedding，除非用户显式允许。
- 发送 embedding 前只发送 chunk 摘要，不发送完整原文。

#### 9.8.6 为什么不直接上独立向量库

MVP 不选独立向量库服务的原因：

- 桌面 `.exe` 要额外管理服务生命周期。
- 需要处理端口、进程、崩溃恢复、数据目录、备份。
- 当前记忆规模小，ANN 优势不明显。
- SQLite 已经承载任务、聊天、记忆、权限，分裂存储会增加一致性问题。

向量库对 LLM 的帮助不是“更方便存东西”，而是“更容易找回语义相近的旧记忆”。这层能力可以先在 SQLite 上实现，不必引入服务型数据库。

## 10. 工作区方案

### 10.1 是否需要工作区

需要。

工作区不是为了模仿 Codex / Claude Code，而是为了：

- 限定工具读写范围。
- 给长期记忆加 scope。
- 让任务、事件、文档、子 agent 产出归属清楚。
- 避免桌宠跨项目混淆。
- 给用户一个可理解的权限边界。

但产品表达不应叫“沙箱”或“cwd 边界”，而应叫：

- 当前项目
- 当前文档
- 当前工作上下文

### 10.2 一个 exe，多工作区

最终应是：

```text
一个全局 desktop exe
一个全局 conductor state
多个 workspace registry 记录
每次对话/任务/工具调用绑定一个 workspace_id
```

不要在每个工作区放一份桌宠副本。

### 10.3 WorkspaceRegistry

建议新增表：

```sql
CREATE TABLE workspaces (
  id TEXT PRIMARY KEY,
  root TEXT NOT NULL,
  name TEXT NOT NULL,
  kind TEXT NOT NULL,
  trust_level TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  last_active_at TEXT,
  metadata_json TEXT
);
```

`kind`：

```text
code
document
office
notes
generic
```

`trust_level`：

```text
trusted
ask_write
read_only
untrusted
```

### 10.4 当前工作区推断

推断优先级：

1. Hook payload 的 `cwd`
   - Claude / Codex 最可靠。
2. 用户手动选择或 pin 的 workspace。
3. 前台应用窗口标题和进程名。
   - VS Code / Cursor：通常能推断项目。
   - Word/WPS：从文件路径或标题推断。
   - Terminal：从 hook 或 shell integration 推断。
4. 最近活跃 workspace。
5. 未知 workspace。

未知时，只允许聊天和只读低风险动作；写入必须要求用户选择工作区。

### 10.5 工作区与文档

文档类工作区不能简单等同于 git repo。对于 Word / PPT / Excel：

- 如果文件在已有 workspace root 下，归属该 workspace。
- 如果文件不在任何 workspace 下，以文件所在目录临时生成 document workspace。
- 如果文件来自云文档或临时目录，需要创建 virtual workspace，只保留文档 ID 和来源，不假装有本地 root。

### 10.6 权限策略

推荐策略矩阵：

| 动作 | trusted workspace | ask_write workspace | unknown workspace |
|---|---|---|---|
| 读取任务/记忆 | 默认允许 | 默认允许 | 默认允许 |
| 读取当前文件 metadata | 默认允许 | 默认允许 | 询问 |
| 读取文件正文 | 可配置允许 | 询问 | 询问 |
| 写入新草稿 | 询问一次 | 询问 | 询问 |
| 修改已有文件 | 预览 + 确认 | 预览 + 确认 | 禁止直到选择 workspace |
| 删除文件 | 永远确认 | 永远确认 | 禁止 |
| 外部发送 | 永远确认 | 永远确认 | 永远确认 |

### 10.7 工作区 UI 表达

桌宠聊天窗口应显示当前上下文：

```text
当前上下文：方案文档 · I:/work/project-a
```

用户可切换：

```text
这次对话绑定到哪个项目？
[I:/work/project-a] [I:/personal-agent] [仅本次临时上下文]
```

不要要求用户理解 cwd，但需要让用户知道“桌宠正在基于哪个项目判断和执行”。

## 11. 主动帮助场景设计

### 11.1 文档写作场景

触发信号：

- 前台应用为 Word / WPS / VS Code / Cursor / 飞书文档窗口。
- 标题包含 `.docx`、`.md`、`PRD`、`方案`、`文档`、`PPT`。
- 用户停留超过 N 分钟。
- 当前没有 quiet mode。
- 同类提示 cooldown 已过。

桌宠提示：

> 我看到你在写“XXX 方案”，要不要我帮你先整理结构或检查上下文？

用户回复：

- “整理结构”
- “帮我润色”
- “先看看这个文档”
- “不用”

后续：

```text
用户同意
  -> 创建 ActionProposal
  -> 如果只读，询问一次或直接执行（取决于配置）
  -> office-cli inspect/export
  -> 生成摘要/建议
  -> 若要写入，必须生成预览并确认
```

### 11.2 编码场景

触发信号：

- 前台应用为 VS Code / Cursor / Terminal。
- 当前 workspace 是 code。
- Codex / Claude 有 in_progress 或 pending 任务。

桌宠提示：

> 这个项目现在有 2 个待审任务。要不要我先帮你把测试点列出来？

后续：

- 读取 task list。
- 调用 Codex 任务摘要。
- 生成测试步骤。
- 如需子 agent，走 proposal。
- 临时切换到 `programmer` 形象，默认 60 分钟或直到退出 coding 模式。

### 11.3 多 agent 积压场景

触发信号：

- pending 任务数量超过阈值。
- 用户刚从 idle 回来。
- 当前不在 quiet mode。

桌宠提示：

> 你离开的这段时间里有 3 个任务完成了。我建议先看 10 分钟内能审完的这个。

这部分和现有 pacer / notify_decider 能直接衔接。

### 11.4 桌宠形象场景切换

MVP 只做三套静态图：

| 形象 | avatar_id | 触发场景 | 视觉定位 |
|---|---|---|---|
| 原设计 | `original` | 默认、休息、无明确垂直场景 | 当前桌宠原始形象 |
| 文档秘书 | `document_secretary` | Word/WPS/Markdown/PPT/飞书文档 | 手持文件夹、笔记本、轻办公元素 |
| 程序员 | `programmer` | VS Code/Cursor/Terminal/Codex 任务 | 轻量程序员装扮、键盘或代码小道具 |

资源建议：

```text
apps/desktop/public/avatar/
  original.webp
  document_secretary.webp
  programmer.webp
```

或如果要保留透明通道更稳定：

```text
apps/desktop/public/avatar/
  original.png
  document_secretary.png
  programmer.png
```

要求：

- 三张图使用同一角色基础设定。
- 同一视角、同一身体比例、同一画风。
- 透明背景。
- 桌宠主轮廓不要差异过大，避免切换时像换了一个人。
- 不在图里放大量文字，状态由 UI badge 和气泡表达。

切换策略：

```text
用户显式要求 > 当前活跃 skill > 当前前台进程 > 默认形象
```

示例：

- 用户说“切程序员模式” -> 立即切 `programmer`，mode 可为 persistent。
- 检测到 VS Code 持续前台 2 分钟 -> 临时切 `programmer`。
- 检测到 Word / WPS / Markdown -> 临时切 `document_secretary`。
- idle / quiet / 没有场景 -> 回到 `original`。

防抖：

- 同一形象切换 cooldown：至少 5 分钟。
- 不同形象自动切换 cooldown：至少 10 分钟。
- 用户手动切换优先级最高，不受自动 cooldown 限制。

状态表达：

```text
avatar_id = programmer
pet_state = working
```

两者分开。`avatar_id` 表示“装扮/领域”，`pet_state` 表示“当前状态”。不要把三张图扩展成三套完整状态图，MVP 只做组合表达。

- diff / patch。
- 实际写入目标。

## 13. 安全与隐私

### 13.1 全局原则

- 默认最小权限。
- 工具必须声明风险等级。
- 写操作必须能解释影响。
- 敏感内容默认不上传云端 LLM。
- 截图、多模态理解必须独立开关。
- 用户随时可以关闭“观察我”。

### 13.2 敏感信息处理

`conductor-sense` 里已有窗口标题脱敏逻辑。后续要扩展：

- token / key 模式识别。
- 邮箱、手机号、身份证等脱敏。
- 私聊窗口白名单外不采集正文。
- 截图默认不落盘。
- 云端 LLM 调用前做敏感检查。

### 13.3 权限记忆

允许用户选择：

```text
允许一次
本工作区内同类只读默认允许
永远询问
禁用这个工具
```

不要给“永远允许写入”这种过大授权。

## 16. 风险清单

| 风险 | 表现 | 对策 |
|---|---|---|
| 工具面过宽 | LLM 想调用任何命令 | ToolRegistry 白名单 + approval |
| 主动性打扰 | 桌宠频繁询问 | cooldown + quiet mode + 用户反馈记忆 |
| 记忆污染 | 错误推断长期保存 | inferred 降权 + 用户确认 |
| 工作区混淆 | A 项目记忆影响 B 项目 | workspace_id 强绑定 |
| 隐私泄露 | 窗口/文档内容上传 LLM | 敏感检测 + 本地摘要 + 明确开关 |
| 写入破坏文件 | office-cli 修改错误 | dry-run + diff + 备份 + 用户确认 |
| MCP 权限绕过 | MCP server 暴露危险工具 | MCP adapter 包装 + allowed_tools |
| 产品变重 | 一次性做全套 agent OS | 分阶段，先只读和提案 |
| 向量记忆污染 | 错误摘要被语义召回 | 只 embedding 摘要和确认记忆，保留 confidence 和来源 |
| 向量检索泄露隐私 | 私密内容被送去云 embedding | sensitivity 过滤，private 默认不用云端 embedding |
| 形象切换突兀 | 三张图像三个人 | 同一角色设定、同画风、同视角 |
| MCP 形象工具被滥用 | 外部传任意图片路径 | 只允许枚举 avatar_id，不接受路径/URL |

## 19. Claude Code 源码能力迁移评估

本节基于本地包 `I:\package` 的源码包调研，重点不是复刻 Claude Code，而是提取适合 Conductor 桌宠交互的能力模型。

调研对象：

- `I:\package\package.json`
  - 包名：`@anthropic-ai/claude-code`
  - 版本：`2.1.88`
  - CLI 入口：`claude -> cli.js`
- `I:\package\sdk-tools.d.ts`
  - 最清晰的公开工具输入/输出 schema 来源。
- `I:\package\cli.js.map`
  - 可还原主要源码模块结构，包括 Tool、权限、Agent、Skill、MCP、Team、Worktree 等实现。

### 19.1 Claude Code 的核心抽象

Claude Code 不是“聊天模型直接执行命令”的结构，而是一个分层执行系统：

```text
模型输出 tool_use
  -> ToolRegistry 查找工具
  -> schema 校验
  -> 工具级 validateInput
  -> hook / permission / policy 决策
  -> tool.call 执行
  -> progress / result / transcript / log
  -> 结果再回到模型上下文
```

这一点对 Conductor 很关键。我们不应该把执行能力塞进 `chat.rs`，而应该把 `chat.rs` 定位为“自然语言入口 + proposal 生成入口”，真正执行在独立 Tool Executor 中完成。

Claude Code 的 `Tool` 抽象包含这些关键字段/能力：

| 能力 | 作用 | Conductor 可迁移方式 |
|---|---|---|
| `name` / `aliases` | 工具唯一名与兼容名 | `ToolSpec.id` + `aliases` |
| `searchHint` | 给 ToolSearch 的短能力描述 | 作为工具检索和 UI 摘要字段 |
| `inputSchema` / `outputSchema` | 参数和结果结构化 | MVP 用 JSON Schema，Rust 侧用 serde 校验 |
| `description` / `prompt` | 给模型看的工具说明 | 分成 `model_description` 和 `user_description` |
| `isEnabled` | 环境/配置开关 | 对应工具 feature flag 和用户开关 |
| `isReadOnly` | 只读分类 | 直接映射到风险等级 |
| `isDestructive` | 删除、覆盖、外发等不可逆动作 | 必须强制 proposal + 二次确认 |
| `shouldDefer` | 渐进式暴露工具 | Conductor 应引入 `tool.search` |
| `alwaysLoad` | 始终暴露的少数工具 | 仅保留 `tool.search`、`ask_user`、低风险内部工具 |
| `validateInput` | 工具自己的参数语义校验 | Rust ToolExecutor 执行前校验 |
| `checkPermissions` | 权限决策 | 对接现有 `proposals.rs` 扩展 |
| `call` | 真正执行 | internal / cli / subagent / mcp provider 统一走这里 |
| `renderToolUseMessage` | 工具执行中 UI 表达 | Tauri 前端显示提案、运行、结果 |
| `mapToolResult...` | 结果回传模型 | Conductor 可用 `ToolRunResult` 摘要化 |
| `maxResultSizeChars` | 大结果落盘，只给模型摘要 | 必须迁移，避免文档/命令输出塞爆上下文 |

建议 Conductor 的 Rust 侧抽象不要一开始做得和 Claude Code 一样复杂，但字段应保留扩展空间：

```rust
pub struct ToolSpec {
    pub id: String,
    pub aliases: Vec<String>,
    pub provider: ToolProviderKind,
    pub model_description: String,
    pub user_description: String,
    pub search_hint: String,
    pub input_schema: serde_json::Value,
    pub output_schema: Option<serde_json::Value>,
    pub risk_level: RiskLevel,
    pub workspace_required: bool,
    pub supports_dry_run: bool,
    pub should_defer: bool,
    pub always_load: bool,
    pub enabled: bool,
}
```

### 19.2 Claude Code 暴露了哪些工具

从 `sdk-tools.d.ts` 和源码 registry 看，Claude Code 的工具面大致分为九类。

第一类：文件与搜索。

- `FileRead`
- `FileEdit`
- `FileWrite`
- `Glob`
- `Grep`
- `NotebookEdit`

特点：

- 文件路径要求绝对路径。
- `FileEdit` 使用 `old_string -> new_string`，并可选 `replace_all`。
- `FileRead` 支持 offset/limit，PDF 支持页范围。
- 写入类工具执行前做路径、deny rule、文件大小、编码、文件是否被外部修改等校验。
- 写入结果会生成 diff/patch，便于用户和模型理解影响。

Conductor 迁移建议：

- 不要直接暴露通用 `FileWrite` 给桌宠聊天。
- MVP 只做领域包装工具，例如：
  - `office.inspect_document`
  - `office.export_text`
  - `office.patch_dry_run`
  - `workspace.read_text`
  - `workspace.create_draft`
- 通用文件写入能力保留在 Executor 内部，不直接给模型作为首批工具。

第二类：Shell/命令。

- `Bash`
- `PowerShell`

特点：

- 参数包含 `command`、`timeout`、`description`、`run_in_background`、`dangerouslyDisableSandbox`。
- 会区分 read/search/list 类命令和修改类命令。
- 支持后台任务，后台任务输出可通过任务输出文件或 TaskOutput 读取。
- 沙箱、权限、命令解析、危险模式是执行前决策的一部分。

Conductor 迁移建议：

- MVP 不暴露通用 shell 给桌宠。
- 只暴露白名单 CLI provider，例如 `office-cli`、`claude -p`、后续 Codex。
- 命令必须由 ToolSpec 模板生成，不接受模型拼接任意 shell 字符串。

第三类：Agent 与后台任务。

- `Agent`
- `TaskOutput`
- `TaskStop`

特点：

- `Agent` 参数包括：
  - `description`
  - `prompt`
  - `subagent_type`
  - `model`
  - `run_in_background`
  - `name`
  - `team_name`
  - `mode`
  - `isolation`
  - 源码中还存在 `cwd` 参数能力。
- Agent 可以同步跑，也可以后台跑。
- 后台 Agent 返回 `agentId`、`outputFile`、`canReadOutputFile` 等，之后通过输出文件或 TaskOutput 追踪。
- 子 Agent 有自己的工具池和权限模式，不简单继承父 Agent 的全部权限。
- 异步 Agent 默认不能弹权限框，会自动避免 permission prompt，除非特殊模式允许冒泡。

Conductor 迁移建议：

- 把现有 `subagent.rs::run_claude_p` 升级为 `subagent.claude_p` 工具。
- 第一阶段仍只允许同步/短超时执行。
- 第二阶段引入 `agent_runs` 或复用 `tasks` 表，让 Agent 后台执行可追踪。
- 后台结果必须写入 `state/tool-runs/` 或 `state/subagent-runs/`，聊天里只展示摘要和链接。
- Agent 调用默认必须走 Action Proposal，除非是只读摘要型内部 Agent。

第四类：Agent Team / Swarm。

- `TeamCreate`
- `TeamDelete`
- `SendMessage`

特点：

- Team 是一个持久化的 team file。
- team file 保存：
  - team name
  - description
  - createdAt
  - leadAgentId
  - leadSessionId
  - members
- member 保存：
  - agentId
  - name
  - agentType
  - model
  - joinedAt
  - tmuxPaneId
  - cwd
  - subscriptions
- `SendMessage` 通过 mailbox 给队友发消息，支持：
  - 指定 teammate name
  - 广播 `*`
  - shutdown request/response
  - plan approval response
- 源码里限制一层 team roster，teammate 不能无限递归 spawn teammate。

Conductor 迁移建议：

- 完整 swarm/team OS 对 MVP 过重，但可以先做轻量 Agent Team 容器。
- 可借鉴的是“后台 agent 有名字、有输出文件、有状态、有消息入口”。
- Conductor 第一阶段需要：
  - `agent_runs`
  - `agent_id`
  - `role`
  - `cwd/workspace_id`
  - `status`
  - `output_ref`
  - `parent_task_id`
  - `team/member/mailbox`
- Team 能力先只做可追踪的 team/member/mailbox，不引入自动调度、跨进程协同和远程 agent。

第五类：计划模式。

- `EnterPlanMode`
- `ExitPlanMode`

特点：

- 计划模式把“先规划、再申请权限、再执行”做成显式流程。
- `ExitPlanMode` 可以带 `allowedPrompts`，描述后续要批准的动作类别，而不是只批准某一条命令。

Conductor 迁移建议：

- 很适合迁移到桌宠交互。
- 用户说“帮我整理这个文档”时，不应立刻执行写入，而应进入：

```text
桌宠：我可以先读取结构并生成修改建议。要继续吗？
用户：可以
桌宠：提案：
  1. 读取当前文档结构
  2. 导出文本摘要
  3. 生成大纲建议
  4. 如需写回，再单独确认
```

- 这比一次性弹复杂权限框更像桌宠，也更容易被用户理解。

第六类：MCP。

- `ListMcpResources`
- `ReadMcpResource`
- 动态 `Mcp` tools

特点：

- MCP tools 通常用 `mcp__server__tool` 命名。
- MCP 工具默认属于 deferred tools，不在初始 prompt 中全部展开。
- Agent 可以声明 required MCP servers。
- Agent 启动时可连接额外 MCP server，并在结束时清理。

Conductor 迁移建议：

- MCP 应作为 `ToolProviderKind::Mcp`，不要成为核心架构。
- 所有 MCP tool 都必须映射为内部 `ToolSpec`，并走同一个 proposal / approval / logging。
- 不允许外部 MCP 绕过：
  - workspace scope
  - risk_level
  - allowed tools
  - tool run log
- `conductor.pet.set_avatar` 是非常适合首个 MCP 工具的低风险样例。

第七类：Skill。

- `Skill`
- 本地/bundled/MCP skills

特点：

- Skill 不是普通工具，它更像“可被模型调用的工作流提示词/命令”。
- Skill 定义包含：
  - `name`
  - `description`
  - `aliases`
  - `whenToUse`
  - `argumentHint`
  - `allowedTools`
  - `model`
  - `disableModelInvocation`
  - `userInvocable`
  - `hooks`
  - `context: inline | fork`
  - `agent`
  - `files`
  - `getPromptForCommand`
- Skill 可以 inline 展开，也可以 fork 到子 Agent 执行。
- bundled skill 的参考文件是懒加载写入磁盘的，模型需要时再读，而不是一开始塞进 prompt。
- Skill 权限可单独允许/拒绝，默认不应无条件执行。

Conductor 迁移建议：

- 继续推进 Skills，但定义要从“人格提示词”升级为“场景工作流”。
- MVP 推荐三个 skill：
  - `document_assistant`
  - `coding_assistant`
  - `pet_avatar_router`
- 每个 skill 都应声明：
  - 触发条件
  - 可用工具
  - 是否允许主动询问
  - 是否允许写入
  - 默认 avatar
  - 需要用户确认的边界

第八类：AskUserQuestion。

- `AskUserQuestion`

特点：

- 是结构化多选问题工具。
- 支持 1-4 个问题。
- 每个问题有：
  - `question`
  - `header`
  - `options`
  - `multiSelect`
- 每个 option 有：
  - `label`
  - `description`
  - `preview`

Conductor 迁移建议：

- 这是最值得迁移到桌宠 UX 的能力之一。
- 桌宠主动问用户时，不要只发自由文本，应该生成可点击选择。
- 但产品层不应该把每个场景都设计成不同按钮。默认保持统一三选一：

```text
检测到你正在写文档，要我帮忙吗？

[可以推进]
我会先读取文档结构并生成建议，不会写入。

[不推进]
本次不处理，短时间内不再提醒。

[有什么不同]
说明推进与不推进的差异、影响范围和风险。
```

- 这可以作为 `ask_user` 内部工具或 proposal UI 的一部分。
- 具体动作选项可以放到“有什么不同”之后，或由系统在用户点“可以推进”后根据上下文选择最小安全动作。

第九类：配置、工作区和 Worktree。

- `Config`
- `EnterWorktree`
- `ExitWorktree`

特点：

- Claude Code 用 cwd / additional working directories / worktree 来定义读写边界。
- Worktree 是 coding 场景的隔离模式，用于让子 Agent 在临时 git worktree 中修改，避免直接污染主工作区。

Conductor 迁移建议：

- 桌宠不应该把产品语言说成“进入 worktree”，但底层需要 workspace scope。
- UI 表达应是：
  - 当前项目
  - 当前文档
  - 当前应用
  - 当前工作上下文
- coding agent 如果要写代码，可后续支持“隔离副本执行”，底层可用 git worktree，但 UI 说“在临时副本中尝试修改”。

### 19.3 工具渐进式暴露机制

Claude Code 的 `ToolSearchTool` 是非常重要的设计。

它解决的问题是：工具太多时，不能把所有工具 schema 都塞进模型上下文。Claude Code 的做法是：

- 少数核心工具始终暴露。
- 大部分工具只暴露名字或延迟暴露。
- 模型需要时先调用 `ToolSearch`。
- `ToolSearch` 根据工具名、MCP server 名、search hint、description 做关键词匹配。
- 搜到后返回完整 JSON Schema，模型才可以调用该工具。
- MCP tools 默认 deferred。
- 工具可以声明 `alwaysLoad` 跳过 deferred。

Conductor 应该迁移这个机制，但做得更轻：

```text
初始可见工具：
  ask_user
  tool.search
  proposal.create
  memory.recall
  pet.set_avatar

延迟可见工具：
  office.*
  workspace.*
  subagent.*
  mcp.*
  future browser/mail/lark tools
```

对桌宠来说，渐进式暴露的价值不只是省 token，还能降低产品风险：

- 用户不会感觉桌宠一上来就“什么都能动”。
- 模型不会被过多工具诱导乱调用。
- MCP 工具不会绕过内部治理。
- 每个场景只暴露该场景允许的工具。

### 19.4 Claude Code 权限模型对 Conductor 的启发

Claude Code 的权限上下文包含：

- 当前 permission mode。
- additional working directories。
- always allow rules。
- always deny rules。
- always ask rules。
- bypass permissions 是否可用。
- auto mode 是否可用。
- background agent 是否应避免弹权限。
- 自动检查是否要先于弹窗完成。

这说明权限不是工具自己的布尔值，而是“工具 + 输入 + 上下文 + 用户规则 + 当前模式”的决策结果。

Conductor 推荐定义：

```rust
pub struct ToolPermissionContext {
    pub mode: PermissionMode,
    pub workspace_id: Option<String>,
    pub active_cwd: Option<PathBuf>,
    pub allow_rules: Vec<PermissionRule>,
    pub deny_rules: Vec<PermissionRule>,
    pub ask_rules: Vec<PermissionRule>,
    pub quiet_mode: bool,
    pub background_agent: bool,
}
```

权限模式建议：

| 模式 | 用途 |
|---|---|
| `observe_only` | 只能观察、记忆、回答，不执行系统动作 |
| `ask_first` | 默认模式，所有非只读动作都提案 |
| `read_allowed` | 当前 workspace 内只读工具可直接执行 |
| `draft_allowed` | 可生成草稿/临时文件，但不覆盖用户文件 |
| `workspace_write_once` | 用户批准的一次写入 |
| `external_side_effect_blocked` | 默认阻断外发消息、邮件、网络写操作 |

MVP 不建议实现 Claude Code 那种复杂的 bypass/auto 模式。桌宠作为常驻个人 agent，默认应该更保守。

### 19.5 Agent / Team 能力是否要移植

结论：

- `Agent` 能力已按 MVP 方式移植。
- 轻量 `Agent Team` / `mailbox` 已落地，用于追踪 team、member 和消息；完整 Swarm / agent OS 仍暂缓。
- `SendMessage` 的“可寻址后台 worker”思想已以 mailbox 形式先做最小闭环。

原因：

- Conductor 当前已经有 `subagent.rs::run_claude_p`，迁移 Agent 能力成本低。
- 个人桌宠的主线是“用户和一个桌宠协作”，不是一开始展示一群 agent。
- 完整 Agent Team 会引入 team file、自动调度、权限同步、任务输出、跨进程通信和远程 agent，MVP 容易过重；因此当前只实现持久化容器和 mailbox，不做自动协作调度。

推荐落地：

```text
Phase A:
  subagent.claude_p
  同步执行
  proposal 批准后运行
  写入 subagent run log

Phase B:
  agent_runs 表
  后台执行
  output_ref
  stop/read output
  桌宠用气泡通知完成

Phase C:
  named agent
  send_message(agent_id, message)
  只做主桌宠 -> 后台 agent 的单向补充指令

Phase D:
  lightweight agent team
  team/member/mailbox 持久化
  不包含自动领取、swarm 调度、远程 agent
```

当前实现状态（2026-05-26）：

- [x] `agent_runs` SQLite 表已落地，包含 `agent_id`、`role`、`workspace_id`、`cwd`、`status`、`pid`、`input_ref`、`output_ref`、`error`、时间戳和 `metadata_json`。
- [x] `agent.start` 已注册为 Subagent 工具，按 workspace trust level 进入 proposal/tool 执行链路。
- [x] `agent.read_output` / `agent.stop` 已注册；桌面端和 CLI 已有 list/read/stop 的薄封装。
- [x] 轻量 Agent Team 已落地：`agent_teams`、`agent_team_members`、`agent_mailbox_messages` 三张表和索引已加入迁移。
- [x] 已注册 `agent.team.create/list/add_member/snapshot` 与 `agent.mailbox.send/list/mark_read` 工具，并补齐 core 测试。
- [x] CLI / Tauri commands / TypeScript IPC 已提供 team 与 mailbox 的薄封装。
- [~] 后台运行当前由宿主进程托管 `claude -p`，stdout/stderr 落到 `state/agent-runs/` sidecar 文件；适合桌面端常驻进程，CLI 触发长跑仍需要后续独立 worker/daemon 化。
- [~] 当前 Agent Team 仅负责 team/member/mailbox 的持久化和手动消息流；自动 scheduler、task watcher 自动领取、远程 agent、完整 Swarm 不在本轮范围。

### 19.6 Skills 如何迁移

Claude Code 的 Skill 机制说明：Skill 不只是“提示词片段”，而是一个带元数据、权限、工具范围、执行上下文的工作流。

Conductor 的 SkillRegistry 建议结构：

```rust
pub struct SkillSpec {
    pub id: String,
    pub name: String,
    pub description: String,
    pub when_to_use: String,
    pub triggers: Vec<SkillTrigger>,
    pub allowed_tools: Vec<String>,
    pub default_avatar_id: Option<String>,
    pub context_mode: SkillContextMode,
    pub proactive_policy: ProactivePolicy,
    pub requires_approval: bool,
    pub reference_files: Vec<PathBuf>,
}
```

SkillContextMode：

```text
inline: 在主聊天中展开提示词，适合轻量建议。
fork: 派给后台 agent，适合长文档分析、代码审查、批量处理。
```

首批 skill 建议：

1. `document_assistant`
   - 触发：Word/WPS/Markdown/PPT/飞书文档前台活跃。
   - 默认形象：`document_secretary`。
   - 工具：`office.inspect_document`、`office.export_text`、`office.patch_dry_run`。
   - 主动权限：只允许问“要不要帮忙”，不允许自动读取正文。
   - 写入权限：必须 preview + approve。

2. `coding_assistant`
   - 触发：VS Code / Cursor / Terminal / Codex / Claude Code 活跃。
   - 默认形象：`programmer`。
   - 工具：`tasks.list`、`subagent.claude_p`、`workspace.read_text`、后续 `codex.run`。
   - 主动权限：可提醒 pending 任务和测试点。
   - 写入权限：编码工具自己走 workspace 权限，桌宠不直接写代码。

3. `pet_avatar_router`
   - 触发：用户显式要求或场景识别。
   - 默认形象：无，负责路由到具体 avatar。
   - 工具：`pet.set_avatar`。
   - 主动权限：低风险，可临时切换。
   - 限制：只允许枚举形象，不接受路径/URL。

### 19.7 MCP 如何迁移

Claude Code 的 MCP 做法值得借鉴的一点是：MCP 是工具来源，不是权限边界。

Conductor 的 MCP 适配规则：

```text
MCP raw tool
  -> MCP adapter
  -> ToolSpec
  -> PermissionContext
  -> ActionProposal
  -> ToolExecutor
  -> ToolRun log
```

禁止：

- MCP server 直接修改 Conductor state。
- MCP server 绕过 workspace scope。
- MCP server 自行声明低风险后直接执行。
- 把所有 MCP tool schema 一次性注入桌宠聊天 prompt。

允许：

- 低风险工具直接映射，如 `conductor.pet.set_avatar`。
- 只读资源读取，如 `read_mcp_resource`。
- 用户批准后执行外部工具。

`conductor.pet.set_avatar` 推荐 MCP schema：

```json
{
  "avatar_id": "original | document_secretary | programmer",
  "mode": "temporary | persistent",
  "ttl_minutes": 60,
  "reason": "切换原因"
}
```

执行规则：

- `avatar_id` 必须是枚举。
- `temporary` 模式默认 60 分钟。
- `persistent` 只允许用户显式要求时使用。
- 外部 agent 调用只能临时切换。
- quiet mode 下只切图，不额外弹通知。

### 19.8 Worktree / Workspace 如何迁移

Claude Code 的修改边界依赖 cwd、additional working directories 和 worktree isolation。桌宠不应在产品层照搬“worktree”话术，但必须有同等边界。

Conductor 推荐：

```text
WorkspaceRegistry:
  id
  root
  name
  kind
  trust_level
  last_active_at
  source

ActiveContext:
  foreground_app
  foreground_title
  inferred_workspace_id
  active_file
  confidence
```

读取 workspace 的来源优先级：

1. Claude / Codex hook 上报的 `cwd`。
2. 当前前台 IDE / Terminal 的项目路径。
3. 最近活跃任务的 `cwd`。
4. 用户手动选择当前项目。
5. 文档场景下用 active file 所在目录临时生成 document workspace。

产品表达：

- 不说“当前 cwd 是...”。
- 对用户说“当前项目：personal-agent”。
- 文档场景说“当前文档：方案.docx”。
- 不要求每个 workspace 放一个桌宠副本。
- `.exe` 全局运行，内部维护多个 workspace。

coding 写入场景的高级隔离：

- 后续可支持 git worktree。
- UI 表达为“在临时副本中尝试修改”。
- 成功后展示 diff，再让用户选择是否应用回主项目。

### 19.10 TaskList V2 可替代当前任务列表

Claude Code 还有一组 TaskList V2 工具，值得作为 Conductor 当前 `tasks.rs` / tasklist 能力的升级方向。

相关源码模块：

- `../src/tools/TaskCreateTool/TaskCreateTool.ts`
- `../src/tools/TaskListTool/TaskListTool.ts`
- `../src/tools/TaskGetTool/TaskGetTool.ts`
- `../src/tools/TaskUpdateTool/TaskUpdateTool.ts`
- `../src/utils/tasks.ts`
- `../src/components/TaskListV2.tsx`
- `../src/hooks/useTaskListWatcher.ts`

它与旧 `TodoWrite` 的区别很明确：

| 能力 | `TodoWrite` | TaskList V2 |
|---|---|---|
| 数据形态 | 每次整体覆盖一个 todo 数组 | 独立 task 实体 |
| 任务 ID | 无稳定实体 ID 或弱 ID | 高水位自增 ID |
| 状态 | `pending/in_progress/completed` | `pending/in_progress/completed` |
| 详情 | `content/activeForm` | `subject/description/activeForm` |
| 所属人 | 无 | `owner` |
| 依赖关系 | 无 | `blocks/blockedBy` |
| 扩展字段 | 无 | `metadata` |
| 并发领取 | 无 | `claimTask` + lock |
| UI 更新 | AppState 内部 | watcher / signal |
| agent 协作 | 弱 | 可被 team / background agent 共享 |

对 Conductor 来说，TaskList V2 更适合替代当前任务列表，因为桌宠接下来会同时承载：

- 用户可见的待办/推进列表。
- Claude / Codex hook 产生的审查任务。
- 桌宠主动拆分出的工作步骤。
- 后台 agent 的子任务。
- 文档/编码/日常事务的跨 session 任务。

产品定调：

- 所有需要用户判断的待审任务都进入 TaskList。
- 用户每次只处理一条当前推荐任务，不面对整屏审批表。
- 当前推荐任务只给三选一：`可以推进 / 不推进 / 有什么不同`。
- `可以推进` 只批准“当前这一步”，不隐式授权后续写入、外发、删除。
- `不推进` 默认只是暂缓或拒绝当前任务，不要求用户解释。
- `有什么不同` 才展开差异、风险、范围、预览、替代方案和技术细节。
- 桌宠负责把复杂任务拆成低负担小步，不能把复杂权限判断转嫁给用户。

待审任务 UI 示例：

```text
待审任务
当前建议推进：整理当前方案文档结构

为什么：你正在编辑方案文档，且这一步只读。
影响：读取标题层级和摘要，不修改文件。

[可以推进] [不推进] [有什么不同]
```

如果有多条待审任务：

```text
还有 4 个待审任务。先看最小的一项：
整理当前方案文档结构

[可以推进] [不推进] [有什么不同]
```

用户不需要逐条审批全部任务；系统按当前上下文、风险、阻塞关系和收益推荐一条。

#### 19.10.1 可直接迁移的任务模型

Claude Code 的任务核心字段可以简化迁移为：

```rust
pub struct AgentTask {
    pub id: String,
    pub task_list_id: String,
    pub subject: String,
    pub description: String,
    pub active_form: Option<String>,
    pub owner: Option<String>,
    pub status: TaskStatus,
    pub blocks: Vec<String>,
    pub blocked_by: Vec<String>,
    pub metadata_json: Option<serde_json::Value>,
    pub workspace_id: Option<String>,
    pub created_at: DateTime<Utc>,
    pub updated_at: DateTime<Utc>,
}

pub enum TaskStatus {
    Pending,
    InProgress,
    Completed,
}
```

当前 `tasks.rs` 里的 `passed/rejected/skipped` 更像审查流状态，不适合作为通用 tasklist 主状态。建议迁移为：

| 当前状态 | 新 tasklist 状态 | 补充字段 |
|---|---|---|
| `pending` | `pending` | 无 |
| `in_progress` | `in_progress` | 无 |
| `passed` | `completed` | `metadata.review_result = "passed"` |
| `rejected` | `completed` 或 `pending` | 若需返工则保留 pending，并写 `metadata.review_result = "rejected"` |
| `skipped` | `completed` | `metadata.resolution = "skipped"` |

这样 tasklist 的状态语义会更稳定：它只表达“是否还需要推进”，而不是混合表达审查结论。

#### 19.10.4 依赖关系和领取机制

Claude Code 的关键逻辑：

- `blocks`: 当前任务完成前，哪些任务不能开始。
- `blockedBy`: 当前任务被哪些任务阻塞。
- unresolved task 指 `status != completed`。
- 可领取任务必须满足：
  - `status == pending`
  - 没有 owner
  - `blockedBy` 中没有未完成任务。
- `claimTask` 在锁内完成，避免多个 agent 同时领取。
- 可选 `checkAgentBusy`，避免一个 agent 同时领多个未完成任务。

Conductor 应迁移这个机制，因为它正好适合桌宠调度后台 agent：

```text
task.list_available(scope)
  -> 找 pending + no owner + blockers resolved
  -> 按 id 升序
  -> 桌宠建议用户处理第一项

task.claim(task_id, owner)
  -> 原子检查 owner/status/blocked_by
  -> 成功后 owner = pet 或 agent_id
  -> status 可保持 pending，真正开始时再 in_progress
```

MVP 可以先不做复杂锁文件，直接依赖 SQLite 事务：

```sql
BEGIN IMMEDIATE;
SELECT ... WHERE id = ?;
-- 检查 owner/status/blockers
UPDATE tasks SET owner = ?, updated_at = ? WHERE id = ? AND owner IS NULL;
COMMIT;
```

这比 Claude Code 的 JSON file + lock 更适合当前项目，因为 Conductor 已经使用 SQLite。

#### 19.10.5 UI 展示规则

Claude Code 的 `TaskListV2` 有几个可迁移 UI 规则：

- 按 ID 升序稳定排序。
- 优先展示：
  - 最近完成的任务
  - in_progress
  - pending
  - 较早 completed
- pending 中未被阻塞的排前面。
- 被阻塞任务显示 `blocked by #id`。
- 有 owner 的任务显示 owner。
- 列表过长时折叠为 “+N pending / in progress / completed” 摘要。

Conductor 桌面 UI 可改成：

```text
当前项目任务
1. [进行中] 整理方案文档结构 · pet
2. [待处理] 生成迁移实施清单
3. [阻塞] 接入 office.patch_dry_run · blocked by #2
```

桌宠气泡只显示最少信息：

```text
当前项目有 2 个待处理任务，下一项是“生成迁移实施清单”。
```

#### 19.10.6 与现有 `tasks.rs` 的替代关系

当前 `tasks.rs` 主要服务于 Conductor 的审查/调度任务，字段包括：

- `kind`
- `artifact`
- `summary_ref`

- `status`
- `session_id`
- `terminal_id`
- `cwd`
- `current_request`
- `last_output_summary`
- `permission_summary`

这些字段不应丢失，而是迁移到 TaskList V2 的 `metadata_json` 或扩展列：

| 当前字段 | 新位置 |
|---|---|
| `source` | `source` 列或 `metadata.source` |
| `kind` | `kind` 列或 `metadata.kind` |
| `artifact.file` | `metadata.artifact_file` |

| `current_request` | `description` 或 `metadata.current_request` |
| `last_output_summary` | `metadata.last_output_summary` |
| `permission_summary` | `metadata.permission_summary` |

建议数据表：

```sql
CREATE TABLE task_lists (
  id TEXT PRIMARY KEY,
  scope TEXT NOT NULL,
  title TEXT NOT NULL,
  workspace_id TEXT,

  updated_at TEXT NOT NULL,

  workspace_id TEXT,

```

说明：

- `task_dependencies.from_task_id blocks to_task_id`。
- 读取时派生 `blocks` 和 `blocked_by`。
- 对模型暴露时仍使用 Claude Code 的 `blocks/blockedBy` 结构，便于迁移工具说明。

#### 19.10.7 与 Tool/Skill/Agent 的关系

TaskList V2 不只是 UI 列表，而应成为 Conductor 调度层的中心状态。

关系如下：

```text
Skill 命中场景
  -> task.create 拆分步骤
  -> task.list 展示给用户
  -> 用户批准某项
  -> task.claim(owner = pet/agent_id)
  -> task.update(status = in_progress)
  -> ToolExecutor / subagent 执行
  -> task.update(status = completed, metadata.output_ref = ...)
```

对桌宠的价值：

- 它能把“主动帮忙”从一次性对话变成可持续推进。
- 它能让后台 agent 的工作有明确归属。
- 它能让记忆层只沉淀任务摘要，不需要保存全部聊天。
- 它能让 `workspace_id` 成为任务、工具、记忆之间的连接点。

```ts
function findAvailableTask(tasks) {
  const unresolvedTaskIds = new Set(
    tasks.filter(t => t.status !== 'completed').map(t => t.id)
  )

  return tasks.find(task => {
    if (task.status !== 'pending') return false
    if (task.owner) return false
    return task.blockedBy.every(id => !unresolvedTaskIds.has(id))
  })
}
```

### 19.11 推荐的交互改造

桌宠交互应采用 Claude Code 的“工具可见、权限可审、结果可追踪”，但表现得更轻。

### 19.12 MVP 取舍

应立即迁移：

- ToolRegistry 的概念。
- ToolSpec schema。
- Tool execution pipeline。
- Proposal/approval 作为权限入口。
- TaskList V2 的 `task.create/list/get/update` 工具模型。
- Task scope / owner / dependency / metadata 模型。
- ToolSearch 的轻量版。
- SkillSpec 的 `when_to_use + allowed_tools + context_mode`。
- `AskUserQuestion` 式结构化询问。
- `subagent.claude_p` 包装。
- `pet.set_avatar` 内部工具和 MCP wrapper。

暂缓迁移：

- 通用 Bash/PowerShell。
- 通用 FileWrite/FileEdit。
- 完整 Agent Team / Swarm。
- Worktree 自动隔离。
- 远程 Agent。
- 自动权限分类器。
- 复杂 hook 系统。
- Task watcher 自动领取。

已按轻量形态落地但不继续扩展：

- Agent Team / mailbox：只做 team、member、mailbox 持久化和手动读写，不做自动 scheduler、swarm 协作或远程 agent。

原因：

- 桌宠是常驻个人 agent，安全预期比命令行 coding agent 更保守。
- MVP 的价值验证在于“观察 -> 询问 -> 提案 -> 执行低风险工具 -> 记录结果”，不是一次性做 agent OS。
- 当前项目已有 proposal/subagent/tasks 基础，先把 `tasks.rs` 升级为工具化 TaskList V2，再把执行链路跑通收益最大。

## 20. 最终结论

Conductor 可以从“任务调度桌宠”演进成“受限行动的个人 agent”，但必须保持下面这条主线：

```text
先观察，再询问；
先提案，再执行；
先限定工作区，再调用工具；
先保存摘要和偏好，再考虑长期记忆；
先接确定工具，再接 MCP 泛化工具。
```

MCP 应该纳入开发方案，但作为工具 provider，而不是核心架构。

Skills 应该推进，而且应成为“桌宠什么时候主动帮忙、如何帮忙、何时停止”的主要载体。

记忆应采用“短期上下文 + 事件日志 + 工作区记忆 + 用户画像”的分层方案，避免把所有历史都塞进 prompt。

MVP 的记忆技术选型明确为 SQLite 事实源 + 本地轻量向量记忆：先用 `memory_chunks` / `memory_embeddings` 和 Rust 进程内 cosine 检索，不引入独立向量数据库服务；SQLite `Vec1` 与 LanceDB 作为后续备选。

工作区方案是必要的，但产品表达应是“当前项目/当前文档/当前上下文”，不要让用户感到每个目录都需要一个桌宠副本。最终形态应是一个全局 `.exe` 管理多个 workspace，并在每次工具调用时绑定明确的 scope 和权限策略。

桌宠形象切换 MVP 只做 `original`、`document_secretary`、`programmer` 三张静态透明图，并通过受限内部工具 / MCP 工具 `conductor.pet.set_avatar` 切换枚举形象，不接受任意图片路径或 URL。
