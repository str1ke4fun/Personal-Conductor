# 项目 Agent 架构状态机与治理范式

> 日期: 2026-05-29  
> 方法: `state-machine-lifecycle` / `agent-boundary + architecture + review`  
> 范围: 项目内运行时 Agent、开发 AgentTeam、工具调用、权限审批、记忆、事件审计、前端状态呈现  
> 结论: 当前项目已经具备 Agent Runtime 的主要构件，但缺少一个统一的 L0 状态账本。后续所有 Agent 能力应先接入本文状态机，再进入实现。

---

## 0. 范式定位

本文不是通用方法论，而是对当前项目 Agent 架构的项目内规范化。

目标只有一个: 把“会跑的 Agent”收敛成“可建模、可授权、可审计、可测试、可回归的 Agent Runtime”。

本项目后续涉及 Agent 的需求、修复、重构、工具接入、前端展示，都必须回答:

| 问题 | 必须产物 |
|---|---|
| 这个需求改变了哪个对象的状态? | 状态机对象与转换 |
| 谁触发转换? | Command/Event/Timer |
| 什么情况下允许? | Guard |
| 会产生什么副作用? | Side Effects |
| 用户或系统如何撤销/补偿? | Recovery/Compensation |
| LLM 是否能越权? | PermissionGrant 与 WorkspaceScope |
| 如何证明没有漂移? | 测试矩阵与审计事件 |

证据标记:

| 标记 | 含义 |
|---|---|
| observed | 代码或现有文档中已存在 |
| inferred | 从现有实现推导，但未显式约束 |
| proposed | 本文给出的后续规范 |

### 1.2 当前运行链路

```text
User
  -> ChatSession
  -> ChatRun(request_id)
  -> LLM planning
  -> ToolCall
  -> tools.execute_tool_with_workspace_async
  -> ToolExecutionUpdate(frontend)
  -> ChatMessage(tool_calls embedded)
```

后台子 Agent 链路:

```text
ToolCall(agent.start / subagent.claude_p)
  -> AgentRun.queued
  -> spawn claude -p
  -> AgentRun.running
  -> AgentRun.succeeded / failed / stopped
  -> optional AgentTask writeback by metadata.task_id
```

审批链路:

```text
ToolCall error: outside current workspace
  -> Proposal.pending
  -> human approve / reject
  -> Proposal.approved
  -> execute_proposal
  -> Proposal.running
  -> ToolRun.completed
  -> Proposal.succeeded / failed
```

这个链路说明: 项目已经有审批对象，但审批不是所有高风险 ToolCall 的统一入口，目前主要由“越工作区路径错误”触发。

## 2. 当前实现的状态机

### 2.1 `AgentRun` 当前状态机

当前代码状态:

```text
queued -> running -> succeeded
queued -> running -> failed
queued -> running -> stopped
running -> stopped
terminal -> terminal(no-op)
```

现状评价:

| 维度 | 结论 |
|---|---|
| 进程生命周期 | 足够表达后台进程是否跑完 |
| 认知生命周期 | 不足，无法表达 planning / awaiting_approval / tool_calling / paused |
| 审计能力 | 有 input_ref/output_ref/error，但缺少逐步 AuditEvent |
| 任务回写 | 通过 `metadata.task_id` best-effort 更新任务状态 |
| 风险 | 成功/失败直接折叠为终态，缺少“需要人工确认输出”的中间态 |

### 2.2 `AgentTeam` 当前状态机

当前代码状态:

```text
AgentTeam: active -> archived
AgentTeamMember: active -> paused / stopped
MailboxMessage: unread -> read
```

消息类型:

```text
message
broadcast
shutdown_request
shutdown_response
plan_approval_request
plan_approval_response
```

现状评价:

| 维度 | 结论 |
|---|---|
| Team 容器 | 已具备 |
| 成员角色 | 已具备 role/run_id/cwd/subscriptions |
| OODA-R | 未强制落到状态机 |
| Plan Approval | 有消息类型，但不是执行门禁 |
| 并行写作用域 | 未建模，无法阻止多个 Agent 写同一文件/目录 |

### 2.3 `ToolSpec` 与 `ToolCall` 当前状态

`ToolSpec` 已有:

```text
provider: internal / cli / subagent / mcp
risk_level: read_only / draft_only / workspace_write / external_side_effect / destructive
permissions:
  - read_workspace
  - write_workspace
  - read_external_path
  - write_external_path
  - network
  - send_message
  - system_control
workspace_required: bool
supports_dry_run: bool
```

前端工具状态:

```text
pending / running / success / error
```

后端事件状态:

```text
started / completed / error / approval_required
```

现状评价:

| 维度 | 结论 |
|---|---|
| 工具元数据 | 较完整 |
| 工作区守卫 | 已有 `workspace_required`、trust level、路径围栏 |
| shell 守卫 | 已有 blocklist 风险拦截 |
| ToolCall 一等对象 | 不完整，对话路径只嵌入 ChatMessage |
| ToolRun 持久化 | 只覆盖 Proposal 执行路径 |
| 状态命名 | 前后端状态命名不统一，`approval_required` 在前端被折叠为 pending |

### 2.4 `Proposal` 当前状态机

```text
pending -> approved -> running -> succeeded
pending -> rejected
approved -> running -> failed
pending / approved / running -> used / expired
```

现状评价:

| 维度 | 结论 |
|---|---|
| 人工审批对象 | 已存在 |
| 风险级别 | 已存在 |
| 工具绑定 | 已存在 `tool_id + tool_input_json` |
| 执行日志 | `execute_proposal` 会写 `tool_runs` |
| 范式缺口 | Proposal 与 PermissionGrant 混用，缺少“授权范围/有效期/复用规则” |

### 2.5 `MemoryEntry` 当前状态

已有字段:

```text
scope: global / workspace / document / session
source: user_confirmed / inferred / tool / summary
sensitivity: normal / private / secret
confidence
expires_at
last_used_at
```

现状评价:

| 维度 | 结论 |
|---|---|
| 分类能力 | 已具备 |
| 检索隔离 | `secret` 不参与搜索，workspace 有过滤 |
| 生命周期 | 缺少 candidate/classified/approved/stored/retrieved/deleted 显式状态 |
| 污染防护 | 不能防止工具输出或 LLM 推断直接变长期事实 |

## 3. 核心判断: 当前最大熵源

### 3.1 状态源分散

当前同一个现实动作可能被多个对象表示:

| 现实动作 | 当前可能落点 |
|---|---|
| Agent 正在跑 | `AgentRun.running`, `AgentTask.in_progress`, `Legacy Task.in_progress`, 前端“运行中” |
| 工具需要审批 | `ToolExecutionUpdate.approval_required`, `Proposal.pending`, 前端 Proposal 卡片 |
| Agent 输出待人工看 | `Legacy Task.pending`, `AgentRun.succeeded`, `AgentTask.completed` |
| 工具执行失败 | `ToolCallRecord.success=false`, `ToolRun.error`, `Proposal.failed`, `events.v2_stage` |

如果不指定 canonical source，后续开发会持续产生“看起来都对，但互相不同步”的状态漂移。

### 3.2 高风险调用没有统一授权账本

现在有三层防护:

```text
ToolSpec.risk_level / permissions
Workspace.trust_level
shell.security.validate_command
```

但缺少统一的:

```text
PermissionGrant
ToolCall
AuditEvent
```

导致问题:

| 问题 | 影响 |
|---|---|
| Chat 直接执行高风险工具 | Proposal 不是普遍门禁 |
| Proposal 代表审批，也代表待执行动作 | 无法表达一次性授权/会话授权/撤销 |
| ToolRun 只记录 Proposal 执行 | 对话内大量工具调用无法完整审计 |
| 子 Agent 权限没有衰减模型 | Child Agent 可能继承过宽上下文 |

### 3.3 AgentTeam 已有通信，但没有工作协议

Mailbox 已经能发:

```text
plan_approval_request
plan_approval_response
shutdown_request
shutdown_response
```

但没有强制:

```text
Observe -> Orient -> Decide -> Act -> Review
```

因此它目前是消息总线，不是工程治理系统。

### 3.4 前端 read model 与后端状态不一致

`useChatSession` 把后端:

```text
started -> running
completed -> success
error -> error
其他 -> pending
```

这会把 `approval_required` 显示为 `pending`。对用户而言，“还没开始”和“需要你批准”是两个不同状态，不能合并。

## 4. 目标范式: 项目 Agent Runtime Meta Model

后续所有 Agent 相关功能必须归入以下对象，不允许散落新增状态枚举。

### 4.1 Canonical Objects

| Canonical Object | 负责什么 | 当前落点 | 后续要求 |
|---|---|---|---|
| `Agent` | 能力、角色、默认边界 | inferred | 新增一等配置对象 |
| `AgentTask` | 可分配工作单元 | `tasklist.rs` | 成为 Agent 工作队列 canonical source |
| `AgentRun` | 一次执行实例 | `agent_runs.rs` | 扩充认知状态 |
| `ToolCall` | 一次工具调用 | partial | 入库，覆盖 Chat/Agent/Proposal 全路径 |
| `PermissionGrant` | 授权 | partial by Proposal | 从 Proposal 拆出 |
| `WorkspaceScope` | 文件/命令/网络边界 | `workspaces.rs`, `tools.rs` | 显式记录 read/write roots |
| `MemoryEntry` | 长期事实/偏好/摘要 | `memory.rs` | 增加生命周期与写入门禁 |
| `AuditEvent` | 可追溯行为 | `events.rs`, `tool_runs` | 标准化事件类型与关联 id |

### 4.2 Canonical Source 规则

| 状态类别 | Canonical Source | Read Model |
|---|---|---|
| Agent 工作状态 | `AgentRun.state` | 任务抽屉、运行中面板 |
| Agent 可领取任务 | `AgentTask.status + owner + blocked_by` | 队列视图 |
| 工具调用状态 | `ToolCall.state` | ToolUseCard |
| 人工授权状态 | `PermissionGrant.state` | Proposal/审批卡片 |
| 工作区权限 | `Workspace.trust_level + WorkspaceScope` | Workspace toolbar |
| 记忆可信度 | `MemoryEntry.state/source/confidence/sensitivity` | memory search/read |
| 审计证据 | `AuditEvent` | 调试面板/安全报告 |

禁止:

| 禁止事项 | 原因 |
|---|---|
| UI 自己发明业务状态 | 会产生不可回放状态 |
| ToolCall 只写 ChatMessage | 无法按 Agent/Task/Permission 回溯 |
| Proposal 同时承担授权和执行日志 | 权限复用、撤销、过期无法表达 |
| AgentTeam 消息直接驱动写操作 | 绕过 Plan Approval 与写作用域 |

## 5. 项目内 AgentRun 状态机

目标状态机:

```text
created
-> configured
-> idle
-> planning
-> awaiting_approval
-> running
-> tool_calling
-> paused
-> succeeded / failed / stopped
-> archived
```

为了兼容当前实现，可以先做两层状态:

| 层 | 字段 | 示例 |
|---|---|---|
| ProcessState | 当前 `AgentRunStatus` | queued/running/succeeded/failed/stopped |
| RuntimePhase | 新增 `phase` 或 `metadata.phase` | planning/awaiting_approval/tool_calling/reviewing |

### 5.1 合法转换

| From | To | Trigger | Guard | Side Effect |
|---|---|---|---|---|
| created | configured | configure_agent | role/workspace/tool policy 完整 | 写 AgentRun config |
| configured | idle | ready | workspace 可访问 | emit agent_ready |
| idle | planning | assign_task | AgentTask 可领取 | task owner set |
| planning | awaiting_approval | request_permission | risk >= workspace_write 或外部副作用 | 创建 PermissionGrant |
| planning | running | plan_accepted | low risk 且无需审批 | 进入执行 |
| awaiting_approval | running | grant_approved | grant 未过期且 scope 匹配 | 记录 AuditEvent |
| running | tool_calling | tool_proposed | ToolSpec 存在且输入合法 | 创建 ToolCall |
| tool_calling | running | tool_succeeded | ToolCall recorded | 回注 LLM/tool result |
| running | paused | pause_or_block | 等待人、等待外部、速率限制 | 保留恢复点 |
| paused | running | resume | grant/scope 仍有效 | 继续 |
| running | succeeded | complete | 所有必要任务完成 | 写 output_ref |
| running | failed | error/timeout | 错误不可自动补偿 | 写 error |
| running | stopped | human_stop/kill | 终止请求 | kill pid/process tree |
| terminal | archived | archive | 输出已审计或过期 | 只读 |

### 5.2 非法转换

| From | Illegal To | 原因 |
|---|---|---|
| created | running | 未配置角色与边界 |
| planning | tool_calling | 未完成风险分类 |
| awaiting_approval | tool_calling | 授权绕过 |
| failed | succeeded | 失败不能静默改成功 |
| stopped | running | 已终止运行不能复活，应创建新 run |
| archived | running/tool_calling | 归档后只读 |

### 5.3 对当前代码的落地要求

短期不必立即重写 `AgentRunStatus`，但新增 Agent 能力必须:

| 要求 | 做法 |
|---|---|
| 保留当前 queued/running/terminal 兼容 | 不破坏现有前端和 DB |
| 认知阶段进入 metadata | `metadata_json.phase = planning/awaiting_approval/...` |
| 所有阶段变化写事件 | `AuditEvent.agent_phase_changed` |
| 终态必须绑定输出证据 | `output_ref` 或 `error` 不可为空 |
| 失败回写任务不能直接变 pending | 应进入 `AgentTask.review_required` 或 metadata.review_required |

## 6. 工具调用与权限范式

### 6.1 ToolCall 状态机

目标状态机:

```text
proposed
-> risk_classified
-> approved / rejected
-> executing
-> succeeded / failed / timed_out / approval_required
-> recorded
```

字段建议:

```yaml
tool_call_id:
request_id:
agent_run_id:
chat_session_id:
agent_task_id:
tool_id:
provider:
risk_level:
permissions:
workspace_scope:
input_ref:
input_summary:
permission_grant_id:
state:
started_at:
finished_at:
output_ref:
result_summary:
error:
audit_ref:
```

### 6.2 PermissionGrant 状态机

目标状态机:

```text
unrequested
-> requested
-> approved_once / approved_session / denied
-> expired / revoked / used
```

`Proposal` 后续应成为 PermissionGrant 的用户展示卡片，而不是权限本体。

| 概念 | 当前 | 后续 |
|---|---|---|
| 用户看到的建议 | Proposal | Proposal |
| 是否允许执行 | Proposal.status | PermissionGrant.state |
| 允许做什么 | tool_id/input_json | grant.scope + policy |
| 执行记录 | tool_runs | ToolCall + AuditEvent |

### 6.3 风险门禁

| RiskLevel | 默认策略 | 是否需要 ToolCall 入库 | 是否需要 PermissionGrant |
|---|---|---|---|
| read_only | 允许 | 建议入库 | 否 |
| draft_only | 允许但不可落盘 | 必须入库 | 否，除非外部路径 |
| workspace_write | 任务 scope 内允许 | 必须入库 | 需要 task scope 或一次性审批 |
| external_side_effect | 默认阻断 | 必须入库 | 必须 |
| destructive | 默认阻断 | 必须入库 | 必须显式人工审批，不支持会话继承 |

### 6.4 WorkspaceScope 规则

后续每个 AgentRun 和 PermissionGrant 都必须有显式 scope:

```yaml
workspace_id:
read_roots:
write_roots:
external_read_roots:
external_write_roots:
allowed_commands:
network_allowlist:
expires_at:
```

规则:

| 规则 | 说明 |
|---|---|
| 子 Agent 权限取交集 | child scope = parent task scope ∩ explicit grant |
| 外部路径不能靠失败后补救长期放行 | 每次外部访问要么一次性批准，要么创建明确 session grant |
| command 执行必须绑定 working_dir | `bash.execute` 不能无工作区执行高风险命令 |
| MCP 工具必须声明 risk/permission | 不能因是 MCP 自动视为 read_only |

## 7. AgentTeam 开发与运行范式

本项目后续会混用多类 Agent。必须区分两侧:

| 类型 | 定义 | 主要风险 |
|---|---|---|
| 开发 AgentTeam | 帮你写项目代码的外部/协作 Agent | 需求漂移、并行写冲突、未 review |
| 项目内 Runtime Agent | 产品里实际运行的 Agent | 越权、记忆污染、工具副作用、审计缺失 |

### 7.1 开发 AgentTeam 必须遵循 OODA-R

```text
Observe -> Orient -> Decide -> Act -> Review
```

对应状态:

```text
idle
-> assigned
-> observing
-> oriented
-> design_ready
-> implementing
-> self_reviewing
-> review_ready
-> accepted / rework_required / blocked
```

### 7.2 项目内 AgentTeam 必须落到 Team 状态机

当前 `AgentTeam` 应扩展为:

```text
created
-> forming
-> planning
-> awaiting_plan_approval
-> executing
-> reviewing
-> succeeded / failed / stopped
-> archived
```

`AgentTeamMember` 应扩展为:

```text
idle
-> assigned
-> observing
-> planning
-> awaiting_assignment_approval
-> executing
-> reporting
-> paused / stopped / completed
```

### 7.3 Team 写作用域规则

每个成员必须在 metadata 中声明:

```yaml
role:
owned_transition:
read_scope:
write_scope:
forbidden_scope:
expected_outputs:
review_required_by:
```

并行规则:

| 情况 | 策略 |
|---|---|
| 写 scope 不重叠 | 可并行 |
| 写 scope 重叠 | 必须串行 |
| 一个 Agent 修改状态机/L0 | 所有实现 Agent 暂停并重新 Observe |
| 一个 Agent 发现需求边界变化 | 回到 Decide，不允许直接实现 |
| Review Agent 与 Implementation Agent | 必须独立，不共享自我结论 |

### 7.4 Plan Approval 必须变成硬门禁

当前有 `plan_approval_request` 消息类型。后续要求:

| 当前 | 后续 |
|---|---|
| 只是 mailbox 消息 | 绑定 `AgentTeam.awaiting_plan_approval` |
| 收到 response 不影响执行 | positive response 才能进入 executing |
| 无计划结构 | plan 必须包含 state impact、write scope、test matrix |
| 无拒绝补偿 | rejected 后进入 rework_required |

## 8. 记忆、事件与审计范式

### 8.1 MemoryEntry 状态机

```text
candidate
-> classified
-> approved
-> stored
-> retrieved
-> expired / deleted / quarantined
```

写入门禁:

| 来源 | 默认策略 |
|---|---|
| user_confirmed | 可直接 approved |
| summary | classified 后可 stored，但 confidence < 1 |
| tool | 默认 candidate，除非工具输出是稳定事实 |
| inferred | 不得直接长期存储，必须人工确认或降级为 session memory |

禁止写入长期记忆:

| 内容 | 原因 |
|---|---|
| LLM 临时推理 | 不可验证 |
| 工具错误输出 | 会污染未来决策 |
| 一次性路径/命令 | 非稳定事实 |
| 未确认偏好 | 容易过拟合用户一时表达 |

P0 事件必须写:

| Event | 触发 |
|---|---|

## 11. 增量落地路线

### Phase 1: 文档与约束先行

目标: 不大改代码，先阻止继续漂移。

动作:

| 动作 | 说明 |
|---|---|
| 将本文作为 Agent 架构 L1 规范 | 后续任务引用本文 |
| 新增 `docs/STATE_MODEL_AGENT.md` | 固化 L0 状态账本 |
| PR/派工模板加入 State Impact Card | 所有 Agent 任务先建模 |
| ToolSpec 注册检查清单 | 新工具必须填写风险与权限 |

### Phase 2: ToolCall 一等对象

目标: 所有工具调用可追溯。

动作:

| 动作 | 说明 |
|---|---|
| 新增 `tool_calls` 表 | 覆盖 Chat/Agent/Proposal |
| `chat.rs` 工具调用前先 create ToolCall | 不再只写 ChatMessage |
| 前端 ToolUseCard 使用 ToolCall.state | 消除状态映射漂移 |
| Proposal 执行复用 ToolCall | `tool_runs` 可逐步合并或作为 output sidecar |

### Phase 3: PermissionGrant 从 Proposal 拆出

目标: 审批从“建议卡片”升级为“授权账本”。

动作:

| 动作 | 说明 |
|---|---|
| 新增 `permission_grants` 表 | scope、risk、expires、revoked |
| Proposal 绑定 grant_id | 用户仍通过 Proposal 操作 |
| workspace_write 以上统一进入权限判断 | 不只外部路径触发 |
| destructive 默认拒绝 | 必须显式人工批准 |

### Phase 4: AgentTeam OODA-R 硬门禁

目标: 多 Agent 开发和项目内 Team 都可控。

动作:

| 动作 | 说明 |
|---|---|
| Team metadata 增加 `phase` | planning/executing/reviewing |
| Member metadata 增加 `write_scope` | 并行冲突检测 |
| plan_approval_response 驱动 Team 状态 | 从消息升级为状态转换 |
| Review Agent 独立记录 verdict | accepted/rework_required/blocked |

### Phase 5: Memory 生命周期

目标: 防止长期记忆污染。

动作:

| 动作 | 说明 |
|---|---|
| MemoryEntry 增加 state | candidate/classified/approved/stored |
| tool/inferred 默认不直接 stored | 需要降级或确认 |
| secret/private 增加 read gate | 默认不注入 LLM |
| memory quarantine | 可隔离污染记忆 |

## 13. 本项目后续使用方式

后续任何涉及 Agent 的开发，我建议直接按下面顺序走:

```text
1. 先写 State Impact Card
2. 确认 canonical object 与状态转换
3. 确认 Tool/Permission/Memory/Audit 是否受影响
4. 派发 AgentTeam 时写 Dispatch Card
5. 实现前先过 OODA-R 的 Observe/Orient/Decide
6. 实现后必须补 ToolCall/Permission/AgentRun/Memory 对应测试
7. Review 时检查状态漂移、权限扩大、读模型不一致
```

最关键的一条: 后续不要再新增孤立的 `status: string`。任何状态字段都必须先回答它属于哪个对象、哪个命名空间、谁是 canonical source、哪些 UI 只是 read model。
