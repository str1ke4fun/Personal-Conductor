## 1. 融合架构 · "清和 × Trae IDE" 统一信息架构

### 1.1 三份文档在产品中的真实角色

三份文档**不是平行方案**，而是同一产品的**三层结构**：

```
┌────────────────────────────────────────────────────────────┐
│  Layer 3 · IDE 容器（Trae 主体）                            │
│  ──────────────────────────────────                        │
│  · 对话区（4 面板优化：swimlane / cycle indicator / 右 panel）│
│  · 任务面板 · 图快照 · 状态条                                │
│  · 用户和 agent 在这里交互                                    │
└────────────────────────────────────────────────────────────┘
                            ▲
                            │  "她在看 IDE 里发生什么"
┌────────────────────────────────────────────────────────────┐
│  Layer 2 · 桌宠（清和）                                     │
│  ──────────────────────                                    │
│  · 文档 A 范围：皮肤 / 动效 / 装饰 / 互动                    │
│  · 文档 B 范围：情绪 / 关系 / 记忆 / 场景                    │
│  · 桌宠 = Layer 3 的"另一个视角 + 情感层"                      │
└────────────────────────────────────────────────────────────┘
                            ▲
                            │  "她的认知来自哪里"
┌────────────────────────────────────────────────────────────┐
│  Layer 1 · 数据底座                                         │
│  ──────────────────                                        │
│  · SQLite（25+ 表）· JSON state · events.ndjson             │
│  · embedding 链路（已修复 80%）· 迁移脚本（缺）· 摘要链路（待通）│
│  · 文档 B 范围：身份/关系/偏好/工作四层持久化                   │
└────────────────────────────────────────────────────────────┘
```

**关键洞察**：
- 文档 A 的 PNG 动态化是 Layer 2 的**皮肤**（最表层、最用户感知）
- 文档 B 的记忆 / 感知分层是 Layer 2 的**灵魂**（数据 + 通道）
- 上一轮的 4 面板优化是 Layer 3 的**容器**（IDE 本身）

如果三份文档独立推进，会出现"皮是新的、魂没接通、容器是旧的"——这正是当前状态。

### 1.2 信息流向

```
                ┌────────────────────────────────────────────┐
                │            用户（看 / 拖 / 点）              │
                └────┬──────────────────────┬────────────────┘
                     │                      │
                     ▼                      ▼
   ┌──────────────────────────┐   ┌──────────────────────────┐
   │  Layer 3 容器（IDE）       │   │  Layer 2 桌宠（清和）      │
   │  ─────────────           │   │  ─────────────           │
   │  对话区  · 任务面板       │◄──┤  皮肤：PNG + 装饰          │
   │  状态条  · 图快照         │   │  灵魂：mood + 关系 + 记忆  │
   │  用户对 agent 下指令      │   │  视角：感知 L1/L2/L3/L4   │
   └──────────┬───────────────┘   └──────────┬───────────────┘
              │                              │
              │  事件流（tool call / OODA）   │  渲染态（CSS 变量）
              ▼                              ▼
   ┌──────────────────────────────────────────────────────┐
   │  Layer 1 数据底座                                       │
   │  ─────────────                                         │
   │  SQLite（tasks/chat/tool/memory/affection/mood/scene）  │
   │  embedding 链路（已 80% 通）                             │
   │  Rust 事件 emit（avatar / expression / scene / initiative）│
   └──────────────────────────────────────────────────────┘
```

**核心抽象**：清和桌宠**不是一个独立 app**，她是 Layer 3 IDE 的"情感镜像层"。所有动效、所有情绪、所有记忆，都最终被 IDE 的容器承载。

## 5. 关键设计决策（融合中的取舍）

### 5.1 桌宠 = Layer 3 的"情感镜像层"，不是独立 app
原文档 A 把桌宠当作"独立前端窗口"设计，文档 B 把桌宠当作"感知层"。融合后：
- 桌宠的 PNG 皮肤是 Layer 3 任务流、对话流、状态流的**情绪反应**
- 桌宠不调用 LLM；她消费 Layer 1 的预计算数据（mood / affection / scene / initiative）
- 桌宠的"主动开口"是 Layer 1 initiative 引擎 emit `pet_self_bubble` 事件后的**前端承载**

### 5.2 三份方案的字体统一为一套
- 上一轮 4 面板的 Fraunces / Bricolage / JetBrains Mono 已经是一套完整 token
- 文档 A 的"清和署名"也用 Fraunces italic（与气泡 header 一致）
- 文档 B 的 L3 记忆簿用 Bricolage Grotesque + Fraunces 标题
- 不再引入第 4 套字体

### 5.3 状态色在三层之间复用
- `--state-running` / `--state-done` / `--state-error` / `--state-quiet` 是一组**统一 token**
- Layer 1（数据底座）的 state 字段、Layer 2（桌宠）的 mood、Layer 3（IDE）的任务状态共用
- 避免出现"桌宠 happy = 黄色，IDE done = 绿色"的色彩不一致

### 5.4 场景 tint 收敛为 7 类
- 文档 A §8 定义了 7 类（morning / afternoon / evening / night / music / work / relax）
- 这些 tint 是 Layer 2 桌宠 + Layer 3 桌面背景**共用**的色温
- 通过 CSS 变量 `--scene-tint-*` 一次定义，Layer 2/3 各自消费

### 5.5 L3 记忆簿是"用户管理面板"，L4 审计是"系统内部"
- 文档 B §3.4/§3.5 把 L3 / L4 分得很清
- L3 = 让用户控制（浏览 / 编辑 / 删除 / 暂停某类写入）
- L4 = 让用户查账（Secret 脱敏、审计导出）
- 融合后：两个 tab 在右 panel 同一 tab group，分清"我能改"和"我能看"

### 5.6 P0 重排：先 IDE 容器后桌宠
- 文档 A 的原始顺序是 A 清理 → B 动态化 → C 互动 → D 气泡 → E 场景 → F 收口 → G 主动开口
- 融合后：先把 Layer 3 IDE 容器（cycle indicator / swimlane / tab panel）做出，**因为这是用户每天 8 小时看的界面**
- 然后再补 Layer 2 桌宠
- 原因：IDE 容器是高频接触面，先把高频做好；桌宠是低频情感接触面，可以 P0 后半段做

## 9. 总结 · 这套融合设计的"一句话"

> **"清和"是 Trae IDE 的"情感镜像层"——她的皮肤（PNG 动态化）随 Layer 1 数据（mood / affection / scene / memory）变化，她的状态（mood / quiet / done）映射到 Layer 3 容器的统一状态色 token；她不调用 LLM，但 LLM 调用产生的所有事件都会通过 `--state-*` CSS 变量让她的光晕、表情、呼吸频率都跟着变。**

三份文档单独看是 3 套方案；融合后是**1 套产品的 3 个层次**。从现状出发，P0 用 10 个工作日可以交付"能用的、有活物感的、有真记忆的清和"。
