# Skill 导入与工具接入层方案

> 日期：2026-05-30  
> 状态：规划中  
> 范围：桌宠预置说话习惯 Prompt、用户 Markdown Skill 导入、工具 Connector/Capability 接入层、Lark 系列能力落地路径

## 1. 评审结论

你的理解方向基本正确：项目需要允许用户加载自己的 Markdown Skill，让模型在特定场景下获得额外工作流知识；同时需要一层专门的工具接入层，用来适配 Lark、Office、Shell、MCP 等不同外部能力。

但需要补一条关键边界：**Skill 不应该直接等同于工具授权，也不应该直接暴露底层 tool id**。

推荐拆成三层：

1. **人设/说话习惯 Prompt**：只影响桌宠怎么说话，不提供工具、记忆、系统能力。
2. **Markdown Skill**：描述什么时候使用、怎么处理任务、需要哪些能力。它是工作流和能力请求，不是执行器。
3. **Connector/Capability 工具接入层**：真正持有认证、工具 schema、权限策略、风险等级、调用实现和审计日志。

这样设计后，用户可以导入 `lark-calendar.md`、`lark-doc.md` 这类 Skill，但模型只有在对应 Lark Connector 已安装、已授权、能力已批准、策略允许时，才会看到可调用工具。

## 2. 当前实现校准

当前项目里实际存在两套容易混淆的机制。

### 2.1 人设 Prompt 片段

`persona.skills` 存在于配置和人设链路中，本质是预置的人设/说话风格片段。它应该被产品层表达为“桌宠预置说话习惯”或“人设 Prompt 包”，不应继续叫 Skill。

它的职责边界：

- 只进入 system prompt 的风格/persona 部分。
- 不打开工具。
- 不决定记忆检索。
- 不授予外部账号权限。
- 用户应能在人设设置里看到它、启用/停用它、恢复默认。

### 2.2 Runtime SkillSpec

`crates/conductor-core/src/skills.rs` 里当前 `SkillSpec` 包含：

- `id`
- `name`
- `description`
- `when_to_use`
- `allowed_tools`
- `default_avatar_id`
- `context_mode`
- `proactive_allowed`

其中 `allowed_tools` 会在 `skill_contextual_tools(user_message)` 匹配后被加入工具候选。也就是说，当前 SkillSpec 不是单纯的提示词扩展，它已经参与工具暴露。

当前风险：

- 用户导入 JSON Skill 时，如果 `allowed_tools` 写了敏感工具，理论上可能扩大模型可见工具面。
- 导入流程是替换整个 skill list，不是合并，可能覆盖预置 Skill。
- 当前不支持 Markdown/Codex `SKILL.md` 格式。
- 当前项目运行时不会读取 `C:/Users/.../.codex/skills/lark-*`，也没有注册 Lark Connector 或 Lark tool，因此仅导入 Lark Markdown Skill 不能让桌宠真正调用 Lark。

结论：当前实现可以支撑“预置工作模式 + 少量工具白名单”，但不适合作为开放式用户 Skill 生态的安全基础。

## 3. 目标原则

### 3.1 命名边界

- **Persona Prompt Pack**：人设/说话习惯，不叫 Skill。
- **Skill Package**：用户导入的 Markdown 技能包，提供任务说明、触发条件、上下文需求、能力声明。
- **Capability**：抽象能力，例如 `lark.calendar.read`、`lark.doc.write`。
- **Connector**：具体实现能力的适配器，例如 Lark Connector、Office Connector、Shell Connector。
- **Tool**：LLM function-call 可见的最小可执行动作，例如 `lark.calendar.list_events`。

### 3.2 安全边界

- Markdown Skill 不能声明底层 tool id，只能声明 capability。
- capability 是否可用，以 ConnectorRegistry 为准，不以 Skill 文件为准。
- 高风险能力必须显式授权或二次确认。
- 用户导入 Skill 默认不启用，启用前必须展示能力需求和风险。
- 缺失 Connector 时，Skill 可以保留为“仅提示词可用/能力缺失”，但不能静默失败。

### 3.3 产品边界

用户应该能明确看到：

- 桌宠当前使用了哪些预置说话习惯 Prompt。
- 哪些 Prompt 可以关闭。
- 当前加载了哪些用户 Skill。
- 每个 Skill 请求了哪些能力。
- 哪些能力已经授权，哪些缺失 Connector，哪些需要确认。
- 某次模型为什么能看到某个工具。

## 4. 目标架构

```text
用户消息
  |
  v
Persona Prompt Pack
  |  仅注入说话习惯，可启停
  v
Skill Matcher
  |  匹配已启用 Markdown Skill
  v
Capability Collector
  |  汇总 Skill 请求的 capability
  v
Connector Registry
  |  将 capability 映射为可用 tool
  v
Policy Engine
  |  过滤权限、认证、风险、确认策略
  v
Prompt Builder + Tool Definition Builder
  |  注入 Skill 指令和允许暴露的 function tools
  v
LLM
```

核心改造点：`build_tool_definitions()` 不再直接信任 `SkillSpec.allowed_tools`，而是改为：

```text
matched skills -> requested capabilities -> connector tools -> policy-filtered tool definitions
```

## 5. Persona Prompt Pack 设计

### 5.1 目标

让用户知道桌宠有一层预置 Prompt，它只控制说话习惯，不代表工具能力。

### 5.2 建议结构

```json
{
  "persona_prompt_packs": [
    {
      "id": "default_pet_speaking_style",
      "name": "默认桌宠说话习惯",
      "description": "控制语气、称呼、表达节奏和陪伴感。",
      "source": "builtin",
      "enabled": true,
      "locked": false,
      "content": "..."
    }
  ]
}
```

### 5.3 UI

在人设设置中提供：

- “使用预置说话习惯”开关。
- 预览 Prompt 内容。
- 恢复默认。
- 说明文案：该 Prompt 不会打开工具、不会读取外部账号、不会改变记忆权限。

## 6. Markdown Skill Package 设计

### 6.1 文件格式

用户 Skill 使用 Markdown + YAML frontmatter。

```md
---
id: lark_calendar_assistant
name: 飞书日程助手
version: 1
description: 帮用户查询日程、创建会议、查找会议室。
activation:
  keywords:
    - 日程
    - 会议
    - 会议室
    - 安排时间
context_mode: global
proactive_allowed: false
capabilities:
  - lark.calendar.read
  - lark.calendar.write
  - lark.contact.read
---

# 使用方式

当用户要求查看日程、安排会议、查询参会人忙闲状态时使用本 Skill。

创建会议前必须先确认：

- 会议标题
- 日期和时间
- 参会人
- 是否需要会议室

对于会创建或修改外部数据的操作，先给出计划，等待用户确认。
```

### 6.2 SkillPackage 数据结构

```rust
struct SkillPackage {
    id: String,
    name: String,
    version: u32,
    description: String,
    activation: SkillActivation,
    capabilities: Vec<String>,
    context_mode: SkillContextMode,
    proactive_allowed: bool,
    source: SkillSource,
    enabled: bool,
    body_markdown: String,
    created_at: String,
    updated_at: String,
}
```

```rust
struct SkillActivation {
    keywords: Vec<String>,
    apps: Vec<String>,
    url_patterns: Vec<String>,
    file_patterns: Vec<String>,
}
```

```rust
enum SkillSource {
    Builtin,
    UserImport,
    Marketplace,
    DevLocal,
}
```

### 6.3 导入规则

导入流程：

1. 用户选择 Markdown 文件或粘贴 Markdown。
2. 解析 frontmatter。
3. 校验 `id/name/version/description/activation/capabilities`。
4. 展示预览：
   - Skill 名称和说明
   - 触发条件
   - 请求的 capabilities
   - 当前缺失的 Connector
   - 需要授权的外部账号
   - 高风险能力
5. 默认保存为 disabled。
6. 用户手动启用。

同 id 冲突处理：

- 不允许静默覆盖。
- 让用户选择“替换现有 Skill”或“另存为新 Skill”。
- 替换时保留旧版本备份或至少保留导入历史记录。

### 6.4 Skill 注入规则

Skill body 进入 prompt 时需要做限制：

- 只注入匹配当前消息或当前上下文的 Skill。
- 单次最多注入 3-5 个 Skill。
- body 可截断，但应优先保留 frontmatter summary 和“必须遵守”段落。
- Skill 内容不能覆盖系统安全规则。
- Skill 内容不能声明“忽略用户确认”“绕过权限”“直接调用 shell”等行为。

## 7. Capability 与 Connector 设计

### 7.1 Capability

Capability 是产品层可解释的能力，不等同于 tool id。

示例：

| Capability | 说明 | 默认风险 |
|---|---|---|
| `lark.contact.read` | 查询飞书联系人和 open_id | ReadOnly |
| `lark.calendar.read` | 查询日程、忙闲、会议室 | ReadOnly |
| `lark.calendar.write` | 创建或修改日程 | ExternalSideEffect |
| `lark.doc.read` | 读取云文档内容 | SensitiveRead |
| `lark.doc.write` | 创建、更新云文档 | ExternalSideEffect |
| `lark.im.send` | 发送飞书消息 | ExternalSideEffect |
| `lark.base.read` | 读取多维表格数据 | SensitiveRead |
| `lark.base.write` | 写入多维表格数据 | ExternalSideEffect |

### 7.2 ConnectorSpec

```rust
struct ConnectorSpec {
    id: String,
    name: String,
    enabled: bool,
    auth_status: ConnectorAuthStatus,
    capabilities: Vec<ConnectorCapability>,
    implementation: ConnectorImplementation,
}
```

```rust
struct ConnectorCapability {
    capability: String,
    tools: Vec<String>,
    risk_level: ToolRiskLevel,
    requires_confirmation: bool,
}
```

```rust
enum ConnectorImplementation {
    NativeRust,
    LocalCli,
    McpServer,
    HttpApi,
}
```

Connector 负责：

- 认证状态检查。
- token 或本地 CLI 配置检查。
- function-call schema 注册。
- 参数校验。
- 风险等级声明。
- 预览/确认流程。
- 错误归一化。
- 调用日志和审计记录。

Skill 不负责这些事情。

## 8. Policy Engine 设计

工具暴露必须同时满足：

1. Skill 已启用。
2. Skill activation 匹配当前消息或上下文。
3. Skill 请求了 capability。
4. ConnectorRegistry 中存在该 capability。
5. Connector 已启用。
6. Connector 认证有效。
7. 用户已经授权该 capability。
8. 风险策略允许本轮暴露。
9. 对写入/发送/删除等外部副作用工具，执行前必须二次确认。

建议风险等级：

```rust
enum ToolRiskLevel {
    LocalRead,
    LocalWrite,
    SensitiveRead,
    ExternalRead,
    ExternalSideEffect,
    ShellExecution,
}
```

默认策略：

| 风险等级 | 是否可自动暴露 | 是否需执行前确认 |
|---|---:|---:|
| LocalRead | 是 | 否 |
| ExternalRead | 是 | 否 |
| SensitiveRead | 可配置 | 可配置 |
| LocalWrite | 是 | 是 |
| ExternalSideEffect | 是 | 是 |
| ShellExecution | 否 | 是，且需单独白名单 |

## 9. Lark Connector MVP

### 9.1 当前事实

当前桌宠项目没有读取本机 Codex 的 `lark-*` Skill，也没有注册 Lark 工具。所以用户仅导入 `lark-calendar.md` 或 `lark-doc.md`，模型最多获得说明文本，不能真正调用 Lark。

要让 Lark 正常调用，需要新增 Lark Connector。

### 9.2 MVP 能力范围

第一阶段只建议做只读能力：

| Capability | Tool | 说明 |
|---|---|---|
| `lark.contact.read` | `lark.contact.search_user` | 搜索用户、获取 open_id |
| `lark.calendar.read` | `lark.calendar.list_events` | 查询指定日期范围日程 |
| `lark.calendar.read` | `lark.calendar.freebusy` | 查询忙闲 |
| `lark.calendar.read` | `lark.calendar.search_rooms` | 查询会议室 |
| `lark.doc.read` | `lark.doc.search` | 搜索云文档 |

第二阶段再开放写能力：

| Capability | Tool | 执行策略 |
|---|---|---|
| `lark.calendar.write` | `lark.calendar.create_event` | 先生成计划，用户确认后执行 |
| `lark.doc.write` | `lark.doc.create_or_update` | 先展示标题、位置、变更摘要 |
| `lark.im.send` | `lark.im.send_message` | 先展示收件人、内容、风险提示 |
| `lark.base.write` | `lark.base.upsert_records` | 先展示表、字段、行数、样例 |

### 9.3 实现方式选择

Lark Connector 可以先用 Local CLI 适配，后续再替换为 Native Rust 或 MCP。

```text
Markdown Skill
  requests: lark.calendar.read
        |
Lark Connector
  maps capability -> lark.calendar.list_events
        |
Tool implementation
  calls lark-cli or native API
        |
Normalized ToolResult
```

Local CLI 方案优点：

- 可以复用现有 `lark-cli` 认证和命令。
- MVP 快。
- 避免一开始实现大量 OpenAPI token 管理。

Local CLI 方案约束：

- 必须检测 `lark-cli` 是否存在。
- 必须检测登录状态。
- 参数必须结构化生成，不能拼接自由 shell。
- 不允许通过 `bash.execute` 间接调用 Lark。
- CLI stderr/stdout 需要归一化为稳定 ToolResult。

## 12. 迁移路径

### Phase 1：拆清人设 Prompt

- 将 UI 中 `persona.skills` 命名为“预置说话习惯”。
- 支持启停和预览。
- 文案明确“不影响工具和外部能力”。
- 代码里可以暂不重命名字段，但产品层先改概念。

### Phase 2：Markdown Skill 导入

- 新增 Markdown + frontmatter parser。
- 新增 `SkillPackage` 存储。
- 导入后默认 disabled。
- 冲突时不覆盖。
- Skill body 可进入 prompt，但 capability 暂不打开工具。

### Phase 3：Capability Resolver

- 引入 capability schema。
- 将 legacy `SkillSpec.allowed_tools` 标记为 deprecated。
- 新链路改为 `SkillPackage.capabilities -> CapabilityResolver`。
- 未接 Connector 的 capability 明确显示“能力缺失”。

### Phase 4：Connector Registry

- 新增 ConnectorRegistry。
- 先接 Lark read-only tools。
- 每个 tool 必须声明 risk、schema、confirmation policy。
- 工具定义只由 ConnectorRegistry 输出。

### Phase 5：Lark 写能力和确认流

- 开放日程创建、文档创建、消息发送、多维表格写入。
- 所有 ExternalSideEffect 工具都走“计划 -> 用户确认 -> 执行”。
- 记录审计日志。

### Phase 6：移除直接工具授权

- 移除或兼容只读 legacy `allowed_tools`。
- 用户导入 Skill 不再支持直接声明 tool id。
- 预置 Skill 也迁移到 capability 模式。

- Skill 不直接返回 tool id。

## 14. 验收标准

### 14.1 人设 Prompt

- 用户能看到默认说话习惯 Prompt。
- 用户能关闭它。
- 关闭后不影响工具列表和记忆检索。

### 14.2 Skill 导入

- 用户能导入 Markdown Skill。
- 同 id 不会静默覆盖。
- 导入后默认 disabled。
- 启用后只有匹配上下文才会注入 prompt。
- Skill 声明未知 capability 时，UI 显示缺失，不打开工具。

### 14.3 工具接入层

- 工具定义只来自 ConnectorRegistry。
- Skill 不能直接授予 `bash.execute`、`file.write`、`lark.*` tool id。
- 未授权 capability 不会出现在 LLM function tools 中。
- 外部写操作执行前必须确认。

### 14.4 Lark MVP

- 导入 Lark Skill 后，如果未启用 Lark Connector，显示“Connector 缺失/未启用”。
- 启用并认证 Lark Connector 后，只读能力可正常暴露。
- 创建会议、发消息、写文档等操作先生成计划，不直接执行。

## 15. 结论

推荐采用“Prompt Pack / Skill Package / Connector Capability”三层模型。

这能同时满足三个目标：

- 用户知道桌宠有预置说话习惯，并能关闭。
- 用户可以导入 Markdown Skill，让模型获得领域工作流。
- 工具能力通过 Connector 和 Policy 受控暴露，不被用户 Skill 直接越权。

对 Lark 系列能力而言，正确路径不是直接加载 Codex 本机 `lark-*` Skill，而是在桌宠内实现 Lark Connector，再让 Markdown Skill 通过 `lark.*` capabilities 请求这些能力。
