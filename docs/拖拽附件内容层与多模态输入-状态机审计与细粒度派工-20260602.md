## 1. 需求归一化

### 1.1 原始需求

```text
希望能直接从 Windows 资源管理器拖 文档 / 代码 / 图片 进入 prompt。
这一层至少能解析文件类型，判断是否可以上传给 LLM。
重点是打通能力，不是让模型猜绝对路径。
用户拖入文件即视为本次上传授权。
允许用户把文件拖到对话草稿文字的任意位置，视为在该处引用该文件。
```

### 1.2 需求类型

- `Feature`
- `Architecture`
- `Capability plumbing`

不是：

- 纯 UI 优化
- 单点工具增强
- 局部重构

### 1.3 五元组

| 项 | 内容 |
| --- | --- |
| `Actor` | User / ChatComposer / Tauri Shell / Conductor Chat Runtime / LLM Provider |
| `Object` | `ComposerDraft`、`AssetDraft`、`InlineReferenceDraft`、`MessageSubmission`、`ProviderAttachmentArtifact` |
| `Current State` | 纯文本对话，消息载体是 `message: string` |
| `Trigger` | 用户拖入或选择本地文件，并发起发送 |
| `Target State` | 对话支持“文本 + 有序文件引用部件”联合发送，模型真正获得图像/文档/代码输入能力，且文件可出现在草稿任意位置 |

### 1.4 非目标

- 不做本地图像理解算法
- 不把拖拽变成任意宿主机路径放权
- 不把 Office 深度解析作为 MVP 前置
- 不在 v1 引入“附件自动写回文件”

## 2. L0 状态账本

这一节是当前方案最缺的部分。没有它，就会出现前端、后端、Provider 编译层各自发明状态的问题。

## 2.1 Object A: `ComposerDraft`

这是用户当前正在编辑的一次消息草稿。

### 状态

| 状态 | 含义 |
| --- | --- |
| `idle` | 无文本、无附件草稿 |
| `editing_text` | 有文本输入，尚未带附件 |
| `drag_hover` | 用户正在窗口或 panel 上方拖拽文件 |
| `targeting_anchor` | 系统正在判断文件将插入草稿中的哪个位置 |
| `inspecting_assets` | 系统正在检查拖入文件 |
| `draft_ready` | 文本和/或附件已可发送 |
| `sending` | 草稿已冻结并进入提交链路 |
| `send_failed` | 发送失败，草稿可回退 |
| `sent` | 本次草稿已成功提交 |

### 合法迁移

| 迁移 ID | 从 | 触发 | 到 |
| --- | --- | --- | --- |
| `CD-T01` | `idle` | 用户输入文本 | `editing_text` |
| `CD-T02` | `idle` / `editing_text` | 用户拖拽文件进入 hover 区 | `drag_hover` |
| `CD-T03` | `drag_hover` | hover 落在草稿文本内部 | `targeting_anchor` |
| `CD-T04` | `targeting_anchor` | drop 发生且锚点已解析 | `inspecting_assets` |
| `CD-T05` | `inspecting_assets` | 至少有一个可接纳 asset | `draft_ready` |
| `CD-T06` | `editing_text` | 用户通过选择器加入附件成功 | `draft_ready` |
| `CD-T07` | `draft_ready` | 用户点击发送 | `sending` |
| `CD-T08` | `sending` | 后端成功返回 | `sent` |
| `CD-T09` | `sending` | 后端返回错误 | `send_failed` |
| `CD-T10` | `send_failed` | 用户继续编辑 | `draft_ready` / `editing_text` |

### Guard

- `CD-T04`: drop 发生时必须已解析出一个合法的插入锚点
- `CD-T05`: 至少一个 `AssetDraft` 不处于 `unsupported`
- `CD-T07`: 文本非空或存在至少一个可发送 content part
- `CD-T07`: 当前无 in-flight request

### Side Effect

- `CD-T04`: 生成一组 `AssetDraft`，并记录 drop anchor
- `CD-T07`: 冻结当前草稿，生成 `MessageSubmission`
- `CD-T09`: 保留用户文本和附件，不丢稿

### Invariant

- `sending` 期间，不能继续修改本次已冻结的发送载荷
- `sent` 后，上一轮附件草稿必须清空
- `drag_hover` 只是视图态，不应直接写入消息历史
- 草稿的 canonical 形态必须是“有序节点流”，而不是 `textarea string + unordered attachments`

### Illegal Transition

- `idle -> sending`
- `drag_hover -> sending`
- `targeting_anchor -> sending`
- `inspecting_assets -> sent`
- `sending -> editing_text`

## 2.2 Object B: `AssetDraft`

这是单个拖入文件在“进入消息前”的草稿对象。

### 状态

| 状态 | 含义 |
| --- | --- |
| `captured` | 已接收到本地路径 |
| `inspecting` | 正在读取 metadata / MIME / workspace 归属 |
| `classified` | 已识别 kind |
| `routed_workspace_ref` | 决定作为 `workspace_file_ref` |
| `routed_attachment` | 决定作为 `attachment` |
| `unsupported` | 当前版本不接纳 |
| `removed` | 用户从草稿区移除 |

### 合法迁移

| 迁移 ID | 从 | 触发 | 到 |
| --- | --- | --- | --- |
| `AD-T01` | `captured` | 开始 inspect | `inspecting` |
| `AD-T02` | `inspecting` | 识别成功 | `classified` |
| `AD-T03` | `classified` | 路由器选择 workspace ref | `routed_workspace_ref` |
| `AD-T04` | `classified` | 路由器选择 direct attachment | `routed_attachment` |
| `AD-T05` | `inspecting` / `classified` | 文件不支持或超限 | `unsupported` |
| `AD-T06` | 任一非终态 | 用户移除 | `removed` |

### Guard

- `AD-T03`: 文件在当前 workspace 内，且类型属于文本/代码可引用集合
- `AD-T04`: 文件类型允许上传给 provider 或允许文本提取 fallback
- `AD-T05`: 类型黑名单、尺寸超限、读取失败

### Side Effect

- `AD-T02`: 写入 `mimeType`、`sizeBytes`、`workspaceRelation`
- `AD-T03` / `AD-T04`: 生成对应 `UserContentPart`
- `AD-T05`: 记录 reject reason

### Invariant

- 一个 `AssetDraft` 最多只能映射成一个最终路由
- `removed` asset 不能进入本次发送载荷
- `unsupported` 不能伪装成 `routed_attachment`

### Illegal Transition

- `captured -> routed_attachment`
- `classified -> captured`
- `unsupported -> routed_workspace_ref`

## 2.3 Object C: `InlineReferenceDraft`

这是“文件在草稿中的那个可见引用 token”。
它不等于原文件，也不等于最终发给模型的 part；它是草稿里的有序节点。

### 状态

| 状态 | 含义 |
| --- | --- |
| `anchor_captured` | 已经解析出插入位置 |
| `binding_asset` | 正在把某个 `AssetDraft` 绑定到该位置 |
| `bound_workspace_ref` | 该位置最终绑定为 `workspace_file_ref` |
| `bound_attachment` | 该位置最终绑定为 `attachment` |
| `unsupported` | 该位置绑定失败，只能展示 reject token 或移除 |
| `removed` | 用户删除了这个引用 token |

### 合法迁移

| 迁移 ID | 从 | 触发 | 到 |
| --- | --- | --- | --- |
| `IR-T01` | `anchor_captured` | 开始绑定 asset | `binding_asset` |
| `IR-T02` | `binding_asset` | 路由为 workspace ref | `bound_workspace_ref` |
| `IR-T03` | `binding_asset` | 路由为 attachment | `bound_attachment` |
| `IR-T04` | `binding_asset` | 文件不支持或绑定失败 | `unsupported` |
| `IR-T05` | 任一非终态 | 用户删除 token | `removed` |

### Guard

- `IR-T01`: 必须已有确定的草稿锚点
- `IR-T02` / `IR-T03`: 只能绑定一个已 `classified` 的 `AssetDraft`

### Side Effect

- `IR-T02` / `IR-T03`: 在草稿节点流中插入一个可见引用 token
- `IR-T04`: 在原位置显示 reject 状态或提示，并允许移除

### Invariant

- 一个 `InlineReferenceDraft` 只能占据草稿中的一个位置
- 一个位置上的 token 不能同时绑定成 `workspace_ref` 和 `attachment`
- 删除 token 只移除该处引用，不应破坏相邻文本节点顺序

### Illegal Transition

- `anchor_captured -> bound_attachment`
- `unsupported -> bound_workspace_ref`
- `removed -> binding_asset`

## 2.4 Object D: `MessageSubmission`

这是一次真正提交给后端和 provider 的发送对象。

### 状态

| 状态 | 含义 |
| --- | --- |
| `draft` | 还在前端草稿态 |
| `normalized` | 已把文本与 `UserContentPart[]` 归一化 |
| `compiled` | 已编译成 provider request |
| `in_flight` | 正在发往 provider |
| `succeeded` | provider 返回成功 |
| `failed` | 提交失败 |

### 合法迁移

| 迁移 ID | 从 | 触发 | 到 |
| --- | --- | --- | --- |
| `MS-T01` | `draft` | 点击发送 | `normalized` |
| `MS-T02` | `normalized` | provider compiler 成功 | `compiled` |
| `MS-T03` | `compiled` | 发起模型调用 | `in_flight` |
| `MS-T04` | `in_flight` | 模型返回成功 | `succeeded` |
| `MS-T05` | `normalized` / `compiled` / `in_flight` | 任一阶段报错 | `failed` |

### Guard

- `MS-T01`: 至少存在一个 `text` 或可发送 part
- `MS-T02`: provider 对所有 part 都有编译策略
- `MS-T03`: 若存在 attachment bytes，必须已准备完成

### Side Effect

- `MS-T01`: 冻结草稿快照
- `MS-T02`: 生成 provider-specific request body
- `MS-T04`: 写入消息历史与附件元数据

### Invariant

- 发送请求是不可变快照
- 编译后的 request 必须可从相同输入重复生成
- `failed` 不能导致历史里出现伪成功消息

### Illegal Transition

- `draft -> compiled`
- `compiled -> normalized`
- `failed -> succeeded`

## 2.5 Object E: `ProviderAttachmentArtifact`

这是 direct attachment 在 provider 侧的中间产物。

### 状态

| 状态 | 含义 |
| --- | --- |
| `not_needed` | 该 part 不需要 provider attachment，例如纯 `workspace_ref` |
| `preparing` | 正在读 bytes / 提取文本 / 编码 |
| `ready_inline` | 已可内联到 request |
| `uploaded_ref` | 已上传为 provider file，拿到 file id |
| `prepare_failed` | 准备失败 |

### 合法迁移

| 迁移 ID | 从 | 触发 | 到 |
| --- | --- | --- | --- |
| `PA-T01` | `not_needed` | 保持 | `not_needed` |
| `PA-T02` | 初始 | 该 part 需要附件处理 | `preparing` |
| `PA-T03` | `preparing` | inline bytes 准备成功 | `ready_inline` |
| `PA-T04` | `preparing` | provider file 上传成功 | `uploaded_ref` |
| `PA-T05` | `preparing` | 读取/转码/上传失败 | `prepare_failed` |

### Invariant

- `workspace_ref` 不得进入 `uploaded_ref`
- 同一 artifact 不能同时处于 `ready_inline` 和 `uploaded_ref`

## 3. MVP 子图

原方案有 Phase，但没有“闭环 MVP 状态图”。补齐如下。

### 3.1 MVP-In

必须覆盖：

```text
ComposerDraft.idle
-> drag_hover
-> targeting_anchor
-> inspecting_assets
-> draft_ready
-> sending
-> sent
```

### 3.2 MVP-Out

必须覆盖失败分支：

```text
AssetDraft.inspecting
-> unsupported

InlineReferenceDraft.binding_asset
-> unsupported

MessageSubmission.in_flight
-> failed
```

### 3.3 MVP 范围

MVP 内：

- 手动添加文件
- Windows 拖拽文件
- 图片 direct attachment
- workspace 内文本/代码 `workspace_ref`
- workspace 外文本/代码 `direct_attachment` 或 `extracted_text`
- PDF direct attachment 或 fallback
- 用户移除附件
- 发送失败不丢稿

MVP 外：

- 批量上百文件拖入
- 压缩包递归展开
- Office 深度结构化解析
- 图片批注
- 历史消息附件重编辑
- 多 provider 缓存复用优化

## 5. 每条原子需求的 acceptance

## 5.1 `REQ-001` 接收窗口级文件拖入

Acceptance:

- 拖入文件到 chat panel 时出现 hover 态
- drop 后进入 inspect，而不是直接发消息
- 仅拖拽视图高亮，不污染消息历史

Illegal:

- 仅凭 hover 就创建历史消息
- drop 后直接调用 provider

## 5.2 `REQ-002` 解析拖拽落点为草稿内插入锚点

Acceptance:

- 当文件拖到正文中部时，系统能解析出稳定锚点，例如 `nodeId + offset` 或等价定位
- 支持三类合法落点：文本节点中间、文本节点边界、已有引用 token 前后
- drop 到正文尾部时，显式解析为“尾部锚点”，而不是隐式兜底
- 锚点未解析成功前，不得进入 `inspecting_assets`

Illegal:

- 无法解析锚点时，静默把文件追加到消息尾部
- rerender 后锚点漂移，导致文件插入到错误文本位置

## 5.3 `REQ-003` 检查单个拖入文件 metadata 与 workspace 归属

Acceptance:

- 能生成 `DroppedAsset` 最小元数据：`absolutePath / displayName / extension / mimeType / sizeBytes`
- 能判断 `in_workspace / outside_workspace / unknown`，并识别 `text / code / image / pdf / office / binary`
- inspect 失败时写出明确 `rejectReason`，并进入 `unsupported`
- metadata 与归属判断完成前，不进入路由决策

Illegal:

- 未 inspect 完成就进入路由
- workspace 外文件被误判为 workspace 内
- metadata 读取失败后被静默吞掉

## 5.4 `REQ-004` 路由策略

Acceptance:

- workspace 内文本/代码默认路由到 `workspace_file_ref`
- 图片默认路由到 `attachment`
- 外部文本/PDF/Office 按策略进入 `attachment` 或 `extracted_text` fallback
- 黑名单二进制被标记 `unsupported`，并可在 UI 上看到当前路由结果
- 多文件拖入时，每个文件独立路由，不要求同一策略

Illegal:

- 图片路由成 `workspace_file_ref`
- 所有文件被强制套用同一条路由
- 不支持文件被静默吞掉

## 5.5 `REQ-005` inline reference token

Acceptance:

- 拖到正文中部时，会在该处插入可见 token，例如 `前文 [a.txt] 后文`
- 删除 token 只移除该处引用，不破坏左右文本与其他 token 顺序
- 不支持文件在原落点显示 reject token 或 inline 错误提示，而不是偷挂到尾部
- 同一条草稿里可同时存在尾部附件与正文内联引用

Illegal:

- 中部 drop 被静默降级成尾部附件
- 删除 token 后相邻文本被错误合并或顺序错乱

## 5.6 `REQ-006` 归一化为 `UserContentPart[]`

Acceptance:

- `ComposerDraftNode[]` 能归一化成有序 `UserContentPart[]`
- 对 `text-left + file-ref + text-right`，归一化结果保持同顺序
- 相同草稿快照必须生成相同 `UserContentPart[]`
- 纯文本草稿仍可映射到旧链路或兼容新链路，不回归

Illegal:

- 归一化阶段按类型重排 part
- 把正文中的文件引用统一压成尾部附件

## 5.7 `REQ-007` IPC 与 Tauri 命令升级

Acceptance:

- FE -> Tauri 的发送 DTO 能携带有序 `user_content_parts` 或等价快照，而不是 `message + unordered attachments[]`
- 文件拖拽接入链路与发送链路解耦：前者负责拿到原始文件，后者负责提交有序内容
- 旧 `send_chat_message_v2(message: string)` 仍可兼容纯文本
- 窗口级 drop 若只能提供文件列表，则编辑器层仍需补齐 anchor 解析能力

Illegal:

- IPC 丢失顺序信息，只传附件集合
- Tauri 只暴露窗口级 drop，却没有给 FE 留出正文锚点绑定能力

## 5.8 `REQ-008 ~ REQ-010` Provider 编译

Acceptance:

- OpenAI / Anthropic 至少各支持：
  - `text`
  - `workspace_file_ref`
  - `image attachment`
  - `text/code/pdf attachment`
- `LlmInputMessage.parts` 保持 `UserContentPart[]` 的顺序，不因 provider 差异重排
- provider 不支持某类文件时，有可解释 fallback 或 `prepare_failed`
- 绝对路径不直接出现在模型主语义输入里

Illegal:

- provider 编译器把宿主机绝对路径原样注入 prompt
- `workspace_ref` 被当成 attachment 乱传
- 编译器把正文里的所有附件统一追加到消息尾部

## 5.9 `REQ-011` 混合发送

Acceptance:

- 支持 `左文本 + 内联文件引用 + 右文本` 在一次发送里成功进入 provider
- 多个 inline file ref 与尾部附件可混合存在，且顺序稳定
- 没有纯文本时，只要存在合法内容 part，也能发送
- 成功消息在历史渲染和回放时能重建原始相对顺序

Illegal:

- 内联引用在发送时被折叠成一句摘要文本
- 成功历史里丢失正文与文件引用的相对位置

## 5.10 `REQ-012` 附件持久化

Acceptance:

- 每条含附件消息都能持久化附件元数据、路由结果和顺序信息
- 历史中能区分 `workspace ref` 和 `direct attachment`
- 若后续支持消息回放/重渲染，能恢复正文内文件引用的相对顺序

Illegal:

- 附件信息只存在临时前端状态里
- 持久化时丢失顺序，只剩文件名列表

## 5.11 `REQ-013` 失败回退与前端恢复草稿

Acceptance:

- provider 或 compiler 失败后，前端能恢复同一份 `ComposerDraftNode[]` 快照
- 用户可直接重试，或先删除某个失败 token / 附件后再重发
- 失败不会提前写入伪成功消息，也不会制造重复 token

Illegal:

- 失败即清空草稿
- 失败后把正文内联引用退化成尾部附件
- 重试时重复插入同一文件引用

## 8. 非法迁移清单

这部分必须写出来，否则实现阶段最容易漂移。

| 对象 | 非法迁移 | 风险 |
| --- | --- | --- |
| `ComposerDraft` | `idle -> sending` | 发送空消息 |
| `ComposerDraft` | `drag_hover -> sent` | 绕过 inspect / normalize / compile |
| `ComposerDraft` | `targeting_anchor -> draft_ready` | 锚点未完成就生成草稿内容，位置语义失真 |
| `ComposerDraft` | `drop@middle -> append@tail` | 用户语义是“此处引用”，系统却偷改成“尾部附件” |
| `AssetDraft` | `captured -> routed_attachment` | MIME / 归属未判断 |
| `AssetDraft` | `unsupported -> routed_workspace_ref` | 不支持文件误入发送 |
| `InlineReferenceDraft` | `anchor_captured -> bound_*` | 跳过 binding / route，token 与 asset 状态失配 |
| `InlineReferenceDraft` | `removed -> bound_*` | 已删除 token 死灰复燃，污染草稿 |
| `MessageSubmission` | `draft -> in_flight` | 跳过 normalize / compile |
| `MessageSubmission` | `normalized(ordered) -> compiled(reordered)` | provider 成功但语义顺序被破坏 |
| `MessageSubmission` | `failed -> succeeded` | 伪成功历史 |
| `ProviderAttachmentArtifact` | `not_needed -> uploaded_ref` | workspace ref 被错误上传 |

## 9. AgentTeam 派工方案

这是 `LC-07 Ready For Dev` 的最小组织方式。

## 9.1 角色

| Agent / 角色 | 职责 | 不负责 |
| --- | --- | --- |
| `Architect` | 固化 `ComposerDraftNode`、`UserContentPart`、`LlmInputPart`、状态账本 | 不直接写 UI 样式 |
| `Frontend Executor` | tokenized composer、锚点解析、附件 UI、失败回退 | 不做 provider 编译 |
| `Tauri Executor` | 文件 drop、命令 DTO 边界、原始 asset 透传 | 不做业务路由规则 |
| `Core Executor` | `asset_inspector`、`asset_loader`、有序归一化、send 主链路 | 不做 provider 差异 UI 处理 |
| `LLM Executor` | provider compiler、多 part message、顺序保持 | 不改前端状态管理 |
| `QA / Reviewer` | 非法迁移、回归矩阵、历史兼容 | 不扩张功能边界 |

## 9.2 OODA-R 顺序

```text
Observe:
Architect + QA 复核状态账本和 REQ

Orient:
Architect 固化 DTO、对象边界、禁止项

Decide:
拆成 FE / Tauri / Core / LLM 四个并行面

Act:
四组并行实现，QA 跟进单测与手工回归用例

Review:
先做 compiler review，再做 end-to-end review，最后做回归 review
```

## 13. 最终派工建议

当前最合理的派工方式不是“按文件分”，而是“按状态迁移分”。

建议以四张主工单推进：

1. `WORK-01 草稿对象与锚点协议冻结`
   - 覆盖 `REQ-002` `REQ-005` `REQ-006`
   - Owner: `Architect + Frontend Executor`
2. `WORK-02 有序发送协议与 provider compiler`
   - 覆盖 `REQ-007 ~ REQ-011`
   - Owner: `Architect + Core Executor + LLM Executor`
3. `WORK-03 文件接入与路由判定`
   - 覆盖 `REQ-001` `REQ-003` `REQ-004`
   - Owner: `Tauri Executor + Core Executor`
4. `WORK-04 失败回退、持久化与回归`
   - 覆盖 `REQ-012` `REQ-013`
   - Owner: `Frontend Executor + Core Executor + QA`

如果要更激进并行，也必须先完成 `WORK-01`。原因很直接：没有 `ComposerDraftNode + anchor` 的冻结定义，FE 做不出“正文中部引用”，Core/LLM 也无法知道顺序语义该如何保留。
