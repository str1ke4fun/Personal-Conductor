# Agent 场景状态机现状差距与整改问题

日期：2026-05-31

定位：基于用户侧状态机和 LLM/工具侧状态机的现状审查 / 下一步整改输入

关联文档：
- `docs/用户侧Agent使用场景与状态机模型-20260531.md`
- `docs/LLM工具与多Agent执行状态机模型-20260531.md`
- `docs/多Agent共享工作区与自治Goal调度方案-20260530.md`
- `docs/运行时可观察性与Agent链路收口-20260531.md`

## 1. 总体判断

当前系统已经有很多正确的底层对象，但用户侧状态机还没有贯通。

已经具备：

- chat v2 的流式回复、thinking-update、tool-execution-update。
- `ToolCall`、`CommandRun`、`AgentRun`、hook task 的记录。
- `GoalRun`、`GoalCycle`、`AgentTask`、`Heartbeat`、`WorkLease`、`AgentMessage`。
- `GoalOrchestrator` 的 OODA 骨架。
- `RuntimeApiServer` 的 HTTP/SSE endpoint。
- `AgentTeam` 生命周期、成员、mailbox。
- 第一轮可观察性修复：`agent_runs_changed`、active AgentRun、active hook task、pending hook review。

没有贯通：

- 前端主工作台没有显式短任务 / 长任务开关。
- `planOnly/capabilityMode` 只是前端状态，没有进入后端工具策略。
- Goal/Agent UI 组件存在但没有挂载到主界面。
- Runtime API server 未看到桌面端启动入口。
- 渐进式工具暴露没有实现；当前是后端预先构造工具列表。
- `agent.start` 默认暴露给短任务/普通聊天子场景，存在静默启动后台 Agent 的风险。
- AgentTeam 是协作容器，但 UI/策略上还没强制绑定真实 executor。

## 2. 场景完成度矩阵

| 场景 | 当前能否完成任务 | 当前能否让用户正确感知 | 主要差距 |
|---|---|---|---|
| 普通 LLM 聊天 | 基本可以 | 部分可以 | 短任务可能拿到过强工具；模式/权限未传后端 |
| 聊天记录/任务记录 | 部分可以 | 部分可以 | 结果投影分散在 chat/task/memory |
| 文档读取/写作 | 部分可以 | 部分可以 | Office 工具存在；Lark connector 未看到运行时注册；写入预览和结果投影不统一 |
| PPTX/脚本处理 | 部分可以 | 弱 | shell/command_run 有状态，但短任务内文档子场景未定义；脚本产物缺统一 artifact 入口 |
| 代码分析/修改 | 部分可以 | 部分可以 | 文件/命令/工具轨迹可见，但短任务内代码子场景、计划审批、写范围边界不清 |
| 后台 `claude -p` | 可以通过 `agent.start` | 已改善但未完备 | 短任务可触发；output tail/结果归档仍需统一 |
| 一次性 `subagent.claude_p` | 可以 | 弱 | 不走完整 AgentRun，可观察性差 |
| AgentTeam 协作 | 只能建容器和 mailbox | 弱 | 不会自动 spawn executor；team executing 和真实工作未绑定 |
| Goal 长任务 | 后端对象和 API 部分可用 | 主界面不可感知 | GoalConsole/AgentLanes 未挂载；Runtime API 未启动；Chat 入口不能创建 Goal |
| 多进程 Runtime 协作 | API/对象已存在 | 基本不可感知 | 外部 agent 回写依赖 Runtime API，但桌面生命周期未接通 |

## 3. P0 问题：用户模式边界缺失

现状：

- `ChatComposer` 有 `capabilityMode` 和 `planOnly`。
- `AgentWorkspacePanel` 保存这两个状态。
- `useChatSession.sendMessage()` 只调用 `sendChatMessageV2(content, sessionId)`。
- 后端不知道用户选择了只读、可写、先计划、长任务。

影响：

- 用户以为自己只是聊天，LLM 仍可能拿到 `agent.start`。
- 用户想要长任务，也没有清晰的 Goal 入口。
- 前端无法解释“现在为什么停住了”：是 LLM、工具、AgentRun、Goal 还是等待审批。

整改要求：

1. Composer 增加显式 `短任务 / 长任务` 选择。
2. `sendChatMessageV2` 或新 API 必须携带 task_mode、capability、planOnly。
3. 后端 `build_tool_definitions` 必须根据短任务/长任务限制工具暴露；文档、代码只是短任务内的 intent/capability，不是前端 mode。
4. 长任务不应走普通 chat send，而应走 `createGoal/startGoal/runCycle` 或明确的 long-task API。

## 4. P0 问题：默认工具暴露过强

现状：

- `default_allowed_tool_ids` 包含 `agent.start`、`agent.read_output`、`agent.stop`。
- policy 结果为空且 `allowed_tool_ids` 非空时，legacy fallback 会把默认工具暴露给 LLM。
- 这不是“用户显式提 Claude 才能调用”，而是“LLM 只要拿到工具定义就可能调用”。

影响：

- 短任务/普通聊天子场景可能启动后台 `claude -p`。
- 用户没有提前感知“将进入后台 Agent 模式”。
- `agent.start` 的副作用级别高于普通问答，应被模式和权限约束。

整改要求：

1. 默认短任务工具集中移除 `agent.start`。
2. `agent.start` 只在长任务、明确后台 Agent 或明确多 Agent 场景暴露。
3. 用户显式提 `claude/subagent/agentteam` 时，先进入工具发现/模式升级提示。
4. `subagent.claude_p` 从短任务工具集中禁用，除非 debug 或显式高级模式。

## 5. P0 问题：Goal 后端存在，但前端入口未贯通

现状：

- 后端有 `GoalRun`、`GoalCycle`、`AgentTask`、`GoalOrchestrator`。
- IPC 有 `createGoal/listGoals/updateGoalStatus/getGoalCycles/listGoalTasks/listGoalEvents/listActiveHeartbeats/writeWorkspaceProjection`。
- 前端有 `GoalConsole`、`AgentLanes`、`OodaTimeline`、`ReviewQueue`、`AgentTranscript`。
- 但递归检索没有发现这些组件被主界面引用。

影响：

- 用户无法从主工作台选择“长任务”。
- Goal 的 planning/running/review 状态不能形成统一感知。
- Agent lanes、心跳、Review queue 没有进入日常操作路径。

整改要求：

1. 在主工作台挂载 Goal Console 或 Goal drawer。
2. Composer 的长任务入口直接创建 Goal，而不是发送普通 chat。
3. Goal 顶部显示状态、elapsed、current cycle、active tasks、blocking reason。
4. Goal 详情页挂载 AgentLanes、OodaTimeline、ReviewQueue、AgentTranscript。

## 6. P0 问题：Runtime API 未接入桌面生命周期

现状：

- `RuntimeApiServer` 实现了 `/runtime/messages`、`/runtime/heartbeats`、task claim/complete/fail/block、goals、permission、events SSE。
- `ClaudePAdapter` 会注入 `RUNTIME_API_URL`、`RUNTIME_TOKEN`、`WORKSPACE_ID`、`TASK_ID`。
- 但在桌面端源码中没有看到启动 `RuntimeApiServer::start()` 的入口。

影响：

- 外部 Claude/Codex/Agent 不能稳定通过 HTTP 回写 heartbeat、message、task result。
- SSE 事件流不能成为 UI 或 agent 的实时同步总线。
- `ClaudePAdapter` 注入 runtime env 但目标服务可能不存在。

整改要求：

1. 桌面启动时按 workspace 启动 Runtime API。
2. 把 bind address、port、token 持久化到 workspace runtime 状态。
3. Adapter 从真实 runtime 状态读取 URL/token，而不是默认常量。
4. UI 订阅 Runtime events 或由后端桥接到 Tauri event。

## 7. P1 问题：文档协作工具链不完整贯通

现状：

- `office.inspect_document/export_text/patch_dry_run` 已作为内置工具注册。
- Lark connector 文件存在，支持 `lark.doc.search`、`lark.doc.create_or_update` 等。
- 但没有看到 `build_lark_connector()` 在运行时注册到 `ConnectorRegistry` 的入口。

影响：

- LLM 可能无法真正拿到 Lark 工具。
- 用户以为“文档协作”包括飞书，但系统工具暴露不稳定。
- Office、Lark、脚本运行的结果没有统一文档 artifact 投影。

整改要求：

1. 明确短任务中文档子场景的工具集合。
2. 启动时注册 Lark connector，并把认证状态投影到 UI。
3. 文档写入必须有 preview/apply/verify 三段状态。
4. 每次输出统一记录 artifact：本地路径、飞书 token、diff、summary。

## 8. P1 问题：AgentTeam 与执行器关系仍容易混淆

现状：

- AgentTeam 有生命周期、成员、mailbox。
- `agent.team.add_member` 可以记录 `run_id`。
- `AgentTeamAdapter` 可以把 team lifecycle/mailbox 桥接到 Goal/Event runtime。
- 但 `agent.team.create/add_member/snapshot` 本身不会 spawn `claude -p`。

影响：

- 用户看到 team executing，可能误以为模型正在工作。
- 实际上没有 executor 的 team 只是协作容器。
- 跨 agent 通信有对象，但没有完整“创建 team -> spawn members -> heartbeat -> messages -> review”的用户闭环。

整改要求：

1. Team executing 必须校验至少一个 active executor。
2. Team snapshot 要展示 member 对应的 `agent_run_id/task_id/heartbeat`。
3. 创建 team 和启动 members 分成两个用户可见动作。
4. Mailbox 消息进入 AgentTranscript，但不要替代真实执行状态。

## 9. P1 问题：渐进式工具发现还没有实现

目标设计：

```text
LLM 先以最小工具集工作
  -> 判断需要更多能力
  -> 查询工具目录
  -> 筛选/申请能力
  -> 用户或 policy 授权
  -> 执行
```

当前实现：

```text
后端根据 config/keyword/skill/policy 预先生成 tool_defs
  -> LLM 在 tool_defs 中自主 tool_call
```

影响：

- 工具暴露和用户意图之间容易过宽。
- “显式提到 claude/agentteam”不会触发一个独立的发现/确认状态，只会影响关键词/skill 匹配。
- 前端无法显示“LLM 正在查找可用工具”。

整改要求：

1. 引入 Tool Catalog 查询工具或后端 tool discovery 阶段。
2. 短任务初始只暴露低风险工具。
3. LLM 需要高级工具时返回 `capability_request`，由 UI 展示。
4. policy 授权后再注入具体工具定义。

## 10. P1 问题：结果落地投影仍然分散

现状：

- chat message 记录最终回复。
- `tool_calls` 记录工具。
- `command_runs` 记录命令。
- `agent_runs` 记录后台 agent。
- `tasks` 记录 hook 和 review。
- `goal_tasks` 记录 Goal task。
- `runtime_events` 记录事件。

影响：

- 用户不能从一个地方回答“刚才到底改了什么、写到哪里、哪个 agent 产出”。
- 长任务结束后缺少统一 Review/Artifact 收口。

整改要求：

1. 增加统一 `Activity/Artifact` 投影层。
2. 每个用户可见工作单元有统一 id：chat turn、doc job、code job、goal cycle。
3. 所有输出归到同一个 activity：assistant message、tool_call、command_run、agent_run、file artifact、summary_ref。
4. 前端提供“正在进行”和“落地记录”两个稳定入口。

## 11. 建议下一步整改顺序

### 第一批：模式边界

1. 前端加 `短任务 / 长任务` 显式选择。
2. 后端 send API 接收 task_mode/capability/planOnly。
3. 按短任务/长任务重写工具暴露白名单；文档/代码作为 intent/capability 层处理。
4. 短任务默认移除 `agent.start`。

### 第二批：Goal 入口

1. 主界面挂载 Goal Console。
2. Goal 模式走 `createGoal/start/pause/cancel/list tasks/events`。
3. 挂载 AgentLanes、OodaTimeline、ReviewQueue、AgentTranscript。
4. Goal running 状态进入会话列表和任务抽屉。

### 第三批：Runtime API 生命周期

1. 桌面启动 Runtime API。
2. Adapter 使用真实 runtime URL/token。
3. heartbeats/messages/leases 与 UI 投影打通。
4. SSE 或 Tauri event 统一刷新 Goal/Agent lanes。

### 第四批：文档/代码能力收口

1. 短任务的文档子场景注册 Lark connector 和 Office 工具。
2. 短任务的代码子场景增加计划审批、写范围、测试结果投影。
3. `subagent.claude_p` 收敛为 debug 或可观察 AgentRun。
4. AgentTeam 强制绑定 executor。

## 12. 审阅结论

当前系统已经具备“可实现多 Agent/Goal”的对象基础，但还不能称为用户侧可控的 Agent 工作台。

核心差距不是缺少表，而是缺少三条贯通链：

1. 用户模式 -> 后端工具策略。
2. Goal/Runtime -> 主界面状态投影。
3. AgentTeam/AgentRun/AgentTask/Artifact -> 统一落地记录。

下一步应优先做模式边界和 Goal 入口，而不是继续增加底层状态对象。
