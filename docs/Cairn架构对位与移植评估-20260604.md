## 0. 评估目的与范围

### 0.1 目的

Cairn 在 [README](../tools/Cairn/README.md) 中给出了一套已经被 CTF 黑客松验证过的"无角色 BlackBoard + OODA"抽象。本项目 (`Personal Conductor`) 的 Agent Runtime 走的是更"工程化"的路(角色化 AgentTeam、ToolCall 状态机、PermissionGrant 账本)。这次评估回答三个问题:

## 1. Cairn 抽象模型回顾

Cairn 的核心抽象只有三个对象(参见 [server/models.py:13-37](../tools/Cairn/cairn/src/cairn/server/models.py#L13-L37)):

```python
Fact   = 已确认的客观事实
Intent = 待执行的探索方向(由 reason 阶段生成)
Hint   = 人类策略输入(任意时刻可注入,下一轮被 agent 读取)
```

加上:

- **Project** = origin + goal + graph 容器
- **三种任务类型**:
  - `bootstrap`: 项目刚启动时尝试直接解题
  - `reason`: 通读整图,判断 goal 是否满足,生成新 intent
  - `explore`: 认领一个 open intent,执行后回报一个 fact
- **OODA 闭环**: Observe(整图) → Orient → Decide(intent 列表) → Act(执行) → 回写 Fact
- **调度器行为**([loop.py:120-267](../tools/Cairn/cairn/src/cairn/dispatcher/scheduler/loop.py#L120-L267)):
  1. 拉所有 active projects
  2. 对每个 project 做 `_try_dispatch_project`:
     - 初始状态 → dispatch bootstrap 或 reason
     - 有 reason 但未领 → claim reason
     - 有 unclaimed intent → dispatch explore
     - 都不满足 → 跳过
  3. 跟踪 `ReasonCheckpoint`(facts/hints/open_intents 计数),只有图状态变化才再次触发 reason

**Cairn 哲学** (与本项目最相关的三点):

1. **不预设角色**: agent 是同一 Worker,通过图状态决定这次跑 bootstrap/reason/explore 中的哪一个
2. **图是公共契约**: 所有 fact/intent 都从 graph 读取,worker 输入是 graph YAML,输出是写回 graph 的 fact/intent
3. **OODA Reason 是核心**: `reason.md` 让 LLM 自己看图决定"goal 满足了吗 / 还要不要新 intent"——它把战略判断完全交给 LLM,而不是 hard-code 规则

## 2. Personal Conductor 现状对位

### 2.1 已有等价能力

| Cairn 概念 | 本项目实现 | 状态 | 评价 |
|---|---|---|---|
| Project / origin / goal | `GoalRun` + `Workspace` | ✅ 已对齐 | [goals.rs](../crates/conductor-core/src/goals.rs) 已有完整 CRUD |
| Fact(已确认事实) | `MemoryEntry` + `AuditEvent` | 🟡 部分对齐 | 没有像 Cairn 那样作为"图节点"建模 |
| Intent(待执行方向) | `AgentTask`(open/unclaimed) | ✅ 已对齐 | [goal_tasks.rs](../crates/conductor-core/src/goal_tasks.rs) 状态机完整 |
| Hint(人类输入) | `agent_mailbox_messages` + `docs/workspace.md` | 🟡 部分对齐 | 走的是消息总线而非独立 Hint 表 |
| OODA Reason | `decide.rs` + LLM planning | 🟡 简化 | 当前是 "execute_goal" 启发式,不是真正的 LLM Reason |
| 三种任务类型 | `agent_kinds`(backend-agent / doc-agent) | ⚠️ 偏离 | 本项目预设了 agent kind,违反 Cairn "无角色" 原则 |
| ReasonCheckpoint(防抖) | 无显式实现 | ❌ 缺失 | 本项目每轮 OODA 都会跑,没有图变化才触发 |
| Worker 选择 | `choose_worker`(Cairn) vs `agent_fit`(本项目) | 🟡 简化 | 我们的 `agent_fit` 不做 LLM 派工 |
| 容器/Docker 隔离 | Tauri 进程 + `claude -p` 子进程 | ✅ 已有 | 不需要移植 Cairn 的 Docker 部分 |
| 协议客户端(API) | `runtime_api.rs` HTTP + SSE | ✅ 已对齐 | 路线同 Cairn,本机 HTTP 通讯 |

### 2.2 关键差距

#### 差距 A: 本项目预设了"Agent Role",Cairn 反对

- 我们的 `agent_team_members.role` 字段(以及派工文档的 planner/executor/reviewer 角色)是固定身份
- Cairn 的实践是:**任务类型在调度时由图状态决定**,agent 本身无类型
- 后果: 我们不能像 Cairn 那样"reason 任务没人跑就转给 explorer 跑"——role 是绑定的

#### 差距 B: 没有 ReasonCheckpoint,OODA 抖动

- Cairn 的 `ReasonCheckpoint`([loop.py:633-647](../tools/Cairn/cairn/src/cairn/dispatcher/scheduler/loop.py#L633-L647))只在 `fact_count / hint_count / open_intent_count` 变化时才触发 reason
- 我们的 `decide.rs` 没有"图是否变化"的判断,会重复派同样的任务

#### 差距 C: Reason 任务不是真"读图判断"

- Cairn 的 reason 任务把整张图 YAML 喂给 LLM,LLM 返回 `{complete | intents | noop}`
- 我们的 `decide.rs` 仍走 `orient.rs → dispatch_plan` 的程序化路径
- LLM 在 Reason 阶段不直接参与战略判断

#### 差距 D: 没有 Graph Tension 概念

- Cairn 用 `fact_count / hint_count / open_intent_count` 做 ReasonCheckpoint
- 但**真正的 Tension**(认知压力)需要更细的指标:contradiction 数、stale 路径、未被认领的紧急任务
- 我们有 `observe.rs` 但 orient.rs 没有把 tension 暴露给调度器

#### 差距 E: Worker 协议不统一

- Cairn 的 worker 三种任务类型共用同一份 Driver(`build_healthcheck` / `build_execute`),通过 prompt group 区分行为
- 我们的 `claude -p`、`codex` 适配是各做一份,没有"任务类型 → prompt 模板" 的统一抽象

## 3. 详细对位映射(可打通点)

下面把 12 个具体可打通点逐一展开。每个点说明:目标 / 当前实现 / Cairn 参考 / 移植步骤 / 破坏面。

### P1. ReasonCheckpoint 与图变化防抖

**目标**: OODA 循环只在 graph 发生变化时重新跑 decide,避免抖动。

**Cairn 参考**: [loop.py:633-647](../tools/Cairn/cairn/src/cairn/dispatcher/scheduler/loop.py#L633-L647) + [models.py:20-25](../tools/Cairn/cairn/src/cairn/dispatcher/models.py#L20-L25)

**本项目当前**:
- `observe.rs` 拉数据,`decide.rs` 直接生成 `DispatchPlan`,**没有"和上次有什么不同"** 这一步
- 每次 OODA cycle 都会跑 `orient → decide`,可能产出和上次一样的 plan

**移植步骤**:
1. 在 `goal_cycles` 表加 `last_graph_hash TEXT`(或单独 `graph_checkpoints` 表)
2. `observe.rs` 末尾计算 hash(facts + hints + open_intents 排序后拼接的 SHA256)
3. `goal_orchestrator/mod.rs` 的 `run_cycle` 加 guard:`if current_hash == last_hash and not force_replan: return skip`
4. cycle 完成后写回 hash

**破坏面**: 无。纯加法。`force_replan` 用于 hint 注入、Plan Approval 拒绝等场景。

**工作量**: 0.5 人天。**难度: 低**。

---

### P2. 三种任务类型的统一 Driver 抽象

**目标**: bootstrap / reason / explore 共用同一 worker 执行路径,只换 prompt。

**Cairn 参考**: [tasks/reason.py](../tools/Cairn/cairn/src/cairn/dispatcher/tasks/reason.py) + [tasks/explore.py](../tools/Cairn/cairn/src/cairn/dispatcher/tasks/explore.py) 共享 [`run_worker_process`](../tools/Cairn/cairn/src/cairn/dispatcher/tasks/common.py) + [`HeartbeatLease`](../tools/Cairn/cairn/src/cairn/dispatcher/runtime/heartbeat.py) + Driver 协议

**本项目当前**:
- `agent_runs.rs` 启动 `claude -p` 子进程
- `codex.rs` 是另一套独立协议
- Goal 派工的 `PlannedTask` 写死 `agent_kind: "backend-agent"`

**移植步骤**:
1. 在 `crates/conductor-core/src/agents/` 新建 `dispatcher/` 子模块,定义 `TaskKind::{Reason, Explore, Bootstrap}` 枚举
2. 复用现有 `agent_runs.rs` 的子进程启动逻辑,但加 `prompt_template` 参数
3. Reason 用一份 prompt(读整图),Explore 用另一份(读 graph + 当前 intent)
4. `PlannedTask` 增加 `task_kind` 字段(默认 `Explore`)

**破坏面**: 小。`agent_kind` 保留为 deprecated 字段,`task_kind` 优先。

**工作量**: 2-3 人天(含三份 prompt 模板)。**难度: 中**。

---

### P3. 真正的 Reason 任务(LLM 读图判断)

**目标**: 在 Decide 阶段真正让 LLM 读整图决定"goal 满足了吗 / 还要不要新 intent"。

**Cairn 参考**: [tasks/reason.py:113-126](../tools/Cairn/cairn/src/cairn/dispatcher/tasks/reason.py#L113-L126) + [prompts/default/reason.md](../tools/Cairn/cairn/src/cairn/dispatcher/prompts/default/reason.md)

**本项目当前**:
- `decide.rs:90-173` 是程序化决策,没有 LLM 调用
- 它的启发式是"如果没任务就跑 execute_goal"

**移植步骤**:
1. 把 `decide.rs` 拆成两个函数:
   - `decide_static(orient, budget)`: 仅做 budget / blockers 校验(保留现有逻辑)
   - `decide_llm(graph_snapshot)`: 调用 LLM 读图,返回 `{complete, intents[], noop}`
2. 设计 Reason Prompt(中文版,适配本项目 Goal/Memory/Task 实体)
3. `goal_orchestrator/decide.rs` 改为:`if orient.blockers.is_empty() && budget ok: call decide_llm`
4. LLM 返回的 intents 写入 `AgentTask` 表(同 Cairn 的 `client.create_intent`)

**破坏面**: 中。`DispatchPlan.tasks` 字段不变,但实际生成路径多了一条 LLM 分支。要做特性开关。

**工作量**: 3-5 人天(prompt 工程占一半)。**难度: 中**。

**前置依赖**: P1 (ReasonCheckpoint) + P2 (TaskKind 抽象)

### P4. Hint 表与 Round 1 注入机制

**目标**: 把人类策略输入从"散落在派工文档"升级为"一等数据对象",可被 Reason 阶段读取。

**Cairn 参考**: [server/models.py:32-36](../tools/Cairn/cairn/src/cairn/server/models.py#L32-L36) + `routers/hints.py`

**本项目当前**:
- 人类输入走 `agent_mailbox_messages` 或 `docs/workspace.md` Markdown
- `workspace.md` 是 Projection(投影),不是状态源

**移植步骤**:
1. 新增 `goal_hints` 表:`id, goal_id, content, creator, created_at, read_at`
2. `observe.rs` 拉 hints 并放入 `ObserveReport`
3. Reason Prompt 里加 `## Hints` 段
4. UI 端加"插入策略输入"按钮 → 写 `goal_hints` 表
5. `workspace.md` 投影层把 hints 也渲染出来

**破坏面**: 无。纯加法。

**工作量**: 1 人天。**难度: 低**。

### P5. "图快照"序列化与导出 API

**目标**: 把当前 goal 的状态序列化为 YAML / JSON,供 Reason Prompt 直接喂给 LLM。

**Cairn 参考**: [protocol/client.py:66-73](../tools/Cairn/cairn/src/cairn/dispatcher/protocol/client.py#L66-L73) `export_project(format=yaml)` + `routers/export.py`

**本项目当前**:
- `runtime_api.rs` 暴露 events / tasks / messages,但**没有**"导出整张图"
- `docs/workspace.md` 是手写/拼装的 Markdown,不是从状态源序列化

**移植步骤**:
1. `runtime_api.rs` 加 `GET /v1/goals/{id}/graph?format=yaml|json` 端点
2. 实现 `GoalGraphSnapshot { facts, intents, hints, recent_events }` 结构体
3. 序列化遵循 Cairn 风格(键名 `facts` / `intents` / `hints`,稳定排序)
4. Reason Prompt 模板用 `{graph_yaml}` 占位符

**破坏面**: 无。新增端点。

**工作量**: 1-2 人天。**难度: 低**。

**前置依赖**: P3 (Reason 任务) 真正会消费这个 API

### P6. Worker 健康检查与不健康熔断

**目标**: worker 连续失败/超时时熔断 N 秒,避免对同一个坏 worker 反复 dispatch。

**Cairn 参考**: [loop.py:671-693](../tools/Cairn/cairn/src/cairn/dispatcher/scheduler/loop.py#L671-L693) `worker_unhealthy_until` / `worker_rejected_until` 字典

**本项目当前**:
- `agent_runs.rs` 失败只是把状态切到 `Failed`
- 没有"worker X 暂时不可用"的记忆

**移植步骤**:
1. 新增 `worker_health` 表:`worker_id, last_failure_at, failure_kind, retry_after`
2. `agent_runs.rs` 写入失败时检查最近 N 次失败率,更新 `retry_after`
3. `goal_orchestrator/dispatch.rs` 派工前过滤冷却中的 worker

**破坏面**: 小。失败语义不变,只是新增调度器侧的过滤。

**工作量**: 1-2 人天。**难度: 中**。

### P7. 任务 claim 协议(防双跑)

**目标**: 一个 open intent 只能被一个 worker 真正执行,避免双跑/写冲突。

**Cairn 参考**: [loop.py:433-454](../tools/Cairn/cairn/src/cairn/dispatcher/scheduler/loop.py#L433-L454) `client.heartbeat(intent_id, worker)` + 服务端 403/409 拒绝

**本项目当前**:
- `goal_tasks.rs` 有 `claim` 概念但没有强约束
- `agent_run` 启动后没有"已 claim 此 task"的反向更新

**移植步骤**:
1. `goal_tasks.rs` 加 `claimed_by` / `claimed_at` 字段
2. `agent_runs.rs` 启动前先 `claim_task(task_id, agent_id)`,数据库事务保证唯一
3. 加超时:`claimed_at` + 5 分钟未 running 视为放弃,自动释放
4. `dispatch.rs` 派工前过滤 `claimed_at IS NULL OR claimed_at < now - 5min`

**破坏面**: 小。`tasks` 表 schema 变更,需要 migration。

**工作量**: 1-2 人天。**难度: 中**。

### P8. Worker 选择策略:从硬编码到声明式

**目标**: 把"哪个 worker 适合哪种任务"从代码里搬到 config(类似 Cairn 的 `dispatch.yaml`)。

**Cairn 参考**: [dispatch.example.yaml](../tools/Cairn/dispatch.yaml) + [scheduler/worker_select.py](../tools/Cairn/cairn/src/cairn/dispatcher/scheduler/worker_select.py) + [loop.py:476-536](../tools/Cairn/cairn/src/cairn/dispatcher/scheduler/loop.py#L476-L536) `_select_worker`

**本项目当前**:
- `goal_orchestrator/orient.rs` 里 `AgentFit` 是简化版
- 没有"worker 注册表 + 任务类型匹配"的声明式机制

**移植步骤**:
1. `config.json` 加 `workers` 段:`[ { name, command, task_types, max_running, priority } ]`
2. `agent_backends.rs` 增加 worker 注册加载逻辑
3. `goal_orchestrator/dispatch.rs` 派工时按 `task_type` 过滤候选 worker
4. 加一个简单的 round-robin / least-loaded 选择器

**破坏面**: 小。`config.json` 字段新增,旧配置仍可用。

**工作量**: 2-3 人天。**难度: 中**。

### P9. Graph Tension 字段与 Tension 驱动调度

**目标**: 在 graph 上加 tension 度量(contradiction / stale / unclaimed_urgent),Decide 阶段优先处理高 tension 区域。

## 6. 移植到本项目的关键改造点详解

### 6.1 ReasonCheckpoint 在本项目的具体落点

**位置**: [crates/conductor-core/src/goal_orchestrator/mod.rs](../crates/conductor-core/src/goal_orchestrator/mod.rs) `run_cycle`

**伪代码**:
```rust
pub async fn run_cycle(&self, goal_id: &str) -> Result<CycleResult> {
    let observe = observe::observe(goal_id).await?;
    let graph_hash = compute_graph_hash(&observe);  // 新增

    let last_hash = crate::goals::get_cycle_hash(goal_id).await?;
    if Some(&graph_hash) == last_hash.as_ref() {
        return Ok(CycleResult::skipped("graph unchanged"));
    }

    let orient = orient::orient(&observe)?;
    let plan = decide::decide(&orient, &self.config.budget, &observe.goal.objective)?;
    let act = act::act(&plan, goal_id).await?;
    let review = review::review(&act).await?;

    crate::goals::set_cycle_hash(goal_id, &graph_hash).await?;  // 新增
    Ok(CycleResult { ... })
}

fn compute_graph_hash(observe: &ObserveReport) -> String {
    let mut hasher = Sha256::new();
    for f in sorted(&observe.facts) { hasher.update(f.id); hasher.update(f.description); }
    for i in sorted(&observe.open_intents) { hasher.update(i.id); hasher.update(i.title); }
    for h in sorted(&observe.hints) { hasher.update(h.id); }
    format!("{:x}", hasher.finalize())
}
```

**关键点**:
- `compute_graph_hash` 只包含"事实"和"开放意图",不含 heartbeat / lease 这些"过程量"
- hint 注入会改变 hash,自动触发下一轮 reason(这是 hint 真正生效的路径)

### 6.2 LLM Reason 任务的 Prompt 契约

```markdown
# 任务
你是一个 Goal 战略判断器。下面是一张"工作区图"快照,包含:
- 已确认的事实 (Facts)
- 开放中的任务 (Open Intents)

{goal_objective}

## Graph (YAML)
{graph_yaml}

## Open Intents
{open_intents}

## Hints
{hints}
```
