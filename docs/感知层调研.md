# Personal Conductor Agent — 感知层调研

> 回答一个问题：**Conductor 能不能像桌宠一样，知道我现在在浏览器 / IDE / 聊天工具里干什么，从而跨应用提醒工作进度？**
>
> 本文不是 MVP 范围，是为「远期愿景」做的可行性预研。

---

## 1. 想要的能力（按野心从低到高）

| 等级 | 能力 | 例子 |
|------|------|------|
| L1 | 知道**当前前台应用**是什么 | 「他切到浏览器 8 分钟了」 |
| L2 | 知道**前台应用的窗口标题** | 「他在看『doc-A.md』」 |
| L3 | 知道**应用内具体内容**（不读屏） | 「他在 VS Code 里编辑 src/foo.ts」 |
| L4 | **读屏理解**（多模态截图） | 「他在 Figma 上调按钮颜色」 |
| L5 | **主动控制其他应用**（写命令） | 「往那个 Claude Code 窗口贴一条 prompt」 |

MVP 一个都不要。但架构要给 L1–L3 留口子。

---

## 2. 三条路径

### 路径 A：OS 级 hook + 白名单

**做法**：
- Windows：`SetWinEventHook(EVENT_SYSTEM_FOREGROUND)` 监听前台窗口切换，配合 `GetWindowText` 拿窗口标题。
- 进一步：UIAutomation API 拿到控件树（能知道编辑框里的文本、按钮 label 等）。
- 写一个白名单：只对「Claude Code 终端」「VS Code」「Chrome 上 docs.google.com 域名」等关心的应用记录事件。

**能拿到什么**（回答你的问题）：
- ✅ 任何应用的**窗口标题**（应用名 + 文档名常包含其中）。
- ✅ 任何应用的**焦点/失焦时间**——能精确算"在浏览器停留了多久"。
- ✅ 通过 UIAutomation，能拿到很多应用的**控件文本**，包括地址栏 URL、编辑器文件路径、聊天工具当前会话名等。**理论上"其他应用的信息"大半能抓到，但取决于该应用是否暴露 UIA。**
- ❌ 拿不到应用的**内部状态/业务数据**（比如 IDE 当前 git branch、Figma 当前选中图层），除非该应用主动提供 API。
- ⚠️ 一些应用（尤其 Electron 套壳的、游戏、DRM 内容）UIA 树残缺，只能拿到标题。

**优点**：
- 不依赖任何应用配合，从 OS 层抓，能覆盖**所有 GUI 应用**的窗口/焦点信息。
- 性能好，事件驱动，不轮询。
- 没有图像识别延迟和成本。

**缺点**：
- Windows 专属，跨平台要重写（macOS 用 Accessibility API，Linux 看 WM）。
- UIA 抓内容有伦理/合规问题：你抓到的可能包括密码框、私信。需要白名单严格控制。
- 反病毒软件可能误报。

**结论**：**这条路是 L1–L3 的主力**。能用相对小的代价覆盖大部分场景。

---

### 路径 B：多模态截图

**做法**：
- 每 N 秒（或在前台切换事件触发时）截一张屏。
- 喂给多模态 LLM（Claude / GPT-4o），让它输出「人现在在做什么」的结构化描述。

**优点**：
- **通用**：不管什么应用，能看见的就能描述。
- 能理解视觉信息（图表、Figma 设计稿、视频画面）。
- 跨平台无差别。

**缺点**：
- **贵**：截图 + LLM 调用，按 5 秒一次算每天数千次调用，token 成本不低。
- **慢**：每次 1–3 秒延迟，不适合实时焦点追踪。
- **隐私重**：截图包含一切，包括密码、私聊。本地缓存策略要谨慎。
- **不稳**：LLM 描述可能漂移。

**结论**：**L4 必须靠它，L1–L3 不该用它**。MVP 之后可以作为可选增强，对"看不懂的应用"兜底。

---

### 路径 C：各应用 SDK / 协议拼接

**做法**：
- Chrome：装一个扩展，实时报告当前 tab URL/title。
- VS Code：装一个扩展，报告当前文件、git branch、selection。
- Slack/Lark/微信：用各自 API 查在线状态、未读数。

**优点**：信号最准，能拿到业务级语义。

**缺点**：每个应用要单独适配，工作量随应用数线性增长，且总有覆盖不到的应用。

**结论**：作为**精度补强**，对最常用的 3–5 个应用做适配。Conductor 不依赖这一层，但有它会更聪明。

---

## 3. 推荐：混合方案（**你已经选了**）

```
L1–L3：路径 A（OS hook + UIA + 白名单）
       └─ 兜底：路径 C（对 Claude Code 终端、VS Code、Chrome 单独适配）
L4：    路径 B（多模态），仅在前两层「看不懂当前应用」时触发
L5：    路径 A 反向：FindWindow + SendInput / Accessibility 写入
```

数据流：

```
OS 事件 ──┐
SDK 信号 ─┼─→ 事件归一化 ──→ state/events/ ──→ Conductor 决策
截图理解 ─┘                                       ↓
                                          (按需) 主动提醒 / 注入
```

所有信号都落到**统一的事件流**（架构文档 §6 已预留），Conductor 主逻辑不感知它是 hook 还是截图来的。

## 4. 关于"反向写命令到 Claude Code"

你提的具体问题：**Conductor 能不能往 Claude Code 窗口写下一步命令？**

技术上 **可以**，三种路子：

1. **文件接口**：写文件到一个 Claude Code 正在监听的目录，触发它读。最干净，不依赖 GUI。
2. **API 接口**：用 Claude Code 的 SDK 或 IPC 接口（如果暴露），程序化下发 prompt。
3. **GUI 注入**：`FindWindow` 找到 Claude Code 终端窗口，`SendInput` 模拟键盘输入。能用，但**脆**——窗口标题变了就找不到，输入法状态会干扰。

**产品上要不要这么做？我的建议是**：

- **MVP 阶段绝对不要**。一旦 Conductor 能"动手"，它就不只是个调度器了，它是个 actor，错误会被放大。
- **远期可以**，但要加**显式确认**：Conductor 生成"下一步建议"，写到 `state/proposed_prompts/`，人点一下"批准"才真的下发。
- **永远不要** GUI 注入。用文件或 API 接口。

## 5. 桌宠形态

PRD 里提到「像桌宠一样」。**已确定形态**：

- **常驻托盘小图标** + **Live2D 桌宠角色**（透明、置顶、点击穿透）。
- 角色状态：`idle` / `new_task` / `pending_review` / `quiet`，由动画/表情切换体现。
- 点击桌宠 → 弹出 `task-panel`；右键 → 菜单（专心 30 分钟 / 静默 / 退出）。
- 系统通知作为**跨应用打断手段**，仅在堆积超阈值时使用。

**技术栈与资产**：
- 桌面壳：Tauri v2 + React/TypeScript，详见 `技术栈选型.md`。
- 桌宠动画：Live2D Cubism 4 Web SDK，免费+自学路线，资产准备详见 `Live2D-资产准备.md`。

> 早期版本里这里曾写过"MVP 用 Toast 通知替代桌宠"，已废弃。MVP 直接走 Tauri + Live2D 路线，不再用 Toast 替代。

## 6. 风险与隐私

- **抓到敏感内容**：UIA 能读到密码框（部分）、私信、邮件。白名单必须严格，且永远不上传到云端 LLM 之前过滤一遍敏感词。
- **截图风险**：路径 B 一旦启用，建议本地 OCR 后只把文本交给 LLM，原图不外传。
- **被动监控感**：哪怕只是给自己用，也要有一个**全局开关**「现在不看我」，让人能临时关掉。

## 7. 决策小结

| 问题 | 答案 |
|------|------|
| OS hook + 白名单能抓其他应用信息吗？ | 能，覆盖窗口标题/焦点 + 大部分应用的 UIA 控件文本。Electron/游戏/DRM 应用有盲区。 |
| 能不能做到「桌宠」体验？ | 能。**已确定走 Tauri + Live2D**，不再用 Toast 替代。资产清单见 `Live2D-资产准备.md`。 |
| Conductor 能往 Claude Code 写命令吗？ | 技术上能。**已决定走复合方案**：P0 用 `claude -p` 子进程，文件接口作为 P1 补强。具体见 `Conductor-反向通道.md`。 |
| 这一切要不要进 MVP？ | **感知层（L1–L5）不进 MVP**。MVP 闭环只跑 Claude 一侧的 Stop hook + 对话调度 + 反向通道 `claude -p`。桌宠 UI 进 MVP 桌面壳阶段（Tauri）。 |
