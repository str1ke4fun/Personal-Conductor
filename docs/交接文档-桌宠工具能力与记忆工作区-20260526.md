# 交接文档：桌宠工具能力、记忆与工作区

> 日期：2026-05-26
> 范围：`crates/conductor-core`、`crates/conductor-cli`、`apps/desktop`

## 1. 当前状态

这轮已经把方案里的核心 MVP 主线推进到可验证状态：

```text
观察桌面/任务状态
  -> 生成任务、提案或主动提示
  -> 用户批准提案
  -> ToolRegistry 执行受限工具
  -> 记录 tool run / chat / memory / task 状态
```

最近一次验证结果：

- `cargo test --workspace` 通过。
- `npm.cmd run build` 通过，仅有 Vite chunk-size warning。
- `cargo check --workspace` 通过，仅有既有 unused/dead_code warning。

## 2. 已落地能力

### 2.1 Workspace

代码位置：

- `crates/conductor-core/src/workspaces.rs`
- `crates/conductor-core/src/db.rs`

已实现：

- `workspaces` 表。
- `WorkspaceKind`: `code`、`document`、`office`、`notes`、`generic`。
- `TrustLevel`: `trusted`、`ask_write`、`read_only`、`untrusted`。
- 从 cwd 推断工作区。
- 从前台进程和窗口标题推断工作区类型。
- proposal 执行时按 `workspace_id` 进入 workspace-aware 权限判断。

未完成：

- 没有完整 UI 管理工作区列表、信任级别和规则。
- 没有 worktree / 临时副本隔离。
- 没有复杂 allow/deny/ask 规则编辑器。

### 2.2 Action Proposal 和工具执行

代码位置：

- `crates/conductor-core/src/proposals.rs`
- `crates/conductor-core/src/tools.rs`
- `apps/desktop/src-tauri/src/commands.rs`
- `apps/desktop/src/windows/TaskPanelContent.tsx`

已实现：

- `action_proposals` 表已扩展：`workspace_id`、`tool_id`、`tool_input_json`、`risk_level`、`dry_run`、`result_ref`、`status`。
- 提案生命周期：`pending -> approved -> running -> succeeded/failed`，另有 `rejected/expired/used`。
- `tool_runs` 表记录工具执行结果。
- Task 面板可以列出提案，并支持 approve / execute / reject。
- CLI 和 Tauri 都有提案执行入口。

未完成：

- 没有完整的前端提案详情页。
- 没有 diff/preview 级别的复杂审核 UI。
- 没有自动权限分类器。

### 2.3 ToolRegistry / ToolExecutor

代码位置：

- `crates/conductor-core/src/tools.rs`

已实现工具类别：

- Demo: `demo.echo`。
- TaskList V2: `task.create`、`task.list`、`task.get`、`task.update`、`task.claim`。
- Tool search: `tool.search`。
- Subagent: `subagent.claude_p`。
- Agent run: `agent.start`、`agent.read_output`、`agent.stop`。
- Agent Team: `agent.team.create`、`agent.team.list`、`agent.team.add_member`、`agent.team.snapshot`。
- Mailbox: `agent.mailbox.send`、`agent.mailbox.list`、`agent.mailbox.mark_read`。
- Avatar: `pet.set_avatar`、`conductor.pet.set_avatar`。
- Office placeholders: `office.inspect_document`、`office.export_text`、`office.patch_dry_run`。

重要边界：

- `office.*` 目前是安全本地占位实现，没有接入真实 `office-cli`。
- 不暴露通用 Bash/PowerShell/FileWrite/FileEdit。
- `conductor.pet.set_avatar` 是 ReadOnly 风险级别，只接受枚举形象，不接受任意图片路径或 URL。

### 2.5 记忆系统

- `crates/conductor-core/src/memory.rs`

## 3. 当前没做完的方案项

高优先级剩余项：

1. 真实 `office-cli` 集成：替换 `office.*` 占位实现，至少完成只读 inspect/export，再做 dry-run patch。
2. Worktree / 临时副本：为高风险修改提供隔离实验区。
3. Workspace 管理 UI：展示当前 workspace、信任级别、读写策略。
4. 提案详情 UI：展示工具输入、风险、workspace、dry-run 结果、执行结果。
5. Agent run daemon 化：让 CLI 和桌面都能稳定托管长任务。

中优先级剩余项：

1. 记忆管理 UI：可查看、编辑、删除、标记敏感记忆。
2. 真 embedding provider：接入本地模型或云端 embedding，并保留本地 fallback。
3. MCP remote execution：完成 provider client、连接状态、错误呈现。
4. Task editor：支持创建、编辑依赖、分配 owner。
5. Agent Team UI：显示 team、members、mailbox。

明确暂缓项：

1. 通用 Bash/PowerShell 工具。
2. 通用 FileWrite/FileEdit 工具。
3. 完整 Agent Team / Swarm。
4. 远程 Agent。
5. 自动权限分类器。
6. 复杂 hook 系统。
7. Task watcher 自动领取。
8. 感情值机制设计和实现。

## 4. 交接注意事项

- 这个目录不是 git 仓库，不能依赖 `git diff/status` 做变更审计。
- 工作区可能有其他 agent 同时修改，编辑前必须重读目标文件。
- 手工编辑用 `apply_patch`，不要覆盖未知改动。
- 运行 shell 命令按本机规则加 `rtk` 前缀。
- 不要把 `office.*`、MCP、memory vector search、Agent Team 写成完全生产完成。
- 不要把形象系统写成 `AvatarTheme`，当前真实 API 是 `AvatarId`。
- 不要实现感情值系统，用户要求先留空评估。

## 5. 推荐下一步

建议下一轮先做“可见闭环”，优先级如下：

1. Workspace/Proposal 详情 UI：用户能看懂工具为什么要执行、在哪里执行、风险是什么。
2. `office.inspect_document` 真实只读接入：这是文档助手 skill 的第一个真实价值点。
3. Agent Team 面板只读 UI：先展示 team/member/mailbox，不做自动调度。
4. 记忆管理面板：先做查看和删除，避免长期记忆成为黑盒。
5. Agent run daemon 设计：明确桌面常驻和 CLI 的职责边界。
