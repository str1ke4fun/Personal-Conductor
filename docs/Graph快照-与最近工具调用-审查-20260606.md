## 2. 推理-图(Reasoning Graph)逐层对位

### 2.1 后端 Runtime HTTP 端点 — ✅ 实通

[crates/conductor-core/src/runtime_api.rs:101-104, 1252-1358](file:///I:/personal-agent/crates/conductor-core/src/runtime_api.rs#L101-L104) 路由注册 + 处理函数都真实在跑:

```rust
.route("/runtime/goals/{goal_id}/graph", get(get_goal_graph_handler))
```

处理函数做的事(无短路、无硬编码):

1. 调 `goals::get_goal(&goal_id)` 拿 Goal 行
2. `goal_tasks::list_tasks_by_goal(&goal_id)` 过滤 `proposed | queued | claimed` 作为 `intents`
3. `sqlx::query(... memory_entries WHERE goal_id = ? ...)` 拿前 50 条作为 `facts`
4. `goal_hints::list_active_hints(&goal_id, None)` 拿 `hints`
5. `events::query_events_db(&goal.workspace_id, None, Some(20))` 拿 `recent_events`(20 条)
6. 用 `chat::get_turn_by_goal_cycle_id(cycle_id)` 反查 `chat_turn_request_id`

输出结构体 [GoalGraphSnapshot](file:///I:/personal-agent/crates/conductor-core/src/runtime_api.rs#L1210-L1221) 字段齐:`goal_id / goal_title / objective / facts / intents / hints / recent_events / chat_turn_request_id`。

### 2.2 Tauri IPC 命令 `get_goal_graph` — ⚠️ 空壳

[apps/desktop/src-tauri/src/commands.rs:1912-1934](file:///I:/personal-agent/apps/desktop/src-tauri/src/commands.rs#L1912-L1934):

```rust
#[tauri::command]
pub async fn get_goal_graph(goal_id: String) -> Result<serde_json::Value, AppError> {
    let hints: Vec<serde_json::Value> = goal_hints::list_active_hints(&goal_id, None)
        .await
        .unwrap_or_default()
        .into_iter()
        .map(|h| serde_json::json!({ "id": h.id, "content": h.content, "hint_kind": h.kind }))
        .collect();

    let tasks = goal_tasks::list_tasks_by_goal(&goal_id).await.unwrap_or_default();
    let intents: Vec<serde_json::Value> = tasks
        .iter()
        .map(|t| serde_json::json!({ "id": t.id, "title": t.title, "status": t.status }))
        .collect();

    Ok(serde_json::json!({
        "goal_id": goal_id,
        "facts": [],          // ← 字面量空数组
        "intents": intents,   // ← 不区分 open/closed,把所有 task 塞进来
        "hints": hints,
        "events": []          // ← 字面量空数组
        // 缺: chat_turn_request_id
    }))
}
```

**实际"假通"的部分**:
- `facts` 永远 0 —— 没有 `sqlx` dep(grep 全文件没 `use sqlx`),只能走 `goal_hints` / `goal_tasks`,但 `memory_entries` 这层没接进来
- `events` 永远 0 —— 不查 `events` 表
- 没有 `chat_turn_request_id` —— 不调 `chat::get_turn_by_goal_cycle_id`
- `intents` 把所有 task 一锅端,没复刻 runtime 端点的 `proposed | queued | claimed` 过滤

历史背景: 在 2026-06-05 那轮派工(E-3)里有 stop 摘要说 "sqlx isn't a direct dep of conductor-desktop. The get_goal_graph command u...",之后被改成了"只 query hints + tasks"的形态,facts / events 留作 TODO。`state/summaries/20260605T034633Z-goalconsole.md`。

### 2.4 任务面板实际显示 — ⚠️ 数字恒为 0

任务面板"图快照"渲染在 [GoalConsole.tsx:386-395](file:///I:/personal-agent/apps/desktop/src/windows/GoalConsole.tsx#L386-L395):

```tsx
{activeTab === 'reasoning' && (
  <>
    {/* Graph snapshot summary */}
    <div className="goal-section">
      <h5>图快照</h5>
      {graph ? (
        <div className="review-meta">
```

走的是 `ipc/invoke.ts:1263-1277` 的 `getGoalGraph`,调 **runtime HTTP**(不是 Tauri 命令),HTTP 拿的是 §2.1 那个全实现版本。但解析层把数据砍成 0:

```ts
// apps/desktop/src/ipc/invoke.ts:1268-1274
open_intents_count:
  raw.open_intents?.length ??
  raw.intents?.filter((i: any) => i.status === 'open').length ??  // ← 永远 0
  raw.open_intents_count ??
  0,
events_count: raw.events?.length ?? raw.events_count,            // ← 永远 undefined
```

| 字段 | 后端 §2.1 实际返回 | 前端读取的 key | 结果 |
|---|---|---|---|
| Intent 状态枚举 | `proposed \| queued \| claimed` | `i.status === 'open'` | 永远 0 |
| Events 字段名 | `recent_events` | `events` / `events_count` | 永远 undefined |
| `chat_turn_request_id` | 有 cycle 就有 | 直接读 | ✅ 可用 |

**用户看到的就是**:`事实 0 · 开放意图 0`(事件那行直接被 undefined 截掉,`三元` 不进)。`图快照不可用(Goal 尚未开始推理)` 这条分支永远走不到,误以为"后端在跑但没东西出来"。

## 4. 最近一次会话的工具调用失败

### 4.1 vitest 4 个失败 = mock 遗漏

最近 stop 摘要档 `state/summaries/20260606T070749Z-agentworkspacepanel.md` 的 transcript 提到 "55 passed, 4 failed",4 个失败全部来自 `GoalConsole.test.tsx` 和 `GoalConsole.integration.test.tsx`。

对照源码,根因清楚:

[GoalConsole.test.tsx:5-24](file:///I:/personal-agent/apps/desktop/src/windows/GoalConsole.test.tsx#L5-L24) 只 mock 了 4 个函数:

```ts
const mocks = vi.hoisted(() => ({
  listGoalsMock: vi.fn(),
  getGoalCyclesMock: vi.fn(),
  listGoalTasksMock: vi.fn(),
  listActiveHeartbeatsMock: vi.fn(),
  listenMock: vi.fn(),
}));

vi.mock('../ipc/invoke', () => ({
  listGoals: mocks.listGoalsMock,
  getGoalCycles: mocks.getGoalCyclesMock,
  listGoalTasks: mocks.listGoalTasksMock,
  listActiveHeartbeats: mocks.listActiveHeartbeatsMock,
}));
```

但 `GoalConsole.tsx` 在 mount 时就调 `useEffect → refreshReasoning → listGoalHints + getGoalGraph`,且后续操作还会调 `createGoalHint / dismissGoalHint`。`vi.mock` 返回的模块对象里**没有这 4 个函数**,从 mock 出来的模块里 import 它们会得到 `undefined`,调用即抛 `TypeError: listGoalHints is not a function`。

### 4.2 真正的"工具调用失败"在最近两轮 agent run

`state/agent-runs/ar-20260605T090501Z-...-output.json` 和 `state/agent-runs/ar-20260605T164702Z-...-output.json` 都是:

```json
{
  "stderr": "claude timed out",
  "stdout": "",
  "written_at": "2026-06-05T..."
}
```

两次子 agent 跑 Claude 整体超时,stdout 为空。属于调度/资源层问题,不是后端具体某个工具(Tauri command / LLM / shell)的失败 —— 因为根本没有调用过任何工具,进程在子 agent 自己内部就死了。

### 4.3 AgentWorkspacePanel + ChatSessionSidebar 改动不是工具失败

`state/summaries/20260606T070749Z-agentworkspacepanel.md` transcript 显示最近一轮的工作主题是修前端 bug:
